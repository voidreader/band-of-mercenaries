# Phase A: 트레잇 라이프사이클 완성 기획서

> 작성일: 2026-04-13
> 유형: 고도화
> 로드맵: `Docs/content-design/roadmap/trait_enrichment_roadmap.md` Phase A
> 선행 완료: Trait-Roadmap Phase 1~5

---

## 개요

트레잇 시스템의 **생성-성장-소멸** 흐름을 완성한다. 현재 후천 트레잇은 한번 획득하면 진화 외에 제거 수단이 없고, 빈 선천 슬롯은 영영 비어 있다. 이 두 가지 구조적 공백을 메운다.

**기대 효과:**
- 원치 않는 트레잇 고착 → 삭제 수단 제공으로 빌드 실험 유도
- 빈 선천 슬롯 → 여행 이벤트로 채울 기회 제공, 이동 의사결정에 전략적 깊이 추가
- 트레잇 촉발점이 퀘스트 완료 1곳 → 퀘스트 완료 + 여행 이벤트 2곳으로 확장

## 레퍼런스 분석

| 게임 | 시스템 | 차용 포인트 |
|------|--------|------------|
| **Darkest Dungeon** | Sanitarium | 부정적 Quirk를 골드 비용으로 제거. 비용이 높아 "어떤 Quirk를 먼저 치료할까"라는 의사결정 발생. 잠금(Lock) 기능으로 좋은 Quirk 보호 가능 |
| **Crusader Kings 3** | 이벤트 특성 획득 | 여행/전쟁/궁정 이벤트를 통해 선천적 특성(Congenital Trait)이 발현. 이벤트 텍스트가 특성 획득의 서사적 근거를 제공 |

## 파트 1: 트레잇 삭제 시스템 (#18)

### 설계 철학

삭제는 **비용이 드는 전략적 선택**이어야 한다. 무료 삭제는 트레잇의 가치를 떨어뜨리고, 실험적 플레이(일단 받아보고 마음에 안 들면 삭제)를 조장한다. Darkest Dungeon의 Sanitarium처럼 "정말 이 트레잇을 치료할 가치가 있나?"라는 고민이 발생해야 한다.

### 삭제 규칙

| 규칙 | 설명 |
|------|------|
| 대상 | 후천 트레잇(`acquired`, `evolved`)만 삭제 가능 |
| 선천 면역 | 선천 트레잇(`innate`)은 삭제 불가 — 용병의 본질적 정체성 |
| 파견 중 불가 | 파견 중인 용병의 트레잇은 삭제할 수 없음 |
| 히스토리 기록 | 삭제된 트레잇은 `traitHistory`에 기록 → 재획득 방지 |
| 비가역 | 삭제한 트레잇은 다시 얻을 수 없음 (히스토리 규칙) |

### 삭제 비용

기본 원칙: **트레잇의 등급(acquired vs evolved)에 따라 비용이 다르다.**

| 트레잇 유형 | 삭제 비용 | 근거 |
|------------|----------|------|
| `acquired` (후천 기본) | 200G | 의무실에서 간단한 교정 — 난이도 3 퀘스트 순수익 1회분 수준 |
| `evolved` (진화 완료) | 500G | 깊이 뿌리내린 습성을 고치는 비용 — 고난도 퀘스트 순수익에 근접 |

> PD 검토 필요: 수치는 `/balance-designer`로 경제 시뮬레이션 후 조정을 권장한다.

### 삭제 조건: 의무실 레벨

삭제 기능은 **의무실(Medical Facility) 레벨 2 이상**에서 해금된다. 현재 의무실은 회복시간 감소 효과만 있어 업그레이드 동기가 약한데, 삭제 기능 해금이 추가 동기를 제공한다.

| 의무실 레벨 | 해금 기능 |
|------------|----------|
| 0 | (기본) 부상 회복만 가능 |
| 1 | 회복시간 -10% |
| **2** | **트레잇 삭제 해금 (acquired만)** |
| 3 | 회복시간 -30% |
| **4** | **evolved 트레잇 삭제 해금** |
| 5 | 회복시간 -50% |

이렇게 하면:
- 초반에는 잘못된 트레잇을 감내해야 함 → 트레잇 획득에 긴장감
- 의무실 업그레이드가 "회복" 외에 "트레잇 관리"라는 새 역할을 가짐
- evolved 트레잇 삭제는 레벨 4에서 해금 → 진화 결정의 무게감 유지

### 삭제 흐름

```
용병 상세 오버레이 → 트레잇 슬롯 탭 → TraitDetailDialog 표시
  → [삭제] 버튼 표시 (후천 트레잇 + 의무실 레벨 조건 충족 시)
  → 확인 다이얼로그: "이 트레잇을 삭제하시겠습니까? 비용: {N}G / 삭제 후 재획득 불가"
  → 확인 → 골드 차감 → traitIds에서 제거 → traitHistory에 추가 → 활동 로그 기록
  → 슬롯 비워짐 → 새 트레잇 획득 가능
```

### 삭제 결과

- 해당 카테고리 슬롯이 비워진다 → 새로운 후천 트레잇 획득 가능
- `traitHistory`에 `deleted` 사유로 기록 → 동일 트레잇 재획득 방지
- 활동 로그에 "김철수의 [트레잇명] 트레잇이 제거되었다" 기록

### UI 설계

**TraitDetailDialog 변경:**
- 다이얼로그 하단에 [삭제] 버튼 추가
- 조건 미충족 시 비활성화 + 사유 표시:
  - 선천 트레잇: "선천 트레잇은 삭제할 수 없습니다"
  - 의무실 레벨 부족: "의무실 레벨 {N} 필요"
  - 파견 중: "파견 중에는 삭제할 수 없습니다"
  - 골드 부족: "골드가 부족합니다 ({현재}G / {필요}G)"
- 삭제 가능 시: 빨간색 [삭제 — {비용}G] 버튼

**확인 다이얼로그:**
- 트레잇 이름, 효과 텍스트, 비용 표시
- "삭제된 트레잇은 다시 획득할 수 없습니다" 경고
- [취소] / [삭제] 버튼

## 파트 2: 여행 이벤트 ↔ 선천 트레잇 연계 (#13)

### 설계 철학

여행은 용병의 "경험"이다. 먼 지역으로의 이동은 단순한 위치 변경이 아니라 **용병이 세상을 경험하는 과정**이어야 한다. Crusader Kings 3에서 여행 이벤트가 캐릭터의 특성을 바꾸듯, 이 게임에서도 여행 중의 사건이 용병의 본질적 특성(선천 트레잇)을 형성할 수 있어야 한다.

단, 선천 트레잇은 영구적이고 강력하므로 **쉽게 얻어져서는 안 된다**. 희소성을 유지해야 한다.

### 핵심 메커니즘

여행 이벤트 중 **특정 조건**을 충족하면, 용병단에 빈 선천 슬롯을 가진 용병이 있을 때 선천 트레잇 획득 기회가 발생한다.

### 발동 조건

```
여행 이벤트 발생 (기존 12종과 별개의 신규 이벤트)
  AND 용병단에 빈 선천 슬롯을 가진 용병이 1명 이상 존재
  AND 해당 카테고리에 부여 가능한 선천 트레잇이 존재
```

### 신규 여행 이벤트

기존 12종 이벤트에 **선천 트레잇 부여** 유형을 추가한다. effectType `trait_innate`를 신설한다.

| ID | 이름 | 설명 | 대상 카테고리 | 티어 범위 |
|----|------|------|-------------|----------|
| `te_harsh_terrain` | 혹독한 지형 | "척박한 환경을 지나며 단련되었다" | Physical | 2~5 |
| `te_old_traveler` | 노련한 여행자와의 조우 | "길에서 만난 노인이 옛 이야기를 들려주었다" | Background | 1~4 |
| `te_natural_talent` | 재능의 발현 | "위기 상황에서 숨겨진 재능이 발현되었다" | Talent | 3~5 |

> 각 카테고리별 1종씩, 총 3종의 신규 이벤트. 향후 확장 가능.

### 이벤트 발생 확률

기존 이벤트와 동일하게 `distance × 0.15` (최대 80%) 확률로 이벤트 풀에 진입한다. 단, 선천 트레잇 이벤트는 **추가 필터링**을 거친다:

```
1. 기본 이벤트 풀에서 랜덤 선택 (기존 12종 + 신규 3종 = 15종)
2. 선택된 이벤트가 trait_innate 타입인 경우:
   a. 빈 선천 슬롯을 가진 용병이 없으면 → 이벤트 무효, 재롤링
   b. 해당 카테고리의 빈 슬롯을 가진 용병이 없으면 → 이벤트 무효, 재롤링
   c. 조건 충족 시 → 이벤트 발동
```

이 방식을 사용하면:
- 빈 선천 슬롯이 많을수록 이벤트가 발동할 확률이 높아짐 (자연스러운 확률)
- 이미 선천 3개를 모두 가진 용병단에는 이 이벤트가 발생하지 않음
- 기존 이벤트 확률에 영향을 주지 않음 (무효 시 재롤링)

### 대상 용병 선택

이벤트 발동 시, 해당 카테고리의 빈 슬롯을 가진 용병 중 **랜덤 1명**에게 트레잇이 부여된다.

```
해당 카테고리 빈 슬롯 보유 용병 목록
  → 랜덤 1명 선택
  → 해당 카테고리의 선천 트레잇 중 랜덤 1개 부여
  → traitIds에 추가
```

### 부여될 트레잇 선택

해당 카테고리(`Physical`, `Background`, `Talent`)의 `type = 'innate'` 트레잇 중 랜덤 선택한다. 단, 다음은 제외한다:

- 해당 용병이 이미 보유한 트레잇 (동일 카테고리에서 중복 불가)
- 해당 용병의 기존 트레잇과 충돌 관계인 트레잇 (`trait_conflicts`)

### 이벤트 결과 UI

기존 여행 이벤트 결과 다이얼로그(HomeScreen의 AlertDialog)를 확장한다.

**기존 이벤트 결과:**
```
[여행 중 사건 발생!]
버려진 보물 발견! (+50G)
```

**선천 트레잇 이벤트 결과:**
```
[여행 중 사건 발생!]
혹독한 지형을 지나며 단련되었다.

김철수가 새로운 선천 트레잇을 획득했습니다!
[강인한 체격] — Physical
"험난한 여정이 그의 육체를 단련시켰다."
```

팝업 구성:
- 이벤트 설명 텍스트
- 대상 용병 이름 강조
- 획득 트레잇 이름 + 카테고리 + 설명
- 용병 카드 탭 시 상세 화면으로 이동 가능

### 활동 로그

트레잇 획득 시 활동 로그에 기록한다:
- "김철수가 여행 중 [강인한 체격] 선천 트레잇을 획득했다"

### travel_events 테이블 추가 데이터 (3행)

```sql
INSERT INTO travel_events (id, name, type, effect_type, magnitude, min_tier, max_tier, description) VALUES
('te_harsh_terrain', '혹독한 지형', 'encounter', 'trait_innate', 0, 2, 5, '척박한 환경을 지나며 단련되었다.'),
('te_old_traveler', '노련한 여행자와의 조우', 'encounter', 'trait_innate', 0, 1, 4, '길에서 만난 노인이 옛 이야기를 들려주었다.'),
('te_natural_talent', '재능의 발현', 'luck', 'trait_innate', 0, 3, 5, '위기 상황에서 숨겨진 재능이 발현되었다.');
```

### 이벤트-카테고리 매핑

`travel_events` 테이블에 `target_category` 컬럼을 추가하거나, 별도의 매핑 방식을 사용한다.

**방법 A: 컬럼 추가 (권장)**
```sql
ALTER TABLE travel_events ADD COLUMN target_category TEXT;
UPDATE travel_events SET target_category = 'Physical' WHERE id = 'te_harsh_terrain';
UPDATE travel_events SET target_category = 'Background' WHERE id = 'te_old_traveler';
UPDATE travel_events SET target_category = 'Talent' WHERE id = 'te_natural_talent';
```

**방법 B: magnitude 활용** — magnitude 값으로 카테고리를 인코딩 (1=Physical, 2=Background, 3=Talent). 모델 변경 불필요하나 가독성이 떨어진다.

> 방법 A를 권장한다. `target_category`는 향후 시설 연계 이벤트 등에서도 재활용 가능하다.

## 밸런스 고려사항

### 트레잇 삭제 비용

- acquired 200G, evolved 500G는 초안이다
- 초반(Tier 1~2 활동) 골드 수입 대비 적절한 부담인지 검증 필요
- 의무실 레벨 2 해금 비용과의 관계 검토 필요

### 선천 트레잇 이벤트 빈도

- 15종 이벤트 풀에서 3종 = 약 20% 확률로 선천 트레잇 이벤트가 선택됨
- 여기에 "빈 슬롯 존재" 필터가 추가되므로 실제 발동률은 훨씬 낮음
- 장거리 이동(거리 5 이상) 시 이벤트 확률 75%+ × 풀 내 20% × 빈 슬롯 필터 → 체감 확률은 적절한 희소성 유지

> 정확한 수치 검증은 `/balance-designer`로 경제/확률 시뮬레이션을 권장한다.
