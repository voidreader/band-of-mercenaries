Recommended Model : Claude Sonnet
** 한국어 스타일 유지 **

## 언제 사용하나요?

- 자동으로 사용되지 않도록 한다.
- 사용자의 자의적 호출에 의해서 사용되도록 한다.
- 게임 기획 문서를 개발 명세서로 변환할 때 사용한다.

## Instructions

기획 문서의 의도를 정확히 파악하고, 현재 코드베이스를 탐색하여 구현에 필요한 구체적인 정보를 담은 명세서를 생성한다.

# 입력

사용자가 `/spec-writer` 호출 시 인자로 기획 문서를 전달한다.

- 호출 예: `/spec-writer @Docs/20260323_신규퀘스트_기획.md`
- 로컬 파일을 Read로 읽어 처리한다.

기획 문서를 찾지 못하면:
- "기획 문서를 찾을 수 없습니다. 로컬 파일 경로를 전달해주세요." 메시지를 출력한다.
- 올바른 인자가 전달될 때까지 진행하지 않는다.

# 수행 절차

## 1단계: 기획 문서 분석

기획 문서를 Read로 읽고 다음을 파악한다:
- 기능의 목적과 핵심 동작
- 사용자(플레이어) 관점의 기대 동작
- 수치/밸런스 데이터 (있는 경우)
- UI/UX 요구사항 (있는 경우)
- 기획 문서에서 모호하거나 누락된 부분

## 2단계: 코드베이스 탐색

기획 문서의 기능과 관련된 기존 코드를 탐색한다. Agent(subagent_type="Explore")를 활용하여 병렬로 탐색할 수 있다. 다음을 **모두 확인할 때까지** 탐색을 계속한다:

1. 기능이 속하는 시스템 식별 (CLAUDE.md의 아키텍처 섹션 참조)
2. 수정 대상 파일을 모두 Read로 열어 실제 구조를 확인했다
3. 기능의 진입점(호출이 시작되는 Provider/Notifier/Widget)을 식별했다
4. 유사 기능이 있다면 해당 구현을 Read로 확인했다
5. 새로 생성해야 할 파일을 식별했다
6. 관련 Hive 박스 또는 Supabase 정적 데이터 테이블을 확인했다 (데이터 모델 변경 필요 여부)
7. build_runner 재실행이 필요한 파일(freezed, json_serializable, riverpod_generator)을 확인했다

## 3단계: 기획 의도 확인

기획 문서에서 다음에 해당하는 항목이 있으면 사용자에게 확인을 요청한다:
- 기존 시스템과 충돌하는 동작 설명
- 여러 가지로 해석 가능한 표현
- 기술적으로 구현이 어렵거나 대안이 필요한 부분
- Hive 박스 스키마 변경 또는 Supabase 테이블 추가가 필요한 경우
- 기획 문서에 없지만 구현 시 결정이 필요한 사항 (에러 처리, 엣지 케이스 등)

모든 확인이 완료될 때까지 명세서를 생성하지 않는다.

## 4단계: 명세서 생성

아래 형식의 명세서를 `Docs/spec/` 디렉토리에 생성한다.

### 파일명 규칙

`[spec]{날짜}_{주제}.md`
- 날짜: YYYYMMDD (오늘 날짜)
- 주제: kebab-case

예: `[spec]20260323_quest-system.md`

### 명세서 형식

```markdown
# {기능명} 개발 명세서

> 기획 문서: {기획 문서 파일 경로}
> 작성일: {날짜}

## 1. 개요

{기능의 목적과 핵심 동작을 2~3줄로 요약}

## 2. 요구사항

### 2.1 기능 요구사항

- [FR-1] 요구사항 설명
  - 상세 동작: ...
  - 조건: ...
- [FR-2] ...

### 2.2 데이터 요구사항 (해당 시)

- 신규/수정 Hive 박스: 박스명 - 추가 필드/변경 내용
- 신규/수정 정적 데이터 모델: 모델명 - 추가 필드/변경 내용 (Supabase 테이블 변경 포함 여부)
- 신규 enum: ...
- 밸런스 수치: ...

### 2.3 UI 요구사항 (해당 시)

- 화면: ...
- 동작: ...
- 연출: ...

## 3. 영향 범위

### 3.1 수정 대상 파일

| 파일 경로 | 수정 내용 | 사유 |
|-----------|----------|------|
| band_of_mercenaries/lib/... | ... | ... |

### 3.2 신규 생성 파일

| 파일 경로 | 역할 |
|-----------|------|
| band_of_mercenaries/lib/... | ... |

### 3.3 코드 생성 필요 파일

| 파일 경로 | 이유 |
|-----------|------|
| band_of_mercenaries/lib/... | freezed 모델 변경 |

(build_runner 재실행이 필요한 파일 목록. 없으면 "없음"으로 표기)

### 3.4 관련 시스템

- {시스템명}: {어떤 영향을 받는지}
- ...

## 4. 기술 참고사항

### 4.1 기존 패턴 참조

- {유사 기능이 구현된 파일 경로}: {어떤 패턴을 참고할지}

### 4.2 주의사항

- {구현 시 주의할 점, CLAUDE.md 금지사항과 관련된 부분 등}

### 4.3 엣지 케이스

- {기획 문서에는 없지만 처리가 필요한 예외 상황}

### 4.4 구현 힌트

- 진입점: {호출이 시작되는 Provider/Notifier/Widget}
- 데이터 흐름: {A → B → C 형태로 데이터가 흘러가는 경로}
- 참조 구현: {유사 기능이 구현된 파일:라인} — {무엇을 참고할지}
- 확장 지점: {기존 코드에서 새 기능을 끼워넣을 위치}

## 5. 기획 확인 사항

- [Q-1] {기획자에게 확인이 필요한 사항} → {확인 결과}
- ...
```

# 출력 규칙

- 명세서는 `Docs/spec/` 디렉토리에 마크다운 파일로 생성한다.
- 기술적인 용어를 유지하며 한국어 스타일을 유지한다.
- 기획 문서의 의도를 변경하지 않는다. 기술적으로 대안이 필요한 경우 명세서의 "기술 참고사항"에 대안을 제시하되 최종 결정은 사용자에게 맡긴다.
- 기획 문서에 없는 내용을 임의로 추가하지 않는다. 구현에 필요하지만 기획에 없는 사항은 "기획 확인 사항"에 기재한다.
- 수정 대상 파일은 실제 코드를 Read로 확인한 결과만 기재한다. 추측하지 않는다.
- 명세서 생성 후 "5단계: 구현 방식 추천 및 실행"을 수행한다.

## 명세서 품질 기준

- 각 FR은 최소 하나의 구현 위치(파일 경로 또는 신규 파일)와 매핑한다.
- 데이터 요구사항은 실제 Hive 박스명/클래스명과 타입을 명시한다.
- 기획 문서의 모호한 표현("적당히", "빠르게" 등)은 그대로 옮기지 않고, "기획 확인 사항"으로 이동시킨다.
- "4.4 구현 힌트" 섹션은 반드시 작성한다. 2단계에서 탐색한 진입점, 데이터 흐름, 참조 구현을 구조화하여 하류 구현 스킬이 코드 탐색을 반복하지 않도록 한다.

## 5단계: 구현 방식 추천 및 실행

생성한 명세서의 내용을 기반으로 적합한 구현 방식을 판단하고 사용자에게 추천한다.

### 판단 기준

| 기준 | implement-spec (올인원) | implement-agent (파이프라인) |
|------|------------------------|---------------------------|
| 수정/생성 파일 수 | 5개 이하 | 6개 이상 |
| 영향 시스템 수 | 1~2개 | 3개 이상 |
| 신규 클래스 생성 | 0~2개 | 3개 이상 |
| 데이터 모델 변경 | 없음 또는 필드 추가 | 신규 모델 생성 또는 Supabase 테이블 추가 |
| UI 작업 | 기존 Widget 수정 | 신규 화면 추가 |
| 기존 시스템 변경 | 소폭 수정 | 구조 변경 또는 새 Provider 흐름 추가 |

### 점수 산정

위 6개 기준 각각에 대해:
- implement-spec 조건에 해당하면 0점
- implement-agent 조건에 해당하면 1점

합산 점수로 추천:
- **0~2점**: implement-spec 추천
- **3~4점**: implement-agent 추천, implement-spec도 가능
- **5~6점**: implement-agent 강력 추천

### 출력 형식

```
## 명세서 생성 완료

파일: {파일 경로}

### 구현 규모 분석

| 기준 | 내용 | 판정 |
|------|------|------|
| 수정/생성 파일 | {n}개 | 소규모 / 대규모 |
| 영향 시스템 | {시스템명 나열} ({n}개) | 소규모 / 대규모 |
| 신규 클래스 | {n}개 | 소규모 / 대규모 |
| 데이터 모델 | {변경 내용 요약} | 소규모 / 대규모 |
| UI 작업 | {변경 내용 요약} | 소규모 / 대규모 |
| 기존 시스템 변경 | {변경 내용 요약} | 소규모 / 대규모 |

**추천: {implement-spec 또는 implement-agent}** ({n}/6점)
- {추천 사유 한 줄}
```

### 실행 선택

구현 규모 분석 출력 후, 추천 스킬의 실행 명령어를 안내한다:

- 추천 항목 옆에 "← 추천"을 표시한다.
- 아래 형식으로 출력한다:

```
구현을 진행하려면 아래 명령어를 실행해주세요:

/implement-spec @{명세서 파일 경로}  ← 추천 (올인원)
/implement-agent @{명세서 파일 경로}  (파이프라인)
```

- `{명세서 파일 경로}`는 실제 생성된 파일의 상대 경로로 치환한다.
- 추천 항목은 구현 규모 분석 결과에 따라 결정한다.
