## 1. 개요

## 2. 구현 기준 확인

본 절은 히든 스탯 효과를 현재 코드의 PassiveEffect, CombatSimulator hook, QuestCompletionService 후처리로 나누어 적용할 때 필요한 구현 기준을 확인한다. 신규 PassiveEffect 타입을 만들지 않고 기존 파서와 스태킹 규칙에 맞추는 것이 핵심이다.

### 2.1 `PassiveEffect` 지원 타입 (passive_effect.dart 기준)

본 리포트가 사용하는 PassiveEffect 타입:

| 타입 | 필드 | 본 리포트 사용 스탯 |
|------|------|-------------------|
| `recoveryTimeReduction` | status, value | 불굴 (injury_recovery, status='injured') |
| `reputationGainModifier` | value | 투지 (reputation_gain, 상한 +0.30) |
| `injuryRateModifier` | value | (미사용) |
| `mercenaryXpBonus` | value | (미사용) |

현재 `PassiveEffect.fromJson`은 18개 지원 타입과 `unknown` fallback을 가진다. 히든 스탯은 이 중 `recovery_time_reduction`과 `reputation_gain_modifier`만 사용하며, 새 PassiveEffect 타입은 추가하지 않는다.

**부호 정책**: `recovery_time_reduction.value`는 양수로 저장한다. `PassiveBonusService.getRecoveryTimeMultiplier()`가 `(1 - Σ).clamp(0.10, 1.0)`을 반환하므로, `value=0.04`는 회복 시간 4% 단축을 의미한다. 음수 `-0.04`를 저장하면 `1 - (-0.04)`가 되어 회복 시간이 증가하므로 금지한다.

### 2.2 운 item_drop_rate 처리

운의 item_drop +0.005/lv는 PassiveEffect 지원 타입에 없는 hook이다. 페이즈 1 #4 §3.4의 **3계층 처리**와 정합한다.
- M8b hook 직접 가산 (death_resistance·critical·evasion·hit·mez_immune)
- PassiveBonusService 통합 (recoveryTimeReduction·reputationGainModifier)
- **QuestCompletionService 전용 후처리** (item_drop 추가 판정)

운의 item_drop은 `QuestCompletionService` 보상 트랜잭션에서 직접 판정. 별도 `PassiveEffect` 신규 추가 없음.

파티에 여러 용병이 참가한 경우 item_drop 보너스는 참가자 중 `luck` 최고 lv 1명만 적용한다. 5인 합산은 재료 드랍 인플레이션을 만들 수 있으므로 금지한다.

## 3. 5 히든 스탯 lv별 효과 매트릭스 (최종)

본 절은 각 히든 스탯의 lv별 효과를 최종 수치로 고정한다. 표의 값은 모두 per-level 누적 결과이며, 실제 적용 단계에서는 각 hook의 기존 clamp와 PassiveBonusService 상한을 통과한다.

### 3.1 불굴 (`fortitude`)

| lv | death_resistance | injury_recovery_time |
|----|------------------|--------------------|
| 1 | +0.02 | ×0.96 (4% 단축) |
| 2 | +0.04 | ×0.92 (8% 단축) |
| 3 | +0.06 | ×0.88 (12% 단축) |
| 4 | +0.08 | ×0.84 (16% 단축) |
| 5 | **+0.10** | **×0.80 (20% 단축)** |

산식:
- `death_resistance_bonus = lv × 0.02`
- `injury_recovery_multiplier = 1.0 - lv × 0.04`

칭호 비교: title_pyegwang_survivor recovery_time_reduction 0.10(표시상 -10%). 본 lv5 ×0.80은 `value=0.20`에 상당 → 칭호 ~2배 강함. 동일 PassiveEffect 채널이므로 양수 reduction value를 합산한 뒤 `PassiveBonusService.getRecoveryTimeMultiplier()`의 하한 0.10을 적용한다.

**솔로 의뢰 cap 0.95 + 불굴 lv5 가산** (페이즈 2 #2 정합):
- `clamp(base + traitBonus + factionBonus + fortitude × 0.02 + emotional_determination × 0.20, 0.20, 0.95)`
- 풀스택 기본 (base 0.40 + 트레잇 0.10 + 불굴 0.10) = 0.60 → cap 0.95 미도달
- 투지 발동 포함 (위 + emotional_determination 0.20) = 0.80 → cap 0.95 미도달
- 세력 보너스 0.05까지 포함 = 0.85 → cap 0.95 미도달
- M8.5 컨셉 "안전망 없음" 정합 — cap 0.95는 T4~T5 base나 추가 보너스가 충분한 용병만 도달 가능

### 3.2 투지 (`grit`)

| lv | emotional_despair 면제 | reputation_gain_modifier |
|----|----------------------|--------------------------|
| 1 | +0.08 (80% → 72%) | +0.015 |
| 2 | +0.16 (80% → 64%) | +0.030 |
| 3 | +0.24 (80% → 56%) | +0.045 |
| 4 | +0.32 (80% → 48%) | +0.060 |
| 5 | **+0.40 (80% → 40%)** | **+0.075** |

산식:
- `despair_immune_bonus = lv × 0.08` (페이즈 2 #3 절망 발동 80%에서 차감)
- `reputation_gain_modifier = lv × 0.015`

적용 순서:
- `TraitEmotionalKeywords.despairImmune` 보유 시 절망 발동 확률은 즉시 0%다.
- 해당 트레잇이 없으면 `despairChance = (0.80 - gritLv × 0.08).clamp(0.0, 0.80)`을 적용한다.
- 투지 lv5만 보유하면 절망 발동 확률은 40%다.

칭호 비교: title_soul_severer reputation_gain 0.05. 본 lv5 +0.075은 칭호 ~1.5배. 단 lv5 도달은 매우 후반(30회 카운터). 적정.

iron_will 트레잇 100% 면제와의 차이:
- iron_will 보유: 절망 발동 0% (완전 면제)
- 투지 lv5만 (트레잇 없음): 절망 발동 40% (절반 면제)
- iron_will + 투지 lv5: iron_will 우선 (100% 면제)

→ 트레잇 면제와 히든 스탯 면제는 자연 차별. 트레잇이 강한 정체성 보호.

### 3.3 운 (`luck`)

| lv | critical | evasion | item_drop |
|----|---------|---------|----------|
| 1 | +0.01 | +0.01 | +0.005 |
| 2 | +0.02 | +0.02 | +0.010 |
| 3 | +0.03 | +0.03 | +0.015 |
| 4 | +0.04 | +0.04 | +0.020 |
| 5 | **+0.05** | **+0.05** | **+0.025** |

산식:
- `critical_bonus = lv × 0.01`
- `evasion_bonus = lv × 0.01`
- `item_drop_chance_bonus = lv × 0.005`

M8b clamp 정합:
- critical clamp [0.05, 0.60]: +0.05 가산은 안전
- evasion clamp [0, 0.75]: +0.05 가산은 안전

item_drop은 QuestCompletionService 후처리 — M5 제작 재료 드랍률에 +2.5% 가산 (lv5 풀스택). 1주 평균 50 의뢰 × +2.5% = 1.25 추가 재료/주. 인플레이션 무시 가능.

파티 단위 적용은 참가자 중 최고 `luck` lv 1명만 사용한다. 예를 들어 5인 파티가 모두 운 lv5여도 추가 판정은 +2.5% 한 번만 적용한다.

### 3.4 공포 저항 (`fear_resistance`)

| lv | mez_immune | 강공격 회피 |
|----|----------|------------|
| 1 | +0.05 | +0.015 |
| 2 | +0.10 | +0.030 |
| 3 | +0.15 | +0.045 |
| 4 | +0.20 | +0.060 |
| 5 | **+0.25** | **+0.075** |

산식:
- `mez_immune_chance = lv × 0.05` (mez 적용 시 차단 확률)
- `strong_attack_evasion_bonus = lv × 0.015` (적 피해 ≥ HP×0.30인 공격에 한정 evasion 추가 가산)

`mez_immune`은 M8b 신규 hook. `CombatSimulator` `mez_stunned` 적용 시점에 `mezRng.nextDouble() < fearResistance × 0.05`이면 적용 차단.

강공격 회피는 evasion clamp [0, 0.75]에 일반 evasion + 본 가산 동시 적용. lv5 풀스택 + 운 lv5: evasion +0.05 + 강공격 +0.075 = +0.125 (강공격에 한정).

### 3.5 전장 감각 (`battle_sense`)

| lv | action_score | featured_score | hit_chance |
|----|------------|---------------|-----------|
| 1 | +0.5 | +0.2 | +0.01 |
| 2 | +1.0 | +0.4 | +0.02 |
| 3 | +1.5 | +0.6 | +0.03 |
| 4 | +2.0 | +0.8 | +0.04 |
| 5 | **+2.5** | **+1.0** | **+0.05** |

산식:
- `action_score_bonus = lv × 0.5` (M8b 행동 순서 정렬에서 가산)
- `featured_score_bonus = lv × 0.2` (M8b 결정적 장면 기여 점수에서 가산)
- `hit_chance_bonus = lv × 0.01` (M8b 명중 hook에 직접 가산)

action_score 정합:
- M8b 평균 action_score ~50-60 + PRNG 노이즈 ±3
- battle_sense lv5 +2.5는 노이즈 범위 안에서 의미 있는 차이
- 풀스택 lv5 용병은 동률·근소차 상황에서 행동 순서 상위 진입 빈도가 증가

featured_score 정합:
- M8b 평균 결정적 장면 기여 점수 5~10
- battle_sense lv5 +1.0는 ~10~20% 가산
- protagonist 후보 동률에서 우위 (페이즈 1 #4 §3.1.1 정합)

칭호 비교: title_escort_master/road_hunter/monster_hunter quest_success_rate_bonus 0.05. battle_sense lv5 hit +0.05는 단일 칭호와 동급 강도. 적정.

### 3.6 5 스탯 풀스택 lv5 누적 효과

5 스탯 모두 lv5에 도달한 용병 1명의 부가 능력:

| 카테고리 | 누적 효과 |
|---------|---------|
| 전투 (M8b hook) | death_resistance +0.10 / despair 면제 +0.40 / critical +0.05 / evasion +0.05 / mez_immune +0.25 / 강공격 evasion +0.075 / action_score +2.5 / featured_score +1.0 / hit +0.05 |
| 비전투 (PassiveBonus) | injury_recovery ×0.80 / reputation_gain +0.075 |
| 후처리 (QuestCompletion) | item_drop +0.025 |

**M6 칭호 합산 정합**:
- 칭호 평균 1개 = 0.05 효과 강도
- 히든 스탯 풀스택 (5 lv5) = 칭호 ~3개 효과 강도

→ "전설적인 베테랑" 수준. 30~50시간 누적 플레이 후 도달. 게임 컨셉 정합.

**솔로 의뢰 cap 풀스택 시나리오**:
- base 0.40 + 트레잇 0.10 + 세력 0.05 + 불굴 0.10 + 투지 발동 0.20 = 0.85
- 솔로 cap 0.95 → 0.85 (cap 미도달)
- 일반 cap 0.80 → 0.80 (cap 도달)

→ 풀스택 용병도 솔로 의뢰에서 cap 0.95에 자주 닿지 않음. 의도된 안전망 없음 컨셉 정합.

## 4. 임계값 1/3/7/15/30 검증

본 절은 5개 히든 스탯이 너무 빨리 풀스택에 도달하지 않으면서도 초반 발견 경험을 제공하는지 확인한다. 임계값은 전 스탯 공통 `[1, 3, 7, 15, 30]`으로 유지한다.

### 4.1 임계값 곡선

| lv | 누적 카운터 | 이전 lv 대비 |
|----|---------|------------|
| 1 | 1 | (첫 사건) |
| 2 | 3 | +2 |
| 3 | 7 | +4 |
| 4 | 15 | +8 |
| 5 | 30 | +15 |

곡선: 3배수 점진 — `next = prev + (prev + 1) × 1` 대략 ≈ ×2.1 평균.

### 4.2 신규 유저 누적 시뮬레이션

신규 유저 평균 4시간/일 × 의뢰 ~10건/시간 = 40 의뢰/일. 각 의뢰가 5 스탯 중 0~2 카운터 +1 가정 시 1일 누적:

| 시간 | 1 스탯 평균 카운터 | 평균 lv 도달 |
|-----|------------------|-----------|
| 1h (10 의뢰) | ~2~3 | lv1~lv2 |
| 3h (30 의뢰) | ~6~9 | lv2~lv3 |
| 5h (50 의뢰) | ~10~15 | lv3~lv4 |
| 10h (100 의뢰) | ~20~30 | lv4~lv5 |
| 20h (200 의뢰) | ~40~60 | lv5 (max) |

→ **10시간 누적 시 평균 1~2 스탯 lv5 도달**. 30~50시간 누적 시 풀스택. 페이즈 1 #4 곡선 정합.

### 4.3 스탯별 카운터 증가 빈도 시뮬레이션

5 스탯 각각의 1주 (50시간 가정) 카운터 증가 추정:

| 스탯 | 1주 평균 카운터 | 1주 lv 진행 |
|-----|---------------|-----------|
| 불굴 | ~25 (사망 저항 발동 ~5 + 투지 발동 ~3 + 솔로 완수 ~3 × 2 = 14, 일반 생존 사건 추가) | lv3~lv4 |
| 투지 | ~15 (절망 면제 ~5 + 솔로 대성공 ~3 × 3 = 14) | lv3 |
| 운 | ~40 (치명타 ~10 + 회피 ~15 + 드랍 ~10 + 결정적 장면 ~5) | lv4~lv5 |
| 공포 저항 | ~20 (mez 면제 ~3 + 유니크 엘리트 ~2 + 강공격 회피 ~15) | lv4 |
| 전장 감각 | ~30 (치명타 처치 ~5 + 결정적 장면 ~10 + 명중 ~15) | lv4 |

→ **1주 1 용병당 평균 2~3 스탯이 lv4~lv5 도달**. 풀스택은 3~4주 소요.

### 4.4 임계값 ±1 sensitivity

| 임계값 변경 | lv5 도달 시간 변화 |
|----------|------------------|
| 1/3/7/15/**25** (-5) | ~20% 빠름 |
| **1/3/7/15/30** (권장) | 기준 |
| 1/3/7/**20**/30 (+5) | ~10% 느림 |
| 1/3/**10**/15/30 (+3) | ~5% 느림 |

권장값은 신규 유저가 5~10시간 안에 lv1 발견을 충분히 경험할 수 있도록 lv1=1로 즉시 해금. lv5는 ~3주 후반 도달로 장기 목표 부여. **유지 확정**.

## 5. 사건 가중치 정합 (페이즈 1 #4 §3.3.2)

본 절은 히든 스탯 카운터를 증가시키는 사건과 가중치를 확정한다. 같은 전투에서 같은 사건이 여러 번 발생할 수 있으므로, 페이즈 4 #3 명세에서는 전투 단위 중복 방지 규칙을 함께 정의한다.

### 5.1 권장 가중치 매트릭스 (페이즈 1 #4 그대로)

| 사건 | 카운터 가중 | 매핑 스탯 |
|------|----------|---------|
| 사망 저항 cap 통과로 살아남 | +1 | 불굴 |
| `emotional_determination` apply | +1 | 불굴 |
| 솔로 의뢰 (`pool.partySizeMax == 1`) 완수 (success/greatSuccess) | +2 | 불굴 |
| `emotional_despair` 면제 | +1 | 투지 |
| 솔로 의뢰 대성공 | +3 | 투지 |
| 체인 주인공 위기(HP<30%) 극복 → 전투 종료 시 생존 | +2 | 투지 |
| 치명타 발동 | +1 | 운 |
| 회피 성공 | +1 | 운 |
| 보상 아이템 드랍 받음 | +1 | 운 |
| 결정적 장면 기여 (점수 ≥ 5) | +1 | 운 |
| mez 면제 | +1 | 공포 저항 |
| 유니크 엘리트와 전투에서 본인 생존 | +1 | 공포 저항 |
| 적 단발 피해 ≥ HP×0.30 회피·방패 막기 | +1 | 공포 저항 |
| 치명타 처치 | +1 | 전장 감각 |
| 결정적 장면 기여 (점수 ≥ 5) | +1 | 전장 감각 |
| 결정적 장면 기여 (점수 ≥ 10) | **+1 추가 (총 +2)** | 전장 감각 |
| 명중 성공 (적 명중률 ≤ 0.50) | +1 | 전장 감각 |

### 5.2 가중 정합 검증

**솔로 완수 +2 vs 일반 사건 +1**:
- 솔로 의뢰 1주 ~3회 완수 × +2 = 6 카운터 (불굴)
- 일반 생존·위기 사건 1주 ~30 카운터 (불굴)
- 솔로 가중 비중 ~17% — 솔로의 희소성에 비해 적정 비중

**솔로 대성공 +3 (단일 가중, +2 추가 누적 아님)**:
- 솔로 의뢰 대성공 1주 ~1회 × +3 = 3 카운터 (투지)
- 절망 면제 1주 ~5회 × +1 = 5 카운터 (투지)
- 솔로 대성공 가중 비중 ~38% — 매우 큰 사건의 정합

**결정적 장면 ≥10 +2 (전장 감각)**:
- 평균 결정적 장면 ≥10 1주 ~3회 × +2 = 6 카운터
- 결정적 장면 ≥5 1주 ~10회 × +1 = 10 카운터
- 큰 사건 가중 비중 ~38% — 적정

### 5.3 가중치 sensitivity

| 변경 | 영향 |
|-----|------|
| 솔로 완수 +1 (현재 +2) | 솔로 비중 ~9% — 솔로 차별 약화 |
| 솔로 완수 +3 (현재 +2) | 솔로 비중 ~22% — 일반 사건과 격차 큼 |
| **솔로 완수 +2** (권장) | ~17% (적정) |

가중치 sensitivity는 미세. 권장 유지.

### 6.2 페이즈 4 #3 spec 정수

```dart
class HiddenStatBonusResolver {
  // 스탯 ID 컨벤션
  static const String fortitude = 'fortitude';
  static const String grit = 'grit';
  static const String luck = 'luck';
  static const String fearResistance = 'fear_resistance';
  static const String battleSense = 'battle_sense';

  // M8b hook 가산 산식 (per lv)
  static double resolveHookBonus({
    required String hook,
    required Map<String, int> hiddenStats,  // mercId의 hiddenStats Map
  }) {
    final fort = hiddenStats[fortitude] ?? 0;
    final gri = hiddenStats[grit] ?? 0;
    final luc = hiddenStats[luck] ?? 0;
    final fear = hiddenStats[fearResistance] ?? 0;
    final battle = hiddenStats[battleSense] ?? 0;

    switch (hook) {
      case 'death_resistance':
        return fort * 0.02;
      case 'despair_immune_chance':
        return gri * 0.08;
      case 'critical_rate':
        return luc * 0.01;
      case 'evasion':
        return luc * 0.01;
      case 'mez_immune_chance':
        return fear * 0.05;
      case 'strong_attack_evasion':
        return fear * 0.015;
      case 'action_score':
        return battle * 0.5;
      case 'featured_score':
        return battle * 0.2;
      case 'hit_chance':
        return battle * 0.01;
      default:
        return 0.0;
    }
  }

  // PassiveBonusService 통합 (per lv)
  static List<PassiveEffect> collectPassiveBonuses(Map<String, int> hiddenStats) {
    final result = <PassiveEffect>[];
    final fort = hiddenStats[fortitude] ?? 0;
    final gri = hiddenStats[grit] ?? 0;

    if (fort > 0) {
      result.add(PassiveEffect.recoveryTimeReduction(
        status: 'injured',
        value: fort * 0.04,
      ));
    }
    if (gri > 0) {
      result.add(PassiveEffect.reputationGainModifier(
        value: gri * 0.015,
      ));
    }
    // luck/fear/battle은 PassiveEffect 미통합
    return result;
  }

  // QuestCompletionService 후처리 (per lv)
  static double itemDropBonus(Map<String, int> hiddenStats) {
    final luc = hiddenStats[luck] ?? 0;
    return (luc * 0.005).clamp(0.0, 0.025);
  }

  // lv 임계 평가
  static const List<int> thresholds = [1, 3, 7, 15, 30];

  static int computeLevel(int counter) {
    int lv = 0;
    for (final t in thresholds) {
      if (counter >= t) lv++;
      else break;
    }
    return lv;
  }
}
```

## 7. 풀스택 누적 효과 검증

본 절은 5개 히든 스탯이 모두 lv5에 도달했을 때 기존 칭호·전투 hook·PassiveBonusService 상한을 넘지 않는지 확인한다. 모든 수치는 한 용병 기준이며, 파티 전체 합산 보너스가 아니다.

### 7.1 5 스탯 lv5 풀스택 시뮬레이션

5 스탯 모두 lv5 도달한 용병 1명의 전투 + 비전투 시뮬레이션:

#### 7.1.1 전투 (M8b)

| hook | 풀스택 가산 | M8b clamp | 효과 |
|------|----------|----------|------|
| death_resistance | +0.10 | [0.20, capForQuest] | 사망률 ~10% 감소 |
| despair 면제 | +0.40 (확률) | (감정 시스템 별도) | 절망 발동 80% → 40% |
| critical | +0.05 | [0.05, 0.60] | 치명타 +5%p |
| evasion | +0.05 + 강공격 +0.075 | [0, 0.75] | 일반 회피 +5%p, 강공격 +12.5%p |
| mez_immune | +0.25 | (mez 차단 확률) | mez 25% 차단 |
| action_score | +2.5 | (정렬 가산) | 행동 우선순위 ~80% 상위 |
| featured_score | +1.0 | (결정적 장면 가산) | protagonist 후보 우위 |
| hit | +0.05 | [0.50, 0.95] | 명중 +5%p |

#### 7.1.2 비전투

| 효과 | 풀스택 가산 |
|-----|----------|
| injury_recovery | ×0.80 (20% 단축) |
| reputation_gain | +0.075 (PassiveBonusService 합산, 상한 +0.30) |
| item_drop | +0.025 (M5 재료 한정) |

### 7.2 칭호와의 합산 시나리오

풀스택 + 5 칭호 보유 용병:
- death_resistance +0.10 (히든) + 칭호 0~5% = ~+0.10~0.15
- evasion +0.05 (히든) + 칭호 ~0% = +0.05
- critical +0.05 (히든) + 칭호 ~0% = +0.05
- recovery_time reduction value 0.20 (히든) + 0.10 (title_pyegwang_survivor) = ×0.70 (30% 단축)
- reputation_gain +0.075 (히든) + +0.05 (title_soul_severer) + +0.03 (title_renowned) = +0.155 (상한 0.30 안)

→ **풀스택 + 5 칭호 = 칭호 ~8개 효과 수준**. 30~50시간 후반 베테랑 정체성. 정합.

### 7.3 풀스택 도달 시점 시뮬레이션

- 1주 1 스탯 lv5 도달 = ~1주 (운/전장 감각이 빠름, 투지가 느림)
- 5 스탯 모두 lv5 = ~3~4주 (1 용병당)
- 100 용병 풀이 모두 풀스택 = ~수개월 (실제 unrealistic)

플레이어가 가장 좋아하는 용병 ~5명 풀스택 = 1~2개월 누적. 컨셉 정합 (페이즈 1 #4 "장기 목표").

## 9. data-generator 수치 가이드

본 절은 data-generator가 SQL 시드와 검증 규칙을 만들 때 사용할 입력을 요약한다. 후속 구현에서 `post_reward_effects_json`은 PassiveEffect가 아니라 QuestCompletionService 전용 후처리 JSON으로 해석한다.

본 리포트의 수치는 페이즈 3 #3 "히든 스탯 정의 시드 5행" SQL 시드 + 페이즈 4 #3 spec 정수 입력.

- **대상 타입**: 별도 타입 스펙 불요 (M8b·M8a 신규 정적 테이블 패턴 재사용)
- **대상 테이블**: `hidden_stats` 5행 INSERT + 신규 테이블 CREATE
- **수치 범위**:
  - `combat_effects_json`: per-lv 값 (death_resistance 0.02, despair_immune 0.08 등)
  - `passive_effects_json`: `{"effects":[...]}` 컨테이너를 사용하는 PassiveEffect 지원 JSON 형식 (recovery_time_reduction, reputation_gain_modifier)
  - `post_reward_effects_json`: 신규 후처리 hook 표현 (item_drop_chance)
  - `level_thresholds`: `[1, 3, 7, 15, 30]` 5행 통일
- **외래 키 제약**: 없음 (정적 카탈로그)
- **balance 근거**: §3~§4 lv별 효과 정밀 + 임계값 시뮬레이션 + §5 사건 가중치 정합 + §7 풀스택 누적 검증
- **특수 요구**:
  - fortitude / grit는 passive_effects_json 포함
  - luck는 post_reward_effects_json (item_drop) 포함
  - fear_resistance / battle_sense는 M8b hook만 사용 (passive_effects_json NULL)
- **검증**:
  - level_thresholds JSON 배열 길이 5
  - combat_effects_json hook key가 M8b hook과 정합 (`death_resistance` / `critical_rate` / `evasion` / `mez_immune_chance` / `hit_chance` / `action_score` / `featured_score` + 신규 `despair_immune_chance` / `strong_attack_evasion`)
  - passive_effects_json 형식이 `PassiveEffect.parseEffects` 컨테이너 형식과 정합
  - `recovery_time_reduction.value`가 양수인지 확인
  - luck item_drop은 파티 합산이 아니라 참가자 최고 lv 1명 기준인지 확인
