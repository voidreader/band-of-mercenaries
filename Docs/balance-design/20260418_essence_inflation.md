## 현재 상태

### 정수가 진입하는 공식 경로

```
effectiveStr = (baseStr × (1 + levelBonusPerLevel × (level-1)) + permanentStr) × fatigueMod
  ↓
  levelBonusPerLevel = 0.1 (+10%/레벨, Lv5 = +40%)
  fatigueMod = 0.8 (tired) 또는 1.0 (normal)
  permanentStr: 정수 소비 누적 기여분, 레벨 증폭 제외 (순수 가산)
  ↓
partyPower = Σ(effectiveStr × w_str + effectiveInt × w_int + effectiveVit × w_vit + effectiveAgi × w_agi)
```

**장비 보정(산출물 1)과의 공식 관계**:
```
effectiveX = (baseX × levelBonus + equipmentX + permanentX) × fatigueMod
```

**장비 vs 정수의 비증폭 위치**:
- 장비 보정(equipmentX)과 정수(permanentX)는 **둘 다 레벨 보너스 외부에 가산**
- 장비는 **탈착 가능**, 정수는 **영구**
- 설계 의도: 정수 > 장비 > 레벨 보너스의 순서로 무게감 차별화 (정수의 희귀성과 영구성이 가장 큰 결단)

## 데이터 분석

### 분석 1 — 상한 설계 철학 검증 (기획서 3항 "T{N+1} base 도달" 주장)

기획서 3항은 T{N} 용병 상한 도달을 "T{N+1} 용병 base 수준"으로 주장했다. 4축별 실제 차이를 보면 주장이 **축마다 일관되지 않음**.

| 축 | T1→T2 차이 | T2→T3 | T3→T4 | T4→T5 | 초안 상한 (통일값) | 일치도 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| STR | +12.6 | +4.8 | +6.3 | +6.4 | +10/+20/+40/+70 | T3 상한 +40 ≫ +6.3 차이 (**과도**) |
| INT | +6.6 | +9.1 | +13.7 | +2.3 | 동일 | T5 상한 +120 ≫ +2.3 차이 (**과도**) |
| VIT | +25.8 | +16.5 | +14.5 | +18.9 | 동일 | T1 상한 +10 < +25.8 차이 (**부족**) |
| AGI | +0.2 | **−2.3** | **−1.9** | **−2.7** | 동일 | T5가 T1보다 AGI **낮음** (역방향) |

**결론**: 상한 초안은 "T{N+1} base 도달" 의미와 축별로 어긋남.

### 분석 2 — 상한 설계 철학 재정의 (컨설팅 Q1 확정)

**확정**: 초안 수치(+10/+20/+40/+70/+120) 유지, 의미 재정의.

**새 프레이밍**:
> "상한은 극한 하드코어 투자자가 도달 가능한 천장이며, 실제로는 M6 승급 비축·여러 용병 분산 투자 때문에 자연스럽게 상한에 못 닿는다. 단일 용병 단일 스탯 상한 도달은 '자랑할 만한 성취' 수준으로 설계."

**선택 근거**:
1. **엔드게임 임팩트 이미 clamp 95 흡수**: T3 warrior Lv5 STR +40 상한 도달 시 raid 파티 partyPower 189 / enemy 80 ratio 2.36 → clamp 95%. 상한 크기가 성공률에 미치는 실효 영향 미미.
2. **AGI 역방향은 "재미 요소"로 수용**: T1 용병에 AGI 정수를 집중 투자하여 고기동 하이브리드 빌드 가능 → "자유 장착/자유 사용" 철학 부합.
3. **M6 승급 비축 자동 유도**: 상한 도달 비용 + 승급 비용 합치면 자연스럽게 부족 체감 → 정수 = 승급 우선 재화 인식.
4. **단순 통일 상한**: 축별 차등 상한은 UX 복잡도 급증. 통일 상한이 플레이어 인지 부담 최소.

**포기한 대안**:
- 스탯별 차등 상한 (복잡) — 기각
- 축별 값 하향 (+8/+15/+25/+40/+60) — 수집 동기 감소 → 기각
- 상대 상한(base의 %) — 공식 복잡 → 기각

### 분석 3 — 효과 곡선 검증 (컨설팅 Q2 확정)

**확정**: 초안 +1/+2/+4/+7/+11 유지.

**스케일 대칭 검증**:

| 항목 | T1 | T2 | T3 | T4 | T5 | T5/T2 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| 장비 곡선 (weapon) | — | +3 | +6 | +10 | +15 | **5.0** |
| **정수 초안** | +1 | +2 | +4 | +7 | +11 | **5.5** |

두 곡선의 T5/T2 비가 5~5.5배로 대칭 → **"교체 가능 장비 vs 영구 소비 정수"의 가속 일관성** 유지.

**M2b 드랍 속도 기준 검증 (5~10개로 의미 있는 강화)**:

T3 용병 warrior, STR 단일 축 집중 투자 시뮬레이션:

| 정수 수집량 | 구성 예시 | permanent STR 증가 | T3 상한 +40 대비 도달률 |

**판정**: 5~10개 구간이 상한의 35~90%에 정확히 매핑 → "의미 있는 강화" 체감 곡선 확보.

**대안 비교 — 더 가파르거나 완만한 곡선의 문제**:

### 분석 6 — M6 승급 비용 선제 시뮬레이션 (컨설팅 Q3 확정)

**확정**: 옵션 **B (승급 우위)**. T1→T2 15개 / T2→T3 25개 / T3→T4 40개 / T4→T5 60개, **T1→T5 총 140개**.

**설계 철학**:
- 승급 비용 **>** 1축 상한 도달 비용 (승급이 더 큰 결단)
- 스탯 강화와 승급 양쪽이 **의미 있는 선택지**로 공존
- 질적 변환(승급) > 양적 강화(permanent +N)의 가치 차별화

**M2b 드랍 속도 역산 (평균 시간당 1.5~2개 정수 가정)**:

|:---:|:---:|:---:|:---:|
| A (대칭) 총 75개 | 5~7h | 37~50h | 너무 빠름 (헤비 1주에 엔드 도달) |
| **B (승급 우위) 총 140개** | **8~10h** | **70~95h** | **일치** (라이트 1개월 내 T2, 헤비 1주 내 T2~T3) |
| C (강한 장벽) 총 230개 | 10~13h | 115~155h | 너무 느림 (헤비 1주에 T2도 어려움) |

**자원 분산 시뮬레이션** — 플레이어가 40시간 플레이, 정수 60개 획득 시:

| 전략 선택 | 결과 | 비고 |
|---|---|---|
| T1 용병 1명 풀강화 | 주력 2축 상한 근접(40~50개) + 여분 | T1 최강이지만 티어 고정 |
| T1→T2 승급 후 부분 강화 | 15(승급) + 45(강화) | T2 + 중간 강화 |
| T1→T3 승급 최소화 | 40(T1→T3) + 20(강화) | T3 용병 + 소량 강화 |

**판정**: 옵션 B에서는 어떤 선택도 명확히 열등하지 않음 → **진짜 딜레마**. 전략 깊이 확보.

### 분석 7 — 사망·방출 전액 소멸 정책의 경제 리스크 평가

**기획서 7항**: 사망·방출 시 `permanent_*` 전액 소멸. 환원·회수 없음 (하드코어).

#### 리스크 A — 리스크 회피 과잉으로 고난도 퀘스트 기피

**시나리오**: T3 용병에 정수 5~10개 투자 완료 (가치 약 5시간 플레이).

| 사망률 구간 | 15% (난이도 3) | 22% (난이도 4) | 30% (난이도 5) |
|:---:|:---:|:---:|:---:|
| 대실패 시 사망률(실패 5%p, 대실패 30%p 기본) | ... | ... | ... |
| 플레이어 체감 | 감당 가능 | 망설임 발생 | 기피 |

**분석**: 정수 투자 용병을 난이도 5에 보내면 **15~30% 확률로 투자 전액 소멸** (사망 시). 합리적 플레이어는 "정수 투자 용병은 난이도 3~4로만 보낸다" 전략을 채택.

**리스크 완화 요소**:
- 성공률 95% 구간 근접 시(T3 풀강화 용병 포함 파티) 대실패 확률 5%p 유지 → 사망률 × 대실패율 = 30% × 5% = **1.5% 퀘스트당 사망** 실질
- 10시간 플레이 연속 성공 가정 시 누적 사망률 ~14% (1 − 0.985^10) → 허용 범위
- 부상·피로 관리로 실제 파견 빈도 낮춤 → 사망 위험 분산

**판정**: 리스크 A는 **설계 의도대로 작동**. "고투자 용병 보호" 전략 발생 → 플레이 깊이 추가.

#### 리스크 B — 신규 플레이어 좌절감

**시나리오**: 신규 유저가 첫 T2 정수 2~3개 투자 후 조기 사망.

**분석**: 정수 수급이 제한적인 초반(M1→M2a 전환기)에 투자 손실 체감은 큼. 기획서 7항의 활동 로그 경고("투입 정수 N개 소실")는 필수.

**완화 제안**:
- **UI 경고 강화**: 고난도 퀘스트 파견 시 "이 용병은 정수 N개가 투입되어 있습니다. 사망 시 모두 소실됩니다."  모달 1회 확인 절차 (정수 투자 용병 한정)
- **활동 로그 로그 가시성**: 사망 로그에 "잃어버린 정수" 섹션 명시 (학습 유도)

**판정**: UI·경고 수준의 완화만 권장. 환원·회수 메커닉은 추가하지 않음 (하드코어 철학 유지).

#### 리스크 C — 방출 악용 차단 확인

**기획서 7항**: 방출 시에도 전액 소멸. 환원 없음.

**분석**: "T5 정수 여러 개 먹였다가 방출로 환원" 같은 우회 경로는 완전 차단. 퇴직금(인건비 × 레벨)만 지급. 플레이어가 정수 투자 용병을 **방출할 이유가 거의 없음** → 자연스럽게 "정수 투자 = 장기 용병 낙점" 전략 형성.

**판정**: 차단 기제 건전. 추가 조치 불요.

### 분석 8 — 상한 초과 손실 UX 수치 기준 (기획서 8항 검증)

**기획서 8항**: 프리뷰 경고 + 사용자 판단. 적용 효과 = `min(정수 효과, 잔량)`. 초과분 손실.

**프리뷰 팝업 기준값 정의**:

| 상황 | 손실량 | UI 표시 |
|:---:|:---:|---|
| 잔량 ≥ 정수 효과 | 0 | 일반 프리뷰 (손실 없음) |
| 0 < 잔량 < 정수 효과 | 정수 효과 − 잔량 | **주의 경고** (손실 예정량 명시, 계속/취소) |
| 잔량 = 0 (상한 완전 도달) | 정수 효과 전체 | **강한 경고** 또는 사용 버튼 비활성 |

**사례 구체화**:

```

[ 주의: 상한 초과 ]
대상: 김철수 (T1, Lv3)
사용: 태초의 힘의 정수 (+11)
현재 STR permanent: +8
T1 STR 상한: +10
상한 잔량: +2
적용 효과: +2 (9 포인트는 소실됩니다)

[취소]  [손실 감수하고 사용]
```

**판정**: 기획서 8항의 정책이 수치 기준으로 그대로 작동. 추가 조정 불요.

### 분석 9 — 인플레이션 종합 시뮬레이션 (장기 성장 곡선)

## 플레이어 체감 분석

### 초반 (M2a 수동 지급 시나리오 / M2b 초기)

- 첫 T1~T2 정수 수령 시 "스탯이 눈에 띄게 오름" 체감 (T1 용병 base STR 7.6에 +2는 +26% 증가)
- 사망·방출 경고로 **신중함** 학습 → 정수 투자 용병을 난이도 3~4 위주로 운용하는 전략 형성
- 프리뷰 팝업이 상한 잔량을 명시 → "지금 쓸지, 모아둘지"의 판단 훈련

### 중반 (T2~T3 용병 중심, 정수 5~20개 축적)

- 단일 용병 1~2축 상한 근접 → 투자 결정의 깊이 체감
- T4~T5 정수 첫 획득 시 "T3 용병에 쓸까, T4 용병 비축용으로 둘까" 결정 긴장감
- M6 승급 비축 인식 형성 → "이 정수 15개를 스탯에 쓸까, T1→T2 승급 재료로 모을까"

### 엔드게임 (T4~T5 용병, 정수 50개+ 축적)

- 95% clamp에서 정수 효과가 성공률에 직접 반영되지 않음 → 정수의 역할이 **"수집 성취 + 승급 재화"** 로 전환
- 상한 도달 용병이 1~2명 존재 → "완성된 에이스" 개념 확립, 이를 핵심 전력으로 운용
- M6 진입 시점에 정수 재고가 곧 승급 가능성 → 장기 진행 동력 유지

### 체감 리스크 정리

- **리스크 A (투자 용병 기피)**: 분석 7에서 설계 의도로 수용. 완화 불필요
- **리스크 B (신규 좌절)**: UI 경고 + 로그 명시로 완화
- **리스크 C (방출 악용)**: 차단 완료

## 조정 제안

### 최종 수치 확정

#### 효과 곡선 (정수 1개 사용 시 `permanent_stat_gain.{stat}` 증가값)

| 티어 | 효과 | 대응 slot |
|:---:|:---:|---|
| T1 | **+1** | `essence_str_t1` / `essence_int_t1` / `essence_vit_t1` / `essence_agi_t1` |
| T2 | **+2** | `essence_*_t2` |
| T3 | **+4** | `essence_*_t3` |
| T4 | **+7** | `essence_*_t4` |
| T5 | **+11** | `essence_*_t5` |

- 타입: `integer` 고정. 소수점 없음
- 4축(STR/INT/VIT/AGI) 동일 곡선 적용

#### 용병 티어별 축당 상한

| 용병 티어 | 축당 `permanent_*` 상한 | 용병 1명 4축 총 잠재 |
|:---:|:---:|:---:|
| T1 | **+10** | +40 |
| T2 | **+20** | +80 |
| T3 | **+40** | +160 |
| T4 | **+70** | +280 |
| T5 | **+120** | +480 |

- 축별 독립 카운트
- 상한 도달 시 프리뷰 경고 후 사용자 판단으로 "손실 감수 사용" 허용 (기획서 8항)

#### effect_json 스키마 (data-generator 입력)

```json
// 정수 아이템 effect_json 구조
{
  "permanent_stat_gain": {
    "<stat_key>": <value>  // stat_key ∈ {str, intelligence, vit, agi}, value = 티어별 곡선 값
  }
}

// 예시
{ "permanent_stat_gain": { "str": 11 } }        // 태초의 힘의 정수 (T5)
{ "permanent_stat_gain": { "vit": 4 } }         // 고대의 수호의 정수 (T3)
{ "permanent_stat_gain": { "agi": 1 } }         // 민첩의 정수 (T1)
```

- **단일 축 정책**: 정수 1개는 1개 스탯 축만 증가. 복수 축 혼합 금지
- 필드 키는 `Mercenary` 모델의 HiveField 이름과 정확히 일치 (`str`, `intelligence`, `vit`, `agi`)

### 스태킹·공식 규칙

| 항목 | 규칙 | 근거 |
|---|---|---|
| permanent 누적 | **가산** (min(현재 + 정수 효과, 티어 상한)) | 기획서 2·8항 |
| effective 공식 | `(base × (1 + levelBonus) + permanent) × fatigueMod` | 기획서 6항, 장비 공식과 동일 구조 |
| 상한 초과 시 | 적용 = `min(효과, 잔량)`, 초과분 손실, 정수는 소멸 | 기획서 8항, 분석 8 |
| 사망·방출 시 | `permanent_*` 전액 소멸, 환원·회수 없음 | 기획서 7항, 분석 7 |
| 다중 소비 | M2a 범위 외 (1회 1개) | 기획서 4-4 |
| 승급 시 상한 확장 | M6 기획에서 확정 (본 리포트 제안: `permanent` 유지 + 상한만 확장) | 기획서 3항, 분석 6 |

### 컨셉 기획서 연동

초기 아이템 세트 기획서(`[content]20260418_initial_item_set.md`)의 명명 체계 준수:
- T1: "힘의 정수" / "지혜의 정수" / "수호의 정수" / "민첩의 정수"
- T2: "오래된 {X}의 정수"
- T3: "고대의 {X}의 정수"
- T4: "태고의 {X}의 정수"
- T5: "태초의 {X}의 정수"

### effect_json 스키마 (고정)

```json
{
  "permanent_stat_gain": {
    "<stat_key>": <int_value>
  }
}
```

- `stat_key`는 정확히 `str`, `intelligence`, `vit`, `agi` 중 택 1 (Mercenary 모델 필드명 일치)
- `int_value`는 위 매트릭스 값 (정수 타입, 소수점 없음)
- 다른 필드 금지 (단일 축 정책)

### 구조적 제약

- **매트릭스 완전성**: 4축 × 5티어 = 20종 모두 생성 필수. 하나라도 누락 시 플레이어 진행 막힘
- **tier 필드**: `items.tier` 컬럼에 1~5 값 정확히 대응
- **slot 값**: `essence_str_t1` ~ `essence_agi_t5` 중 택 1 또는 `slot=essence_str` + `tier` 조합 (spec-writer 페이즈 4에서 네이밍 최종 결정). **둘 중 하나의 규약 선택 후 20종 전체 일관 적용**
- **category 값**: `consumable` 고정
- **flavor_text 톤**: 초기 아이템 세트 기획서의 시간·연대 계열 서사. 예: T5 태초의 → "세계가 형체를 갖기 전부터 존재한 결정"

### 수치 출처

본 리포트 **분석 3**(효과 곡선) 및 **컨설팅 Q2 확정**(+1/+2/+4/+7/+11 유지)에서 도출.

### 특수 요구

- **T1 정수 4종 생성 필수**: M2a 인프라 검증에서 T1 정수가 가장 먼저 쓰이므로, "T1 정수는 M2b에서 드랍하지 않음" 정책이 있어도 데이터는 존재해야 함 (operation-bom 수동 지급 경로 활용)
- **효과 곡선 일관성**: 4축 모두 동일 곡선 적용 → 플레이어 인지 부담 최소화
- **승급 재료 이중 역할**: M6 기획에서 동일 아이템 풀이 재사용됨. data-generator는 M2a 용도만 생성, M6 기획 시 추가 메타 필드가 필요하면 별도 migration

### 페이즈 4 개발 명세 반영 필수 항목

`/spec-writer` 호출 시 반드시 반영:

1. **Mercenary 모델 확장**: HiveField 18~21에 `permanentStr/Int/Vit/Agi` (int, default 0) 추가 + migration
2. **effective 공식 확장**: `(baseX × (1 + levelBonus) + equipmentX + permanentX) × fatigueMod`. 장비(산출물 1)와 정수가 둘 다 레벨 증폭 외부 가산
3. **EssenceService 구조**:
   - `canApply(mercenary, essenceItem) → bool` (상한 판정)
   - `previewApply(mercenary, essenceItem) → PreviewResult` (현재/사용 후/손실 계산)
   - `apply(mercenary, essenceItem) → ApplyResult` (min(효과, 잔량) 적용, 아이템 소비)
4. **프리뷰 팝업 공통 컴포넌트**: 경로 A(용병 상세)/B(인벤토리) 양쪽에서 동일 팝업 재사용
5. **사망·방출 시 전액 소멸 확정**: QuestCompletionService(사망 처리), 방출 다이얼로그(경고 추가 + 소실 확정)
6. **활동 로그 타입 추가**: `ActivityLogType.essenceApplied` / `essenceLostOnDeath` / `essenceLostOnRelease`
7. **T{N} 승급 후 permanent 유지 정책**: M6 기획 시 확정. 본 리포트 제안은 **유지** (승급은 상한만 확장, permanent 값 보존)
