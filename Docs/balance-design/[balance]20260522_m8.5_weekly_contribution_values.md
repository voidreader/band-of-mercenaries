## 1. 개요

본 리포트는 페이즈 1 #6에서 정의된 5항목 점수 가중치(지역 안정화 50/200/500 / 세력 difficulty×50 / 전투 100/10/1000 / 제작 tier×30 / 위업 category별)를 정밀화한다. 검토 과정에서 페이즈 1 #6 권장값으로 시뮬레이션한 결과 **전투 활약 항목이 베테랑 1주 점수의 약 62% 비중을 차지**해 한 영역 편중이 발생했다. critical 발동 가중치를 10→3으로 약화하고 솔로 의뢰 대성공 점수를 1000→750으로 낮춰 전투 비중을 약 52%까지 낮춘다. 또한 위업 categories에 누락된 `settlement_trust_belonging`을 200점으로 추가한다.

### 1.1 주요 보완 결과

| 항목 | 페이즈 1 #6 권장 | 보완 결과 | 근거 |
|------|----------------|---------|------|
| 전투 critical 발동 점수 | 10 | **3** | 베테랑 1주 critical ~250회 × 10 = 2500점 → 전투 약 62% 비중 (편중) |
| 솔로 의뢰 대성공 점수 | 1000 | **750** | 베테랑 1주 대성공 ~2회 기준 2000점 → 1500점으로 완화 |
| 위업 `settlement_trust_belonging` | (미정의) | **200** | reputation_rank와 동급 (1회성 큰 사건) |
| 위업 `craft_first_rare` | 100 | **100** (유지) | 페이즈 1 #6 권장 적정 |
| 항목별 주 cap 도입 | (없음) | **combat 5500 포함 5항목 cap 권장** | 한 영역 점수 폭주 방지 (보조 안전망) |
| 동률 처리 정책 | (미정의) | **total_score desc → updated_at asc** | RLS SELECT 결정성 보장 |

### 3.3 전투 활약 (`combat`) — frequent event 가중 약화

페이즈 1 #6 권장 critical 10점 → **본 리포트 3점**. 솔로 의뢰 대성공은 1000점 → **750점**으로 낮춘다. decisive scene은 사용자에게 직접 노출되는 대표 전투 사건이므로 100점을 유지한다.

| 사건 | 점수 | 1주 평균 (베테랑) | 1주 점수 |
|------|------|--------------|--------|
| decisive scene 기여 ≥ 5 | 100 | ~25회 | 2,500 |
| critical 발동 | **3** (조정 ↓) | ~250회 | 750 |
| 솔로 의뢰 대성공 | **750** (조정 ↓) | ~2회 | 1,500 |
| **합계** | | | **4,750** |

페이즈 1 #6 권장(critical 10, solo 1000): 베테랑 전투 7,000점 → 전체 약 62% 비중. **편중**.
본 리포트(critical 3, solo 750): 베테랑 전투 4,750점 → 전체 약 52% 비중. 전투가 핵심 루프임을 반영하되, 단일 항목이 60%를 넘지 않도록 조정한다.

#### 3.3.1 critical 가중치 sensitivity

| critical 점수 | 베테랑 1주 critical 점수 | 전투 합계 | 전투 비중 |
|------------|-------------------|---------|---------|
| 10 (페이즈 1 #6, solo 1000) | 2,500 | 7,000 | 62% |
| 5 (solo 750) | 1,250 | 5,250 | 55% |
| **3 (권장, solo 750)** | 750 | 4,750 | 52% |
| 0 (solo 750) | 0 | 4,000 | 48% |

critical 3점은 "발동 자체에 가산이 있되 비중을 압도하지 않는" 균형선이다. critical은 전투마다 반복 발생할 수 있는 raw event이므로 decisive scene보다 낮은 점수를 부여한다.

신규 유저 (1주 critical ~100, decisive ~10, solo 0):
- decisive 10 × 100 = 1,000
- critical 100 × 3 = 300
- solo 0
- **합계 1,300**

## 4. 항목별 cap 도입 검토 (페이즈 1 #6 보완)

### 4.1 cap 미도입 시 risk

페이즈 1 #6 §3.1.2 "항목별 cap 없음 (자연 비중 분포)" 정책. 그러나 다음 인플레이션 시나리오 가능:

- **전투 활약 폭주**: 베테랑 유저가 1주 50시간+ 플레이 시 critical ~500회 × 3 = 1,500점. decisive·솔로 대성공까지 합산하면 전투 합계 ~8,000점. 비중 60%+
- **세력 의뢰 폭주**: 모든 세력 d5 완수 1주 ~5회 = 1,250점. 합계 ~3,500점

### 4.2 항목별 주 cap 권장

페이즈 1 #6 권장 안과 사용자 명령서 "한 영역 편중 방지" 정합:

| 항목 | 주 cap |
|------|------|
| 지역 안정화 | 2,500 |
| 세력 의뢰 | 2,500 |
| 전투 활약 | **5,500** |
| 제작 | 1,500 |
| 위업 | 2,000 |

cap 도달 시 추가 사건은 점수 가산 안 함 (UserData buffer 누적 시 min(...))

베테랑 50시간+ 시뮬레이션:
- 전투: ~8,000 → cap 5,500으로 차단
- 다른 영역: cap 미도달 → 자연 누적
- **합계 cap 적용 후 ~13,500 (cap 없을 시 ~15,500)**

→ cap 도입으로 1주 점수 상한을 13,000~14,000점 부근으로 안정화한다. 인플레이션 방지. **확정 권장**.

### 4.3 cap 도입 sensitivity

| cap 정책 | 베테랑 50h 1주 점수 | 비고 |
|---------|------------------|------|
| cap 없음 (페이즈 1 #6) | ~17,000 | 전투 편중 |
| cap 없음 (본 조정 가중치) | ~15,500 | 전투 편중 완화, 장시간 플레이 편중 잔존 |
| **본 권장 cap** | ~13,500 | 균형 + 자연 누적 |
| 항목별 cap 1,500 균등 | ~7,500 | 너무 낮음, 베테랑 동기 약화 |

본 권장 cap은 "베테랑 동기 유지 + 폭주 방지" 균형. **확정**.

## 5. 시간당 점수 효율 검증

### 5.1 1시간당 점수 효율

| 유저 | 1주 점수 | 1주 시간 | 시간당 점수 |
|-----|---------|---------|---------|
| 신규 | 3,275 | 10h | 328 |
| 중급 | 6,400 | 15h | 427 |
| 베테랑 | 9,125 | 20h | 456 |
| 베테랑+ (cap 적용) | 13,500 | 50h | 270 |

→ 시간당 효율은 5~10시간 구간에서 328→427점/시간으로 가속, 그 이후 베테랑 단계에서 456점/시간으로 정점. 50시간+ 구간은 cap으로 270점/시간까지 감속한다.

곡선 의도:
- 신규 유저: 학습 단계 — 시간당 효율 낮음
- 중급/베테랑: 본격 진행 — 시간당 효율 최고
- 50시간+ : cap 도달 — 감속

→ 자연스러운 progression curve. 인플레이션 우려 없음.

### 5.2 다른 채널과의 정합

페이즈 2 #2 §7.3 솔로 의뢰 골드 비중 33%와 정합:
- 솔로 의뢰 점수 (전투 활약 750점/회) × 1주 2회 = 1,500점 (전체 점수의 ~16%)
- 일반 의뢰 점수 (decisive 100점/회 + critical 3점/회) × 1주 50회 = ~4,000+ 점

솔로 의뢰는 골드 비중 33%, 점수 비중 16% — 골드 보상은 강하게, 랭킹 점수는 과도하게 끌어올리지 않는 구조다. 적정.

## 6. 동률 처리 정책

### 6.1 top 100 정렬 정합

페이즈 1 #6 §3.5.2 SELECT 쿼리:
```sql
SELECT user_id, total_score FROM weekly_contributions
WHERE week_start_utc = '...'
ORDER BY total_score DESC LIMIT 100;
```

동률 시 user_id alphabetical 정렬은 결정성 없음. **본 리포트 권장**:

```sql
ORDER BY total_score DESC, updated_at ASC, user_id ASC LIMIT 100;
```

- 1순위: total_score DESC
- 2순위: updated_at ASC (먼저 도달한 사용자 우선 — "선착순 영광")
- 3순위: user_id ASC (deterministic fallback)

### 6.2 자기 위치 lookup

자기 위치는 별도 쿼리:
```sql
SELECT COUNT(*) + 1 AS my_rank
FROM weekly_contributions
WHERE week_start_utc = '...'
  AND (total_score > my_score
       OR (total_score = my_score AND updated_at < my_updated_at)
       OR (total_score = my_score AND updated_at = my_updated_at AND user_id < my_user_id));
```

페이즈 4 #6 spec에서 RPC `get_my_weekly_rank()` 함수로 캡슐화.

### 7.2 weekly_contributions 테이블 INSERT 정합

페이즈 1 #6 §3.5.1 스키마 + 본 리포트 cap 적용:

```dart
void incrementScoreWithCap(String category, int delta) {
  final cap = WeeklyContributionScoreConfig.weeklyCategoryCaps[category]!;
  final current = userData.weeklyScoreBuffer[category] ?? 0;
  if (current >= cap) return;  // cap 도달 시 가산 안 함
  final newValue = math.min(current + delta, cap);
  userData.weeklyScoreBuffer[category] = newValue;
  scheduleUploadIfDirty();
}
```

MVP에서는 클라이언트 측 buffer에서 cap을 적용한 절대값을 Supabase RPC로 업로드한다. RPC는 5항목 값을 upsert하고 `total_score`는 GENERATED 컬럼 또는 SQL 계산식으로 산출한다. 서버측 사건 검증과 delta 재계산은 M9로 위임한다.

## 8. 시뮬레이션 정확도 검증

### 8.1 가중치 ±10% sensitivity

| 변경 | 베테랑 1주 점수 |
|-----|--------------|
| 본 권장 (critical 3 / solo 750) | 9,125 |
| critical 5 / solo 750 | 9,625 |
| critical 10 / solo 1000 (페이즈 1 #6) | 11,375 |
| decisive 150 (+50%) | 10,375 |
| solo 1000 (+250점) | 9,625 |
| 세력 difficulty×70 | 9,745 |

권장 가중치는 베테랑 ~10,000점 안정선. 변경 시 ±10% 영향.

### 8.2 cap 도입 sensitivity

| cap 정책 | 베테랑 50h 점수 |
|---------|--------------|
| cap 없음 (페이즈 1 #6) | ~17,000 |
| cap 없음 (본 조정 가중치) | ~15,500 |
| **본 권장** | ~13,500 |
| 절반 cap | ~8,500 |

본 권장 cap은 "성장 동기 유지 + 폭주 방지" 균형.

### 8.3 항목 비중 검증

본 권장 가중치 베테랑 비중:
- 안정화 15% / 세력 17% / 전투 52% / 제작 9% / 위업 7%

전투 52%는 "전투가 게임의 핵심 루프"임을 반영한다. 60% 이상 편중은 페이즈 1 #6 권장에서 발생하므로 critical 3점, solo 750점 조정으로 해소한다.

## 9. data-generator 수치 가이드

본 리포트는 SQL 시드 직접 생성 불필요. 페이즈 3 #5 "주간 기여도 점수 메타데이터"에서 `weekly_contributions` 테이블 + RPC + RLS + pg_cron SQL을 작성하며, 본 리포트 §3·§7의 매트릭스를 입력으로 사용.

- **대상 타입**: 별도 타입 스펙 불요 (사용자 데이터 테이블, 신규 INSERT 데이터 없음)
- **대상 테이블**:
  - `weekly_contributions` (사용자별 행, 마이그레이션 시드 0행 — 사용자 데이터)
  - `weekly_contribution_categories` (선택 — 5항목 메타데이터, Dart 정적 상수로 대체 권장)
- **수치 범위**: 본 리포트 §7.1 `WeeklyContributionScoreConfig` 그대로
- **외래 키 제약**: RLS 본인 INSERT/UPDATE + 모든 SELECT
- **balance 근거**: §3~§5 1주 누적 시뮬레이션 + §4 cap 도입 + §5 시간당 효율 검증
- **특수 요구**:
  - `total_score`는 GENERATED ALWAYS AS (...) STORED — 클라이언트가 별도 보내지 않음
  - 항목별 cap은 **클라이언트 buffer 적용** (서버는 단순 UPSERT)
- **검증**:
  - 5항목 합산 = total_score
  - GENERATED 컬럼 자동 계산
