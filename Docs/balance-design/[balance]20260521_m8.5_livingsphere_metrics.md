## 1. 개요

본 리포트는 생활권 완성도 대시보드의 6개 지표를 0~100%로 정규화하고, 통합 완성도와 30분/8시간 목표 추천 점수식을 확정한다. 검토 과정에서 기존 초안의 중복 집계, 실제 데이터와 맞지 않는 분모, 임박도 점수 방향 오류를 수정했다.

### 1.1 주요 보완 결과

다음 항목은 구현 명세 입력 전에 반드시 반영해야 하는 수정 사항이다.

| 항목 | 초안 문제 | 보완 결과 |
|------|----------|----------|
| 사건 완료율 | 체인 완료, 인프라, 신뢰도 위업을 별도 사건으로 더해 다른 지표와 중복 집계 | 사건 분모를 체인 6 + 발견 3 + region_state 의뢰 2 = 11로 고정 |
| 자원·제작 | 특산품 5종 설명이 `region_exclusive=3` 4종과 중복되는 재료를 섞어 불명확 | 5종 allowlist를 명시하고 레시피는 region 3 unlock 조건 10개 전체를 집계 |
| 영향력 | 미접촉 세력의 `reputation=0`을 36%로 환산해 신규 유저 시작 완성도가 과대 표시 | `FactionRelationStage.untouched`는 0점, `noticed`부터 점수 부여 |
| 위업 달성률 | `tier >= 3` 조건으로 발급되지 않는 T2 제작 위업 2개를 분모에 포함 | 실제 발급 가능한 region 3 위업 5개로 분모 고정 |
| 목표 추천 | 임박도 산식이 일부 반대로 되어 목표가 멀수록 높은 점수를 받음 | 모든 gap 기반 factor를 `1 - gap / threshold` 형태로 통일 |

### 1.2 목표 곡선

M8.5 MVP는 region 3 더스트플레인 단독 대시보드다. 시작 거점이 이미 region 3이므로 신규 유저 시작값은 0%가 아니라 `dangerScore=0`에 따른 안정도 기초값을 포함한다.

| 구간 | 목표 통합 완성도 | 의도 |
|------|----------------|------|
| 신규 시작 직후 | 8~12% | 대시보드가 비어 보이지 않되, 대부분 미완료 상태로 보인다 |
| 1~3시간 플레이 | 25~30% | 신뢰도 2단계와 초반 체인 진행이 체감된다 |
| 5~10시간 플레이 | 60~70% | 체인 완주와 인프라 Tier 3 진입이 큰 진척으로 보인다 |
| Tier 4 도달, 평정 직전 | 88~94% | 막판 목표가 남아 있음을 보여준다 |
| 완전 클리어 | 100% | 6개 지표가 모두 닫힌다 |

## 2. 데이터 기준

본 절은 산식 분모를 고정하기 위해 사용하는 region 3 관련 데이터 기준을 정리한다. 코드와 마이그레이션 기준으로 확인 가능한 항목만 분모에 포함한다.

### 2.1 region 3 기준 분모

region 3 MVP에서 직접 집계하는 핵심 분모는 다음과 같다.

| 카테고리 | 분모 | 근거 |
|----------|------|------|
| 거점 체인 단계 | 6 | `settlement_3_pyegwang_reopen` 6단계 |
| region 3 발견 | 3 | `disc_dustvile_pyegwang_normal` / `hidden` / `deepest` |
| region 3 M7 상태 의뢰 | 2 | `qp_m7_r3_cave_bats`, `qp_m7_r3_safe_escort` |
| 특산품 allowlist | 5 | §3.4의 명시 allowlist |
| region 3 제작 레시피 | 10 | `recipe_dustvile_*` 10개 |
| region 3 위업 | 5 | §3.6의 명시 allowlist |

### 2.2 region 3 제작 레시피 분포

`crafting_recipes`의 region 3 레시피는 총 10개다. `first_acquired_item` 조건도 사용자가 해금 상태로 체감하는 제작 목표이므로 레시피 해금 분모에 포함한다.

| unlock 조건 | 개수 | 레시피 |
|-------------|------|--------|
| `trust_level >= 2` | 4 | `banner_repair`, `hide_bundle`, `ore_polished`, `armor_solid` |
| `trust_level >= 3` | 2 | `herbalist_seal`, `herb_pouch` |
| `chain_step settlement_3_pyegwang_reopen step 1` | 2 | `miner_dagger`, `rusty_pickaxe` |
| `chain_step settlement_3_pyegwang_reopen step 6` | 1 | `pyegwang_relic` |
| `first_acquired_item mat_relic_pyegwang_shard` | 1 | `miner_charm` |
| **합계** | **10** | |

### 2.3 region 3 위업 분모 기준

현재 `CraftingService.craft()`는 결과 아이템 `tier >= 3`일 때만 `craft_first_rare:$recipeId`를 발급한다. 따라서 T2 결과물인 `recipe_dustvile_banner_repair`와 `recipe_dustvile_miner_charm`은 현 코드 기준 위업 분모에 넣지 않는다.

| 위업 templateId | 포함 여부 | 근거 |
|-----------------|----------|------|
| `region_pacified:region_3` | 포함 | region 3 첫 평온 또는 안정권 진입 |
| `infrastructure_tier:tier_4` | 포함 | 더스트빌 인프라 Tier 4 |
| `settlement_event_completed:settlement_3_pyegwang_reopen` | 포함 | 폐광길 재개방 완주 |
| `settlement_trust_belonging:region_3` | 포함 | 더스트빌 신뢰도 4단계 |
| `craft_first_rare:recipe_dustvile_pyegwang_relic` | 포함 | T3 결과물 제작 |
| `craft_first_rare:recipe_dustvile_banner_repair` | 제외 | 결과물 T2, 현 코드에서 미발급 |
| `craft_first_rare:recipe_dustvile_miner_charm` | 제외 | 결과물 T2, 현 코드에서 미발급 |

## 3. 6 지표 정규화 산식

각 지표는 0~100%로 환산한다. 분모가 고정된 지표는 allowlist를 사용하고, 분모가 상태값인 지표는 clamp로 범위를 보정한다.

### 3.1 안정도 (`stability_pct`)

안정도는 `dangerScore`가 낮을수록 높아진다. `dangerScore`가 null이면 region 3 기본값 0으로 처리한다.

```text
stability_pct = clamp(((100 - dangerScore) / 200) * 100, 0, 100)
```

| dangerScore | stability_pct | 단계 라벨 |
|-------------|---------------|-----------|
| -100 | 100% | 안정 |
| -50 | 75% | 안정/평온 경계 |
| 0 | 50% | 평온 |
| +50 | 25% | 긴장 |
| +100 | 0% | 위협 |

### 3.2 거점 발전 (`infrastructure_pct`)

거점 발전은 `infrastructureTier` 1~4를 선형 환산한다. null이면 Tier 1로 처리한다.

```text
infrastructure_pct = clamp(((infrastructureTier - 1) / 3) * 100, 0, 100)
```

| infrastructureTier | infrastructure_pct |
|--------------------|--------------------|
| 1, 고립 | 0% |
| 2, 연결 | 33.3% |
| 3, 거점화 | 66.7% |
| 4, 변방의 중심 | 100% |

펼침 영역에는 신뢰도 보조 진행률을 함께 표시한다.

```text
trust_progress_pct =
  clamp((currentTrust - currentTrustThreshold) /
        (nextTrustThreshold - currentTrustThreshold) * 100, 0, 100)
```

### 3.3 사건 완료율 (`event_completion_pct`)

사건 완료율은 장기 완료 이벤트만 집계한다. 위업 자체는 별도 지표에서 집계하므로 사건 완료율에는 중복 포함하지 않는다.

| 카테고리 | 분모 | 분자 산출 |
|----------|------|-----------|
| 거점 체인 단계 | 6 | `ChainQuestProgress.currentStep`, 완주 시 6 |
| region 3 발견 | 3 | `RegionState.triggeredDiscoveries` 중 region 3 allowlist |
| region 3 M7 상태 의뢰 | 2 | `questPoolCompletionCounts`에서 대상 pool 완료 수 |
| **합계** | **11** | |

```text
event_completion_pct = clamp((completed_events / 11) * 100, 0, 100)
```

구현 시 `settlement_event_completed:settlement_3_pyegwang_reopen`은 체인 완주 여부를 보정하는 fallback marker로만 사용한다. 체인 6단계와 별도 점수로 더하지 않는다.

### 3.4 자원·제작 (`resource_craft_pct`)

자원·제작은 특산품 획득과 레시피 해금을 50:50으로 반영한다. 특산품은 `firstAcquiredMaterialIds` 전체가 아니라 region 3 대시보드용 allowlist만 집계한다.

| 구분 | 분모 | allowlist |
|------|------|-----------|
| 특산품 | 5 | `mat_herb_dust_resin`, `mat_relic_pyegwang_pickaxe_head`, `mat_relic_pyegwang_shard`, `mat_monster_giant_bat_fang`, `mat_relic_ancient_seal_piece` |
| 레시피 | 10 | `recipe_dustvile_*` 10개 |

```text
resource_craft_pct =
  clamp((acquired_materials / 5 * 50) + (unlocked_recipes / 10 * 50), 0, 100)
```

`mat_hide_faded_cloth`는 더스트빌 제작에 중요하지만 신뢰도 보너스와 이동 선택지 보상으로 지급되는 보조 재료다. 특산품 5종에는 넣지 않고 레시피 입력재로만 반영한다.

### 3.5 영향력 (`influence_pct`)

영향력은 M8a 활성 3세력의 관계 단계 평균으로 계산한다. 미접촉 세력을 0점으로 처리하여 신규 유저 시작값이 과대 표시되지 않게 한다.

대상 세력은 다음 3개로 고정한다.

| factionId | 이름 |
|-----------|------|
| `faction_adventurers_guild` | 모험가 길드 |
| `faction_merchants_alliance` | 상인 연합 |
| `faction_warriors_guild` | 전사 길드 |

```text
faction_score(fid) =
  if stage == untouched:
    0
  else if stage == noticed:
    20
  else if stage == patronage:
    20 + clamp(reputation, 0, 10) / 10 * 20
  else if isJoined:
    clamp(50 + reputation / 100 * 50, 0, 100)
  else if reputation < 0:
    clamp((reputation + 100) / 100 * 20, 0, 20)

influence_pct = average(faction_score for 3 active factions)
```

이 산식은 `FactionRelationStage.resolve()`의 의미를 따른다. `FactionState`가 없거나 접촉점이 비활성인 `reputation=0` 세력은 `untouched`로 간주한다.

| 상태 | 점수 |
|------|------|
| 미접촉 | 0 |
| 접촉 확인, 평판 0 | 20 |
| 미가입 평판 10 | 40 |
| 가입 평판 0 | 50 |
| 가입 평판 50 | 75 |
| 가입 평판 100 | 100 |

### 3.6 위업 달성률 (`achievement_pct`)

위업 달성률은 region 3 관련 위업 5개를 고정 분모로 사용한다. `BandAchievement.regionId == 3`을 우선하되, 과거 데이터 호환을 위해 templateId allowlist를 함께 적용한다.

| 위업 templateId | 분류 |
|-----------------|------|
| `region_pacified:region_3` | 지역 평정 |
| `infrastructure_tier:tier_4` | 인프라 성장 |
| `settlement_event_completed:settlement_3_pyegwang_reopen` | 거점 사건 |
| `settlement_trust_belonging:region_3` | 거점 신뢰도 |
| `craft_first_rare:recipe_dustvile_pyegwang_relic` | 희귀 제작 |

```text
achievement_pct = clamp((region_3_achievements / 5) * 100, 0, 100)
```

향후 T3 이상 더스트빌 제작 결과물이 추가되면 `craft_first_rare:*` allowlist와 분모를 함께 늘린다. T2 제작 결과물만 추가하는 방식은 현 코드의 발급 조건과 맞지 않으므로 권장하지 않는다.

## 4. 통합 완성도 산식

통합 완성도는 6개 지표의 가중평균이다. 안정도, 거점 발전, 사건 완료율을 핵심 진행축으로 두고 자원·제작과 위업을 중간 비중으로 둔다.

```text
total_completion_pct =
    stability_pct        * 0.20
  + infrastructure_pct   * 0.20
  + event_completion_pct * 0.20
  + resource_craft_pct   * 0.15
  + influence_pct        * 0.10
  + achievement_pct      * 0.15
```

| 지표 | 가중치 | 근거 |
|------|--------|------|
| 안정도 | 0.20 | 생활권 상태 변화의 핵심 축 |
| 거점 발전 | 0.20 | M7 인프라 성장의 대표 진행도 |
| 사건 완료율 | 0.20 | 체인, 발견, 상태 의뢰의 실제 완료량 |
| 자원·제작 | 0.15 | 반복 루프와 대장간 목표 |
| 영향력 | 0.10 | region 3 외부 시스템이므로 보조 비중 |
| 위업 달성률 | 0.15 | 장기 목표와 연대기 보상 |

### 4.1 곡선 시뮬레이션

보완된 분모와 산식으로 대표 상태를 다시 계산한다.

#### 신규 시작 직후

시작 직후에는 region 3 상태가 기본값으로 존재하지만 진행성 데이터는 없다.

| 지표 | 값 | 산정 |
|------|----|------|
| 안정도 | 50.0% | dangerScore 0 |
| 거점 발전 | 0.0% | Tier 1 |
| 사건 완료율 | 0.0% | 0/11 |
| 자원·제작 | 0.0% | 0/5 + 0/10 |
| 영향력 | 0.0% | 3세력 모두 미접촉 |
| 위업 달성률 | 0.0% | 0/5 |

```text
total_completion_pct = 10.0%
```

#### 1~3시간 플레이

신뢰도 2단계에 진입하고 체인 초반과 첫 발견을 진행한 상태다.

| 지표 | 값 | 산정 |
|------|----|------|
| 안정도 | 60.0% | dangerScore -20 |
| 거점 발전 | 33.3% | Tier 2 |
| 사건 완료율 | 27.3% | 3/11 |
| 자원·제작 | 20.0% | 1/5 * 50 + 2/10 * 50 |
| 영향력 | 10.0% | 1세력 후원 단계, 2세력 미접촉 가정 |
| 위업 달성률 | 0.0% | trust 2는 위업 미발급 |

```text
total_completion_pct = 28.1%
```

#### 5~10시간 플레이

체인을 완주하고 인프라 Tier 3에 도달한 중급 진행 상태다.

| 지표 | 값 | 산정 |
|------|----|------|
| 안정도 | 80.0% | dangerScore -60 |
| 거점 발전 | 66.7% | Tier 3 |
| 사건 완료율 | 81.8% | 체인 6 + 발견 2 + 상태 의뢰 1 = 9/11 |
| 자원·제작 | 55.0% | 3/5 * 50 + 5/10 * 50 |
| 영향력 | 56.7% | 가입 2세력 rep 30, 미가입 1세력 rep 10 |
| 위업 달성률 | 60.0% | 3/5 |

```text
total_completion_pct = 68.6%
```

#### Tier 4 도달, region_pacified 직전

후반부에는 인프라와 사건이 대부분 닫히고 평정 위업만 남는다.

| 지표 | 값 | 산정 |
|------|----|------|
| 안정도 | 95.0% | dangerScore -90 |
| 거점 발전 | 100.0% | Tier 4 |
| 사건 완료율 | 100.0% | 11/11 |
| 자원·제작 | 95.0% | 5/5 * 50 + 9/10 * 50 |
| 영향력 | 80.0% | 3세력 가입, 평균 rep 60 |
| 위업 달성률 | 80.0% | 4/5 |

```text
total_completion_pct = 93.3%
```

#### 완전 클리어

모든 지표가 닫힌 상태는 100%로 수렴한다.

```text
total_completion_pct = 100.0%
```

### 4.2 곡선 평가

보완 산식은 초안보다 신규 시작값을 낮추고, 중후반 진척은 충분히 크게 보여준다.

| 단계 | 통합 % | 평가 |
|------|--------|------|
| 신규 시작 직후 | 10.0% | 안정도 기본값만 반영되어 적정 |
| 1~3시간 플레이 | 28.1% | 초반 진행 체감이 충분함 |
| 5~10시간 플레이 | 68.6% | 체인 완주 보상이 크게 보임 |
| Tier 4 도달, 평정 직전 | 93.3% | 막판 목표가 명확함 |
| 완전 클리어 | 100.0% | 정상 수렴 |

## 5. 목표 자동 추천 점수식

목표 추천은 표시용 완성도와 분리한다. 30분 슬롯은 완료까지 가까운 진행 중 작업을 우선하고, 8시간 슬롯은 다음 큰 임계값에 가까운 장기 목표를 우선한다.

```text
goal_score = base_weight * progress_factor + value_factor
```

공통 규칙은 다음과 같다.

| 규칙 | 내용 |
|------|------|
| factor 범위 | 모든 `progress_factor`는 0.0~1.0으로 clamp |
| value 범위 | `value_factor`는 0~50 권장 |
| 정렬 | `goal_score desc`, `base_weight desc`, `id asc` |
| 핀 처리 | 핀 목표가 유효하면 자동 추천보다 우선 |
| 후보 cap | 자동 추천 1개 + 후보 최대 3개, score > 0만 노출 |

### 5.1 30분 슬롯

30분 슬롯은 이미 시작된 행동이나 아주 가까운 임계값을 추천한다.

| 후보 | base_weight | progress_factor | value_factor |
|------|-------------|-----------------|--------------|
| 진행 중 의뢰 | 100 | `1 - minutesRemaining / 30` | `difficulty * 5` |
| 진행 중 이동 | 80 | `1 - minutesRemaining / 30` | `distance * 2` |
| 진행 중 조사 | 80 | `1 - minutesRemaining / 30` | `regionTier * 3` |
| 진행 중 건설 | 80 | `1 - minutesRemaining / 30` | `facilityLevel * 4` |
| 임박 신뢰도 임계 | 60 | `1 - trustGap / 30` | `nextTrustLevel * 8` |
| 임박 인프라 flag | 60 | `1 - flagGap / 2` | 30 |
| 임박 명성 랭크 | 50 | `1 - reputationGap / 200` | 40 |
| fallback | 10 | 1 | 0 |

임박 조건은 다음과 같다.

| 임계 종류 | 조건 |
|-----------|------|
| 신뢰도 | `nextTrustThreshold - currentTrust <= 30` |
| 인프라 | 다음 tier까지 flag 1개 부족 |
| 명성 | `nextRankRep - currentRep <= 200` |
| 체인 다음 단계 | `nextStepAvailableAt - now <= 30 minutes` |

### 5.2 8시간 슬롯

8시간 슬롯은 장기 진척과 다음 큰 보상을 추천한다.

| 후보 | base_weight | progress_factor | value_factor |
|------|-------------|-----------------|--------------|
| 다음 신뢰도 단계 | 100 | `1 - trustGap / 80` | `nextTrustLevel * 10` |
| 다음 인프라 Tier | 100 | `unlockedFlags / nextTierFlagCount` | `nextTier * 12` |
| 다음 명성 랭크 | 90 | `1 - reputationGap / 8000` | 40 |
| 활성 체인 완주 | 80 | `1 - remainingSteps / 6` | 45 |
| 핵심 제작 레시피 해금 | 60 | `1 - missingMaterials / 3` | `recipeTier * 10` |
| 신규 세력 가입 조건 | 50 | `currentRep / requiredRep` | 30 |
| region_pacified 임박 | 100 | `1 - (dangerScore + 100).abs() / 100` | 50 |
| fallback | 10 | 1 | 0 |

8시간 조건은 다음과 같다.

| 임계 종류 | 조건 |
|-----------|------|
| 신뢰도 | `nextThreshold - currentTrust <= 80` |
| 인프라 | `nextTierFlagCount - unlockedFlags <= 2` |
| 명성 | `nextRankRep - currentRep <= 8000` |
| 체인 완주 | `remainingSteps <= 3` |
| 제작 | 핵심 입력재 부족 수가 2개 이하 |
| 평정 | `dangerScore <= -80` |

### 5.3 의사코드

구현은 후보 수집, 점수 계산, 핀 검증, fallback 순서로 처리한다.

```dart
GoalRecommendation recommendGoal({
  required GoalSlot slot,
  required LivingsphereDashboardSnapshot snapshot,
}) {
  final candidates = slot == GoalSlot.short30Min
      ? _collectShortTermCandidates(snapshot)
      : _collectLongTermCandidates(snapshot);

  for (final candidate in candidates) {
    candidate.score =
        candidate.baseWeight * candidate.progressFactor.clamp(0.0, 1.0) +
        candidate.valueFactor.clamp(0.0, 50.0);
  }

  final pinId = slot == GoalSlot.short30Min
      ? snapshot.userData.shortGoalPinId
      : snapshot.userData.longGoalPinId;
  if (pinId != null) {
    final pinned = candidates.firstWhereOrNull(
      (candidate) => candidate.id == pinId && candidate.isValid,
    );
    if (pinned != null) return pinned;
  }

  candidates.sort((a, b) {
    final scoreCompare = b.score.compareTo(a.score);
    if (scoreCompare != 0) return scoreCompare;
    final weightCompare = b.baseWeight.compareTo(a.baseWeight);
    if (weightCompare != 0) return weightCompare;
    return a.id.compareTo(b.id);
  });

  if (candidates.isEmpty || candidates.first.score <= 0) {
    return GoalRecommendation.fallback(slot);
  }
  return candidates.first;
}
```

### 5.4 추천 시나리오 검증

다음 예시는 보완된 factor 방향이 의도대로 작동하는지 확인한다.

| 시나리오 | 후보 | score | 결과 |
|----------|------|-------|------|
| 의뢰 12분 남음, difficulty 2 | 진행 중 의뢰 | `100 * 0.60 + 10 = 70` | 30분 슬롯 추천 |
| 명성 랭크 200 남음 | 임박 명성 | `50 * 0.00 + 40 = 40` | 완료 직전이 아니므로 의뢰보다 낮음 |
| 명성 랭크 20 남음 | 임박 명성 | `50 * 0.90 + 40 = 85` | 진짜 임박 시 상위 후보 |
| 인프라 Tier 4까지 flag 1개 부족 | 다음 인프라 Tier | `100 * 0.83 + 48 = 131.3` | 8시간 슬롯 강후보 |
| dangerScore -90 | region_pacified 임박 | `100 * 0.90 + 50 = 140` | 평정 직전 최우선 |

### 6.1 구현 주의사항

다음 주의사항은 초안에서 발견된 오류가 코드로 넘어가지 않도록 막는 기준이다.

| 항목 | 구현 기준 |
|------|-----------|
| 사건 완료율 | 위업 templateId는 체인/상태 보정용 fallback으로만 사용하고 분자에 중복 가산하지 않는다 |

| 목표 추천 | 모든 gap factor는 가까울수록 커지는 형태로 작성한다 |
