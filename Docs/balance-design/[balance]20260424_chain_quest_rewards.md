# 연계 퀘스트 확정 보상 강도 밸런스 리포트

> 작성일: 2026-04-24
> 유형: 밸런스 분석 + 수치 조정 제안
> 분석 대상: M3 연계 퀘스트(7체인/24단계)의 최종 확정 보상 강도와 M2b 엘리트 반복 파밍 대비 시간 효율
> 입력: `Docs/content-design/[content]20260423_chain_quests.md`, `Docs/content-data/[item]20260420_m2a-equipment.csv`, `elite_monsters`/`elite_loot_tables`(Supabase), `QuestCalculator`, `IdleRewardService`
> 후속 입력: 페이즈 3-0-1 `types/chain-quest.md`, 페이즈 3-1 체인 벌크 생성, 페이즈 4-2 `ChainQuestService` spec

## 결론 요약

- **체인 2~6 보상 강도 기존 유지** — 확정 보상(T2~T4 장비) + 단계 골드 합계가 동일 active 시간 엘리트 파밍 대비 이미 우세. 조정 불필요
- **체인 1 보상 소폭 상향** — 입문 체인의 성취감이 "1시간 파밍"보다 약함. 단계 골드 120/300 → **150/400**, 최종 단계 XP 40 유지
- **체인 7 엔드게임 보상 기존 유지 + 완주 명성 보너스** — 멸혼결은 엘리트 경로로는 실질 획득 기대값 < 100시간 파밍. 현 설계가 "확정 경로"로서 핵심 역할. 단, 체인 7 완주 시 명성 +2,000 지급 추가 권장
- **체인 5~7 D5 단계 `death_rate × 0.5` 하드 캡** — 주인공 용병(`protagonistMercId`) 사망으로 서사 훼손 최소화. injury_rate는 그대로 유지하여 재도전 허들은 살림
- **모든 체인 완주 시 명성 보너스 지급** — 단계 수 × 150 + 티어 가중(T2~T5 = 1.0~1.8 배수). 체인 7은 약 2,000, 체인 1은 약 300
- **Q-1(휴면 기간) 기존 7일 → 14일 확장** — 방치형 게임 특성 + 엔드게임 체인 6시간 delay 감안
- **Q-2(인벤 사전 차단) 확정** — 최종 단계 수행 전 "인벤 여유 1칸 필요" 경고 강제
- **체인 7 최종 단계 delay 6시간 → 4시간 단축** — 접속 주기 현실화, 엔드게임 호흡 유지. 총 delay 15시간 → **13시간**

## 1. 현재 상태

### 1-1. 보상 공식 (`QuestCalculator.calculateReward`)

```
reward = baseReward × rewardMultiplier × (1 + stackedBonus)
stackedBonus = clamp(trackBonus + passiveRewardBonus, 0.0, 0.80)
greatSuccess → reward × 2
```

### 1-2. 파견 비용 / 소요 시간 / 인건비

```
duration(sec) = baseDuration × (1 + (difficulty-1) × 0.2) / (speedMul × partyAgi/50)
cost = minCost + (maxCost - minCost) × clamp(duration / 144, 0, 1)
wage_total = Σ tier-wages [tier1:10, tier2:25, tier3:50, tier4:100, tier5:200]
netProfit = totalReward - totalWage - dispatchCost
```

### 2-2. 엘리트 파밍 EV (active 시간당)

**가정**: 
- 일반 퀘스트 1회 평균 2분 (D3~D4)

### 2-5. 실패 재도전 비용 (체인 5~7)

D5 단계 실패 시:
- `injury_rate` 60% → 파티 3명 중 평균 1.8명 부상 (회복 50분)
- `death_rate` 30% → 파티 3명 중 평균 0.9명 사망 (영구 손실)
- **주인공 용병 사망 시**: 체인 계승자로 폴백 → 서사 훼손

체인 5~7 모두 D5 단계 포함. 완주까지 1~2명 사망 각오 → 엔드게임 체인이 실질적으로 "용병 풀 고갈 게임"이 될 위험.

**리스크**: 플레이어가 "체인 포기 불가능"인데 용병이 계속 사망하면 **좌절 루프**. 로드맵 "실패 패널티 최소화" 원칙과 충돌.

## 4. 플레이어 체감 분석

- **체인 1(폐사당)**: "10분 기다려서 420G + 철투구(T2). 1시간 게임하면 4,000G 버는데…" → 서사 부족. 입문 체인 **성취감 보강** 필요
- **체인 2~4(중급)**: T3 확정 장비 좋음. 지역 이동 포함된 서사 리듬도 좋음 → **유지**
- **체인 5(상인, 용병단)**: 황금 저울은 영구 +3% 골드 → 투자 회수 시간 짧음. "이걸 놓치면 바보다" 인식 형성 → **유지**
- **체인 6(장인)**: 단련의 은반지 강력. "왕복 이동"의 원형 서사 + T4 최강 accessory → **유지**
- **체인 7(혼)**: 15시간 대기 + 5단계 + 주인공 사망 위험. "나의 김철수가 죽었다" → 서사는 가능하지만 유저 좌절 크면 재도전 단념. **death_rate 감산 필요**
- **체인 완주 성취감**: 현재 "확정 보상 받았다" 단일 이벤트. 명성 보너스 + "서사 완주 배지" 추가 시 장기 진행 동기 강화

| 단계 1 (탐험 D2) | 120G | **150G** |
| 단계 2 (토벌 D3 + 철투구) | 300G | **400G** |
| 체인 총 | 420G + 철투구 | **550G + 철투구** |

**근거**: 입문 체인 10분 대기 + 3분 active로 "1시간 파밍(~4,400G)의 12.5%" 가치 제공. 1/8 시간으로 1/8 가치 → 납득 가능.

### 5-2. 체인 완주 명성 보너스 추가 (신규)

```
chain_complete_reputation = (단계 수 × 150) × tier_weight
tier_weight = {T2: 1.0, T3: 1.2, T4: 1.5, T5: 1.8}
```

| 체인 | 단계 | 최종 티어 | 계산 | 보너스 명성 |
|---|---|---|---|---|
| 1 폐사당 | 2 | T2 | 2×150×1.0 | **300** |
| 2 질풍 | 3 | T3 | 3×150×1.2 | **540** |
| 3 철갑 | 3 | T3 | | **540** |
| 4 국경 | 3 | T3 | | **540** |
| 5 상인 | 4 | T3 | 4×150×1.2 | **720** |
| 6 장인 | 4 | T4 | 4×150×1.5 | **900** |
| 7 혼 | 5 | T5 | 5×150×1.8 | **1,350** |
| **합계** | | | | **4,890** |

**근거**: 랭크 D(2,000) → C(8,000) 도달에 6,000 명성 필요. 체인 4~5개 완주 = D→C 진입 보조. 체인 7 = 1,350으로 엔드 랭크 여정에 기여 (B:25,000 / A:80,000 기준).

**대조**: 일반 퀘스트 명성 획득(미구체화) 대비 "체인이 강력한 랭크 진행 경로"로 자리잡음.

### 5-3. 체인 5~7 D5 단계 death_rate 하드 감산

**제안**: 체인 단계 퀘스트 완료 시 `death_rate × 0.5` 적용 (injury_rate는 그대로).

```
// QuestCompletionService or ChainQuestService
if (quest.isChainStep) {
  effectiveDeathRate = baseDeathRate * 0.5;
}
```

| 난이도 | 기존 death | 체인 단계 death | 기존 injury | 체인 단계 injury |
|---|---|---|---|---|
| D3 | 15% | **7.5%** | 30% | 30% |
| D4 | 22% | **11%** | 45% | 45% |
| D5 | 30% | **15%** | 60% | 60% |

**근거**:
- 서사적 정당화 — "주인공의 여정은 운명이 보호한다" / "부상당해도 생환"
- 게임플레이 정당화 — 체인 7은 D5 단계 4회 연속, 기존 death_rate 30% 적용 시 완주까지 주인공 사망 확률 1-(0.7)⁴ = **76%**. 조정 시 1-(0.85)⁴ = **48%**로 감소
- injury_rate 유지 → 재도전 허들은 살아 있음(용병 회복 대기)
- 주인공 아닌 용병도 혜택 → "체인 서사 중엔 파티 전체가 운명의 보호를 받는다"

### 5-4. 인벤토리 사전 차단 (Q-2 확정)

**정책**: 체인 최종 단계의 `ChainQuestService.canAdvanceToFinal()` 체크에 인벤 여유 1칸 확인. 부족 시 "최종 보상을 받으려면 인벤토리에 1칸 이상 필요합니다" 모달로 차단.

**구현 메모** (페이즈 4-2 spec 반영):
- `UserData.inventoryCapacity` (M2a 이미 존재) vs `currentItemCount` 비교
- 용병단 장비 체인(상인)은 용병단 슬롯 확인
- 모달 dismiss 시 체인 상태 유지, 유저가 인벤 정리 후 재시도 가능

### 5-5. Q-1 휴면 기간 7일 → 14일

**근거**:
- 방치형 특성상 주간 단위 접속 가능성 — 7일은 짧음
- 체인 7 delay 합계만 15시간 → 유저가 2일 쉬면 delay 도달 ×2건 누적 가능. 그 사이 미수행이면 체인 진행 감각 희미해짐 (14일이 더 적절)
- `ChainQuestProgress.dormantAfter: Duration(days: 14)`

### 5-6. 체인 7 최종 단계 delay 단축

**제안**: 체인 7(혼을 끊는 자) 단계 4→5 delay **6시간 → 4시간**.

| 구간 | 기존 | 제안 |
|---|---|---|
| 단계 1 → 2 | 3h | 3h (유지) |
| 단계 2 → 3 | 2h | 2h (유지) |
| 단계 3 → 4 | 4h | 4h (유지) |
| 단계 4 → 5 | **6h** | **4h** |
| **합계** | 15h | **13h** |

**근거**:
- 엔드게임 체인의 "긴 호흡"은 유지 (13시간은 여전히 하루 호흡)
- 6시간 단계는 낮/밤 접속 주기 2회를 요구 → 현실적 벽. 4시간은 한 세션 외 1회 접속으로 도달 가능
- 체인 7 전체 감성은 유지(총 13시간 ≈ 하루), 마지막 호흡만 현실화
- data-generator: 단계 5의 `next_step_delay` = 이전 단계(단계 4)에서의 대기 = **14,400초(4h)**

## 6. 시뮬레이션 — 조정 후 주인공 생존율

**가정**: 체인 7 완주, D4 1회 + D5 4회, 주인공 매 단계 파티 포함(기여도 최상위).

**조정 전**:
- 단계별 주인공 death 확률: D4 22% / D5 30%
- 완주까지 생존: (1-0.22) × (1-0.30)^4 = 0.78 × 0.24 = **18.7%**
- 즉 주인공 교체(폴백) 확률 81%

**조정 후**(death × 0.5):
- 단계별 주인공 death 확률: D4 11% / D5 15%
- 완주까지 생존: (1-0.11) × (1-0.15)^4 = 0.89 × 0.52 = **46.2%**
- 주인공 교체 확률 54%

**해석**: 여전히 절반은 교체되지만, "이야기를 처음 시작한 용병이 끝까지 함께할 수 있다"는 체감 가능성이 18% → 46%로 상승. 서사 보존률 대폭 개선.

**수치는 injury_rate는 그대로**이므로 "체인 단계 실패 시 회복 대기 → 재도전" 구조는 유지되어 난이도 감소 체감은 미미함 (재도전 빈도는 거의 동일).

## 7. data-generator 수치 가이드

> 페이즈 3-1(`chain_quests` 벌크 생성) 시 이 가이드를 준수한다.

- **대상 타입**: `chain-quest` (페이즈 3-0-1에서 타입 스펙 선행)
- **대상 테이블**: `chain_quests` (신규)
- **행 수**: **24행** (체인 7개 × 각 2~5단계, 기획서 §1 표 그대로)
- **수치 범위 (필드별)**:
  - `combat_power`: 기획서 §2 표 각 체인의 전투력 범위 중앙값 사용 (예: 20~30 → 25)
  - `reward_gold`: 본 리포트 §5-1 반영 (체인 1 단계 1=150, 단계 2=400). 다른 체인은 기획서 §2 값 그대로
  - `next_step_delay`: 초 단위(int), 기획서 §2 표 값을 초로 환산 (10분=600, 6시간=21,600). **단, 체인 7 단계 4의 `next_step_delay`는 §5-6 반영하여 14,400(4h)** — 기획서 §2의 6h(21,600)에서 변경
  - `duration_minutes`: 난이도별 공식 `baseDuration × (1+(diff-1)×0.2) × 60 / avgAgi×50` — 또는 기획서의 `D2=1~2분 / D3=2~3분 / D4=3~4분 / D5=4~5분` 수동 입력
  - `final_reward`: bool, 마지막 단계 true 외 false
- **외래 키 제약**:
  - `chain_id`는 §1 표 7개 ID 고정
  - `region_id`는 페이즈 3-2에서 결정되는 `region_discoveries.hidden_quest` 발동 리전 ID와 일치
  - `target_region_id`는 타 리전 단계에서만 다른 값, 동일 리전 단계는 `region_id`와 동일
  - `reward_items` JSONB는 최종 단계에서만 값, 비최종 단계는 `NULL` 또는 `{}`
  - 최종 단계 `reward_items`는 §6 표의 `items.id` 정확히 참조:
    - `equip_helmet_iron_helm`(체인 1), `equip_boots_gale_leather`(체인 2), `equip_armor_chain_mail`(체인 3), `equip_weapon_steel_sword`(체인 4), `guild_artifact_golden_scale`(체인 5), `equip_accessory_forge_silver_ring`(체인 6), `equip_accessory_soul_seal`(체인 7)
- **특수 요구**:
  - 체인 완주 명성 보너스(§5-2)는 **데이터가 아닌 `ChainQuestService` 로직**에서 계산 — `chain_quests`에는 표기 불필요. 단, 체인 최종 행에 `final_reputation_bonus: int` 컬럼이 선행 DDL로 추가되어 있다면 표 값 입력 (권장: 컬럼 추가)
  - 체인 단계 `death_rate × 0.5` 하드 감산은 **런타임 로직**(`QuestCompletionService` or `ChainQuestService`)에서 처리 — 데이터 변경 없음
- **balance 근거 요약**:
  - 체인 1 골드 상향: 입문 active 시간당 가치 보강 (§2-1, §5-1)
  - 체인 완주 명성: 랭크 D→A 여정 보조 (§5-2)
  - 인벤 사전 차단: Q-2 해결, 유저 좌절 최소 (§5-4)
  - 휴면 14일: 방치형 감안 (§5-5)

## 8. 페이즈 4 spec 반영 사항

페이즈 4-2 `ChainQuestService` spec 작성 시 다음을 포함한다:

1. **완주 명성 보너스 로직**
   - `ChainQuestService.onChainCompleted(chainId, finalStep)` → `tier_weight` 테이블(§5-2)에서 조회 → `userDataNotifier.addReputation(bonus)` 호출
   - 활동 로그: `ActivityLogType.chainCompleted`에 명성 보너스 수치 포함

2. **체인 단계 death_rate 하드 감산**
   - `ActiveQuest.isChainStep: bool` 필드 추가
   - `QuestCompletionService._rollDamage()` 호출 시 `isChainStep` true면 `effectiveDeathRate *= 0.5` 적용
   - 관련 테스트: `death_rate × 0.5` 적용 여부 단위 테스트

3. **인벤 사전 차단**
   - `ChainQuestService.canAdvanceToFinal(chainId)` → `inventoryNotifier.hasFreeSlot(finalRewardItemId)` 체크
   - UI: 체인 카드에 "인벤 여유 필요" 경고 배지

4. **휴면 14일**
   - `ChainQuestProgress.currentStepAvailableAt` 이후 14일 미수행 → 상태 `dormant` 전환
   - 휴면 체인은 파견 화면에서 상단 고정 해제 (일반 세션 목록 최하단 노출)

## 9. 오픈 질문 / 후속 메모

- **Q-A**: 체인 완주 명성 보너스가 일반 퀘스트 명성 획득과 비교해 과도한가? — 일반 퀘스트 명성 지급 수치가 아직 공식적으로 확정 안 됨(M2a 명성 시스템 완료 후 측정 필요). **페이즈 4-4 서사 통합 spec 시점에 일반 퀘 명성 평균/시간 측정 후 재조정 여지**
- **Q-B**: 체인 단계 death_rate 감산이 체인 단계에만 적용되는 특별 규칙 → 플레이어 인지 비용. UI에서 "이 단계는 운명의 보호를 받습니다" 배지로 명시해야 좌절 방지 — **페이즈 1-6 공존 정책 추가 반영** 권장(카드 배지 1종 추가)
- **Q-C**: 체인 7 멸혼결 엘리트 경로가 거의 닫혀 있는 상황 — 이것이 "체인 = 유일 경로"라는 설계 의도와 일치한다면 유지. 다만 M4+에서 "엔드게임 파밍 빌드" 유저에게 대안 경로 필요할 수 있음 → **M4 기획 시 재검토**
- **Q-D**: 본 리포트는 `ranks` 기반 passive(+0.80 cap)을 대부분 고려하지 않음. 체인 완주 시점의 유저 랭크가 D~C라면 실제 체인 골드 총합이 +30~60% 상승 → 본 분석보다 유리. **리포트 수치는 "랭크 미포함 베이스"로, 실제 체감은 더 풍요롭다**는 점을 밸런스 결정자에게 전달
- **Q-E**: 체인 7 delay 총합 15시간 → 14일 휴면 규칙과 상충 없음(delay 내 수행 시 휴면 진입 X). 단 delay 누적 중 플레이어가 접속 안 하면 시간이 "흘러버려" 손실감 있음 → UX 측면 추후 관찰
