## 0. 적용 방법 및 사전 확인

### 적용 방법

| 방법 | 명령 / 절차 |
|------|-----------|
| **Supabase MCP** (권장) | `apply_migration` 또는 MCP `execute_sql`로 각 SQL 블록을 순서대로 실행 |
| **Supabase CLI** | `supabase migration new m8.5_combat_emotion_hidden_stats` → 생성된 파일에 본 SQL 붙여넣기 → `supabase db push` |
| **Supabase Studio** | SQL Editor → 본 SQL 블록을 순서대로 붙여넣기 실행 |

**실행 순서**: SQL (1) → (2) → (3) → (4). 각 블록은 독립 트랜잭션으로 실행 가능.
`combat_status_effects` / `combat_report_templates`는 **기존 테이블** → ALTER+INSERT만.
`hidden_stats` / `battle_memory_templates`는 **신규 테이블** → CREATE+INSERT.

### ⚠ Q-1: combat_report_templates scope CHECK 기존 목록 미확정

본 SQL (2)에서 `combat_report_templates_scope_check`를 DROP 후 재선언할 때 **기존 CHECK의 전체 scope 목록을 반드시 SELECT로 확인** 후 누락 없이 포함해야 한다.

```sql
-- 적용 전 반드시 실행하여 현재 scope CHECK 목록을 확인한다:
SELECT pg_get_constraintdef(oid)
FROM pg_constraint
WHERE conname = 'combat_report_templates_scope_check';
```

본 SQL은 명세 §2.2와 탐색 결과 9 scope에 `'emotional'`을 추가한 상태로 기재했다.
실제 DB의 CHECK 목록이 다르면 해당 SQL을 수정 후 실행한다.

### ⚠ Q-2: combat_report_templates emotion 매칭 컬럼 미확정

본 SQL (2)의 `emotional` scope 20행은 `tags_json` JSONB 컬럼에 `emotion_id` 키를 포함하는 구조로
**잠정 작성**되었다. 실제 `combat_report_templates` 테이블의 컬럼 구조(tags_json 존재 여부, 
emotion 매칭용 별도 컬럼 유무)를 확인 후 정합에 맞춰 수정이 필요하다.

```sql
-- 적용 전 반드시 실행하여 현재 컬럼 구조를 확인한다:
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'combat_report_templates'
ORDER BY ordinal_position;
```

`tags_json` 컬럼이 없거나 emotion 매칭 방식이 다른 경우, INSERT 행의 `tags_json` 인자를 
적절히 수정하거나 삭제한다.

### 검증 쿼리

```sql
-- (1) 적용 후 감정 상태 효과 확인
SELECT id, kind, display_label, default_intensity, default_duration_turns
FROM combat_status_effects
WHERE kind = 'emotional'
ORDER BY id;
-- 기대: 4행 (emotional_rage, emotional_despair, emotional_sorrow, emotional_determination)

-- (2) 감정 보고서 템플릿 확인
SELECT scope, COUNT(*) AS cnt
FROM combat_report_templates
GROUP BY scope
ORDER BY scope;
-- 기대: emotional 20행 포함

-- (3) 히든 스탯 확인
SELECT id, name, level_thresholds
FROM hidden_stats
ORDER BY id;
-- 기대: 5행 (battle_sense, fear_resistance, fortitude, grit, luck)

-- (4) 전투 기억 템플릿 확인
SELECT entry_type, COUNT(*) AS cnt
FROM battle_memory_templates
GROUP BY entry_type
ORDER BY entry_type;
-- 기대: emotional_apply 12 / hidden_stat_unlock 10 / solo_great_success 3 / unique_elite_first_kill 5
```

### 롤백 SQL

```sql
-- (1) 롤백: combat_status_effects emotional 4행 + CHECK 원복
DELETE FROM combat_status_effects WHERE kind = 'emotional';
ALTER TABLE combat_status_effects DROP CONSTRAINT IF EXISTS combat_status_effects_kind_check;
ALTER TABLE combat_status_effects ADD CONSTRAINT combat_status_effects_kind_check
  CHECK (kind IN ('buff','debuff','mez','dot'));
-- ↑ 원래 CHECK 목록은 실제 DB 확인 후 수정

-- (2) 롤백: combat_report_templates emotional 20행 + CHECK 원복

ALTER TABLE combat_report_templates DROP CONSTRAINT IF EXISTS combat_report_templates_scope_check;
ALTER TABLE combat_report_templates ADD CONSTRAINT combat_report_templates_scope_check
  CHECK (scope IN ('chain_final','chain_step','settlement_event','unique_elite','elite',
                   'faction_named','quest_type','scene','combat_skill'));
-- ↑ 원래 CHECK 목록은 실제 DB 확인 후 수정

DROP TABLE IF EXISTS hidden_stats;

-- (4) 롤백: battle_memory_templates 테이블 삭제
DROP TABLE IF EXISTS battle_memory_templates;
```

## 5. 로컬 캐시 JSON 처리

소스 트리(`band_of_mercenaries/`)에 정적 캐시 fixture 디렉토리 또는 `assets/cache`가
존재하지 않는 것을 확인했다 (앱 문서 디렉토리에 런타임 생성되는 구조).

`hidden_stats` / `battle_memory_templates`는 `optionalTables`로 등록 예정이므로,
캐시가 비어도 앱 기동·기존 기능이 막히지 않는 fail-soft가 보장된다.

**결론**: 로컬 캐시 JSON은 Supabase 동기화(SyncService)로 런타임 생성되므로
별도 fixture 파일 불요.

## 6. 적용 후 추가 확인 사항

- `combat_status_effects` 4행 적용 후 Flutter 앱의 `staticDataProvider`가
  `kind='emotional'` 행을 파싱하는지 확인 (기존 `CombatStatusEffect` 모델이 kind 필드를 
  String으로 저장하므로 자동 호환).
- `hidden_stats` / `battle_memory_templates` 테이블 적용 후 SyncService `optionalTables`에
  등록이 완료된 빌드에서 첫 sync 시 캐시 JSON이 정상 생성되는지 확인.
- `battle_memory_templates` 템플릿 변수(`{merc.name}` / `{ally.name}` / `{enemy.name}` / 
  `{quest.name}`)는 `TemplateEngine` 기존 컨텍스트와 정합 확인 필요.
