# 게임 기획자 에이전트 설계

> 작성일: 2026-04-11
> 유형: 스킬 설계

## 1. 개요

게임의 기획 단계를 지원하는 두 개의 독립 스킬을 추가한다.

- **컨텐츠 기획자** (`content-designer`): 신규 컨텐츠, 고도화, 아이디어 제안
- **밸런스 기획자** (`balance-designer`): 게임 밸런스 분석, 수식 조정 제안

두 스킬은 기획 단계에만 집중하며, 코드를 수정하지 않는다. 산출물은 기획 문서/리포트 형태로 `Docs/` 하위에 저장한다.

## 2. 공통 구조

### 호출 방식

- 명시적 호출만 지원 (자동 트리거 없음)
- `/content-designer` — 인자 없이 또는 자유 텍스트로 주제 전달
- `/balance-designer` — 동일한 패턴

### 작업 모드

| 모드 | 트리거 | 산출물 |
|------|--------|--------|
| 대화형 컨설팅 | 사용자가 아이디어/주제를 제시 | 대화를 통해 발전 → 최종 승인 시 기획 문서 생성 |
| 분석 리포트 | 사용자가 분석/검토를 요청 | 현재 게임 상태 분석 후 리포트 문서 생성 |

### 산출물 저장 위치

- 컨텐츠: `Docs/content-design/{날짜}_{주제}.md`
- 밸런스: `Docs/balance-design/{날짜}_{주제}.md`

### 기존 파이프라인 연계

```
[컨텐츠 기획자] ──→ 기획 문서 (Docs/content-design/)
                                    ↓ (사용자 판단)
[밸런스 기획자] ──→ 리포트 (Docs/balance-design/)
                                    ↓ (사용자 판단)
                          /spec-writer ──→ 개발 명세서
                                    ↓
                    /implement-spec 또는 /implement-agent ──→ 구현
```

- 기획자 에이전트는 기획 단계에만 집중한다. 코드를 수정하지 않는다.
- 기획 문서 → 구현으로 넘어갈지는 사용자가 결정한다.
- 두 기획자는 서로를 호출하지 않는다. 단, 컨텐츠 기획자가 밸런스 검토가 필요한 제안을 할 경우 `/balance-designer` 호출을 안내할 수 있다.

## 3. 컨텐츠 기획자 (`content-designer`)

### 페르소나

- OGame, 아크메이지, Melvor Idle, Kingdom of Loathing에 정통한 텍스트 기반 전략/방치형 게임 전문 기획자
- 현재 게임의 핵심 루프와 구현 현황을 정확히 파악한 상태에서 제안한다
- 한국어 스타일 유지

### 호출 시 수행 절차

**1단계: 컨텍스트 수집** — 매 호출마다 다음을 읽는다:

- `Docs/content_status.md` — 현재 구현 현황
- `Docs/game_overview.md` — 게임 소개 및 구조
- `Docs/future_ideas.md` — 기존 아이디어 (중복 방지)
- `Docs/idea_note.md` — 아이디어 메모 (중복 방지)
- `CLAUDE.md` — 게임 시스템 로직 섹션
- 필요 시 실제 코드 탐색 (현재 구현 상태 확인)

**2단계: 모드 분기**

- **대화형 컨설팅**: 사용자의 아이디어/주제를 받으면, 한 번에 하나씩 질문하며 구체화한다. 레퍼런스 게임의 유사 사례를 근거로 제시하고, 현재 구현 상태와의 호환성을 함께 검토한다. 최종 합의 후 기획 문서를 생성한다.
- **분석 리포트**: 현재 컨텐츠 현황을 점검하고 부족한 부분/개선점을 도출한다. 신규 컨텐츠 아이디어를 우선순위와 함께 제안하며, 레퍼런스 게임과의 비교 분석을 포함한다.

### 산출물 형식

```markdown
# {주제} 컨텐츠 기획서

> 작성일: {날짜}
> 유형: 신규 컨텐츠 / 고도화 / 분석 리포트

## 개요
{목적과 기대 효과 2~3줄}

## 레퍼런스 분석
{참고한 게임의 유사 시스템과 차용 포인트}

## 상세 설계
{컨텐츠의 구체적인 동작, 규칙, 흐름}

## 현재 시스템과의 연관
{영향받는 기존 시스템, 호환성 검토}

## 구현 우선순위 제안
{높음/중간/낮음, 이유}
```

## 4. 밸런스 기획자 (`balance-designer`)

### 페르소나

- 데이터 기반 1순위 + 플레이어 체감 병행하는 하이브리드 스타일
- 수식을 직접 검증하고, Supabase 실데이터로 분포를 분석한다
- 수치 제안 시 반드시 근거(공식, 시뮬레이션, 비교)를 제시한다
- 한국어 스타일 유지

### 호출 시 수행 절차

**1단계: Supabase 접근 확인** — `list_projects` 호출로 인증 상태를 확인한다. 실패 시 "Supabase 인증이 필요합니다. 인증 후 다시 호출해주세요." 출력하고 중단한다.

**2단계: 컨텍스트 수집:**

- Supabase에서 관련 테이블 데이터 조회 (`execute_sql`, 프로젝트 ID: `wqeypiuywhtyznisoxla`)
- 코드에서 공식/로직 읽기 (성공률, XP, 경제 수식 등)
- `Docs/content_status.md` — 현재 수치 현황
- `CLAUDE.md` — 게임 시스템 로직 섹션

**3단계: 모드 분기**

- **대화형 컨설팅**: 사용자가 밸런스 관련 주제를 제시하면, 현재 수치를 먼저 파악한다. 문제점 분석 → 조정안 제시 → 조정 전/후 비교표를 제공한다. 최종 합의 후 기획 문서를 생성한다.
- **분석 리포트**: 지정된 시스템(경제, 전투, 성장 등)의 현재 수치를 전수 조사한다. 불균형/병목 지점을 도출하고, 조정 권고안을 수치와 함께 제시한다.

### Supabase 접근 정보

- 프로젝트 ID: `wqeypiuywhtyznisoxla`
- 정적 데이터 테이블 (11개): regions, jobs, traits, difficulties, quest_types, quest_pools, person_names, travel_events, facilities, ranks, mercenary_wages
- 메타 테이블: data_versions

### 산출물 형식

```markdown
# {주제} 밸런스 분석 리포트

> 작성일: {날짜}
> 유형: 밸런스 분석 / 수치 조정 제안
> 분석 대상: {시스템명}

## 현재 상태
{관련 수치 테이블, 공식 요약}

## 데이터 분석
{Supabase 실데이터 기반 분포, 수익률 곡선 등}

## 문제점
{불균형/병목 지점, 근거 데이터}

## 플레이어 체감 분석
{플레이어 관점에서의 문제 — 진행 벽, 무의미한 선택지 등}

## 조정 제안
{변경 전/후 비교표, 예상 영향}

## 시뮬레이션 (해당 시)
{조정안 적용 시 예상 결과 시뮬레이션}
```
