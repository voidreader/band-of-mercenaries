## 1. 개요

M4~M8a에서 분산 구현된 거점 신뢰도·세력 도감·체인 의뢰·M7 인프라·M8a 세력 수직 절편의 6개 지표를 region 3(더스트플레인) 한정으로 통합 가시화한다. 홈 화면 `LivingsphereSummarySection`(요약 카드)과 정보 탭 `LivingsphereDetailScreen`(상세 화면) 두 위치에서 동일한 `LivingsphereDashboardSnapshot`을 공유하고, 30분/8시간 목표 자동 추천 + 수동 핀을 제공한다. 신규 시스템이 아니라 기존 데이터의 변환·표시 계층이므로 신규 도메인 hook·산식·마이그레이션은 없다.

핵심 산출물 3종:
- 6 지표 계산 서비스 `LivingsphereDashboardService` + 가중평균 통합 완성도
- 30분/8시간 목표 추천 서비스 `GoalRecommendationService` + `UserData` 핀 영속(HiveField 28~29)
- 6 점프 동선 (`currentTabProvider` + 신규 3 target provider)

## 2. 요구사항

### 2.1 기능 요구사항

#### A. 6 지표 계산 (`LivingsphereDashboardService`)

- **[FR-1] 안정도 (`stability_pct`)**
  - 산식: `clamp(((100 - dangerScore) / 200) * 100, 0, 100)`
  - 입력: `RegionState.dangerScore` (null → 0 fallback)
  - 단계 라벨: `DangerLevelResolver.resolveLevel(dangerScore)` 결과 그대로 사용 (stable/peaceful/tension/threat)
- **[FR-2] 거점 발전 (`infrastructure_pct`)**
  - 산식: `clamp(((infrastructureTier - 1) / 3) * 100, 0, 100)`
  - 입력: `RegionState.infrastructureTier` (null → 1 fallback)
  - 펼침 보조: `trust_progress_pct = clamp((currentTrust - currentTrustThreshold) / (nextTrustThreshold - currentTrustThreshold) * 100, 0, 100)` (Tier 4 도달 시 100%로 고정)
- **[FR-3] 사건 완료율 (`event_completion_pct`)**
  - 산식: `clamp((completed_events / 11) * 100, 0, 100)`
  - 분모 = 11 (체인 6 + 발견 3 + region 3 M7 상태 의뢰 2)
  - 분자 산출:
    - 체인: `ChainQuestProgress(chainId='settlement_3_pyegwang_reopen')`의 `currentStep` (완주 시 6으로 clamp)
    - 발견: `RegionState.triggeredDiscoveries`에서 region 3 allowlist 3종(`disc_dustvile_pyegwang_normal` / `_hidden` / `_deepest`) 교집합 개수
    - M7 상태 의뢰: `RegionState.questPoolCompletionCounts`에서 `region3StateQuestPoolIds` allowlist 2종(`qp_m7_r3_cave_bats` / `qp_m7_r3_safe_escort`) 완료 횟수 합산 (각 1회 cap)
  - **제약**: 위업 templateId(`settlement_event_completed:settlement_3_pyegwang_reopen` 등)는 fallback marker로만 사용. 체인 6단계와 별도 점수로 분자에 가산 금지.
- **[FR-4] 자원·제작 (`resource_craft_pct`)**
  - 산식: `clamp((acquired_materials / 5 * 50) + (unlocked_recipes / 10 * 50), 0, 100)`
  - 특산품 allowlist 5종 (분모 5):
    - `mat_herb_dust_resin`, `mat_relic_pyegwang_pickaxe_head`, `mat_relic_pyegwang_shard`, `mat_monster_giant_bat_fang`, `mat_relic_ancient_seal_piece`
  - 분자 산출: `RegionState.firstAcquiredMaterialIds`와 allowlist의 교집합 개수
  - **제약**: `firstAcquiredMaterialIds.length`를 직접 사용 금지. 항상 allowlist 교집합 적용.
  - 레시피 allowlist 10종 (분모 10): `recipe_dustvile_*` 10개 (banner_repair, hide_bundle, ore_polished, armor_solid, herbalist_seal, herb_pouch, miner_dagger, rusty_pickaxe, pyegwang_relic, miner_charm)
  - 레시피 분자 산출: `CraftingService.evaluateState(recipe)`가 `RecipeState.locked`가 아닌 레시피 개수 (즉 `insufficient` 또는 `ready` 상태)
- **[FR-5] 영향력 (`influence_pct`)**
  - 활성 3 세력 평균. Provider 계층에서는 `FactionRelationStage.resolveFromProviderRef(factionId, ref)` 신규 helper 또는 동일 로직의 순수 helper 결과를 기준으로 점수 산출:
    - `untouched`: 0
    - `noticed`: 20
    - `patronage`: `20 + clamp(reputation, 0, 10) / 10 * 20`
    - `joined`(joined/trusted/core 단계): `clamp(50 + reputation / 100 * 50, 0, 100)`
    - `hostile` (reputation < 0): `clamp((reputation + 100) / 100 * 20, 0, 20)`
  - 대상 세력 3종 고정: `faction_adventurers_guild` / `faction_merchants_alliance` / `faction_warriors_guild`
  - **제약**: `FactionState`가 없거나 `FactionContactService.isActive()`가 false인 세력은 항상 `untouched`로 간주. `reputation=0`만으로 점수 부여 금지.
- **[FR-6] 위업 달성률 (`achievement_pct`)**
  - 산식: `clamp((region_3_achievements / 5) * 100, 0, 100)`
  - 분모 = 5 (region 3 위업 allowlist 5종)
  - allowlist:
    - `region_pacified:region_3`
    - `infrastructure_tier:tier_4`
    - `settlement_event_completed:settlement_3_pyegwang_reopen`
    - `settlement_trust_belonging:region_3`
    - `craft_first_rare:recipe_dustvile_pyegwang_relic`
  - 분자 산출: `bandAchievementsProvider`에서 `templateId`가 allowlist에 포함된 위업 개수 (`regionId==3`을 1차 fallback, 비어있으면 templateId 패턴 매칭으로 보정)
  - **제약**: T2 결과물 `craft_first_rare:*`는 현 `CraftingService.craft()` 발급 조건(tier >= 3) 미달이므로 allowlist에서 제외.
- **[FR-7] 통합 완성도 (`total_completion_pct`)**
  - 가중평균 산식:
    ```
    total_completion_pct =
        stability_pct        * 0.20
      + infrastructure_pct   * 0.20
      + event_completion_pct * 0.20
      + resource_craft_pct   * 0.15
      + influence_pct        * 0.10
      + achievement_pct      * 0.15
    ```
  - 가중치 합 = 1.00, 결과는 0~100 clamp
  - 곡선 검증 기준 (단위 테스트 권장):
    - 신규 시작(dangerScore=0, Tier 1, 진행 0): 약 10.0%
    - 1~3시간 플레이(dangerScore=-20, Tier 2, 3/11 사건, 1특산품 + 2레시피, 1세력 후원): 약 28.1%
    - 5~10시간 플레이(dangerScore=-60, Tier 3, 9/11 사건, 3특산품 + 5레시피, 2가입+1미가입, 3/5 위업): 약 68.6%
    - Tier 4 직전(dangerScore=-90, Tier 4, 11/11 사건, 5특산품 + 9레시피, 3가입 평균 rep 60, 4/5 위업): 약 93.3%

#### B. 30분/8시간 목표 추천 (`GoalRecommendationService`)

- **[FR-8] 후보 수집 (30분 슬롯, `_collectShortTermCandidates`)**
  - 진행 중 의뢰: `ActiveQuest(status=inProgress)` (`base_weight=100`, `progress_factor=1 - minutesRemaining/30`, `value_factor=difficulty * 5`)
  - 진행 중 이동: `UserData.moveEndTime != null` (`base_weight=80`, `progress_factor=1 - minutesRemaining/30`, `value_factor=distance * 2`)
  - 진행 중 조사: `UserData.investigationEndTime != null` (`base_weight=80`, `progress_factor=1 - minutesRemaining/30`, `value_factor=regionTier * 3`)
  - 진행 중 건설: `UserData.constructionEndTime != null` (`base_weight=80`, `progress_factor=1 - minutesRemaining/30`, `value_factor=facilityLevel * 4`)
  - 임박 신뢰도 임계: `nextTrustThreshold - currentTrust <= 30` (`base_weight=60`, `progress_factor=1 - trustGap/30`, `value_factor=nextTrustLevel * 8`)
  - 임박 인프라 flag: 다음 tier까지 flag 1개 부족 (`base_weight=60`, `progress_factor=1 - flagGap/2`, `value_factor=30`)
  - 임박 명성 랭크: `nextRankRep - currentRep <= 200` (`base_weight=50`, `progress_factor=1 - reputationGap/200`, `value_factor=40`)
  - fallback: 후보 없거나 모두 score ≤ 0 → `GoalRecommendation.fallback(GoalSlot.short30Min)` → "다음 의뢰를 시작하세요" stub + `DispatchScreen` 점프
- **[FR-9] 후보 수집 (8시간 슬롯, `_collectLongTermCandidates`)**
  - 다음 신뢰도 단계: `nextThreshold - currentTrust <= 80` (`base_weight=100`, `progress_factor=1 - trustGap/80`, `value_factor=nextTrustLevel * 10`)
  - 다음 인프라 Tier: `nextTierFlagCount - unlockedFlags <= 2` (`base_weight=100`, `progress_factor=unlockedFlags / nextTierFlagCount`, `value_factor=nextTier * 12`)
  - 다음 명성 랭크: `nextRankRep - currentRep <= 8000` (`base_weight=90`, `progress_factor=1 - reputationGap/8000`, `value_factor=40`)
  - 활성 체인 완주: `remainingSteps <= 3` (`base_weight=80`, `progress_factor=1 - remainingSteps/6`, `value_factor=45`)
  - 핵심 제작 레시피 해금: 부족 입력재 ≤ 2 (`base_weight=60`, `progress_factor=1 - missingMaterials/3`, `value_factor=recipeTier * 10`)
  - 신규 세력 가입 조건: `currentRep > 0 && stage in {noticed, patronage}` (`base_weight=50`, `progress_factor=currentRep / requiredRep`, `value_factor=30`)
  - region_pacified 임박: `dangerScore <= -80` (`base_weight=100`, `progress_factor=1 - (dangerScore + 100).abs() / 100`, `value_factor=50`)
  - fallback: 후보 없거나 모두 score ≤ 0 → `GoalRecommendation.fallback(GoalSlot.long8Hour)` → "위업 컬렉션을 채워보세요" stub + `ChronicleScreen` 점프
- **[FR-10] 점수 계산 + 정렬 + 핀 검증**
  - `score = base_weight * clamp(progress_factor, 0.0, 1.0) + clamp(value_factor, 0.0, 50.0)`
  - 정렬: `score desc`, `base_weight desc`, `id asc`
  - **핀 우선 처리**: `pinId != null && candidate.id == pinId && candidate.isValid` 후보가 발견되면 score와 무관하게 즉시 선택
  - **핀 무효화**:
    - 의뢰 핀: `ActiveQuest`가 사라졌거나 `status==completed`
    - 체인 핀: `ChainQuestProgress.status==completed` 또는 dormant
    - 레시피 핀: `CraftingService.evaluateState(recipe)`가 `RecipeState.ready`로 도달 또는 입력재 부족이 아닌 ready
    - 신뢰도/인프라/명성 핀: 해당 임계값 이미 도달
    - 가입 핀: 이미 가입 완료
  - 무효화된 pinId는 `recommendGoal()` 호출 시 자동 fallback 처리하되, 서비스 내부에서 `UserDataNotifier`를 직접 호출하지 않음
  - 무효 pinId가 감지되면 `GoalRecommendation.invalidatedPinId`에 해당 pinId를 담아 반환하고, UI 또는 provider listener가 post-frame cleanup으로 핀을 제거
  - 자동 추천 후보 1개 + 후보 최대 3개 노출, `score > 0`만 노출
- **[FR-11] 핀 영속 + 무효화**
  - `UserData.shortGoalPinId` HiveField 28 (String?) 신규 추가
  - `UserData.longGoalPinId` HiveField 29 (String?) 신규 추가
  - pinId 포맷: `'quest:{questId}'`, `'chain:{chainId}'`, `'craft:{recipeId}'`, `'trust:{regionId}:{nextLevel}'`, `'infra:{regionId}:{nextTier}'`, `'rank:{nextGrade}'`, `'pacify:{regionId}'`, `'faction_join:{factionId}'`
  - `UserDataNotifier.setShortGoalPin(String? pinId)` / `setLongGoalPin(String? pinId)` 메서드 신규 추가 (`save()` 호출 포함)
  - `GoalRecommendationService.recommendGoal()`은 무효 pinId를 결과에 표시만 한다. `LivingsphereSummarySection`과 `LivingsphereDetailScreen`은 `invalidatedPinId != null`을 감지하면 `WidgetsBinding.instance.addPostFrameCallback` 또는 `ref.listen`에서 `UserDataNotifier.set...Pin(null)`을 호출해 자동 정리한다.

#### C. UI — 홈 요약 카드 (`LivingsphereSummarySection`)

- **[FR-12] 카드 위치**
  - `home_screen.dart:386` 위치 (ChronicleHomeCard 직후, `_ActivityLog` 직전) 삽입
  - 배치 순서: FlagshipHomeCard → ChronicleHomeCard → **LivingsphereSummarySection** → _ActivityLog
  - 카드 padding: `EdgeInsets.symmetric(horizontal: 14, vertical: 4)` (기존 카드와 동일)
- **[FR-13] 카드 내용**
  - 헤더: 🏘️ "더스트플레인 생활권"  + 우측 정렬 "완성도 {total}%" (16~18sp)
  - 6 지표 미니 행: 각 지표명 + 값(%/Tier/n/m) + 진행 바 (Flutter `LinearProgressIndicator` 8px 높이)
    - 안정도, 거점 발전, 사건 완료, 자원·제작, 영향력, 위업 6행
    - 색상: §4.4 색상 매핑 표 참조
  - 30분 목표 행: 🎯 "30분: {label}" (자동/핀 모두 동일 포맷)
  - 8시간 목표 행: 🎯 "8시간: {label}"
  - 자세히 보기 버튼: 우측 정렬 텍스트 버튼 "자세히 보기 →" → `currentTabProvider`를 5(정보 탭)로 변경 + `infoScreenAutoShowLivingsphereProvider`(StateProvider<bool>)를 true로 설정 → InfoScreen이 감지하여 자동으로 `LivingsphereDetailScreen` 진입
- **[FR-14] 카드 빌드 임계 케이스**
  - region 3 외 위치: 헤더 "더스트플레인 생활권" 유지, 데이터는 항상 region 3 기준 (위치 무관)
  - 6 지표 중 데이터 부족: "—" 텍스트 + 0% 진행 바 (예: FactionState 없음 → 영향력 "—" + 0%)
  - 진행 중 작업 0개 + 임계값 멀 때: 30분 목표 stub "다음 의뢰를 시작하세요", 8시간 목표 stub "위업 컬렉션을 채워보세요"
- **[FR-15] 카드 reactivity**
  - watch 대상: `userDataProvider` / `livingsphereDashboardProvider` / `livingsphereGoalProvider(GoalSlot.short30Min)` / `livingsphereGoalProvider(GoalSlot.long8Hour)`
  - 30분/8시간 목표는 시간 진행에 따라 진행률이 변하므로 `gameTickProvider` watch 필요

#### D. UI — 정보 탭 상세 화면 (`LivingsphereDetailScreen`)

- **[FR-16] 진입 동선**
  - `InfoScreen`에 `_showLivingsphere = false` 상태 변수 추가 (라인 21~25 영역)
  - 명성 카드와 세력 도감 카드 사이에 신규 ListTile "🏘️ 더스트플레인 생활권" 추가 (`onTap: () => setState(() => _showLivingsphere = true)`)
  - `_showLivingsphere=true`일 때 `LivingsphereDetailScreen(onBack: () => setState(() => _showLivingsphere = false))` 렌더링 (기존 `_showChronicle` 패턴과 동일)
  - 자동 진입: `infoScreenAutoShowLivingsphereProvider` watch → non-null/true이면 `WidgetsBinding.instance.addPostFrameCallback`로 `_showLivingsphere=true` 설정 + provider 즉시 null/false 리셋 (factionCodexScrollTargetProvider 패턴 동일)
- **[FR-17] 상세 화면 구조**
  - 헤더: ← 뒤로가기 + "🏘️ 더스트플레인 생활권" + 우측 "완성도 {total}%"
  - 6 지표 펼침 카드 (각 카드 ExpansionTile 또는 자체 상태 기반 펼침):
    - 접힘 상태: 지표명 + 값 + 진행 바 + [▼] 펼침 아이콘
    - 펼침 상태: 상세 텍스트 + 점프 버튼 1~2개
    - 펼침/접힘 상태: SessionState (영속 X, 화면 이탈 시 초기화) — 로컬 `Map<MetricKey, bool> _expandedMap` 사용
  - 6 지표 상세 펼침 텍스트:
    - **안정도**: "{단계 라벨} 단계 / 임박 사건 N건" + 사건 요약 텍스트 ≤ 2줄
    - **거점 발전**: "신뢰도 {level}단계 ({trust}/{nextThreshold})" + "인프라 flag {n}/{m} (다음 후보: ...)"
    - **사건 완료**: "체인 진행 {step}/6 ({chainId})" + "발견 {n}/3, 상태 의뢰 {n}/2"
    - **자원·제작**: "특산품 {n}/5 (미보유: ...)" + "레시피 해금 {n}/10 (다음 부족 재료: ...)"
    - **영향력**: 세력 3행 "{이름} {stageLabel} ({reputation})" (`untouched`는 "미접촉")
    - **위업**: allowlist 5종 각각 "{name} {1/1 또는 0/1}"
  - 점프 버튼 (지표별):
    - 안정도 → `MovementScreen` (region 3 포커스)
    - 거점 발전 → 현재 위치가 region 3 village 섹터이면 `ChiefHouseScreen`, 아니면 `MovementScreen`에서 region 3 sector 1 포커스
    - 사건 완료 → `DispatchScreen` (활성 체인 또는 첫 region 3 fixed quest pool 포커스)
    - 자원·제작 → `InventoryScreen MaterialTab` + 현재 위치가 region 3 village 섹터이면 `OldSmithyScreen`, 아니면 `MovementScreen`에서 region 3 sector 1 포커스
    - 영향력 → `FactionCodexScreen`
    - 위업 → `ChronicleScreen`
  - 30분 목표 카드: §FR-19 참조
  - 8시간 목표 카드: §FR-20 참조
- **[FR-18] 카드 펼침/점프 동작 정책**
  - 현재 앱 셸은 `IndexedStack`이 아니라 `_screens[currentTab]` 단일 렌더링 구조이므로, 탭 전환 시 `InfoScreen`과 `LivingsphereDetailScreen`은 unmount될 수 있음

#### B. 신규 enum 및 모델 (모두 도메인 코드, Hive type 미생성)

- **`GoalSlot`**: `enum GoalSlot { short30Min, long8Hour }` — Hive 영속 미사용 (`UserData`에 pinId 문자열로 저장).
- **`MetricKey`**: `enum MetricKey { stability, infrastructure, eventCompletion, resourceCraft, influence, achievement }` — UI/서비스 공통 식별자.
- **`LivingsphereDashboardSnapshot`**: 불변 freezed 클래스 (Hive 영속 X, Provider 출력 전용)
  - `regionId: int` (MVP는 항상 3)
  - `metrics: Map<MetricKey, MetricValue>`
  - `totalCompletionPct: double`
- **`MetricValue`**: freezed
  - `percent: double` (0~100)
  - `displayMode: MetricDisplayMode` (percent / tierLevel / countOverTotal / averageStage)
  - `currentValue: num?` (예: Tier 3 → 3, n/m → n)
  - `totalValue: num?` (예: n/m → m)
  - `label: String?` (예: 단계 라벨 "평온", "Tier 3")
  - `expandedSummary: String?` (펼침 본문 요약, 화면에서 추가 텍스트 조합)
- **`GoalCandidate`**: freezed
  - `id: String` (pinId와 호환되는 포맷)
  - `slot: GoalSlot`
  - `label: String`
  - `kind: GoalCandidateKind` (enum: inProgressQuest / movement / investigation / construction / imminentTrust / imminentInfra / imminentRank / chainCompletion / craftUnlock / factionJoin / pacification / fallback)
  - `baseWeight: double`
  - `progressFactor: double`
  - `valueFactor: double`
  - `score: double` (산출 결과)
  - `jumpTarget: GoalJumpTarget?` (어디로 점프할지)
  - `isValid: bool` (핀 무효화 판정)
- **`GoalJumpTarget`**: freezed sealed union (구현은 freezed `@Freezed(unionKey: 'type')`)
  - `Movement(int? regionId)` — `livingsphereMovementTargetProvider` + tab=0
  - `Dispatch(String? questPoolId)` — `dispatchFocusQuestPoolIdProvider` + tab=1
  - `SettlementFacility(VillageFacility facility, int regionId)` — `settlementFacilityTargetProvider` + tab=0
  - `Inventory(String? itemId)` — `materialJumpTargetItemIdProvider` + tab=5
  - `Smithy` — `settlementFacilityTargetProvider→oldSmithy` + tab=0
  - `Faction(String? factionId)` — `factionCodexScrollTargetProvider` + tab=5
  - `Chronicle` — `infoScreenAutoShowChronicleProvider` + tab=5
- **`GoalRecommendation`**: freezed
  - `slot: GoalSlot`
  - `primary: GoalCandidate?` (null이면 fallback)
  - `pinned: bool`
  - `alternatives: List<GoalCandidate>` (최대 3개)
  - `isFallback: bool`
  - `invalidatedPinId: String?` (무효 pinId 감지 시 설정, UI/post-frame cleanup에서 핀 제거)

### 4.2 주의사항

- **CLAUDE.md 제약 준수**:
  - 화면 전환은 `Navigator.push` 대신 상태 기반 렌더링
  - 공용 위젯은 동일 패턴 3개 이상 반복 시 `shared/widgets/`로 (본 명세 위젯은 feature 내부에 유지 — 단일 사용)
  - 스타일은 `core/theme/` 사용, `shared/styles/` 생성 금지
- **HiveField 충돌 방지**: HiveField 28~29만 사용. 30~33은 페이즈 4 #6(주간 기여도)이 점유 예정이므로 침범 금지.
- **typeId 미생성**: MVP는 신규 Hive type을 만들지 않음. `LivingsphereGoalPin`, `LivingsphereDashboardSnapshot` 등 모두 freezed 또는 일반 클래스로만 정의. typeId 31+는 M9+로 위임.
- **gameTickProvider watch 분리**: `livingsphereDashboardProvider`는 watch 안 함. `livingsphereGoalProvider`는 watch 필요. 두 provider를 분리하여 6 지표 재계산 비용을 시간 진행과 무관하게 0으로 만듦.
- **Provider build 중 side effect 금지**: `GoalRecommendationService.recommendGoal()`은 핀 삭제, save, dialog enqueue 같은 상태 변경을 수행하지 않는다. 무효 pinId는 `GoalRecommendation.invalidatedPinId`로 반환하고 UI/listener가 post-frame에서 정리한다.
- **region 3 상태 반응성**: `currentRegionSectorChangesProvider`는 현재 유저 위치만 추적하므로 생활권 대시보드 입력으로 부적합하다. region 3 전용 version/watch provider를 별도로 둔다.
- **fail-soft**: snapshot 계산 중 `RegionState`가 null이면 안정도만 50%, 나머지는 0%로 기본값. 절대 예외 throw 금지.
- **freezed 모델 변경 시 build_runner 필수**: `lib/core/models/user_data.dart`와 신규 `livingsphere_dashboard_models.dart` 두 곳 변경 후 `dart run build_runner build --delete-conflicting-outputs` 실행. 누락 시 컴파일 실패.
- **펼침 상태는 SessionState**: `_LivingsphereDetailScreenState._expandedMap`은 영속 안 함. 화면 dispose 시 자동 초기화. 현 앱 셸은 탭 전환 시 현재 탭 화면만 렌더링하므로 점프 후 정보 탭 재진입 시 펼침 상태 보존을 보장하지 않는다.
- **점프 후 target provider null 리셋**: consume 화면(`MovementScreen` 등)에서 watch → consume 직후 즉시 null 리셋. 이중 publish 시 마지막 publish가 유효.

### 4.3 엣지 케이스

- **신규 유저(region 3 진입 직후 + RegionState null)**: 안정도 50% (fallback dangerScore=0), 나머지 0%, 통합 10%. ChainQuestProgress 비어있어 활성 체인 0개. 30분 fallback = "다음 의뢰를 시작하세요" + DispatchScreen 점프. 8시간 fallback = "위업 컬렉션을 채워보세요" + ChronicleScreen 점프. **단 사용자가 region 3 미진입 상태에서도 카드는 항상 표시**(기획서 § 3.4 빌드 임계 케이스).
- **region 3 외 위치**: 헤더 유지, 데이터는 항상 region 3 기준. 안정도 점프 시 region 3로 자동 이동(`livingsphereMovementTargetProvider→3`).
- **FactionState 일부 누락**: `region3ActiveFactionIds` 중 일부가 FactionState에 없는 경우 → `untouched`로 간주, 점수 0. 평균은 항상 3 세력 기준 분모.
- **active 체인 동시 다수**: 8시간 슬롯에서 `remainingSteps`가 가장 작은 체인 1개만 후보로 등록. settlement_ prefix 체인도 동일 처리.
- **핀 의뢰가 완료됨**: 다음 `recommendGoal()` 호출 시 isValid=false 감지 → `invalidatedPinId` 반환 + 자동 추천/fallback 노출. UI/listener가 post-frame에서 `UserDataNotifier.setShortGoalPin(null)`을 호출한다.
- **레시피 핀이 ready로 도달**: 동일하게 `invalidatedPinId` 반환 후 UI/listener가 핀을 자동 해제. 단 사용자가 다시 핀을 걸려고 시도하면 ready인 레시피는 "후보로 안 나타남"(score 0 또는 후보 미수집).
- **인프라 flag 1개 부족이나 그 flag가 핀**: 사용자가 flag 잠금 해제 액션 후 → 임박 인프라 후보 score 100 → 핀 유지 또는 자동 추천.
- **점프 대상에 데이터 없음**: dispatchFocusQuestPoolIdProvider에 quest_pool_id를 publish했으나 해당 quest_pool이 없거나 DispatchScreen에서 발견 못함 → 무시 + null 리셋 + DispatchScreen은 기본 상태로 표시. 사용자에게 토스트 등 알림 불필요(silent skip).
- **도감/인벤토리 전체 보기 점프**: 특정 factionId/itemId가 없으면 null target provider를 publish하지 않는다. 대신 `infoScreenAutoShowCodexProvider` 또는 `infoScreenAutoShowInventoryProvider`를 true로 설정한다.
- **시설 점프 시 현재 위치가 마을이 아님**: `settlementFacilityTargetProvider`는 시설을 즉시 열지 않고 region 3 sector 1 포커스만 맞춘다. 사용자가 실제 이동을 완료한 뒤 다시 시설 버튼을 누르거나 목표 점프를 재실행하면 해당 시설을 연다.
- **gold/스테이트가 빈 사용자(`userDataProvider == null`)**: 카드 자체를 `SizedBox.shrink()`로 처리 (CircularProgressIndicator는 home 상위에서 이미 처리됨, 라인 155).
- **신뢰도 1단계(설정 안 됨)**: nextThreshold가 정의 안 됐을 경우 fallback (default 30, 무한대 처리). `SettlementInfrastructureConfig.infraTierThresholds`에서 누락 케이스 회피.

## 5. 기획 확인 사항

- [Q-1] **현재 `AppTheme`에 자원·제작 전용 색상이 없음**: 자원·제작 진행 바 색상을 `materialAccent`(예: 0xFF8D6E63 갈색)로 신규 추가할지, 아니면 fallback으로 `AppTheme.textSecondary`나 `AppTheme.chainGold`(이미 사건 완료에 사용 — 중복 가능성)를 임시 사용할지? → **권장**: `materialAccent` 신규 추가 (M5 베이스 색상 명시 미루기보다 명세 시 결정).
- [Q-2] **30분 임박 명성 후보의 `currentRep`와 `nextRankRep`은 어디서 가져오는가?** `ReputationService`(또는 동일 기능)에 `getCurrentRankInfo(reputation)`/`getNextRankThreshold(reputation)` 함수가 있는지 구현 시 확인 필요. 없으면 기존 `RankInfoScreen` 표시 로직에서 동일 함수 활용. → **권장**: 구현 1차 단계에서 `ReputationService` 함수 확인. 없으면 본 명세 범위에서 `ReputationService.getNextRankInfo()` 신규 추가.
- [Q-3] **dispatchFocusQuestPoolIdProvider 점프 시 quest_pool_id 선택**: 사건 완료 점프 시 우선순위는 (a) 활성 체인 다음 단계의 quest_pool_id (b) region 3 fixed quest_pool 첫 ID (c) target 미발행 + 파견 탭 기본 화면이다. 활성 체인의 step에서 quest_pool_id 매핑이 명확한가? → **권장**: `chain_quests.step_data`에 quest_pool_id가 있으면 (a), 없으면 staticData에서 region 3 fixed quest_pool 첫 ID를 찾아 (b), 둘 다 없으면 `dispatchFocusQuestPoolIdProvider`를 건드리지 않고 `currentTabProvider=1`만 수행한다.
- [Q-4] **점프 후 LivingsphereDetailScreen 자동 복귀**: 사용자가 안정도 점프(이동 탭으로 전환) 후 다시 정보 탭으로 돌아왔을 때 _showLivingsphere=true가 유지되어 자동으로 LivingsphereDetailScreen이 보이는가? → **확인 결과**: 현재 `app.dart`는 `IndexedStack`이 아니라 `_screens[currentTab]` 단일 렌더링 구조다. 탭 전환 시 `InfoScreen`이 unmount될 수 있으므로 펼침 상태 보존은 요구하지 않는다. 상세 화면 재진입이 필요하면 `infoScreenAutoShowLivingsphereProvider`를 true로 설정한 뒤 정보 탭 진입 시 소비한다.
- [Q-5] **CraftingService.evaluateState** 호출 시 ref 필요 여부: 본 명세는 `LivingsphereDashboardService.computeSnapshot(ref)` 정적 메서드 안에서 `staticData.craftingRecipes`를 순회하며 각 레시피의 unlock 상태를 확인해야 함. `CraftingService`가 ref 기반 인스턴스 메서드일 경우 호출 방식 검토. → **확인 결과**: 구현 시 `craftingServiceProvider`로 `CraftingService` 인스턴스 획득 후 `evaluateState(recipe)` 호출. `computeSnapshot` 시그니처는 `(WidgetRef ref)` 또는 `(Ref ref)`. ref.read로 craftingService 획득.
- [Q-6] **펼침 카드의 "다음 부족 재료" 표시**: 자원·제작 펼침 카드에서 "다음 부족 재료" 1~2개를 표시하려면 모든 region 3 레시피 중 부족한 재료를 어떻게 우선순위 매기는가? (예: 가장 가까운 ready 도달 가능한 레시피의 부족 재료, 또는 가장 자주 등장하는 부족 재료) → **권장**: ready로 도달까지 부족 재료 개수가 가장 적은 레시피 1개의 첫 부족 재료를 표시. 구현 시 단순화 가능.
- [Q-7] **carving `recipeTier` 의미**: 8시간 슬롯의 "핵심 제작 레시피 해금" 후보의 `value_factor = recipeTier * 10`에서 recipeTier는 무엇? 결과 아이템의 tier, 또는 recipe 자체의 unlock 조건 tier. → **권장**: 결과 아이템의 `items.tier`(1~5). recipe → output item → `tier` 조회.
- [Q-8] **핀 무효화 자동 정리 시 race condition**: `recommendGoal` 호출 안에서 `UserDataNotifier.setShortGoalPin(null)` 호출은 Provider의 build 중 발생할 수 있음. 이는 watch loop 가능성. → **결정**: `recommendGoal`은 무효 pinId를 `GoalRecommendation.invalidatedPinId`로 반환만 한다. `LivingsphereSummarySection`/`LivingsphereDetailScreen` 또는 provider listener가 post-frame에서 `setShortGoalPin(null)`/`setLongGoalPin(null)`을 호출한다.
- [Q-9] **CHANGELOG 및 docs**: 본 기능 구현 완료 시 `Docs/changelog-fragments/`에 fragment 작성하고 향후 `/merge-changelog`로 합산. CLAUDE.md "feature 모듈 구조" 섹션의 새 디렉토리(`info/domain/livingsphere_*`)는 추가하지 않아도 됨(기존 info feature 확장).
