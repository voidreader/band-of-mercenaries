## 1. 개요

용병의 능력은 STR/INT/VIT/AGI 4 기본 스탯과 트레잇으로 완전히 노출되어 있다. 그러나 "그 용병이 어떤 위기를 겪고 살아남았는가"는 시뮬레이션 결과로 휘발된다. M8.5 히든 스탯은 그 위기·극복 사건을 **점진적 정수 레벨로 누적**하여, 시간이 흐를수록 "이 용병은 어떤 잠재력을 갖고 있는가"를 드러내는 두 번째 정체성 채널을 만든다.

5 히든 스탯(**불굴**·**투지**·**운**·**공포 저항**·**전장 감각**)을 도입하되, 각 스탯은 0~5 레벨로 점진 성장한다. 레벨업 트리거는 M8.5 페이즈 1 #3에서 정의한 감정 반응 발동 사건을 직접 hook으로 사용한다(예: 투지 발동 → fortitude 카운터 +1). 효과는 M8b CombatSimulator hook(사망 저항·명중·회피·mez 면제 등), 기존 PassiveBonusService로 표현 가능한 보너스(회복 시간·명성 등), QuestCompletionService 전용 보너스(드랍 추가 판정)로 나누어 적용한다.

핵심 결정 사항:
1. **데이터 구조**: `Mercenary.hiddenStats: Map<String, int>` 신규 HiveField 26. 5 키 × 0~5 레벨. 런타임 이벤트 모델은 만들지 않고, 정적 정의만 `hidden_stats` 5행으로 관리한다.
2. **레벨업 임계값**: 누적 카운터 3배수 점진 — **1 / 3 / 7 / 15 / 30**. lv1 도달은 1회 사건으로 즉시, lv5는 30회 누적. M6 칭호 단계 임계값보다 부드러운 곡선.
3. **트리거 hook**: `Mercenary.stats: Map<String, int>`에 5 신규 카운터 (`fortitude_event_count` 등). M6 action_stat hook 패턴 100% 재사용. 페이즈 1 #3 감정 발동 사건이 직접 카운터 +1.
4. **효과 3계층**: M8b CombatSimulator hook 직접 가산(사망 저항·명중·회피·mez 면제·치명타·action_score·featured_score 등) + 기존 PassiveBonusService 효과 활용(회복 시간·명성 등) + QuestCompletionService 전용 후처리(아이템 드랍 추가 판정). 지원하지 않는 PassiveEffect 타입을 새로 가정하지 않는다.
5. **표시 정책**: 용병 상세 화면 신규 `HiddenStatsSection`. **lv1+만 노출**, 미해금(lv0)은 완전 숨김. 다음 lv 진행도 바 (현재 카운트 / 다음 임계). lv1 해금 시 발견 다이얼로그 (medium priority).
6. **트레잇과의 차별**: 트레잇 = 슬롯 7개 한정 + 직접 능력 + 충돌·진화·삭제. 히든 스탯 = 5 슬롯 무제한 + 점진 잠재력 + 충돌·진화·삭제 없음 + 사건 누적.

종료 조건 매핑:
| roadmap M8.5 #4 종료 조건 | 본 문서 충족 |
|---|---|
| "불굴, 투지, 운 등 히든 스탯이 전투 사건으로 해금된다" | 5 스탯 정의 + 사건 누적 트리거 + lv1 해금 다이얼로그 |
| "용병 상세 화면에서 히든 스탯을 확인할 수 있다" | §3.5 `HiddenStatsSection` UI 명시 |
| "M8b CombatSimulator 자연 통합, 신규 시스템 최소화" | `Mercenary.hiddenStats`/`stats` Map 키 + 정적 카탈로그 1개만 추가 |

## 2. 레퍼런스 분석

| 레퍼런스 | 차용 메커니즘 | 본 시스템 적용 |
|---------|-------------|---------------|
| **The Elder Scrolls V: Skyrim — Skill Usage Leveling** | 행동 사용 횟수 누적 → 스킬 레벨 자동 증가 (검술 100회 사용 → 검술 lv +1) | 누적 카운터 기반 점진 레벨업의 직접 영감. 본 시스템은 5 스탯 한정 + 5 레벨 cap |
| **Mount & Blade II Bannerlord — Perks Unlock by Skill Level** | 스킬 레벨 도달 시 특정 perk 해금. 점진적 발견의 즐거움 | 히든 스탯 lv1 해금 시 발견 다이얼로그 (medium priority) — "이 용병에게 새로운 잠재력을 발견했습니다" |
| **Crusader Kings III — Hidden Traits / Lurking Traits** | 인물의 일부 트레잇이 숨겨져 있다가 특정 사건 후 발현 | "히든" 컨셉의 차용. lv0은 완전 숨김 — 노출 자체가 발견 |
| **Darkest Dungeon — Affliction Acceptance → Virtue Probability** | 위기 사건 누적 → 영구 캐릭터 변화의 가능성 | "위기는 누적되어 잠재력이 된다" 컨셉. 본 시스템은 5 카테고리로 분류된 누적 |
| **Pillars of Eternity II — Disposition Levels (Aggressive 1 → 5)** | 행동 누적 → 성향 레벨 + 대화·전투 hook | 정수 레벨 0~5의 직접 영감. 본 시스템은 능력 hook으로 확장 |
| **Wartales — Trauma & Recovery Stats** | 전투 사건 누적 → 외상/회복 정수 | 누적 카운터의 정수 표현 차용. 본 시스템은 5 카테고리 별도 트래킹 |
| **Battle Brothers — Background Traits → Latent Stats** | 출신 배경에 따라 일부 스탯이 시간 지나서 드러남 | 트레잇과의 차별점 — 트레잇은 슬롯, 히든 스탯은 시간 누적 |

**핵심 설계 원칙**:
- **"위기는 시간 위에서 잠재력이 된다"** — 사건이 휘발되지 않고 누적되어, 시간이 흐를수록 용병의 정체성이 더 풍부해진다. M3 트레잇이 "어떤 능력을 가진 용병인가"라면, 히든 스탯은 "어떤 위기를 겪고 어떻게 성장했는가"이다.
- **"히든"의 의미 보존** — lv0 스탯은 완전히 숨겨져 있다. 첫 lv1 해금이 "발견" 사건이 되며 그 자체로 게임 보상. Skyrim/CK3 패턴의 점진적 발견.
- **"트레잇과 다른 채널"** — 트레잇은 슬롯 7개 한정 + 충돌·진화·삭제 가능. 히든 스탯은 5 슬롯 무제한 + 점진 + 영구 + 사건 기반. 두 시스템이 한 용병의 두 정체성 면을 보여준다.
- **"기존 데이터 모델 최소 확장"** — 영속 데이터는 `Mercenary.hiddenStats` HiveField 1개와 `Mercenary.stats` Map 키만 추가한다. 정적 데이터는 `hidden_stats` 5행 카탈로그로 분리하고, 신규 Hive 박스나 신규 런타임 이벤트 모델은 만들지 않는다.

## 3. 상세 설계

### 3.1 5 히든 스탯 정의

#### 3.1.1 표

| ID | 한국어명 | 의미 | M8b hook 효과 | 비전투/후처리 효과 |
|----|---------|------|-------------|----------------------|
| `fortitude` | 불굴 | 끈기와 회복력 | death_resistance +0.02/lv | injury_recovery_time ×(1.00 - 0.05/lv) |
| `grit` | 투지 | 정신력과 위기 극복 | emotional_despair 면제 확률 +0.10/lv | reputation_gain +0.02/lv (위기 극복 평판) |
| `luck` | 운 | 행운과 우연 | critical_rate +0.01/lv + evasion +0.01/lv | 보상 드랍 추가 판정 +0.01/lv (`QuestCompletionService` 전용) |
| `fear_resistance` | 공포 저항 | 침착함 | mez_immune_chance +0.05/lv + 적 강한 공격에서 추가 회피 +0.02/lv | (없음) |
| `battle_sense` | 전장 감각 | 전투의 흐름을 읽는 능력 | action_score +1/lv + featured_score +0.25/lv (소수점 누적) + hit_chance +0.01/lv | (없음) |

본 표의 효과 강도는 권장 default. 페이즈 2 #4 시뮬레이션 검증 후 확정.

#### 3.1.2 효과의 누적 의미

5 스탯이 모두 lv5에 도달한 용병 (총 25 레벨 누적):
- death_resistance +0.10 + emotional_despair 면제 +0.50 (사실상 절망 면제) + critical_rate +0.05 + evasion +0.05 + mez_immune +0.25 + action_score +5 + featured_score +1.25 + hit +0.05
- 비전투/후처리: injury_recovery_time ×0.75 + 보상 드랍 추가 판정 +0.05 + reputation_gain +0.10

이는 "전설적인 베테랑" 수준의 부가 능력. M6 칭호 효과(2~5%)와 결합하면 lv5 풀스택 용병은 단순한 T5 신규 용병과 구분되는 정체성을 가진다.

### 3.2 데이터 구조

#### 3.2.1 Mercenary 모델 확장

```dart
@HiveType(typeId: 1)
class Mercenary extends HiveObject {
  // ... 기존 0~25 ...

  @HiveField(26)
  Map<String, int> hiddenStats;  // 신규 — 히든 스탯 ID → lv (0~5)
}
```

**HiveField 점유 갱신**:
- 현재 `Mercenary`는 `titleIds` HiveField 24, `recruitedAt` HiveField 25까지 점유한다.
- 본 문서는 `hiddenStats`에 **HiveField 26**을 사용한다.
- 구현 후 `Mercenary` 다음 HiveField는 **27**이다.

**기존 데이터 호환**: nullable 또는 default 빈 Map. Hive 자동 마이그레이션. 기존 용병은 `hiddenStats == {}`로 시작.

#### 3.2.2 Mercenary.stats 신규 카운터

기존 `Mercenary.stats: Map<String, int>`에 5 신규 키:

| 카운터 키 | 증가 사건 | 매핑되는 히든 스탯 |
|----------|---------|---------------|
| `fortitude_event_count` | 사망 저항으로 살아남 / 투지 발동 / 솔로 의뢰 완수 | 불굴 |
| `grit_event_count` | 절망 면제 / 솔로 의뢰 대성공 / 체인 주인공 위기 극복 | 투지 |
| `luck_event_count` | 치명타 발동 / 회피 성공 (적의 명중에서) / 보상 아이템 드랍 / 결정적 장면 기여 | 운 |
| `fear_resistance_event_count` | mez 면제 / 유니크 엘리트 대면 후 생존 / 적 강한 공격 (피해 ≥ HP×0.30) 회피 | 공포 저항 |
| `battle_sense_event_count` | 치명타 처치 (`criticalKill`) / 결정적 장면 기여 (점수 ≥ 5) / 명중 성공 (적 명중률 ≤ 0.50) | 전장 감각 |

신규 모델 변경 없음. Map 값 추가만. CLAUDE.md "행동 지표" 패턴 그대로 확장.

#### 3.2.3 신규 정적 카탈로그 vs 코드 상수

5 히든 스탯의 정의(이름·효과·임계값)는 어디에 저장하는가:
- 옵션 A: 신규 Supabase 테이블 `hidden_stats` 5행
- 옵션 B: Dart 정적 상수 (`HiddenStatCatalog`) — 5 스탯이 운영 중 변경 가능성 낮음
- 옵션 C: `trait_categories` 테이블에 카테고리 추가 (트레잇 시스템과 다른 채널이므로 부적합)

**권장**: 옵션 A — 신규 `hidden_stats` 테이블 (41번째). M8b·M8a 카탈로그 패턴 정합. 운영 도구에서 효과 미세 조정 가능. 페이즈 3 #3 시드로 5행 INSERT.

대안 옵션 B(Dart 정적 상수)는 페이즈 4 #3 spec 단계에서 운영 정책 결정 후 확정. 본 문서는 옵션 A를 권장.

#### 3.2.4 Supabase `hidden_stats` 테이블 (옵션 A 채택 시)

| 컬럼 | 타입 | 제약 | 의미 |
|------|------|------|------|
| `id` | TEXT | PRIMARY KEY | 히든 스탯 ID (예: `fortitude`) |
| `name` | TEXT | NOT NULL | 한국어 표시명 |
| `description` | TEXT | NOT NULL | 1~2문장 설명 |
| `counter_key` | TEXT | NOT NULL | Mercenary.stats 카운터 키 (예: `fortitude_event_count`) |
| `level_thresholds` | JSONB | NOT NULL | `[1, 3, 7, 15, 30]` (5 임계값 배열) |
| `combat_effects_json` | JSONB | NOT NULL | M8b hook → per-lv 효과 매핑 (예: `{"death_resistance": 0.02}`) |
| `passive_effects_json` | JSONB | NULL | 기존 PassiveEffect 직렬화. NULL이면 PassiveBonus 가산 없음 |
| `post_reward_effects_json` | JSONB | NULL | QuestCompletionService 전용 후처리 효과. 예: `{"drop_bonus": 0.01}` |
| `icon_key` | TEXT | NOT NULL DEFAULT 'default' | UI 아이콘 식별자 |
| `narrative_hint` | TEXT | NULL | 운영자용 톤 가이드 |

**SyncService 등록**: 현재 M8b 기준 40개 테이블 뒤에 **41번째 테이블**로 등록한다. `hidden_stats`는 배포 순서 리스크를 줄이기 위해 `optionalTables`에도 포함하고, 캐시가 비어 있으면 히든 스탯 UI와 효과 적용을 fail-soft skip한다. data_versions 엔트리 + DataLoader 캐시 + `StaticGameData.hiddenStats` 필드를 추가한다.

### 3.3 레벨업 임계값

#### 3.3.1 임계값 곡선 — 3배수 점진

| lv | 누적 카운터 |
|----|------------|
| 1 | 1 |
| 2 | 3 |
| 3 | 7 |
| 4 | 15 |
| 5 | 30 |

3배수 점진(1·3·7·15·30)은 다음 의미:
- lv1: 첫 사건 즉시 해금 ("발견" 사건)
- lv2: 3회 누적 (단기 도달)
- lv3: 7회 누적 (의도된 플레이로 도달)
- lv4: 15회 누적 (장기 플레이어)
- lv5: 30회 누적 (전설적 베테랑)

M6 칭호 임계값(예: raid_count 30 → 칭호 1회 해금)과 비교:
- M6 칭호는 단계 1개 (보유 vs 미보유)
- 히든 스탯은 단계 5개로 더 부드러운 곡선
- 같은 30회 누적이 칭호 1개 vs 히든 스탯 lv5 두 다른 의미

#### 3.3.2 사건당 가중치

기본 +1 가산. 일부 사건은 가중치 차별화 (페이즈 2 #4에서 확정 위임):

| 사건 | 카운터 가중 | 근거 |
|------|----------|------|
| 사망 저항으로 살아남 (사망 저항 롤 성공) | +1 | 일반 |
| 투지 발동 (`emotional_determination` apply) | +1 | 일반 |
| 솔로 의뢰 완수 (성공·대성공) | +2 | 큰 사건 |
| 솔로 의뢰 대성공 | +1 추가 (총 +3) | 매우 큰 사건 |
| 치명타 발동 | +1 | 일반 |
| 결정적 장면 기여 (점수 ≥ 5) | +1 | 일반 |
| 결정적 장면 기여 (점수 ≥ 10) | +2 | 큰 사건 |

본 가중치는 권장 default. 페이즈 2 #4에서 시뮬레이션 검증 — 신규 유저 누적 10시간 시점 평균 lv 분포 시뮬레이션.

### 3.4 효과 적용 — 3계층 통합

#### 3.4.1 M8b CombatSimulator hook 직접 가산

`CombatSimulator.simulate` 진입 시 partyMercs의 hiddenStats를 평가 → hook별 추가 보너스 산출 → 기존 M8b hook 계산식에 가산.

현재 `CombatSimulator`는 `partyMercs` 원본과 `CombatantSnapshot`을 모두 입력으로 사용할 수 있다. 히든 스탯 보너스는 전투 중 런타임 `_Combatant`에 `hiddenStats` Map을 들고 들어가 적용한다. 보고서용 `CombatantSnapshot`에는 필드를 추가하지 않는다.

```dart
// CombatSimulator helper (개념)
double hiddenStatBonusFor(Mercenary merc, String hook) {
  final stats = merc.hiddenStats;
  switch (hook) {
    case 'death_resistance':
      return (stats['fortitude'] ?? 0) * 0.02;  // +0.02/lv
    case 'critical_rate':
      return (stats['luck'] ?? 0) * 0.01;
    case 'evasion':
      final luck = stats['luck'] ?? 0;
      final fear = stats['fear_resistance'] ?? 0;
      return luck * 0.01 + fear * 0.02;
    case 'mez_immune_chance':
      return (stats['fear_resistance'] ?? 0) * 0.05;
    case 'action_score':
      return (stats['battle_sense'] ?? 0) * 1.0;
    case 'featured_score':
      return (stats['battle_sense'] ?? 0) * 0.25;
    case 'hit_chance':
      return (stats['battle_sense'] ?? 0) * 0.01;
    case 'emotional_despair_immune':
      return (stats['grit'] ?? 0) * 0.10;
    default:
      return 0.0;
  }
}
```

페이즈 4 #3 spec에서 정밀화. 위 helper는 M8b `CombatSimulator` 내부 또는 별도 `HiddenStatBonusResolver` 정적 서비스.

#### 3.4.2 PassiveBonusService 통합

`PassiveBonusService.collect(..., hiddenStatEffects: ...)`에 기존 PassiveEffect로 표현 가능한 보너스만 가산한다. 현재 코드의 PassiveEffect에는 `recovery_time_reduction`과 `reputation_gain_modifier`가 있으므로 불굴·투지 보너스는 이 경로로 처리한다. `item_drop_rate` 같은 미지원 PassiveEffect 타입은 만들지 않는다.

```dart
List<PassiveEffect> collectHiddenStatPassives(Mercenary merc, StaticGameData data) {
  final result = <PassiveEffect>[];
  for (final entry in merc.hiddenStats.entries) {
    final lv = entry.value;
    if (lv == 0) continue;
    final stat = data.hiddenStats.firstWhere((s) => s.id == entry.key);
    final passiveJson = stat.passiveEffectsJson;
    if (passiveJson == null) continue;
    // passiveJson은 {"effects":[...]} 컨테이너. 각 value를 lv배 스케일링한 뒤 기존 parser 재사용.
    result.addAll(PassiveEffect.parseEffects(scalePassiveEffects(passiveJson, lv)));
  }
  return result;
}
```

`scalePassiveEffects`는 `effects` 배열의 각 `value`를 lv에 비례해 조정한다. 페이즈 4 #3 spec에서 정확한 직렬화 정합을 결정한다.

#### 3.4.3 QuestCompletionService 후처리

`luck`의 보상 드랍 추가 판정은 PassiveBonusService가 아니라 의뢰 보상 확정 직후의 QuestCompletionService 전용 helper에서 처리한다. 기존 `PassiveEffect` sealed 타입에 없는 `item_drop_rate` 타입을 추가하지 않는다.

개념:
```dart
double hiddenStatDropBonus(Mercenary merc) {
  final luckLv = merc.hiddenStats['luck'] ?? 0;
  return (luckLv * 0.01).clamp(0.0, 0.05);
}
```

파티 보너스는 참가자 중 최대값 1개만 적용한다. 합산 적용은 5인 파티에서 드랍률이 과도하게 상승하므로 페이즈 2 #4에서 별도 검증하기 전까지 금지한다.

#### 3.4.4 효과 동시 적용 정합

같은 hook(예: evasion)에 luck lv4 (+0.04) + fear_resistance lv2 (+0.04) 동시 가산:
- M8b hook 계산 시 모든 가산 합산 → 기존 hook별 clamp 통과
- 트레잇 보너스·세력 보너스와 별도 채널 (페이즈 4 #3에서 명시)

    "이 용병은 위기 속에서도 결코 무너지지 않는 끈기를 보입니다."

- 메시지: `"✨ {merc.name} 히든 스탯 '불굴' lv2 → lv3"`

#### 3.7.2 카운터 증가 처리 시점

`CombatSimulator` 내부 사건과 `QuestCompletionService` trailing의 두 경로로 분리:

- **`CombatSimulator` 내부 사건**: simulate 종료 시 `CombatSimulationResult`에 `hiddenStatEvents: Map<String, Map<String, int>>` (mercId → counterKey → delta) 결과 반환. 현재 `CombatSimulationResult`는 HiveField 12까지 점유하므로 신규 필드는 HiveField 13을 사용한다.
- **`QuestCompletionService` trailing**: 보상 처리 직후 fail-soft try/catch로 카운터 증가 + lv 임계 평가 + 임계 통과 시 hiddenStats lv +1 + `hiddenStatUnlockedProvider` enqueue.

#### 3.7.3 lv 임계 평가

```dart
void evaluateHiddenStatLevelUp(Mercenary merc, String hiddenStatId, StaticGameData data) {
  final stat = data.hiddenStats.firstWhere((s) => s.id == hiddenStatId);
  final counterKey = stat.counterKey;
  final currentCount = merc.stats[counterKey] ?? 0;
  final currentLv = merc.hiddenStats[hiddenStatId] ?? 0;
  if (currentLv >= 5) return;  // max
  final nextThreshold = stat.levelThresholds[currentLv];  // currentLv=0 → thresholds[0]=1
  if (currentCount < nextThreshold) return;
  // lv up
  merc.hiddenStats[hiddenStatId] = currentLv + 1;
  if (currentLv + 1 == 1) {
    // lv1 해금 → 발견 다이얼로그
    enqueueHiddenStatUnlocked(merc, stat);
  }
  activityLog.add(ActivityLogType.hiddenStatLevelUp, ...);
}
```

페이즈 4 #3 spec에서 정밀화. fail-soft 처리.

### 4.2 호환성 검토

- **기존 Mercenary 데이터**: `hiddenStats == {}` (빈 Map) default. Hive 자동 호환. lv0 = 키 없음으로 처리.
- **기존 Mercenary.stats**: 5 신규 카운터 키 default = 0 또는 키 없음. `stats[key] ?? 0` 패턴.
- **기존 MercenarySnapshot 데이터**: 본 문서는 snapshot 필드 추가를 필수로 요구하지 않는다. M8.5 #5에서 보존을 채택할 경우 다음 가용 HiveField 6을 nullable로 추가하면 기존 데이터와 호환된다.
- **M6 칭호 시스템**: 영향 없음. 칭호 효과와 히든 스탯 효과 별도 채널 가산.
- **M8b CombatSimulator**: hook 가산 helper 추가는 기존 hook 계산 정합. death_resistance·hit·evasion 등 각 hook의 기존 clamp 범위를 그대로 유지.
- **M8.5 페이즈 1 #2 솔로 의뢰**: 솔로 의뢰 완수가 fortitude·grit 카운터 증가 → 자연 시너지.
- **M8.5 페이즈 1 #3 감정 반응**: 4 감정 발동 사건이 hiddenStat 카운터의 핵심 hook → 자연 결합.

### 4.3 호환성 리스크

- **낮음**: 41번째 테이블 추가는 SyncService 등록 부담 1회. `optionalTables` 등록 시 마이그레이션 전 클라이언트도 빈 리스트 fallback으로 기동 가능.
- **중간**: 효과 적용 순서 — M8b hook 가산 단계에서 트레잇·세력·랭크·히든 스탯 모두 가산. 누적 cap 도달 (예: evasion 0.75 cap) 시 분배 정책 결정 필요. 페이즈 4 #3 spec에서 명시.
- **중간**: `Mercenary.stats` Map 키 폭증 — M3 기존 26 키 + M6 3 키 + M8.5 페이즈 1 #2 4 키 + 본 문서 5 키 = 38+ 키. 모델 변경 없으나 디버깅·로깅 부담. 페이즈 4 #4 UI에서 카운터 그룹화 검토.
- **낮음**: lv 임계 평가 fail-soft. 카운터 증가 실패해도 시뮬레이션 무결성 유지.
- **중간**: PassiveBonusService 통합 시점 — 히든 스탯 보너스가 기존 collect 호출부에 자연 수집되어야 함. 페이즈 4 #3에서 `hiddenStatEffects` 인자 주입 경로를 결정한다.

## 6. data-generator 지시사항

본 문서의 5행 `hidden_stats`는 페이즈 3 #3 또는 페이즈 4 #3 인라인 SQL 처리 권장. 텍스트량 적음.

- **대상 타입**: 별도 타입 스펙 불요 (M8b·M8a 신규 테이블 패턴 정합)
- **대상 테이블**: `hidden_stats` 5행
- **생성 수량**: 5행 (§3.1.1 표 그대로 고정)
- **톤/세계관 가이드**:
  - 한국어 판타지 톤. 각 스탯은 "이 용병의 어떤 잠재력인가" 톤
  - description 1~2문장. 발견 다이얼로그에서 노출되므로 감성적 문체
  - narrative_hint는 운영자용 톤 가이드 (예: "위기 극복 사건의 누적", "치명타·회피·드랍의 행운")
  - 고유명사 저작권 금칙
- **구조적 제약**:
  - 5행 id: `fortitude` / `grit` / `luck` / `fear_resistance` / `battle_sense`
  - 5행 counter_key: 각각 `{id}_event_count` 컨벤션
  - level_thresholds 전 행: `[1, 3, 7, 15, 30]` 통일
  - combat_effects_json 행마다 다름 (§3.1.1 표)
  - passive_effects_json은 fortitude/grit 2행만 사용하고, luck/fear_resistance/battle_sense는 NULL
  - post_reward_effects_json은 luck 1행만 `{"drop_bonus": 0.01}`로 사용
- **수치 출처**: 페이즈 2 #4에서 효과 강도·임계값 시뮬레이션 검증
- **특수 요구**:
  - `combat_effects_json`은 hook name → per-lv 값 매핑 (예: `{"death_resistance": 0.02}` lv 1당 가산)
  - `passive_effects_json`은 기존 PassiveEffect 직렬화. lv별 scale은 HiddenStatService/PassiveBonusService 연결부에서 처리
  - `post_reward_effects_json`은 PassiveEffect가 아니며 QuestCompletionService 후처리 helper만 해석
- **검증**:
  - 5행 모두 id PK 중복 없음
  - level_thresholds JSON 배열 길이 5
  - combat_effects_json hook key가 M8b CombatSimulator 인지 hook 이름과 정합

## 7. 오픈 질문

- **Q-1 (효과 강도 수치 정확성)**: §3.1.1 표 권장값(death_resistance +0.02/lv 등). **권장**: 페이즈 2 #4에서 시뮬레이션 검증. 누적 cap 도달 시 분배 정책도 함께 결정.
- **Q-2 (사건 가중치 정확성)**: §3.3.2 가중치 (솔로 완수 +2, 솔로 대성공 +3 등). 신규 유저 10시간 시점 평균 lv 분포 시뮬레이션 검증 필요. **권장**: 페이즈 2 #4 위임.
- **Q-3 (정적 카탈로그 vs 신규 테이블)**: **결정**. 옵션 A(신규 `hidden_stats` 테이블)를 채택한다. 단 `hidden_stats`는 `optionalTables`에 포함해 배포 전후 빈 캐시를 허용한다.
- **Q-4 (MercenarySnapshot 보존 정책)**: 사망/방출 시 hiddenStats를 MercenarySnapshot에 보존하는가? **권장**: M8.5 페이즈 1 #5 "용병 전투 기억" 작성 시점에 결합 결정. 보존 시 연대기에 "(故 {name}은 불굴 lv5, 운 lv3을 가졌습니다)" 표시.
- **Q-5 (효과 동시 가산 분배 정책)**: 같은 hook에 트레잇 + 세력 + 칭호 + 히든 스탯 4 채널 가산 시 분배. **권장**: 페이즈 4 #3에서 가산 → clamp 순서 명시. M6 success rate 분해 패턴 정합.
- **Q-6 (lv5 도달 후 카운터 누적 처리)**: lv5 도달 후에도 카운터는 계속 증가? **결정**: 증가 (영구 카운터). lv5 = max, 추가 사건은 카운터에만 누적. UI에 100% 충만 + 카운트 표시.
- **Q-7 (페이즈 1 #3 감정 발동 카운터와 페이즈 1 #4 hiddenStat 카운터의 중복 처리)**: 동일 사건(예: 투지 발동)이 페이즈 1 #3 측 emotional_history와 페이즈 1 #4 측 fortitude_event_count 둘 다에 기록되는가? **결정**: 둘 다 기록 (다른 채널). 페이즈 1 #3은 시각화/보고서용, 페이즈 1 #4는 영속 카운터.
- **Q-8 (operation-bom 편집 폼의 JSONB 편집 UX)**: combat_effects_json 같은 복잡 JSONB 편집은 관리자 도구 부담. **권장**: 페이즈 4 #4 별도 작업으로 위임. M8.5 본 마일스톤 범위 외.
