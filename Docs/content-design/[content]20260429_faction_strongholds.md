## 개요

14개 세력이 패시브(M1) → 전용 시설(M4 #1) → 전용 트레잇(M4 #4) 순으로 깊어지는 흐름에서, **거점**은 "현장 인터랙션" 축을 담당한다. 시설은 야영지에서 자동 작동하는 백그라운드 보너스, 패시브는 상시 적용되는 정적 보너스인 반면, **거점은 "그 리전에 직접 가야만 발동하는 능동형 컨텐츠"의 진입점**이다.

거점이 채우는 빈자리는 다음과 같다:

1. **이동의 의미 회복**: 현재 이동 시스템은 퀘스트 변동을 위한 수단일 뿐, 특정 리전에 직접 가야 할 강한 동기가 약하다 (변형 섹터·체인 퀘스트가 일부 제공). 거점은 "가입 세력의 본거지로 출장 가야 한다"는 새 동기를 부여한다.
2. **세력 가입의 게이미파이드 영구 보상**: 가입 자체는 추상적이지만, 거점 첫 방문은 시각적 도장 + 영구 깃발(stronghold flag)로 "내가 이 세력에 발을 들였다"는 표지를 남긴다.
3. **위치 기반 게이트의 자연스러운 도입**: 트레잇 해금에 평판·행동 지표 외에 "거점 방문 깃발" AND 조건을 결합하여 단순 누적 게임에서 **"한 번이라도 거기에 가야 한다"** 라는 능동적 액션을 요구한다.

**M4 거점 설계 3원칙:**

1. **방치형 UX 보존**: 거점 도착은 자동 인지 (별도 입장 액션 없음). 도착 즉시 다이얼로그 큐 medium priority로 "거점 도착 팝업" 자동 발생.
2. **영구 도장 + 실시간 인터랙션 분리**: 첫 방문은 영구 깃발(트레잇 해금 자격) → 영속화. 상점·시설 효과 강화·트레잇 조건 재평가는 "현재 == 거점"일 때만.
3. **자유 이동 보존**: 충돌 세력 거점도 진입 자체는 자유. 단 "환영받지 못함" 서사 1회 + 모든 거점 효과(상점·트레잇 검증·시설 강화) 차단.

## 상세 설계

### 1. 거점 리전 지정 원칙

#### 1.1 데이터 모델

```
factions 테이블 신규 컬럼:
  stronghold_region_id   INTEGER NULLABLE   -- regions.id FK
                                             -- null 허용 (M4 도입 전 호환)
                                             -- M4 도입 후 14개 세력 모두 NOT NULL
```

리전 단위로만 지정한다. **섹터(0~9)까지 지정하지 않는다** — 자유도가 너무 떨어지고, 거점 진입이 "특정 섹터에 일시 위치"가 되어 자동 인지 UX가 부자연스러워진다.

- M3 변형 트리거 대상 18개 리전과 **겹치지 않게** 권장 (M3 변형 서사와 M4 거점 서사가 한 리전에서 겹치면 인지 부하 증가). 운영 입력 시 충돌 검증.

### 2. 거점 가시성 규칙

거점 리전 노출은 세력 발견 단계에 종속된다. 세력 도감의 정보 공개 단계와 동일한 규칙을 따른다.

| 세력 유형 | 가시성 단계 | 이동 화면 마커 | 도감 거점 정보 |
|---------|-----------|--------------|--------------|
| 공개 (6개) | 발견 즉시 = 항상 표시 | 게임 시작부터 마커 표시 | 도감에 거점 리전명 항상 표시 |
| 비밀 (4개) | clue level 1 = 미공개 / clue level 2 = 도감에 "거점: ???" / clue level 3 = 마커 + 리전명 공개 | clue3 도달 후 마커 표시 | clue3 도달 후 리전명 공개 |
| 지역 (4개) | 동일 (clue level 단계) | 동일 | 동일 |

**활동 로그 메시지:**

- clue3 도달 시점: 기존 "거점 발견: {세력명}의 전초기지 위치를 파악했다" 활용 (M1에서 이미 구현된 메시지). 단 M4부터는 이 시점에 **이동 화면 마커가 실제 점등**되도록 연동.
- 첫 방문 시점: 신규 메시지 "{세력명} 거점 도착 — 환영의 깃발을 받았다" (트레잇 해금 자격 영구 확보 알림).

### 3. 방문 메커니즘

#### 3.1 도착 판정

```
정의:
  isAtStronghold(factionId) =
    userData.region == factions[factionId].strongholdRegionId
    && !userData.isMoving
```

- 섹터(0~9) 무관. 거점 리전에 진입한 순간 도착으로 간주.
- 이동 중(`isMoving == true`)에는 도착 처리 보류. 이동 완료 시점에 평가.

#### 3.2 영속화 — 방문 깃발

```
FactionState 신규 필드:
  firstVisitedAt: DateTime?  // null = 미방문, value = 첫 방문 시각
  lastVisitedAt: DateTime?   // null = 미방문, value = 마지막 방문 시각 (재방문마다 갱신)
```

**저장 규칙:**

- **첫 방문 (firstVisitedAt == null + 도착 트리거)**:
  1. `firstVisitedAt = now()` 영구 기록
  2. `lastVisitedAt = now()`
  3. 활동 로그 "환영의 깃발" 기록
  4. 다이얼로그 큐 medium priority로 환영 팝업 (튜토리얼 1회)
  5. 트레잇 해금 자격 영구 확보 (페이즈 1 #4의 "거점 방문" 게이트 통과)
- **재방문 (firstVisitedAt != null + 도착 트리거)**:
  1. `lastVisitedAt = now()` 갱신만
  2. 다이얼로그 팝업 없음 (조용히 도착)
  3. 홈 화면에 "{세력명} 거점에 있음" 인디케이터 활성화
  4. 상점 UI 접근 가능 (페이즈 1 #3 영역)
  5. 트레잇 조건 재평가 트리거 (`TraitAcquisitionService.evaluate(mercId)` 호출)

#### 3.3 트레잇 해금 게이트의 거점 조건

페이즈 1 #4 (세력 전용 트레잇 컨셉)에서 정의될 AND 조건의 한 축:

```
세력 트레잇 획득 조건 (AND):
  1. 가입 세력 일치 (joined && factionId match)
  2. 평판 구간 ≥ 임계값 (예: 31 / 61 / 91)
  3. 거점 방문 깃발 보유 (firstVisitedAt != null)
  4. 행동 지표 임계값 (예: raid_count ≥ 50)
```

**중요한 설계 결정**:

- **3번 조건 = 영구 깃발 (firstVisitedAt) 한 번만 충족하면 영원히 통과**. "현재 == 거점" 실시간 조건이 아님.
- 단 **조건 재평가 트리거 = 거점 도착 시점 + 퀘스트 완료 시점 + 평판 변동 시점**. 행동 지표가 거점 밖에서 누적되어도 다음 거점 방문 시 일괄 검증되어 트레잇 자동 부여.
- 이 설계가 "한 번이라도 가야 함" 액션과 "방치형 누적" UX를 동시에 만족.

### 4. 거점 트리거 메커닉 정리

거점 도착 = 다음 메커닉의 게이트:

| 메커닉 | 게이트 조건 | 페이즈 |
|------|----------|--------|
| **트레잇 해금 자격 확보** | 첫 방문 1회 (firstVisitedAt != null) | 페이즈 1 #4 |
| **트레잇 조건 재평가 트리거** | 재방문 시점마다 발화 | 페이즈 4 #1 (TraitAcquisitionService 확장) |
| **세력 상점 UI 접근** | 현재 위치 == 거점 | 페이즈 1 #3 |
| **상점 구매 인터랙션** | 현재 위치 == 거점 + 평판 구간 충족 | 페이즈 1 #3 |
| **시설 효과 강화 (선택적)** | 본 기획서에서 도입 안 함 | — |

**시설 효과는 거점 방문과 무관**: 페이즈 1 #1(시설 14종)에서 명시한 대로 시설 효과는 "가입 + 비적대 상태"만 요구. 거점 방문은 별개. 이는 방치형 UX 보존 의도.

### 5. 충돌 세력 거점 진입 정책

가입 세력 A의 충돌 세력 B의 거점에 진입하는 경우:

#### 5.1 진입 자체는 자유

이동 자체를 차단하지 않는다. 방치형의 자유 이동 원칙 유지. 차단은 사용자가 우회 경로를 학습해야 하는 부담을 만들고, 위치 기반 컨텐츠를 망친다.

#### 5.2 첫 진입 = 환영받지 못함 서사 (1회)

```
첫 진입 (firstVisitedAt == null + 가입 충돌 세력 보유):
  1. firstVisitedAt 기록 안 함 (깃발 미발급)
  2. 다이얼로그 큐 low priority로 "환영받지 못함" 서사 1회 팝업
     "{거점주 세력명}의 거점에 도착했지만, {가입 세력명}의 깃발을 보고
      문이 닫혔다. 환영받지 못한 손님은 발길을 돌려야 한다."
  3. 활동 로그 "{거점주 세력명} 거점 진입 거부됨"
  4. 평판 추가 차감 없음 (가입 시 -100 페널티가 이미 충분히 패널티)
```

#### 5.3 재진입 = 조용한 차단

```
재진입 (firstVisitedAt 기록 안 됨 + 가입 충돌 세력 보유):
  1. 팝업 없음
  2. 활동 로그 기록 없음
  3. 거점 효과(상점·트레잇 검증·환영 인디케이터) 모두 차단
  4. 홈 화면 인디케이터에 "{거점주 세력명} 거점 — 진입 거부" 표시 (회색 처리)
```

#### 5.4 충돌 세력 탈퇴 후 재방문

탈퇴 후 더 이상 충돌이 아닐 때 재방문하면:

```
재방문 (탈퇴 완료 + 거점 도착):
  1. firstVisitedAt 기록 (정상 첫 방문 처리)
  2. 환영 팝업 정상 발화 (튜토리얼)
  3. 트레잇 해금 자격 확보
```

즉 **충돌 상태에서의 진입은 깃발이 안 찍히는 "헛걸음"**. 충돌이 해제된 후에야 정상 첫 방문으로 간주.

#### 5.5 가입자 본인 거점 진입 (정상 케이스)

```
정상 첫 진입 (가입 세력 == 거점주 세력):
  → 3.2 영속화 절차 그대로 적용
정상 재진입:
  → 3.2 재방문 절차 그대로 적용
```

#### 5.6 가입하지 않은 세력 거점 진입

가입하지 않은 세력의 거점은 **단순한 "관광지"**:

```
미가입 세력 거점 진입:
  1. 도감에 표시되어 있는 경우(공개 또는 clue3 달성):
     - 첫 진입 시 "{세력명} 거점에 도착했다 — 가입하지 않았으므로
       환영의 깃발을 받지 못한다" 서사 팝업 (low priority, 1회)
     - 활동 로그 기록
     - firstVisitedAt 기록 안 함 (가입 후 재방문 시 정상 첫 방문 처리)
  2. 도감에서 비공개 상태(clue level < 3 비밀/지역 세력):
     - 일반 리전과 구분 안 됨. 팝업 없음, 로그 없음
     - "내가 우연히 비밀 거점을 지나갔다"는 사실을 플레이어가 알 수 없음 (의도적)
```

#### 5.7 정책 요약 매트릭스

| 상태 | 첫 진입 결과 | 재진입 결과 | 깃발 발급 | 거점 효과 |
|------|------------|----------|---------|---------|
| 가입자 (정상) | 환영 팝업 + 깃발 발급 | 조용한 도착 + 인디케이터 | ✅ | ✅ |
| 충돌 세력 가입 중 | 환영받지 못함 팝업 | 조용한 차단 | ❌ | ❌ |
| 미가입 (도감 공개) | 미환영 팝업 1회 | 조용함 | ❌ | ❌ |
| 미가입 (clue<3) | 일반 리전과 동일 | 일반 리전과 동일 | ❌ | ❌ |
| 충돌 탈퇴 후 가입 | 환영 팝업 + 깃발 발급 | 조용한 도착 + 인디케이터 | ✅ | ✅ |

### 6. 방문 UX

#### 6.1 이동 화면 마커

**M3 변형 섹터 시각 패턴 차용 + 확장:**

| 표시 요소 | 패턴 |
|---------|------|
| 거점 리전 카드 우상단 | 세력 깃발 아이콘 (🏰) + 세력 컬러 작은 도트 |
| 거점 리전 카드 테두리 | 세력 컬러 1.5px 테두리 (변형 섹터 2px와 차별화) |
| 가입 세력 거점 | 깃발 아이콘 옆 ✅ 마이크로 배지 |
| 충돌 세력 거점 (가입자) | 깃발 아이콘 옆 ⛔ 마이크로 배지 + 카드 opacity 0.85 |
| 미공개 비밀/지역 거점 (clue<3) | 표시 안 함 (일반 리전과 동일) |

**다중 거점이 한 리전에 겹치는 경우** — 본 기획서 운영 입력 가이드에서 "한 리전당 거점 최대 1개" 권장. 운영 입력 시 검증 필요.

#### 6.2 거점 도착 팝업 (환영 팝업)

```
타입: medium priority 다이얼로그 (다이얼로그 큐 통합, dialogType = factionStrongholdWelcome)
표시 조건:
  - 가입 세력 거점 첫 진입 (firstVisitedAt 기록 직후)
  - barrierDismissible: true (방치형 부담 최소화)

내용:
  - 헤더: "{세력명}의 깃발 아래" (세력 컬러 헤더)
  - 일러스트: 세력 컬러 그라디언트 + 세력 아이콘
  - 본문 (TemplateEngine 렌더):
    "{merc.name}{이/가} 그대를 안내한다. {faction.name}의 본거지에 발을
     디딘 그대는 깃발을 받아 들었다. 이 깃발은 영원히 그대의 자격을
     증명할 것이다."
  - "환영의 깃발 획득" 강조 라인
  - 액션 버튼:
    [ 세력 상점 보기 ]   ← 페이즈 1 #3 연동, 첫 방문 튜토리얼 진입점
    [ 잠시 둘러본다 ]    ← 단순 닫기

세력별 환영 서사 변주 (TemplateEngine [pick] 활용):
  - 모험가 길드: 의뢰 게시판이 빼곡한 방으로 안내
  - 도둑 길드: 술집 뒷문을 통과해 비밀 회합실
  - 화산 심장단: 용암 흐르는 단련장 한가운데

```

### 7. 데이터 입력 가이드

본 기획서로부터 페이즈 3 데이터 생성이 별도로 필요하지 않다 (14행은 운영팀 직접 입력). 대신 운영 입력 가이드를 명세한다.

#### 7.1 `factions.stronghold_region_id` 입력

| 작업 | 비고 |
|------|------|
| 14개 세력 모두 region_id 지정 | 1.2 권장 후보 표 + 운영팀 199개 리전 검토로 확정 |
| M3 변형 트리거 18개 리전과 겹침 검증 | 겹치면 충돌 — 운영팀이 수동 회피 |
| 두 충돌 세력 거점 거리 6~8리전 이상 검증 | 거리 검증 쿼리 운영 도구에 추가 |
| 한 리전당 거점 최대 1개 검증 | UNIQUE 제약 권장 |

#### 7.2 환영 서사 텍스트 입력

페이즈 4 데이터 입력 단계에서:

- `quest_narratives`와 별도로 신규 테이블 `faction_stronghold_narratives` 또는 `factions` 테이블에 컬럼 추가:
  - `welcome_narrative_template` TEXT — 첫 방문 환영 서사 (TemplateEngine 변수 사용)
  - `rejected_narrative_template` TEXT — 환영받지 못함 서사 (충돌 세력 거점)
- 14개 세력 × 환영 서사 14개 + 환영받지 못함 서사 14개 = **28개 서사 텍스트** 입력 필요
- 서사 톤: 페이즈 1 #1 시설 설명과 일관 (한국어 1~2문장, 세력 정체성 반영)

**M3 변형 섹터/체인 퀘스트와의 호환:**

- 변형 섹터(M3)와 거점(M4)이 한 리전에서 겹칠 가능성 → 운영 입력 시 회피 권장. 만약 겹치면 우선순위 = 변형 섹터 시각 (2px 테두리) > 거점 마커 (1.5px 테두리).
- 체인 퀘스트 대상 섹터 금색 테두리(M3)와 거점 마커는 다른 리전에 표시되므로 충돌 가능성 낮음. 같은 리전에 겹치면 둘 다 표시(레이어 누적).

**M3 다이얼로그 큐와의 호환:**

- 환영 팝업은 medium priority — 건설 완료/조사 완료/이동 자동 이벤트와 동일 우선순위.
- 환영받지 못함 + 미가입 팝업은 low priority — 다른 컨텐츠 팝업이 누적된 상태에서 후순위로 처리 (방치형 UX 보존).

## 결정 사항

본 기획서에서 굳힌 핵심 정책 (페이즈 4 #3 명세에서 그대로 적용):

1. **거점 단위 = 리전**. 섹터 지정 안 함.
2. **트레잇 해금 게이트의 "거점 방문" = 영구 도장형** (firstVisitedAt 기록). "현재 == 거점" 실시간 조건이 아님.
3. **재방문 시점 = 트레잇 조건 재평가 트리거**. 행동 지표가 거점 밖에서 누적되어도 재방문 시 일괄 검증되어 트레잇 자동 부여.
4. **상점 이용 = 현재 위치 == 거점**. 영구 깃발과 별개로 실시간 위치 게이트.
5. **시설 효과는 거점 방문과 무관**. "가입 + 비적대 상태"만 요구 (페이즈 1 #1 결정 유지).
6. **충돌 세력 거점 진입 = 효과만 차단**. 이동 자체 자유. 깃발 미발급. 첫 진입 시 환영받지 못함 팝업 1회.
7. **충돌 탈퇴 후 거점 재방문 = 정상 첫 방문 처리** (firstVisitedAt 기록).
8. **미가입 세력 거점 = 미환영 팝업 1회** (도감 공개 상태에서만). 깃발 미발급.
9. **clue level < 3 비밀/지역 세력 거점 = 일반 리전과 시각 동일**. 의도적으로 미공개.
10. **방문 깃발은 영구**. 탈퇴해도 firstVisitedAt 기록 유지. 재가입 시 즉시 트레잇 해금 자격 보유.

## 미해결 사항 (페이즈 2/4에서 결정)

1. **세력별 거점 region_id 정확 매핑**: 1.2 권장 후보 + 운영팀 199개 리전 검토로 확정. 본 기획서에서는 매칭 가이드만.
2. **이동 화면 마커 표시 방식 세부**: 변형 섹터(2px 테두리) / 체인 대상 섹터(금색 2px) / 거점(1.5px)의 시각 우선순위 — 한 리전에서 겹칠 때 어느 것을 우선 표시할지 페이즈 4 #3 명세에서 확정.
3. **다중 가입 시 거점 환영 팝업 누적**: 가입 직후 3개 세력 거점을 연달아 방문하는 시나리오에서 환영 팝업 3개가 큐에 쌓임. 이대로 직렬 처리하는지, 첫 1개만 표시하는지 페이즈 4 결정.
4. **거점 리전 변경 가능성**: 향후 컨텐츠 추가로 거점 리전이 운영자에 의해 변경되는 경우, 기존 firstVisitedAt 깃발은 그대로 유지할지(권장) 무효화할지. 페이즈 4 #3 명세에서 정책 명문화.
5. **상점 진입 UX의 세부 (페이즈 1 #3 영역)**: 거점 진입 시 환영 팝업 → 상점 자동 진입인가, 명시적 버튼 누름 후 진입인가. 페이즈 1 #3에서 결정.
6. **트레잇 자동 부여 시점의 UI (페이즈 1 #4 영역)**: 거점 도착 시 조건 충족된 트레잇이 즉시 자동 획득 → 환영 팝업 직후 별도 트레잇 획득 팝업이 추가 표시. 다이얼로그 누적 정합성 페이즈 4 #1 + #4에서 확정.
