> 선행 의존: 페이즈 1-2 연계 퀘스트(`Docs/content-design/[content]20260423_chain_quests.md`), 페이즈 1-3 지역 변형(`Docs/Archive/20260424_region-transform-system/design.md`), 페이즈 1-4 퀘스트 서사(`Docs/Archive/20260424_quest-narrative-integration/design.md`), 페이즈 1-5 이동 선택지(`Docs/Archive/20260425_travel-choice-system/design.md`), M1 세력 전용 퀘스트 UI, M2b 엘리트 퀘스트 UI
> 후속 페이즈 의존: 페이즈 4-6(SyncService + 파견 화면 공존 정렬 spec)

## 개요

M3가 마무리되는 시점에 파견 화면·도착 팝업·홈 탭에서 **5개 레이어의 컨텐츠**가 동시 존재하게 된다: 체인 퀘스트 / 가입 세력 전용 / 엘리트 / 변형 섹터 / 일반. 거기에 이동 선택지 회상·건설 완료·랭크업·조사 완료·변형 발동까지 합쳐지면 한 번의 도착에 **최대 6~8개 팝업**이 줄 설 수 있다.

본 문서는 레이어 간 **우선순위·시각 규칙·팝업 순서·인터럽트 큐**를 확정한다. 개별 레이어 기획은 이미 페이즈 1-2~1-5에서 각자 완료됐고, 이 문서는 **그들이 만날 때의 규칙**만 다룬다. 페이즈 1-5 Q-8(도착 팝업 순서)과 페이즈 1-3 §5-4(체인·변형 동시 도달)를 여기서 최종 해결한다.

## 레퍼런스 분석

| 게임 | 참고 포인트 | 차용/변형 |
|------|-----------|----------|
| **Melvor Idle** | 전역 모달 큐: 레벨업·업적·발견이 줄 서서 순차 노출. "Close All" 버튼 | 다중 팝업 큐 개념 차용. M3 MVP는 "Close All" 생략, 순차만 |
| **Kingdom Rush — 타워 목록 정렬** | 카드 목록에 계층별 색상·배지. 정렬 우선순위 명시적 | 파견 화면 카드 시각 규칙 모델 |
| **Stardew Valley — 아침 보고** | 하룻밤 사이 누적된 이벤트를 **아침에 단일 화면으로 요약** | "오늘의 기록" 아이디어의 원형. MVP는 보류하고 M4+ 검토 |
| **Darkest Dungeon — 타운 알림** | 탑 아이콘 배지로 미확인 이벤트 수 표시 → 클릭 시 상세 | 홈 탭 진입 시 알림 배지 모델 |

**핵심 설계 원칙**: "**정렬은 한 기준, 강조는 한 강도, 배지는 모두 보여준다.**" 사이드바 색상은 가장 무거운 계층 1개만, 배지는 가진 모든 속성을 동시 표시. 시각 혼란과 정보 손실의 균형.

## 상세 설계

### 1. 파견 화면 정렬 — 5계층 + 상단 고정 섹션

#### 1-1. 5계층 우선순위

```
[상단 고정 섹션]
  Tier 0 — 체인 퀘스트 다음 단계 (chainQuestProgress 기반)
           현재 리전 수행 가능 카드 + 타 리전 필요 안내 카드(비활성)

[일반 목록 — 아래 순서로 정렬]
  Tier 1 — 가입 세력 전용 퀘스트 (is_faction_exclusive=true + joined_factions 교집합)
  Tier 2 — 엘리트 퀘스트 (is_elite=true, 유니크 먼저 → 보통)
  Tier 3 — 변형 섹터 전용 퀘스트 (sector_type IS NOT NULL AND 현재 섹터 일치)
  Tier 4 — 일반 퀘스트 (그 외)
```

**같은 Tier 내 정렬**: 보상 기대값 내림차순 → 난이도 오름차순 → id 사전순 (결정론적).

#### 1-2. Tier 0 상단 고정 섹션 규칙

- 최대 표시 **3장 카드** (진행 중 체인이 3개 초과면 가장 최근 진행 3개만. Q-1에서 미해결 시 기본 3)
- 현재 리전에서 수행 가능한 카드는 **활성**
- 현재 리전에서 수행 불가(다른 리전 필요)는 **비활성 카드 + "이동 필요" 안내**
- 0장이면 섹션 자체 생략 (빈 박스 노출 안 함)
- 상단 고정 섹션과 일반 목록 사이에 **구분선** 1px + 회색 라벨 "진행 중인 체인"

**근거**:
- 체인 퀘스트는 서사적 연속성이 핵심 → 유저가 **언제나 즉시** 다음 단계를 인지해야 함
- 일반 목록에 묻히면 체인 완주율 급락 우려
- 3장 상한은 스크롤 없이 표시 가능한 최대치 (mobile viewport 430px 기준)

#### 1-3. 일반 목록 섹션 내부 정렬 (Tier 1~4)

**단일 스크롤 목록**. 섹션 구분선 없이 정렬만 우선순위 반영.

- 유저가 특정 Tier만 보고 싶은 경우(예: "세력 전용만")는 **M3 범위 외 필터 UI**. 운영 데이터 축적 후 M4+ 검토
- 대신 각 카드의 **상단 배지**로 Tier를 즉시 인식 가능하도록 설계 (§2)

#### 1-4. 섹터 이동 시 정렬 재계산

유저가 섹터를 이동하면 정렬이 즉시 갱신된다:
- Tier 3(변형 섹터 전용)은 현재 섹터가 변형된 섹터일 때만 노출
- Tier 0 체인 카드의 "활성/비활성" 플래그도 현재 리전 기반으로 재평가
- 리전 이동은 기존 로직(`movement_provider`)에 의해 이동 완료 후 갱신, 섹터 이동은 즉시 반영

### 2. 카드 시각 규칙 — 사이드바 1개 + 배지 복수

#### 2-1. 좌측 사이드바 (3px 세로 막대) — 가장 상위 계층 1개만

**우선순위 (위에서 강함)**:

| 계층 | 색상 | 근거 |
|------|------|------|
| 체인 다음 단계 | **#d4af37 (금색)** | M3 간판 서사, 최강 주목성 |
| 엘리트 유니크 | #7b1fa2 (진보라) | M2b 기존 |
| 엘리트 보통 | #e65100 (주황) | M2b 기존 |
| 변형 섹터 — hidden | #d4af37 (금색)? → **#b8860b (어두운 금)**으로 변경 | 체인과 충돌 방지 |
| 변형 섹터 — ruins | #6a1b9a (보라 톤) | 고대·신비 |
| 변형 섹터 — village | #2e7d32 (녹색) | 안정·커뮤니티 |
| 세력 전용 | **세력 고유 `FactionData.color`** | M1 기존 |
| 일반 | 없음 (사이드바 생략) | 중립 |

**색상 충돌 해결**:
- 체인(금색 #d4af37)과 변형 섹터 hidden(원 제안 금색)이 충돌 → **hidden을 #b8860b(어두운 금)로 재지정**
- 유사 보라(엘리트 유니크 #7b1fa2 vs 변형 ruins #6a1b9a)는 채도 차이로 분간 가능. 유저 피드백 시 재조정

**규칙**: 한 카드의 사이드바는 **위 순위에서 가장 먼저 해당되는 1개 색**만 적용.

#### 2-2. 상단 배지 — 복수 표시

배지는 **왼쪽에서 오른쪽**으로 이 순서:

1. **체인 단계 배지** — `[체인명 · N/M]` 형태 (예: `잃어버린 서고 · 2/3`)
2. **엘리트 배지** — 🔥(보통) / ★(유니크)
3. **변형 섹터 배지** — 🏘️(village) / 🏛️(ruins) / ✦(hidden)
4. **세력 원형 배지** — 세력 컬러 원 + 세력명 (M1 기존)

**공간 제약**: 한 줄 표시. 4개 모두 켜지면 배지 간 간격 2px로 좁힘. 세력명은 "정예 기수단" 같은 긴 이름은 **3글자 축약** ("정예기") 규칙 추가 필요(Q-2).

#### 2-3. 이름 색상

| 계층 | 이름 색 |
|------|--------|
| 체인 다음 단계 | `primary` (Material 3 theme primary) |
| 엘리트 유니크 | `#c084fc` (밝은 보라) |
| 엘리트 보통 | `#fb923c` (밝은 주황) |
| 나머지 (변형·세력·일반) | **기본 `onSurface` 유지** |

**규칙**: 이름 색은 **체인·엘리트만** 강조. 변형·세력 전용은 사이드바+배지로 충분하므로 이름 색은 기본 유지. 과다 강조 방지.

#### 2-4. 테두리 강조

현 M1 규칙 유지 (확장 없음):
- **체인 다음 단계 카드**: 사이드바에 추가로 카드 전체에 2px `primary` 테두리
- **세력 전용 퀘스트**: 사이드바에 추가로 카드 전체에 1.5px 세력 컬러 테두리
- 그 외: 테두리 없음

**이유**: 테두리까지 모든 계층에 적용하면 시각 과포화. 가장 강한 감정적 주목이 필요한 체인·세력 전용만.

### 3. 중첩 케이스 매트릭스

같은 퀘스트가 복수 Tier에 해당하는 경우의 시각 결정:

| 케이스 | 사이드바 | 배지 | 이름 색 | 테두리 |
|--------|---------|------|--------|--------|
| 체인 × 일반 | 금색 | [체인 N/M] | primary | 2px primary |
| 체인 × 세력 전용 | 금색 | [체인 N/M] + [세력] | primary | 2px primary + 1.5px 세력 컬러 (double border) |
| 체인 × 엘리트 보통 | 금색 | [체인 N/M] + 🔥 | primary | 2px primary |
| 체인 × 엘리트 유니크 | 금색 | [체인 N/M] + ★ | primary | 2px primary |
| 세력 전용 × 엘리트 보통 | 세력 컬러 | [세력] + 🔥 | 엘리트 보통 색(#fb923c) | 1.5px 세력 컬러 |
| 세력 전용 × 엘리트 유니크 | 세력 컬러 | [세력] + ★ | 엘리트 유니크 색(#c084fc) | 1.5px 세력 컬러 |
| 변형 섹터 × 세력 전용 | 세력 컬러 (세력 우선) | [세력] + 🏘️/🏛️/✦ | 기본 | 1.5px 세력 컬러 |
| 변형 섹터 × 엘리트 | 엘리트 색 | 🔥/★ + 🏘️/🏛️/✦ | 엘리트 색 | 없음 |
| 변형 섹터 × 일반 | 변형 유형 색 | 🏘️/🏛️/✦ | 기본 | 없음 |

**핵심 결정**:
- **사이드바는 정렬 우선순위(§1-1)의 "상위"를 따른다.** 세력 전용이 변형 섹터보다 상위이므로 중첩 시 사이드바는 세력 컬러 우선
- **이름 색은 체인·엘리트만 적용**(§2-3). 세력/변형은 이름 색 건드리지 않음
- **테두리는 체인·세력 전용만 적용**. 중첩 시 `double border` 허용 (체인 금색 + 세력 컬러)

#### 3-1. 체인 단계가 엘리트/세력/변형과 결합하는 경우 (Q-3 대비)

**현 체인 기획(페이즈 1-2)에는 체인 단계가 엘리트·세력 전용·변형 섹터에 배정되지 않음**. 단 M4+ 확장에서 가능:
- 체인 X의 3단계가 세력 Y의 거점에서 진행 → 체인 + 세력 전용
- 체인 Z의 최종 단계가 엘리트 몬스터 처치 → 체인 + 엘리트 유니크

본 문서는 **확장 시 사용할 규칙을 매트릭스에 선반영**. 실제 적용은 M4+에서.

### 4. 상단 고정 섹션 세부

#### 4-1. 체인 카드 컴포넌트

```
┌──────────────────────────────────────────┐
│ ▌ [잃어버린 서고 · 2/3]                   │
│ ▌                                          │
│ ▌ 고서관 탐사                              │
│ ▌ "낡은 지도의 다음 표식을 따라 깊은 서고로" │
│ ▌                                          │

  (금색 2px 테두리, 금색 사이드바)
```

```
│ ▌ [상인의 비망록 · 3/4]                   │
│ ▌                                          │
│ ▌ 눈보라 고개 탐사                         │

│ ▌ [이동 화면으로]                          │
└──────────────────────────────────────────┘
  (반투명 + 금색 사이드바 + 회색 테두리)
```

#### 4-2. 체인 0장 시

상단 고정 섹션 **생략**. 일반 목록이 화면 상단에 붙는다.

#### 4-3. 타 리전 체인이 3개 이상인 경우

현재 리전 수행 가능 카드 우선 → 타 리전 필요 카드(비활성) 나머지. 총 3장 상한. 4개 이상은 표시하지 않음(활동 로그에서 확인).

### 5. 도착 팝업 순서 (페이즈 1-5 Q-8 해결)

#### 5-1. 공식 순서

이동 완료 시점에 누적된 팝업을 **순차 노출**한다:

```
1. 파견 완료 퀘스트 결과 (QuestResultDialog)
   └ 트레잇 획득 → 트레잇 진화 선택 (기존 체이닝)
2. 자동 이벤트 결과 (AutoEventDialog — 기존 te_* 요약)
3. 이동 선택지 회상 (TravelChoiceRecallDialog — 2단: 상황→선택→결과)
4. 건설 완료 (ConstructionCompleteDialog)
5. 조사 완료 (InvestigationResultDialog) — 이동 중 조사 불가이므로 실제 충돌 드묾
6. 세력 랭크업 오버레이 (RankUpOverlay)
7. 체인 진행 알림 (ChainProgressDialog — 다음 단계 해금 or 체인 완주)
8. 변형 발동 팝업 (RegionTransformDialog — 페이즈 1-3 §7-1)
```

#### 5-2. 순서의 근거

- **1 먼저**: 파견 결과는 용병 생사·보상이 걸린 가장 무거운 정보. 최우선
- **2~3**: 이동 중 실제 일어난 일 (자동 이벤트는 과거형, 선택지 회상은 결정 요청). 시간 순서
- **4~5**: 배경에서 경과 시간에 발생한 일. 중요도 중간
- **6**: 랭크업은 오버레이(전체화면 축하). 다른 팝업들 중 하나라도 "평판 +" 효과를 주면 랭크업 판정이 먼저 처리되어도 유저 체감상 **결과 확인 후** 축하가 맞음 → 6번 위치
- **7~8**: 파생 이벤트. 앞 팝업들의 결과로 촉발된 시스템 레이어 알림

**충돌 해소 원칙**: 팝업 A의 결과가 팝업 B를 트리거하는 경우, A → B 순서 유지. 예: 퀘스트 완료(1)의 평판 +가 랭크업(6) 트리거 → 1 먼저, 6 나중. 이벤트 처리 순서는 자연스럽게 정합.

#### 5-3. 다중 팝업 처리 (MVP)

- **3개 이하**: 순차 팝업. 확인 클릭 → 다음 팝업
- **4개 이상**: **M3 MVP는 순차 팝업 유지**. "오늘의 기록" 요약 화면은 M4+ 검토 (§6)
- 순차 팝업 중 유저가 **앱을 닫고 다시 열면**: 큐가 초기화되지 않고 `dialogQueueProvider`에 보존 (§7). 복귀 시 남은 팝업부터 재개

#### 5-4. 선택지 회상 UI 타이밍

페이즈 1-5 §10에서 2단 팝업(상황→결과) 정의. 큐에서 선택지 이벤트 차례가 되면:

```
a. 상황 팝업 표시 (선택지 대기)
b. 유저 선택 클릭
c. 결과 팝업 표시 (효과 반영)
d. 확인 → 큐 다음 단계
```

중간에 앱 종료 시: **상황 팝업 단계**에서 종료되면 선택지 미결. 복귀 시 재표시. 선택 후 결과 전 종료는 드묾(거의 즉시). 결과도 ActivityLog 기록 후 큐 다음으로 이동.

#### 5-5. 트레잇 팝업 체이닝 위치

기존 M1/M2b 규칙 유지:
- **QuestResultDialog** (1번) 내부 체이닝 → 트레잇 획득 → 트레잇 진화 선택
- 팝업 1이 닫힌 후에야 팝업 2로 이동

변경 없음. 단 체인 퀘스트 완료로 얻은 트레잇도 동일 체이닝 적용.

### 6. "오늘의 기록" 요약 화면 — M3 MVP 미포함, M4+ 검토

#### 6-1. 결정

**M3 MVP는 구현하지 않는다.**

#### 6-2. 근거

- 유저 1회 도착에 팝업 4개 이상 누적되는 빈도는 **앱 장시간 미접속 복귀 시에만** 현실적
- 해당 케이스는 `ActivityLog`로 이미 커버 (최대 100개 엔트리, 시간순)
- 요약 화면 UI는 신규 컴포넌트 + 섹션별 확장/축소 로직 필요 → 개발 비용 중간
- 운영 데이터(도착 시점 팝업 누적 통계)가 없는 상태에서 선제 구현은 과잉

#### 6-3. M4+ 검토 조건

다음 중 하나 충족 시 재검토:
- 유저 세션 중 4개 이상 팝업 누적 빈도가 **세션당 평균 0.5회 이상**
- 유저 피드백에 "팝업이 너무 많다" 수렴 신호

측정은 M3 운영 데이터 분석 기반. 본 문서 범위 외.

### 7. Global Dialog Queue — 인터럽트 처리

#### 7-1. 큐 구조

```dart
enum DialogPriority { critical, high, medium, low }

class DialogRequest {
  final String id;                    // 중복 방지용
  final DialogPriority priority;
  final Widget Function(BuildContext) builder;
  final dynamic payload;              // 팝업 내부에서 쓰는 데이터
}

final dialogQueueProvider =
  StateNotifierProvider<DialogQueueNotifier, List<DialogRequest>>(...);
```

#### 7-2. 우선순위 분류

| Priority | 포함 팝업 | 동작 |
|---------|---------|------|
| **critical** | 용병 사망 통지, 랭크업 오버레이 | 큐 최우선, 다른 팝업 대기 |
| **high** | 퀘스트 결과, 변형 발동, 체인 완주 | 도착 팝업 순서(§5) 내 우선 |
| **medium** | 자동 이벤트 결과, 이동 선택지, 건설 완료, 조사 완료, 체인 진행 | 순차 |
| **low** | (향후 확장) 시설 이정표 해금, 소규모 알림 | 활동 로그에만 기록, 팝업 생략 가능 |

#### 7-3. 큐 처리 규칙

1. 새 `DialogRequest`가 들어오면 **priority 내림차순** 삽입
2. 동일 priority 내 **FIFO**
3. **중복 id** 무시 (같은 이벤트 2번 트리거 방지)
4. 큐에 항목이 있는 동안 **`currentDialog`** 표시. 확인 시 pop → 다음 항목
5. 앱 종료 시 큐 상태는 Hive `dialogQueue` 박스(신규)에 저장 → 복귀 시 복원

#### 7-4. 기존 `RankUpOverlay` 통합

- 현재 `reputationRankUpProvider`가 독립 채널. app.dart가 `ref.listen`으로 전역 감지
- 본 기획 이후: **큐 통합**. `reputationRankUpProvider` 발동 시 큐에 `critical` priority로 enqueue
- 단 사용자가 파견 화면/이동 화면에 있을 때 오버레이가 뜨면 **현재 팝업 처리 완료 후** 표시 (기존 즉시 표시 변경)

**호환성 판단**: 즉시 표시 → 큐 경유로 변경 시 체감 지연은 거의 없음(현재 팝업 하나 닫는 시간). 그러나 퀘스트 결과가 진행 중일 때 랭크업이 끼어들지 않는 UX가 더 자연스러움 → 변경 수락.

#### 7-5. 큐와 활동 로그 관계

- **팝업으로 표시된 이벤트**: ActivityLog에도 기록 (이중 기록, 참조 가능)
- **low priority 팝업 생략된 이벤트**: ActivityLog에만 기록
- ActivityLog의 기존 100개 상한 유지

### 8. 홈/정보 탭 인터럽트 처리

#### 8-1. 홈 탭 진입 시

- 앱 실행·포그라운드 복귀 시 큐에 팝업이 있으면 **홈 진입 → 큐 처리 시작**
- 이동·파견·시설 화면으로 **바로 진입 시도**해도 큐가 먼저 처리됨 (모달 특성)
- 큐 처리 중 유저가 Android back 누르면: critical은 무시 (강제 확인), 나머지는 **활동 로그로 이동** 후 dismiss

#### 8-2. 홈 우측 상단 알림 배지 (M3 MVP 범위)

현재 구현: 없음. M3 범위에서는 **팝업 순차 처리로 충분하므로 배지 신설 안 함**.

M4+ 검토: 다크스트 던전 스타일 탑 배지(미확인 이벤트 수). 본 문서 범위 외.

#### 8-3. 정보 탭 인터럽트

정보 탭(`InfoScreen` → `FactionCodexScreen`)은 진입 시점에 `factionCodexScrollTargetProvider`·`factionRefreshProvider`로 자동 스크롤/갱신. 본 기획에 영향 없음.

단 정보 탭에 있는 중 변형 발동 팝업이 트리거되면 큐가 즉시 모달 표시 → 유저 경험상 자연(**현재 보는 화면과 무관하게 중요 알림은 즉시**).

### 9. 파견 화면 외 다른 화면의 컨텐츠 레이어

M3 후에도 **파견 화면이 공존 정책의 주 대상**. 다른 화면은 영향 미미하거나 이미 정의:

| 화면 | 레이어 영향 |
|------|----------|
| 이동 화면(MovementScreen) | 변형 섹터는 섹터 아이콘·색상(페이즈 1-3 §7-3) 그대로. 체인 다음 단계 리전은 **이동 UI 섹터에 금색 하이라이트**(M3 MVP 범위 — 간단) |
| 모집 화면 | 영향 없음 |
| 시설 화면 | 영향 없음 |
| 정보 탭 | 세력 도감은 기존 M1 규칙 유지. 변형 발동 이력을 세력 도감에 노출하지 않음(별개 레이어) |

#### 9-1. 이동 화면 섹터 시각 — 체인 하이라이트 추가

페이즈 1-3 §7-3이 변형 섹터 아이콘을 정의. 본 기획에서 추가:

- **체인 다음 단계 리전/섹터**: 금색 테두리 + "체인" 배지 작게
- 변형 섹터 × 체인 리전 중첩: 체인 강조 우선 (금색 테두리) + 변형 섹터 아이콘 유지

### 10. 활동 로그 확장

M3에서 새 로그 타입 추가:

| ActivityLogType | HiveField | 용도 |
|----------------|----------|------|
| `regionTransform` (페이즈 1-3에서 이미 예약) | 15 | 변형 발동 |
| `chainProgressed` | 16 | 체인 단계 완료 |
| `chainCompleted` | 17 | 체인 완주 |
| `travelChoiceCompleted` (페이즈 1-5에서 이미 예약) | 18 | 이동 선택지 이벤트 완료 |

**로그 메시지 예시**:
- `regionTransform`: "{region.name}의 섹터가 개척 마을로 변형되었다"
- `chainProgressed`: "[잃어버린 서고] 2/3 단계 완료"
- `chainCompleted`: "[잃어버린 서고] 체인 완주, 확정 보상 획득"
- `travelChoiceCompleted`: "길에서 부상자를 도왔다 — 명성 +10"

**표시 규칙**: 기존 역순 정렬 유지. 신규 타입 전용 아이콘 4종 추가 (페이즈 4-6 spec).

- **중간**: `RankUpOverlay`가 즉시 표시 → 큐 경유로 변경. 기존 체감 UX에서 "0.5초 지연" 수준 차이. 유저 체감 거의 없음
- **중간**: 파견 화면 정렬 5계층 + 상단 고정 섹션 추가는 기존 UI 리팩터링 요구. 페이즈 4-6에서 spec

- 큐 우선순위 정렬 유닛 테스트
- 큐 persistence round-trip 테스트 (저장→복원)
- 도착 팝업 순차 표시 통합 테스트 (4개 팝업 시뮬레이션)

## MVP vs 확장 가능 지점

| 범주 | MVP (페이즈 4-6에서 구현) | 확장 가능 (M4+) |
|------|--------------------------|----------------|
| 파견 화면 정렬 | 5계층 + 상단 고정 | 유저 필터 UI (세력만 / 엘리트만 등) |
| 카드 시각 | 사이드바 1개 + 배지 복수 + 테두리 제한 | 카드 아래 소형 트리뷰(시너지 role chip 확장) |
| 배지 | 4종 아이콘 + 세력 원형 | 커스텀 배지 운영 (시즌 이벤트 등) |
| 팝업 순서 | 고정 8단계 순차 | "오늘의 기록" 요약 화면 |
| 큐 | 4 priority + Hive persistence | 큐 상태 디버그 도구(개발용) |
| 홈 알림 배지 | 없음 | 상단 카운터 배지 |
| 이동 화면 체인 하이라이트 | 금색 테두리 + 배지 | 체인 루트 라인 오버레이 (지도 위 경로 시각화) |
| 활동 로그 | 4종 신규 타입 | 타입별 필터 탭 |

## 구현 우선순위 제안

**우선순위: 높음 (M3 통합 릴리스의 핵심)**

사유:
- M3의 4개 컨텐츠 레이어(체인·변형·서사·선택지)를 **유저가 동시에 마주치는** 유일한 기획 문서
- 개별 레이어가 아무리 잘 설계돼도 공존 규칙 없이는 UI 혼란 발생
- 페이즈 4의 다른 spec들(4-2/4-3/4-4/4-5)이 각각 팝업 큐에 enqueue하는 계약 필요 → 본 문서가 그 **통합 계약**

**M3 내 착수 순서 권장**:
1. 본 기획서 확정 (완료)
2. 페이즈 2 밸런스 검토(1~3) — 공존 정책은 수치가 없으므로 무관
3. 페이즈 3 데이터 생성 — 공존 정책은 데이터가 없으므로 무관
4. 페이즈 4-6 spec 작성 — 다른 spec들과 연계하여 "큐 경유 알림" 계약을 모든 신규 다이얼로그에 주입

**본 기획은 페이즈 4-2~4-5의 spec과 병렬 진행 가능**. 단 4-6 spec에서 큐 API 확정 후 다른 spec들이 enqueue 방식을 반영.

## 오픈 질문

- **Q-1 (체인 상단 고정 섹션 최대 표시 카드 수)**: 현재 3장 제안. 체인 동시 진행 상한은 페이즈 1-2에서 정하지 않음(체인 포기 불가 + 7일 휴면만 명시). 다중 체인이 5개 쌓이면 3장 제한으로 일부 가려짐. → **권장 유지 3장**. 5개 쌓이는 케이스는 희귀하고, 나머지는 활동 로그 확인. M4+에서 "체인 목록" 별도 화면 추가 검토

- **Q-2 (긴 세력명 축약 규칙)**: "은빛 용병단의 잔재" 같은 긴 세력명이 카드 배지에 들어가면 overflow. → **권장**: 3글자 축약 규칙 도입 ("은빛잔"). 축약 규칙은 `FactionData.short_name` 컬럼 추가 방식 또는 클라이언트 측 자동 축약. 페이즈 4-6 spec에서 확정

- **Q-3 (M4+ 체인 × 엘리트/세력/변형 중첩 대비)**: §3-1에서 매트릭스 선반영. 실제 발생 시점에 세부 UI 조정 필요성 있을 수 있음. → **M3 범위 외**

- **Q-4 (큐 persistence 만료 정책)**: 앱 종료 후 3일 뒤 복귀 시 누적된 팝업 복원 → 유저 혼란. → **권장**: 큐 항목 만료 24시간. 24h 지난 항목은 복원 시 **자동 dismiss + ActivityLog만 유지**. 페이즈 4-6에서 기본값으로 도입

- **Q-5 (조사 완료와 이동 완료 동시성)**: 현재 이동 중 조사 불가(상호 배제). 따라서 이동 완료 도착 시점에 조사 완료 큐는 비어 있음. 혹 버그로 동시성 발생하면 §5-1 순서 유지. → **현 구조상 드묾**, 테스트 대상

- **Q-6 (엘리트 보통/유니크와 변형 hidden 색 유사성)**: 엘리트 유니크 #7b1fa2와 변형 ruins #6a1b9a가 진보라 계열로 유사. → **권장**: 변형 ruins를 #5e35b1(중보라)로 재조정 또는 유지. 페이즈 4-6 시각 프로토타입에서 최종 확인

- **Q-7 (상단 고정 섹션 토글)**: 유저가 체인에 흥미 없을 때 상단 섹션을 **접을 수 있게** 할지. → **권장 MVP 제외**. 체인은 M3 간판이므로 강제 노출 유지. 피로 시그널 나오면 M4+ 토글 추가

- **Q-8 (파견 필터 UI)**: "세력 전용만 보기", "엘리트만 보기" 같은 필터 탭. → **M3 범위 외**. 정렬 5계층으로 충분

- **Q-9 (변형 섹터 × 엘리트 중첩의 배지 공간)**: 한 카드에 [체인] + [🔥] + [🏘️] + [세력] 4배지가 모두 켜지면 overflow. → **권장**: 세력명 3글자 축약 + 배지 아이콘만(텍스트 제거) 방식. Q-2와 연계

- **Q-10 (dialogQueue Hive 박스 persistence 복원 중 예외)**: 복원 중 데이터 손상/deserialization 실패 시 큐 초기화 + 로그. fail-safe 기본값 필요. → **권장**: 예외 발생 시 큐 비움 + `ActivityLog`에 "알림 일부 유실됨" 기록. 페이즈 4-6에서 spec
