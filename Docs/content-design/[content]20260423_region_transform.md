# 지역 변형 컨텐츠 기획서

> 작성일: 2026-04-23
> 유형: 신규 컨텐츠 (M3 페이즈 1-3)
> 후속 페이즈 의존: 페이즈 2-2(유적지·마을 보상/난이도 밸런스), 페이즈 3-0-2(`types/region-transform.md` 타입 스펙), 페이즈 3-2(`region_discoveries.transform` 트리거 18행), 페이즈 3-3(섹터 변형 전용 `quest_pools` 34행), 페이즈 4-3(RegionState 확장 + QuestGenerator sector_type 분기 spec)

## 개요

지역 조사를 **최고 임계**까지 밀어붙이면 리전의 한 섹터가 **영구적으로 변형**된다. 폐광이 재건되어 마을이 되고, 안개 낀 숲에서 고대 유적이 드러나며, 외딴 협곡에 숨겨진 지하도가 열린다. 변형된 섹터에서는 **전용 퀘스트 풀**이 생성되어 새로운 플레이 사이클이 열린다. M3의 표어 "**지역을 끝까지 파면 세계가 변한다**"의 구현체다.

이 문서는 **3유형(village/ruins/hidden)의 포지셔닝**, **18개 변형 대상 리전 선정**, **트리거 임계값**, **전용 퀘스트 풀 컨셉**, **연계 퀘스트(페이즈 1-2)와의 상호작용**, **영속성·UX 규칙**을 확정한다. 구체 퀘스트 데이터는 페이즈 3-3에서 data-generator가 생성, 수치 밸런스는 페이즈 2-2에서 balance-designer가 검토한다.

## 레퍼런스 분석

| 게임 | 참고 포인트 | 차용/변형 |
|------|-----------|----------|
| **Melvor Idle — Township** | 플레이 누적으로 마을 건물이 단계적으로 성장. "세계가 성장한다"는 감성 | 마을 변형 후 퀘스트 종류가 바뀌는 방식 참고 |
| **Caves of Qud — Historic Sites** | 지역별 고대 유적이 플레이 중 발견되고 전용 루트 생성 | 유적지 변형 컨셉. 단 생성이 아닌 "조사 결과로 드러남" |
| **Kingdom of Loathing — 지역 해금** | 메인 퀘스트 진행으로 새 지역이 열림 | 변형을 "섹터 자체를 새로 여는" 구조로 차용 |
| **Darkest Dungeon — 햄릿 진척도** | 지역의 특수 이벤트가 햄릿의 건물을 해금 | 변형으로 전용 퀘스트 풀 해금 (햄릿 건물 ↔ 섹터) |
| **Stardew Valley — 커뮤니티 센터/조-마트 분기** | 한 번 선택한 변형 경로는 되돌릴 수 없음 | 변형 영속성(되돌릴 수 없음) 차용 |

**핵심 설계 원칙**: "**변형은 조사의 마침표이자 새 문장의 시작**". 체인 퀘스트(페이즈 1-2)가 "이 지역의 숨은 이야기"라면, 변형은 "**조사를 끝까지 밀어붙여 지역에 흔적을 남긴다**"는 결과물이다. 그리고 변형된 섹터의 전용 퀘스트 풀이 **다음 플레이 사이클**로 이어준다.

## 상세 설계

### 1. 3유형 포지셔닝 — "4축 매트릭스"

마을·유적지·숨겨진 3유형은 **난이도 × 보상 × 반복성 × 특수성** 4축에서 서로 다른 포지션을 가져야 엘리트(M2b) 파밍·체인 퀘스트(M3)와 겹치지 않는다.

| 유형 | 난이도 | 보상 | 반복성 | 특수성 | 핵심 포지션 |
|------|--------|------|-------|-------|---------|
| **village** (마을) | 낮음 (난이도 2~3) | 낮음~중간 (골드 위주) | 높음 (정기적 호위·의뢰) | 없음 (일반 보상) | "**안전한 반복 소득처**" |
| **ruins** (유적지) | 높음 (난이도 4~5) | 높음 (장비·정수) | 중간 (갱신 주기 느림) | 중간 (전용 풀) | "**1회성 고난도 챌린지**" |
| **hidden** (숨겨진) | 중간~높음 (난이도 3~5) | 특수 (트레잇 영감·희귀 유물) | 낮음 (희소 출현) | 높음 (트레잇·서사) | "**성장 영감의 원천**" |

**엘리트(M2b)와 유적지의 차별화 축**:
- **엘리트**: 퀘스트 갱신마다 확률 출현, 반복 파밍이 핵심 루프, 정수 드랍 주축 (40~60%)
- **유적지**: 섹터가 변형되면 전용 퀘스트 풀에서 **상시 생성**되나 각 퀘스트는 **일반 갱신 주기**를 따른다 (반복 파밍 X). 개인 장비·XP 주축, 정수는 부차. "이 섹터에 가면 언제나 유적 탐사 퀘스트가 걸린다"는 감성

즉 엘리트는 **사냥**, 유적지는 **탐사**다. 동일 리전에 엘리트와 유적지 변형이 공존할 수 있으며 서로 다른 자원을 준다.

**숨겨진 섹터와 엘리트의 차별화 축**:
- **엘리트**: 전투력 수치 중심의 파밍
- **숨겨진**: **트레잇 학습 기회** 위주. 퀘스트 완료 시 `MercenaryStatService`의 행동 지표에 특수 가중치를 주거나(트레잇 획득 조건 가속화), 희귀 유물(M2a 용병단 장비 후보)이 드문 확률로 나오는 "특별한 섹터"

### 2. 변형 대상 리전 18개 — 테마 매칭

199개 리전 중 18개를 **리전 이름·환경 태그·티어 분포**를 근거로 선정. 각 유형 6개씩 균형. 실제 `region_id`는 페이즈 3-2에서 `[region-environment-tag]20260423_m2b-regions.csv`와 교차 확인 후 확정.

**선정 원칙**:
- 유형당 최소 5개 이상 (다양성 확보). 각 6개 채택
- 리전 이름과 변형 서사가 **자연 정합**하는 곳만 선정
- 티어 분포: T1~T2(입문 경험용), T3(중급 본진), T4~T5(엔드 도전) 혼합
- `environment_tags`와 변형 유형 제약:
  - village ← plains / coast / forest (사람이 살 수 있는 지형)
  - ruins ← ruins / underground / mountain (황폐하거나 깊숙한 지형)
  - hidden ← swamp / desert / underground / forest (감추어진 지형)

### 3. 변형 트리거 — `region_discoveries.transform`

**스키마 활용** (기존 `region_discoveries` 테이블):

```
discovery_type: 'transform'
discovery_data: {
  "transform_type": "village" | "ruins" | "hidden",
  "sector_index": 3,                -- 리전 내 어느 섹터가 변형되는지 (0~9)
  "transformed_name": "개척 마을",   -- UI에 표시될 이름
  "narrative_template": "용병단의 흔적을 따라 개척민들이 {region.name}에 정착했다. {merc.name}이 그 첫 손님을 맞이한다."
}
description: "지역에 깊은 변화의 기운이 감돈다"
```

#### 3-1. `knowledge_threshold` 분포 — "변형은 체인 완주 뒤에 온다"

| 체인(페이즈 1-2) | 변형(본 기획) |
|-----------------|--------------|
| 임계 60~95 | 임계 **98** (고정) |

변형은 **체인 발동보다 늦게** 트리거된다. 근거:
- 서사적으로 "체인을 끝까지 따라간 유저만 변형을 본다"는 흐름이 자연스러움
- 체인(95)과 변형(98) 사이에 작은 간격을 두어 조사 후반부에 체인 진행 → 완주 → 변형 순서로 리듬 형성
- 지식 포인트 0~100 스케일이므로 98은 **거의 최후 관문**

**체인이 없는 리전에서도 변형은 발생한다**. 변형 대상 18개 리전 중 체인 발동 리전과 일치하는 곳은 일부뿐. 체인이 없는 리전은 일반 조사 경로만으로 98까지 쌓아 변형에 도달.

#### 3-2. 섹터 선정 규칙

**한 리전당 1섹터만 변형**(M3 MVP 제약). 이유:
- `RegionState.sectorChanges: Map<int, String>`은 복수 섹터 변형을 데이터 구조상 허용하나, 기획 측면에서 1섹터만 변형해야 "이 섹터가 특별해졌다"는 포커스가 살아남
- 여러 섹터 변형은 M6(완성과 균형)에서 확장 검토

**변형 섹터 인덱스**: 각 리전별로 `discovery_data.sector_index` 고정(페이즈 3-2에서 결정). 섹터 0~9 중 리전의 특성을 고려하여 자연스러운 위치 선정 (예: 해안 리전은 섹터 9번, 산악 리전은 섹터 0번 등).

### 4. 전용 퀘스트 풀 — `quest_pools.sector_type` 활용

**`sector_type` 필드** (M1 추가, 현재 nullable): 변형 유형과 동일한 값을 가짐 — `village` / `ruins` / `hidden` / NULL(일반).

**QuestGenerator 분기 로직**:
```
현재 섹터가 변형된 섹터인가?
  ├─ 예 → quest_pools 중 sector_type = {변형 유형}인 풀에서 생성
  └─ 아니오 → sector_type IS NULL인 일반 풀에서 생성
```

#### 4-1. 유형별 풀 규모 (페이즈 3-3 data-generator 목표)

| village | **12개** | 호위 5 / 탐험 4 / 약탈 2 / 토벌 1 | 난이도 2~3 중심 |

#### 4-2. 유형별 대표 퀘스트 예시 (3~4개씩)

**village (마을)**:
- "장터 호위" — 난이도 2, 호위, 보상 100~150G, 안정적
- "개척민 분쟁 조정" — 난이도 2, 탐험, 보상 90~130G + 평판
- "정착지 식량 수급" — 난이도 3, 호위, 보상 120~180G
- "도둑고양이 소탕" — 난이도 2, 토벌, 보상 100~140G + XP 소량

**ruins (유적지)**:
- "잠든 수호 기계 파괴" — 난이도 5, 토벌, 보상 600~800G + 장비 확률
- "유적 심층 진입" — 난이도 4, 탐험, 보상 500~650G + XP 다량
- "부활한 경비병 토벌" — 난이도 5, 토벌, 보상 650~850G + 정수 소량 확률
- "은닉된 보물실 약탈" — 난이도 4, 약탈, 보상 550~700G + 용병단 장비 극희소 확률

**hidden (숨겨진)**:
- "은둔자의 가르침" — 난이도 3, 탐험, 보상 300G + **트레잇 획득 가속 버프** (행동 지표 특수 가중치. 당해 퀘스트에 참가한 용병 한정 24시간 지속)
- "기억의 파편 회수" — 난이도 4, 탐험, 보상 400G + **희귀 유물 극희소 확률** (M2a 용병단 장비)
- "별의 조각 채집" — 난이도 5, 탐험, 보상 500G + 트레잇 영감 기회
- "금기된 의식 중단" — 난이도 4, 토벌, 보상 450G + 평판 하락 위험

**"트레잇 획득 가속 버프"는 M3 신규 메커니즘**. `MercenaryStatService`의 행동 지표에 24시간 한시적 가중치 +50%를 주어 후천 트레잇 획득 조건이 빨리 충족되도록 한다. 구현은 페이즈 4-3 spec에서 정교화. **트레잇 학습**이라는 숨겨진 섹터의 특수성을 수치 아닌 "기회"로 제공.

#### 4-3. 퀘스트 풀 공유 규칙

- 전용 퀘스트 풀은 **모든 변형 섹터가 공유** (리전별로 별도 풀이 아님). 즉 "마을 변형된 6개 리전 어디를 가든 동일 마을 풀에서 생성"
- 이것은 데이터 밀도(34개)를 유지하기 위한 선택. 페이즈 3-3의 34개가 18개 변형 섹터에 충분한 다양성 제공
- 향후 M4~M6에서 리전별 고유 퀘스트가 필요해지면 `quest_pools.region_id` 필드로 확장 가능

### 5. 연계 퀘스트(페이즈 1-2)와의 상호작용

**핵심 결정: 변형은 체인과 독립적**. 단, 일부 체인은 완주 시 변형 트리거를 앞당길 수 있다.

#### 5-1. 일반 규칙 — "독립 발동"

대부분의 경우 변형은 조사 지식 포인트 98 누적만으로 트리거된다. 체인 유무와 관계없다.

#### 5-2. 예외 — "체인 완주 보너스 트리거"

체인 6 **장인의 유산**, 체인 5 **상인의 비망록** 등 **특정 체인 완주 시 그 리전의 변형 임계가 조기 만족**될 수 있다.

| 체인 | 완주 시 효과 |
|------|-----------|
| 체인 5 (상인의 비망록) | 체인 5의 단계 2/3/4 중 어느 리전에서든 knowledge 값이 실제보다 +10 더 쌓인 것으로 간주 (변형 임계 조기 충족 가능) |
| 체인 6 (장인의 유산) | 1단계 시작 리전의 knowledge +15 보너스 |

**왜 모든 체인에 적용하지 않는가**:
- 모든 체인 완주가 변형 트리거를 앞당기면 "체인=변형 지름길"이 되어 변형의 독립성이 희석됨
- 체인 5·6만 **주거·장인 테마**로 마을/유적지 변형과 서사 정합이 뚜렷함. 나머지 체인은 지역을 떠나는 서사가 많아 그 리전 변형과 자연 정합이 약함
- **체인 7(혼을 끊는 자) 완주 시 트리거 효과는 없음.** 이미 엔드게임 임계(95)에서 발동되어 완주 이후에 변형이 올 자연스러운 여지가 있기 때문

이 예외는 페이즈 3-2(region_discoveries 세팅)에서 체인 리전의 `knowledge_threshold`를 상대적으로 낮추는 방식으로 구현 (별도 신규 메커니즘 없음).

#### 5-3. 역방향 — 변형이 새 체인을 여는가?

**M3 MVP에서는 아니다.** "변형 → 체인 해금"을 추가하면 복잡도가 커진다. 변형된 섹터의 전용 퀘스트 풀만으로 충분히 "새 컨텐츠 해금" 감성이 전달된다.

M4 이후 "세력 전용 체인" 확장 시 세력 거점 변형이 세력 체인을 여는 구조로 확장 가능 (선택). 본 문서에서는 범위 외.

#### 5-4. 트리거 우선순위

같은 리전에서 체인과 변형이 **동시 임계 도달 시** 순서:
1. 체인 발동 팝업 → 체인 정보 UI
2. 변형 발동 팝업 → 변형 팝업 (체인 팝업 닫은 뒤 자동 순차 노출)

이는 페이즈 1-6(공존 정책)과 페이즈 4-3(변형 시스템 spec)에서 UI 중첩 처리로 상세화.

### 6. 변형 영속성

- **영구 변형**: 한 번 변형된 섹터는 **절대 되돌릴 수 없다**. 이것이 "흔적"의 감성
- **되돌리기 불가 근거**:
  - Stardew Valley의 커뮤니티 센터/조-마트처럼 플레이어의 선택과 조사가 세계에 영구 흔적을 남긴다
  - 되돌리기 허용 시 "어떤 유형으로 변형할지 재롤링"하려는 시도가 발생 — 서사 무게 상실
- **선택 UI 없음**: 유형은 `region_discoveries.discovery_data`에 **사전 고정**. 플레이어가 "이 리전을 마을로 변형할지 유적지로 변형할지" 선택하지 않는다. 이것은 의도적 — 플레이어는 조사를 계속할지 멈출지만 결정한다.
- **저장 위치**: `RegionState.sectorChanges: Map<int, String>` (Hive `regionStates` 박스, 유저별 로컬). 서버 공유 아님 (싱글플레이이므로 자명)
- **변형 이전 섹터 대기 퀘스트 처리**: 변형 발생 시점에 해당 섹터에서 발급된 대기 퀘스트는 **유지**. 이미 발급된 것은 일반 풀 퀘스트라도 유저 소유. 다음 퀘스트 갱신부터 전용 풀로 교체

### 7. 변형 알림 UX

#### 7-1. 발동 팝업

```
┌──────────────────────────────────┐
│  ✨ 지역 변형                    │
│                                   │
│  {region.name}에 변화가 일어났다. │
│  [서사 텍스트 — TemplateEngine 치환] │
│                                   │
│  [유형 배지] 개척 마을            │
│                                   │
│  이 섹터에서 새로운 종류의 의뢰가  │
│  생성됩니다.                      │
│                                   │
│   [ 확인 ]   [ 이동 화면으로 이동 ]│
└──────────────────────────────────┘
```

- **즉시 노출**: 조사 완료 시점에 발동 조건 만족 시 바로 팝업. 체인 팝업과 중첩되면 체인 우선 → 변형 순서
- **서사 텍스트**: `discovery_data.narrative_template` + TemplateEngine 치환. `merc.*`는 **조사 수행 용병**(지역 조사 슬롯의 용병)으로 바인딩 — 체인의 protagonistMercId와는 별개. 변형은 "이 리전을 조사한 용병"이 주역
- **이동 유도 버튼**: "이동 화면으로 이동" 누르면 바로 변형된 섹터 위치 카메라 이동 (섹터 선택 UI에서 강조 표시)

#### 7-2. 활동 로그

- `ActivityLogType`에 **`regionTransform`** 신규 엔트리 추가 (HiveField 15)
- 메시지 예: "{region.name}의 섹터가 개척 마을로 변형되었다"
- 기존 `discoveryFound` 타입과 별개로 취급 (변형은 더 큰 사건)

#### 7-3. 이동 화면 시각 구분

- 변형된 섹터는 **전용 아이콘**으로 표시 (🏘️ village / 🏛️ ruins / ✨ hidden)
- 섹터 이름이 `discovery_data.transformed_name`으로 변경 (예: "섹터 3" → "개척 마을")
- 색상 강조: village = 녹색 tint / ruins = 보라 tint / hidden = 금색 tint

#### 7-4. 퀘스트 풀 갱신 타이밍

- 변형 발생 즉시 반영: 발동 팝업 확인 후 해당 섹터의 **다음 퀘스트 갱신 주기(1시간 게임 시간)** 부터 전용 풀 적용
- 즉시 강제 갱신은 하지 않음 (이미 대기 중 퀘스트 존중 — §6 영속성 항목과 정합)

### 8. 데이터 구조 요약 — `RegionState.sectorChanges`

로드맵에 이미 명시된 필드. 본 기획은 이 필드를 이렇게 채운다:

```dart
// 예: region_id=42의 섹터 3이 마을로 변형된 경우
regionState.sectorChanges[3] = 'village';
```

- **Key**: 섹터 인덱스(int, 0~9)
- **Value**: 변형 유형 문자열(`village` / `ruins` / `hidden`)
- **현재는 섹터당 하나의 유형**. 한 리전이 여러 섹터가 다른 유형으로 변형되는 경우도 구조상 가능하나 M3 MVP에서는 리전당 1섹터만 변형

페이즈 4-3 spec에서 `RegionState` Freezed 모델에 이 필드를 추가하고 Hive 어댑터 재생성.

## 현재 시스템과의 연관

### 영향받는 기존 시스템

| 시스템 | 영향 내용 |
|--------|---------|
| `RegionState` (Hive typeId:8) | `sectorChanges: Map<int, String>` HiveField 추가 (기존 `knowledge`/`triggeredDiscoveries`와 공존) |
| `InvestigationNotifier` / `InvestigationService` | 조사 완료 시 knowledge 누적 → `region_discoveries`의 transform 임계 도달 체크 → `RegionState.sectorChanges` 갱신 |
| `QuestGenerator` | 섹터 인덱스 + `RegionState.sectorChanges`를 조회하여 `quest_pools.sector_type` 필터 적용 |
| `quest_pools` | `sector_type` 컬럼 기존(M1 추가) 활용. NULL이면 일반, 그 외는 전용 풀 |
| `MovementScreen` | 변형된 섹터의 아이콘/이름/색상 표시 |
| `ActivityLog` + `ActivityLogType` | `regionTransform` HiveField 15 추가 |
| TemplateEngine | 변형 팝업 서사 렌더 (`merc.*` 조사 용병 바인딩, `region.*` 변형 대상 리전) |
| 체인 퀘스트(페이즈 1-2) | 체인 5·6 완주 시 knowledge 보너스 적용으로 변형 조기 트리거 가능 |
| 엘리트(M2b) | 영향 없음 (독립 시스템). 엘리트 출현 리전이 변형 리전과 중첩돼도 둘 다 동작 |

### 호환성 리스크

- **낮음**: `RegionState` 추가 필드는 기존 `triggeredDiscoveries`와 같은 패턴. Hive 마이그레이션은 nullable default로 처리
- **중간**: `QuestGenerator` 분기 로직 추가 — 섹터 인덱스를 활용한 sector_type 필터링. 현재 QuestGenerator가 섹터 인덱스를 받는지 여부 확인 필요. 안 받으면 signature 변경 필요
- **낮음**: 이동 화면 섹터 시각 구분은 순수 UI 변경. 기존 색상/아이콘 체계와 조화롭게 배치

### 신규 인프라 (페이즈 4-3 spec 대상)

- `RegionState` Freezed 모델 확장 — `sectorChanges: Map<int, String>` HiveField
- `InvestigationService` 확장 — knowledge 임계 도달 시 transform 트리거 분기
- `QuestGenerator.generate()` 확장 — `sectorType: String?` 파라미터 또는 `RegionState` 전체 주입
- `RegionTransformService` (신규) — 변형 발동·팝업·로그 처리
- `ActivityLogType.regionTransform` 추가
- `ActiveQuest`에 `buff_hint` 같은 임시 필드 추가? → **불필요**. 트레잇 학습 가속 버프는 별도 `PendingTraitBuff` 값 객체로 처리 (페이즈 4-3 상세화)

## 구현 우선순위 제안

**우선순위: 높음 (M3 크리티컬 패스)**

사유:
- M3의 "지역 조사 최종 보상" 정체성을 연계 퀘스트와 함께 완성하는 축
- 페이즈 3-2(region_discoveries transform 18행)와 페이즈 3-3(sector_type quest_pools 34행)의 입력

## data-generator 지시사항

본 기획서는 **두 개의 data-generator 호출**을 유발한다:

### (A) `region_discoveries` 변형 트리거 18행

- **대상 타입**: `region-transform` (신규 — 타입 스펙 작성 필요, 페이즈 3-0-2)
- **대상 테이블**: `region_discoveries` (기존 테이블, `discovery_type='transform'` 행 18개 INSERT)
- **생성 수량**: **18행**
- **톤/세계관 가이드**:
  - `description` 필드: "지역에 깊은 변화의 기운이 감돈다" 수준의 짧은 예고문
  - `discovery_data.narrative_template`: TemplateEngine 문법 활용 (`{region.name}`, `{merc.name}`, `[if has_trait:scholar]...[/if]` 등)
  - 한국어, 판타지 톤, 저작권 금칙 준수
- **구조적 제약**:
  - 18개 모두 `knowledge_threshold = 98`
  - `discovery_type = 'transform'` 고정
  - `discovery_data`에 `transform_type`(village/ruins/hidden), `sector_index`(0~9), `transformed_name`(한국어), `narrative_template`(TemplateEngine 문법) 포함
  - 유형당 6행 균등
  - 체인 5(chain_merchant_ledger)·체인 6(chain_forge_masters) 연관 리전에는 knowledge 보너스 처리를 위해 `knowledge_threshold`를 88~90으로 낮추는 옵션 (페이즈 2-2 밸런스 검토 시 확정)
  - `region_id`: §2 테마 매칭 기준으로 실제 리전 18개 선정. `[region-environment-tag]20260423_m2b-regions.csv` 참조
- **수치 출처**: balance-designer 페이즈 2-2 결과 대기 (임계값 98, 보너스 트리거 임계치 조정)
- **특수 요구**: 체인 연계 리전 보너스 처리는 data-generator 출력 시 주석 처리하고 최종 INSERT 시 결정

### (B) `quest_pools` 변형 섹터 전용 34행

- **대상 타입**: `quest-pool-sector` (신규 — `types/region-transform.md`의 일부로 포함 가능)
- **대상 테이블**: `quest_pools` (기존 테이블, `sector_type IS NOT NULL` 행 34개 INSERT)
- **생성 수량**: **34행** (village 12 / ruins 12 / hidden 10)
- **톤/세계관 가이드**:
  - `name`: 한국어 3~6자 퀘스트 명 (예: "장터 호위", "잠든 수호기계 파괴")
  - `description`: 1~2문장, TemplateEngine 치환 가능
  - 저작권 금칙 준수
- **구조적 제약**:
  - `sector_type ∈ {village, ruins, hidden}` 필수
  - village: 난이도 2~3 중심(평균 2.4), 호위 5/탐험 4/약탈 2/토벌 1, 보상 90~180G
  - ruins: 난이도 4~5 중심(평균 4.4), 토벌 5/탐험 4/약탈 3, 보상 500~850G
  - hidden: 난이도 3~5 혼합(평균 3.8), 탐험 4/토벌 3/호위 2/약탈 1, 보상 300~500G + 특수 보상 플래그
  - hidden 유형의 `reward_data` JSONB에 `special_bonus` 필드 포함 가능 (`trait_learning_buff`, `artifact_rare_drop` 등 — 페이즈 4-3 spec에서 구체 매커니즘 결정)
- **수치 출처**: 페이즈 2-2 balance-designer 결과
- **특수 요구**:
  - 34개 풀은 18개 변형 섹터가 공유 — 리전별 중복 생성이므로 반복성 피로 방지 위해 서사 훅이 너무 리전 특정하지 않을 것
  - hidden의 10개 중 2~3개는 "희귀 유물 확률 드랍" 플래그 (M2a 용병단 장비 아이템 ID 참조)

## 오픈 질문

- **Q-1 (마을 변형의 경제 영향)**: 마을 변형이 "안정 소득처"로 자리잡으면 엘리트 파밍보다 마을 반복이 우위가 될 가능성. 마을 보상이 너무 안정적이면 유저가 어려운 콘텐츠를 회피. → **페이즈 2-2 밸런스 검토 시 마을 시간당 골드 기대치가 엘리트 대비 30~50% 수준이 되도록 조정**

- **Q-2 (트레잇 학습 가속 버프 메커니즘)**: hidden 섹터 퀘스트 완료 시 주는 "행동 지표 가중치 +50% 24시간"을 어떻게 구현할지. 현재 `MercenaryStatService`는 이벤트마다 정수를 누적하는 구조. 가중치 승수는 기존 로직을 확장해야 함. → **페이즈 4-3 spec에서 확정**. 권장: `Mercenary.traitLearningBoostUntil: DateTime?` 필드 추가, 퀘스트 완료 시 행동 지표 증가분에 배수 적용

- **Q-3 (체인 5·6 조기 트리거 보너스 수치)**: knowledge +10~15 보너스가 실제로 체감될지. 조사 소요 시간 총량과 비교하여 "체인 완주가 변형을 의미있게 앞당기는가" 검증 필요. → **페이즈 2-2 balance-designer 검토**

- **Q-4 (hidden 섹터 희귀 유물 드랍 확률)**: M2a 용병단 장비(깃발·뿔피리·수호자방패장식) 중 체인 확정 보상으로 커버되지 않은 3종의 획득 경로가 엘리트 레어 드랍과 hidden 퀘스트 두 곳으로 분산. 확률 설계가 엘리트 드랍 테이블과 충돌하지 않아야 함. → **페이즈 2-2 balance-designer + M2b 드랍 시뮬레이션 교차 확인**

- **Q-5 (변형 섹터 이름 표시 범위)**: 이동 화면에서 `transformed_name` 표시 시 원래 "섹터 3" 명칭을 완전히 대체할지, 병기할지. → **권장 완전 대체** (변형은 영구적이므로 원래 이름 유지 필요 없음). 페이즈 1-6 공존 정책에서 확정

- **Q-6 (섹터 인덱스 고정 vs 동적)**: 같은 리전이라도 유저마다 다른 섹터가 변형되면 다양성이 좋지만 구현 복잡. → **권장 고정** (페이즈 3-2에서 `sector_index` 미리 결정). 동적은 M6 이후 고려

- **Q-7 (엘리트와 유적지 변형이 같은 리전에서 겹치는 경우)**: 예를 들어 T4 고대유적 리전에 이미 엘리트가 출현하는데 변형 R4도 발동되면 퀘스트 풀이 혼잡. → **분리 동작 권장**: 엘리트는 리전 전체에서 출현(기존 로직), 유적지 변형 전용 퀘스트는 변형 섹터에서만 생성. 두 풀이 독립적으로 돌아감. 페이즈 4-3 spec에서 확인

- **Q-8 (변형 이전에 획득한 퀘스트가 변형 후에도 수행 가능한지)**: §6에서 "유지"로 결론 지었으나 UI에서 해당 퀘스트가 여전히 "섹터 3" 명칭으로 표시될 수 있음. 혼란 방지 표기가 필요할지 → **권장**: 이미 발급된 퀘스트는 원래 리전/섹터 정보로 수행 가능하되 UI에 "(변형 이전)" 레이블 부가. 페이즈 4-3 spec 검토
