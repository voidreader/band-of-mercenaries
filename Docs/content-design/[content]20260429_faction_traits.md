## 개요

세력의 깊이를 만드는 4축 — **패시브(M1) / 전용 시설(M4 #1) / 거점·상점(M4 #2·#3) / 전용 트레잇(M4 #4)** — 중 마지막 축은 **용병 단위 영구 보상**이다. 패시브·시설·상점은 모두 용병단(전체 또는 다수)에 작용하지만, 전용 트레잇은 **"이 세력의 깃발 아래 싸운 한 명의 용병"**의 정체성에 새겨진다. 이는 영입한 용병 자체에 세력 색깔을 입히는 마지막 단추다.

**전용 트레잇이 채우는 빈자리:**

1. **세력 정체성의 미세 표현 자리**: 패시브·시설은 거시 보너스(보상 +%, 시간 -%, 확률 가산)로 표현되지만 트레잇은 **"한 용병이 이 세력에서 받은 훈련"**이라는 서사를 담는다. 도감의 트레잇 슬롯에 세력 색이 박힌 트레잇이 자리잡는 것 자체가 외형 증명.
2. **장기 누적 게이트의 끝점**: 평판(누적) × 거점 방문(액션) × 행동 지표(시간) 3축이 동시에 충족되어야 풀리는 **"가장 늦게 풀리는 보상"**이다. 평판 91~100 + 거점 도장 + 누적 행동 지표 = "이 용병은 이 세력의 핵심 멤버"라는 게이미파이드 증명.
3. **다중 가입자의 분산 전략 결정점**: 1세력 집중 풀투자 (정수 트레잇 1개 + 강화 + 기본 = 3종 보유) vs 3세력 분산 (각 기본 트레잇 1개씩 = 3종 보유). 동일 슬롯 점유라 **"누구에게 어떤 세력 도장을 찍을 것인가"** 결정이 발생.

**M4 전용 트레잇 설계 4원칙:**

1. **AND 게이트 4중 조건 (가입 + 거점 + 평판 + 행동 지표)** — 단순 누적이 아닌 액션·소속·시간 결합.
2. **평판 3티어 자동 승격 (기본 → 강화 → 정수)** — 진화 메커닉 차용 안 함. 평판 게이트 통과 시 자동 교체로 단순화.
3. **세력 슬롯 분리 (후천 4슬롯과 별도 1슬롯)** — 기존 빌드를 강제 침해하지 않음. 단 다중 가입자도 동시 1개만 활성 (3슬롯이 아님).
4. **탈퇴 시 잠금 + 즉시 재활성화** — 시설 정책과 동일. 데이터는 영구 보존, 효과만 가입 상태 게이트.

## 상세 설계

### 1. AND 게이트 4중 조건

세력 전용 트레잇은 다음 4개 조건을 **모두** 충족해야 자동 부여된다 (수동 액션 없음):

```
세력 트레잇 자동 부여 = 
    (1) joined_factions contains target_faction_id
  AND (2) factionState[target].firstVisitedAt is not null
  AND (3) factionState[target].reputation >= tier_threshold (31 / 61 / 91)
  AND (4) mercenary.stats[target_stat] >= behavior_threshold
```

#### 1.1 조건별 설계 의도

| # | 조건 | 의미 | 데이터 출처 |
|---|------|------|-----------|
| 1 | 가입 세력 일치 | "이 세력의 일원이 되어야" | `FactionState.joined == true` (M1 구현) |
| 2 | 거점 방문 깃발 | "이 세력의 본거지에 한 번이라도 가야" | `FactionState.firstVisitedAt != null` (페이즈 1 #2 결정) |
| 3 | 평판 임계값 | "이 세력에 충분히 기여해야" | `FactionState.reputation` (M1 구현) |
| 4 | 행동 지표 임계값 | "이 세력 정체성에 부합하는 행동을 누적해야" | `Mercenary.stats[]` (26개 지표, 일부 M4 신규 필요) |

조건 1·2는 **용병단 단위** (FactionState), 조건 3은 **용병단 세력 평판**, 조건 4는 **용병 단위** (Mercenary.stats). 즉:

- 용병단이 가입 + 거점 방문 + 평판 충족 → 그 시점부터 **모든 가입자 용병이 후보**
- 행동 지표만이 용병별 차이 → 같은 시점에 다른 용병이 다른 단계 트레잇 보유 가능

#### 1.2 평판 3티어 자동 승격

| 평판 구간 | 명칭 | 트레잇 등급 | 행동 지표 임계값 (예시) |
|---------|------|-----------|--------------------|
| 11~30 (신참) | 가입 직후 | 트레잇 미해금 (가입+거점만 통과한 단계) | — |
| 31~60 (인정) | **기본** | Tier 1 | 30~80 (지표별) |
| 61~90 (신뢰) | **강화** | Tier 2 (기본 자동 교체) | 80~150 |
| 91~100 (핵심) | **정수** | Tier 3 (강화 자동 교체) | 200~400 |

**자동 승격 규칙:**

```
조건 (1)~(4) 충족 시점에 부여된 트레잇이 보유 상태에서:

   평판 31 도달 + 행동 지표 ≥ 임계값(기본) 충족
      → 기본 트레잇 자동 부여

   평판 61 도달 + 행동 지표 ≥ 임계값(강화) 충족
      → 기본 트레잇 자동 제거 + 강화 트레잇 부여 (슬롯 동일)

   평판 91 도달 + 행동 지표 ≥ 임계값(정수) 충족
      → 강화 트레잇 자동 제거 + 정수 트레잇 부여 (슬롯 동일)
```

**자동 승격 vs 수동 진화의 차이:**

- M2a 후천 트레잇 진화: 플레이어가 카드 비교형 팝업에서 경로 선택 (`TraitEvolutionDialog`)
- M4 세력 트레잇 승격: **플레이어 결정 없음. 조건 충족 시 자동 교체**.
- 이유: 평판 게이트가 자동 노출 게이트라 "어느 등급으로 갈지" 선택지가 없음. 진화 같은 분기 메커닉 불필요.
- UI: 자동 부여 알림 팝업 (M2a `TraitAcquisitionDialog` 재사용. low priority dialogQueue)

**평판 하락 시 정책 (예: 충돌 강제 탈퇴 -100):**

- 트레잇은 슬롯에 남되 **비활성화** (효과 0)
- 평판 회복 시 자동 재활성화 (등급은 회복된 평판 구간에 따라 자동 재계산)
- 단계 강등은 발생하지 않음 (예: 91에서 정수 획득 후 89로 떨어져도 강화로 강등되지 않음 — 누적 보상 보존). 단 행동 지표가 정수 기준을 넘은 상태가 유지되어야 함.

#### 1.3 거점 방문 게이트의 영구 도장 정책

페이즈 1 #2 결정 사항 인용:

> firstVisitedAt 깃발은 영구. 탈퇴해도 기록 유지. 재가입 시 즉시 트레잇 해금 자격 보유.

→ **재가입자는 가입 즉시 평판/행동 지표만 충족하면 트레잇 자동 부여**. "한 번 발 들이면 영원히 자격 보유" 정책.

#### 1.4 조건 재평가 트리거

페이즈 1 #2 결정 사항 인용:

> 재방문 시점 = 트레잇 조건 재평가 트리거. 행동 지표가 거점 밖에서 누적되어도 재방문 시 일괄 검증되어 트레잇 자동 부여.

본 문서는 이를 보완한다:

| 재평가 트리거 시점 | 처리 |
|----------------|------|
| 거점 도착 (`firstVisitedAt`/`lastVisitedAt` 기록 시점) | 모든 세력 트레잇 조건 재평가 |
| 퀘스트 완료 시점 (행동 지표 누적 직후) | 해당 용병 + 그 용병이 가입 세력의 트레잇만 재평가 (효율) |
| 평판 변동 시점 (`FactionStateRepository.addReputation` 직후) | 해당 세력의 트레잇만 재평가 (모든 용병 대상) |
| 가입/탈퇴 시점 | 해당 세력 트레잇 활성/비활성 일괄 처리 |
| 시그니처 정수 사용 시점 (페이즈 1 #3 연동) | 해당 용병 + 시그니처 매칭 세력 트레잇만 재평가 |

기존 `TraitAcquisitionService`(M2a)에 세력 트레잇 평가 로직 추가. 페이즈 4 #1 데이터 모델 확장에서 평가기 인터페이스 확정.

### 2. 14세력 × 트레잇 매핑 (28~42개)

#### 2.1 구성 방침

각 세력별 **2~3종**:

| 등급 | 세력당 개수 | 14세력 합계 |
|------|---------|----------|
| 기본 (Tier 1, 평판 31 게이트) | 1 | 14 |
| 강화 (Tier 2, 평판 61 게이트) | 1 | 14 |
| 정수 (Tier 3, 평판 91 게이트) | 0~1 (정체성 직결 세력만) | 7 |

= **합계 35종** (사용자 요청 28~42 범위 중간).

**정수 트레잇 7세력 선택 기준**: 패시브·시설 효과의 정체성 강도 기준 상위 7세력. 나머지 7세력은 강화까지만 (= 페이즈 2/4에서 추가 가능).

| 정수 보유 7세력 | 정체성 강도 근거 |
|--------------|--------------|
| 모험가 길드 | 6대 공개 세력 중 가장 보편 정체성 (탐험) — 정수 보상으로 보편성 회수 |
| 전사 길드 | raid/hunt 정체성 강한 패시브 + 시설(대성공) → 정수가 끝물 도달 표지 |
| 마탑 연합 | 트레잇 시스템 가속 정체성 (시설 = 행동 지표 누적 +50%) — 정수가 트레잇 메타 자체 강화 |
| 태양 교단 | 부상→사망 회피 정체성 (시설 = 부상 회피 40%) — 정수가 사망 회피 |
| 금지된 서고 | 트레잇 진화 정체성 (시설 = 진화 분기 추가) — 정수가 금기 트레잇 |
| 화산 심장단 | 파괴 정체성 (raid 보상 +15%, 파견비 -30%) — 정수가 위험-보상 극단 |
| 혈계 귀족회 | T4~T5 정체성 (시설 = 레벨업 +40%) — 정수가 만렙 도달 표지 |

#### 2.2 14세력 트레잇 카탈로그

각 트레잇은 다음 6개 항목을 갖는다:
- **이름 (한글)**
- **등급 (기본/강화/정수)**
- **설명 (세력 정체성 반영, 1~2문장)**
- **효과 방향성** (max_effect 후보값, 페이즈 2 확정)
- **행동 지표 게이트** (지표명 + 임계값 후보)
- **세력 패시브·시설과의 차별화** (중복 방지)

---

##### 1. 모험가 길드 — "탐험" 정체성

| # | 등급 | 이름 | 설명 | 효과 | 행동 지표 게이트 | 차별화 |
|---|------|------|------|------|-------------|------|

### 3. 슬롯 점유 정책

#### 3.1 후천 슬롯 4개와 분리된 "세력 슬롯" 1개

| 슬롯 영역 | 용량 | 비고 |
|---------|----|------|
| 선천 (Physical / Background / Talent) | 3 | M2a 그대로 |
| 후천 (CombatStyle / Survival / Behavior / Mental / Experience 중 4개) | 4 | M2a 그대로 |
| **세력 (M4 신규)** | **1** | **세력 트레잇 1개. 가입 세력이 다중이어도 1개만 활성** |

= 용병 1명당 최대 보유 트레잇 = 3 + 4 + 1 = **8개** (M2a 7개에서 1 증가).

#### 3.2 다중 가입자 정책

```
용병이 2~3개 세력 가입자라면:
  → 가입 세력별 조건 (1)~(4) 중 충족하는 트레잇 후보 N개
  → 그중 1개만 세력 슬롯 점유
  → 어느 트레잇이 활성될지 → "최우선 매칭 정책"으로 자동 결정
```

**최우선 매칭 정책 결정 안:**

A. **평판 최고 세력 우선**: 가장 높은 평판 세력의 트레잇 우선. 동률 시 가입일 빠른 순.
B. **플레이어 명시 선택**: UI에 "활성 세력 트레잇 선택" 토글. 다중 가입자에게만 노출.
C. **자동 등급 우선**: 정수 > 강화 > 기본 우선, 그 안에서 평판 최고 우선.

**본 문서 권장: C (자동 등급 우선)**. 이유:
- 가장 강한 보상이 자동 활성 = 직관적 "정수 트레잇은 무조건 켜진다"
- 동일 등급 다중 보유 시 평판 최고 세력 우선 = 충성도 자동 반영
- B(명시 선택)는 후속 폴리시로 보류 (페이즈 4에서 검토)

```
활성 세력 트레잇 = 
   max(정수 > 강화 > 기본, 평판 desc, 가입일 asc)
```

비활성 트레잇은 슬롯에 보이되 회색 처리 (`isActive = false`). UI는 `MercenaryDetailOverlay` 트레잇 슬롯 그리드에 신규 "세력" 행 추가.

#### 3.3 카테고리 1개 룰 면제

세력 트레잇은 후천 5개 카테고리(CombatStyle/Survival/Behavior/Mental/Experience)와 별개의 신규 카테고리 `Faction`. M2a `category 1개 룰`은 5개 카테고리 내 1개씩이라 세력 슬롯과 충돌 없음.

#### 3.4 충돌 관계 처리

세력 트레잇과 M2a 트레잇 간 충돌 관계는 **본 문서에서 도입하지 않음** (단순화). 이유:
- 14×3 = 42개 후보 트레잇이 기존 106종과 N:M 충돌 매트릭스를 만들면 페이즈 4 구현 비용 폭증
- 세력 트레잇은 슬롯 분리되어 있어 후천 빌드와 자연 공존
- 세력 트레잇 사이 충돌도 도입 안 함 (활성 1개 정책으로 자연 해결)

단 향후 페이즈 2/4 검토 가능: "용암의 후예(12-3) ↔ 자연 결사 견습(9-1)" 같은 정체성 충돌은 페이즈 4에서 추가 가능.

### 4. 진화/소멸 정책

#### 4.1 진화 불가

세력 트레잇은 M2a `trait_transitions` / `trait_combo_evolutions`에 등록되지 않는다. 이유:
- 평판 자동 승격(2.1)이 진화 메커닉을 대체
- M2a 진화 분기는 행동 패턴 분기를 표현 (예: trauma → hardened vs broken). 세력 트레잇은 단일 충성 경로

#### 4.2 자동 교체 (등급 승격)

```
평판 31 → 60 도달 + 강화 행동 지표 충족
  → 슬롯 내 기본 트레잇 즉시 제거
  → 강화 트레잇 부여 (slot 동일)
  → ActivityLog "{세력명} 충성도 인정 — '{기본}'이 '{강화}'(으)로 승격되었다"
  → 다이얼로그 큐 low priority "트레잇 승격" 알림 (M2a TraitAcquisitionDialog 재사용)

평판 91 도달 + 정수 행동 지표 충족
  → 동일 패턴으로 강화 → 정수 교체
```

#### 4.3 탈퇴 시 정책 (자발 / 충돌 강제)

| 상태 | 트레잇 슬롯 | 효과 적용 | 비고 |
|------|---------|----------|------|
| **자발 탈퇴** | 🔒 비활성 (슬롯 유지) | ❌ 효과 0 | 평판 0 회복 + 재가입 시 자동 복구 |
| **충돌 강제 탈퇴 (-100)** | 🔒 비활성 | ❌ 효과 0 | 평판 0 회복 + 재가입 시 복구 |
| **재가입 (firstVisitedAt 보유)** | ✅ 활성 (즉시 재계산) | ✅ 적용 | 평판 구간에 따라 등급 재산출. 행동 지표 그대로 |

**핵심 규칙:**

1. 트레잇 슬롯 데이터(`Mercenary.factionTraitId` 신규 필드)는 영구 보존. 잠금 상태에서도 데이터 유지.
2. 잠금 중 효과 0 — `TraitEffectService` 적용 시 가입 상태 게이트 검증.
3. 재가입 시 평판 회복 정도에 따라 등급 자동 재산출 (예: 재가입 후 평판 50이면 기본만 활성).
4. 충돌 강제 탈퇴 시 행동 지표는 그대로 유지. 평판만 -100. 충돌 해소 후 평판 회복하면 트레잇 복구.

#### 4.4 트레잇 삭제 (M2a Phase A 시스템 호환)

세력 트레잇은 **삭제 불가**. 이유:
- 세력 트레잇은 평판 자동 승격으로 자연 교체
- 의무실 삭제 비용 매트릭스(acquired 200G / evolved 500G)에 추가하면 무게감 분산
- 탈퇴로 비활성화 가능 → 삭제 수단 불필요

`TraitDeletionService.canDelete()`에서 `traitType == 'faction'` 체크 후 즉시 false 반환. UI에서 [삭제] 버튼 미노출.

### 5. 페이즈 1 #3 시그니처 정수와의 정합성

페이즈 1 #3 미해결 사항 #3 (시그니처 정수의 trait_innate 부여 정책) + #8 (시그니처 깃발이 trait를 부여하는 시그니처가 본 문서와 정합하는지) 해소.

#### 5.1 시그니처 정수 (consumable) 의 트레잇 부여 정책

페이즈 1 #3에서 14세력 시그니처 consumable이 정의되었다 (페이즈 1 #3 표 2.3 참조). 그중 일부는 트레잇 부여 효과를 가질 수 있다 (예: "혈계 영약(불멸판) — 가문 표식 트레잇 부여").

**본 문서 결정**: 시그니처 정수가 트레잇을 부여하는 경우, **부여 대상은 본 문서 정수 트레잇 (Tier 3, 평판 91)에 한정**.

**근거**:
- 정수 트레잇은 7세력만 보유. 일치 세력에서만 시그니처 정수가 트레잇 부여 효과 가능
- 시그니처 정수 (페이즈 1 #3 한정 91~100 상품, 1회 영구 구매)는 가격 12,000~50,000G 범위 → 정수 트레잇 = 동급 보상
- 시그니처 정수의 트레잇 부여 = 행동 지표 게이트 우회 단축 경로

**구체 정책**:

| 시그니처 정수 사용 시점 | 효과 |
|--------------------|------|
| 평판 91 + 거점 방문 충족 + 행동 지표 미충족 | **행동 지표 게이트 우회 부여** (정수 트레잇 즉시 활성) |
| 평판 91 + 거점 방문 + 행동 지표 충족 | (이미 자동 부여된 상태이므로 정수 효과는 다른 보너스 또는 매수 의미만 존재) |
| 평판 91 미달 (예: 60대) | 정수 효과만 적용. 트레잇 부여 안 됨 (게이트 미통과) |
| 가입 안 된 세력의 시그니처 정수 | (애초에 비가입 세력 상점 진입 차단으로 구매 불가) |

#### 5.2 7세력 정수 트레잇 ↔ 시그니처 정수 매핑

| 세력 | 시그니처 정수 (페이즈 1 #3) | 정수 트레잇 (본 문서) | 트레잇 부여 정수인지 |
|------|--------------------------|--------------------|----------------|
| 모험가 길드 | 모험가의 휴식 정수 | 전설의 모험가 (1-3) | ⭕ 한정 시그니처 정수 사용 시 부여 |
| 전사 길드 | 전사의 피 | 전쟁군주 (3-3) | ⭕ |
| 마탑 연합 | 지혜의 정수(상위) | 대마법사 (5-3) | ⭕ |
| 태양 교단 | 신성 약초 | 태양의 사도 (6-3) | ⭕ |
| 금지된 서고 | 금단의 정수 | 금단의 현자 (8-3) | ⭕ |
| 화산 심장단 | 분노의 정수 | 용암의 후예 (12-3) | ⭕ |
| 혈계 귀족회 | 혈계 영약 | 혈계 영주 (13-3) | ⭕ |

**나머지 7세력의 시그니처 consumable은 트레잇 부여 효과 없음** — 단순 회복/스탯 부여만. 페이즈 1 #3 큐레이션 가이드(2.3 표)와 정합.

#### 5.3 페이즈 2 검증 항목

- 시그니처 정수 가격(12,000~50,000G) vs 정수 트레잇 ROI 비교
- 행동 지표 누적 시간(예: raid_count 200 도달 시간) vs 시그니처 정수 단축 가치
- 시그니처 정수가 트레잇 부여하는 7세력 vs 안 하는 7세력의 가격 차이 (트레잇 부여 = +5,000~15,000G 가치 추가 권장)

### 6. 페이즈 1 #5 데이터 모델 사전 확정 요구사항

본 문서가 페이즈 1 #5에 전달하는 데이터 모델 요구사항:

#### 6.1 `traits` 테이블 신규 컬럼

```sql
ALTER TABLE traits ADD COLUMN faction_id TEXT NULLABLE FK → factions.id;
   -- null: M2a 일반 트레잇 (선천/후천 acquired/evolved)
   -- value: M4 세력 트레잇 (faction별)

ALTER TABLE traits ADD COLUMN faction_tier INTEGER NULLABLE;
   -- null: M2a 일반
   -- 1: 기본 (평판 31)
   -- 2: 강화 (평판 61)
   -- 3: 정수 (평판 91)
```

- `category_id` 신규 카테고리 `Faction` (id=9 권장, 기존 1~8 다음)
- `slot_type` 신규 enum 값 `faction` (기존 `innate` / `acquired` 와 별개)
- `type` enum 그대로 사용 또는 `faction_basic` / `faction_advanced` / `faction_essence` 신규 도입 — 페이즈 1 #5 결정

#### 6.2 acquisition_condition_json 확장

기존 M2a `acquisition_condition` JSONB는 단일 행동 지표 임계값(예: `{"solo_dispatch_count": 30}`)만 표현. 본 문서의 4중 AND 조건 표현 위해 확장:

```json
{
  "type": "and",
  "clauses": [
    { "type": "faction_match", "faction_id": "faction_warriors_guild" },
    { "type": "stronghold_visited", "faction_id": "faction_warriors_guild" },
    { "type": "reputation_at_least", "faction_id": "faction_warriors_guild", "value": 31 },
    { "type": "stat_at_least", "stat_key": "raid_count", "value": 50 }
  ]
}
```

평가기 `TraitConditionEvaluator`(M2a 기존) 확장 시 `clauseTypes`:

| clauseType | 데이터 출처 | 평가 |
|---------|----------|------|
| `faction_match` | `FactionStateRepository.getJoinedFactionIds()` | 가입 세력 목록에 faction_id 포함 |
| `stronghold_visited` | `FactionState.firstVisitedAt` | non-null |
| `reputation_at_least` | `FactionState.reputation` | >= value |
| `stat_at_least` | `Mercenary.stats[]` | >= value |
| `evolved_trait_count_at_least` (신규) | `Mercenary.allTraitIds`에서 type=evolved 카운트 | >= value (8-1, 8-2, 5-3 등에서 사용) |

#### 6.3 신규 행동 지표 (페이즈 1 #5에서 결정)

본 문서가 사용한 지표 중 M2a 23개 + M3 추가에 없는 것:

| 신규 지표 | 사용처 | 추적 방식 |
|---------|------|----------|
| `guild_equipment_quest_count` | 11-2 룬 장인 | 길드 장비 장착 상태로 퀘스트 완료 시 +1 |
| `large_party_dispatch_count` | 14-2 부족 전사 | 5명+ 파티 파견 시 +1 |
| `evolved_trait_acquired_count` | 5-3 / 8-1 / 8-2 / 8-3 | derived (현재 보유 evolved 카운트) |
| `chain_quest_completed_count` | (미사용, 향후 검토) | 체인 퀘스트 완주 시 +1 |

→ 페이즈 1 #5에서 `Mercenary.stats` Map 키 추가 (또는 derived getter로 처리). 기존 23개 + M3 추가 → 최종 약 26~28개 지표.

- 비활성 트레잇 회색 처리 (다중 가입자만 표시)
- 트레잇 카드 탭 → `TraitDetailDialog` (M2a) 재사용
- 세력 트레잇 다이얼로그 표시 차이:
  - 헤더: 세력 컬러 + 세력 깃발 아이콘
  - "획득 조건" 섹션: 4중 AND 조건 진행도 (각 조건 ✅/❌ + 진행도 바)
  - "다음 등급" 섹션: 강화 트레잇 미해금 시 "평판 X / 행동 지표 Y 더 필요"
  - "삭제" 버튼 미노출

### 9. 결정 사항

본 기획서에서 굳힌 핵심 정책:

1. **AND 게이트 4중 조건** = 가입 + 거점 방문(`firstVisitedAt`) + 평판 임계값(31/61/91) + 행동 지표 임계값
2. **평판 자동 승격** = 31 → 61 → 91 도달 시 자동 교체. 플레이어 선택지 없음
3. **세력 슬롯 1개 분리** = 후천 4슬롯과 별개 1슬롯. 다중 가입자도 활성 1개
4. **다중 가입자 활성 우선순위** = 정수 > 강화 > 기본 / 동률 시 평판 desc / 가입일 asc
5. **카테고리 1개 룰 면제** = 세력 트레잇은 신규 카테고리 `Faction`. 후천 5개와 무관
6. **충돌 관계 도입 안 함** = 세력↔M2a 간 / 세력↔세력 간 모두 단순화 (활성 1개 정책으로 자연 해결)
7. **진화 메커닉 도입 안 함** = `trait_transitions` / `trait_combo_evolutions` 미사용. 평판 승격으로 대체
8. **삭제 불가** = `TraitDeletionService.canDelete()` faction 타입 거부
9. **탈퇴 시 잠금** = 슬롯 유지 + 효과 0. 재가입 시 자동 재활성화. 행동 지표는 그대로 유지
10. **시그니처 정수 = 7세력 정수 트레잇만 부여** = 행동 지표 게이트 우회 단축 경로 (페이즈 1 #3 미해결 #3 / #8 해소)
11. **신규 카테고리 `Faction` (id=9) + 신규 slot_type `faction`** = 페이즈 1 #5 데이터 모델 입력
12. **35종 (정수 7세력 한정)** = 사용자 요청 28~42 범위 중간

### 10. 미해결 사항 (페이즈 2/4에서 결정)

1. **명칭 충돌 검토**: M2a `warlord`(CombatStyle evolved) ↔ 본 문서 3-3 "전쟁군주". M2a `madman`(Mental combo evolved) ↔ 12-3 "용암의 후예" 컨셉 유사. 페이즈 1 #5에서 트레잇 key 충돌 회피 (예: `faction_warlord_warriors` vs `warlord`).
2. **3개 신규 행동 지표 도입** (`guild_equipment_quest_count` / `large_party_dispatch_count` / `evolved_trait_acquired_count`): 페이즈 1 #5에서 `Mercenary.stats` Map 확장 또는 derived getter 처리 결정.
3. **다중 가입자 활성 우선순위 자동 vs 명시 선택 UI**: 본 문서는 자동 권장. 페이즈 4에서 명시 선택 UI 도입 여부 검토.
4. **시그니처 정수 사용 시 행동 지표 게이트 우회의 정확한 효과**: 페이즈 2에서 가격(12,000~50,000G) vs 행동 지표 누적 시간(raid_count 200 도달 ≈ 30~50시간 플레이) 매핑 산정.
5. **금단의 현자(8-3) 정수의 "금기 카테고리" 신규 트레잇 풀**: 본 문서는 "강력한 효과 + 부작용"만 제시. 구체 풀(3~5개?)은 페이즈 4 #4에서 정의.
6. **자연 결사(9) / 황혼 공학회(10) / 도둑 길드(4) / 송곳니 결사(14) / 균형 감시자(7) / 상인 연합(2) / 심층 망치단(11) 정수 트레잇 추가 가능성**: 본 문서는 정수 7세력만 부여. 페이즈 2/4에서 추가 검토 가능 (사용자 요청 42 상한까지 여유).
7. **태양의 사도(6-3) 부활 1회 ↔ 자연 제단 시설 사망→부상 강등 처리 순서**: 두 보호가 동시 활성화될 때 어느 것이 먼저 적용되는가? 페이즈 4 #4에서 처리 순서 결정 (권장: 부활 우선 → 부활 실패/소진 시 강등).
8. **세력 트레잇 effect_json 스키마**: 본 문서는 효과 방향성만. 페이즈 2 밸런스 + 페이즈 4 데이터 모델에서 JSON 스키마 확정.
9. **인스턴스 단위 정수 트레잇 (예: 사도 부활 1회) 영속화**: `Mercenary` 신규 필드 `factionTraitChargesUsed: Map<String, int>?` HiveField 24 다음 슬롯에 추가 검토. 페이즈 4 #4 결정.
10. **8-3 금단의 현자의 부작용**: "사망률 +5%" 컨셉. 페이즈 2에서 기존 `madman`(M2a)과 비교하여 강도 차별화.

## data-generator 지시사항

본 기획서로부터 **페이즈 3 #3 (`faction-trait` 데이터 35행 생성)**이 호출된다.

- **대상 타입**: `faction-trait` (data-generator 신규 타입 — 타입 스펙 작성 필요: `Docs/superpowers/data-generator/types/faction-trait.md`)
- **대상 테이블**: `traits` (M2a 기존, 109 → 144행)
- **생성 수량**: 35행 (기본 14 + 강화 14 + 정수 7)
- **컬럼 구조** (기존 traits 스키마 + 신규 컬럼 2개):
  ```
  id                       TEXT PK              -- 'fac_trait_<faction_id_short>_<tier>'
  key                      TEXT UNIQUE          -- 'guild_apprentice' / 'seasoned_adventurer' / ...
  name                     TEXT                 -- 한글 트레잇명
  category_id              INT (=9, Faction)
  type                     TEXT                 -- 'faction_basic' / 'faction_advanced' / 'faction_essence'
  description              TEXT                 -- 한글 1~2문장
  effect_text              TEXT                 -- 한글 효과 설명
  effect_json              JSONB                -- 효과 스키마 (페이즈 4 #4에서 확정)
  acquisition_condition    JSONB                -- 4중 AND 조건 (본 문서 6.2 스키마)
  faction_id               TEXT FK              -- factions.id
  faction_tier             INTEGER              -- 1 / 2 / 3
  ```
- **톤/세계관 가이드**:
  - 14세력 정체성에 직결되는 한국어 트레잇명 (본 문서 2.2 컨셉 표 참조)
  - description 1~2문장. 세력 정체성 + 용병 단위 서사 (예: "전사 길드 무도장에서 첫 검을 받았다")
  - effect_text는 명확한 보너스/패널티 표현 (예: "raid 성공률 +6%p")
  - "견습 → 베테랑 → 정수" 등급 어휘 일관 (예: "X 견습" / "노련한 X" / "전설의 X")
  - 외래어 남발 금지 (한국어 위주)
- **구조적 제약**:
  - 14세력 모두 기본 1종 + 강화 1종 (28행 보장)
  - 정수 7세력만 추가 1종 (7행)
  - faction_id 분포: 14세력 균등 (각 2~3행)
  - faction_tier 분포: 1 = 14 / 2 = 14 / 3 = 7
