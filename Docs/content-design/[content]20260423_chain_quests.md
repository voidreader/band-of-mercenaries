# 연계 퀘스트 시나리오 컨텐츠 기획서

> 작성일: 2026-04-23
> 유형: 신규 컨텐츠 (M3 페이즈 1-2)
> 후속 페이즈 의존: 페이즈 2-1(확정 보상 강도 밸런스), 페이즈 3-0-1(`types/chain-quest.md` 타입 스펙), 페이즈 3-1(`chain_quests` 벌크 생성 15~30단계), 페이즈 4-2(ChainQuestService spec)

## 개요

지역 조사를 끝까지 밀어붙이면 **1회성 서사 체인**이 열리고, 완주 시 **확정 장비**가 지급된다. 엘리트 반복 파밍(M2b)이 "자원 수집 루프"라면, 연계 퀘스트는 "**이 지역을 끝까지 판 사람만 보는 이야기**"다. 한 체인은 2~5단계로 이어지며, 단계마다 서사 훅과 보상이 제공되고, 단계 사이에는 "시간이 흘러 다음 단서가 드러난다"는 호흡이 있다.

이 문서는 **7개 체인(총 24단계)** 의 시나리오, 트리거, 단계 지연, 실패 재도전 규칙, 최종 보상 매핑, 대표 용병 선정 규칙(Q-2)을 확정한다. 구체 서사 텍스트는 페이즈 1-4(서사 매트릭스)와 통합되며, 수치 밸런스는 페이즈 2-1에서 balance-designer가 검토한다.

## 레퍼런스 분석

| 게임 | 참고 포인트 | 차용/변형 |
|------|-----------|----------|
| **Fallen London** | 스토리렛(Storylet) 분기 구조. 선택·진행에 따라 단서 조각이 쌓이며 "장기 서사" 형성 | 체인 단계별 완료 조건 + "다음 단서가 드러난다" 감성 |
| **Melvor Idle — Astrology 단계 해금** | 단계별로 선행 요구치를 쌓아 다음 단계를 연다 | 체인 단계의 knowledge_threshold 트리거는 1단계에만, 이후는 직전 단계 완료만 조건 |
| **Caves of Qud — 역사적 임무** | 지역 깊이 탐사가 최종 보물/전설 아이템으로 연결 | 체인 최종 단계 **확정 아이템** 드롭 — 확률 아님 |
| **Darkest Dungeon — Heirloom 수집** | 특정 지역 반복 탐사로만 얻는 전용 보상 자원 | 체인 보상은 엘리트 파밍과 다른 경로 (확정 vs 확률) |
| **특성 쌓는 김전사, 용마검전** | "주인공 서사를 따라가는 감성" — 프로듀서 레퍼런스 | 체인 `protagonistMercId` 고정으로 **"이 용병의 이야기"** 감성 |
| **Hades — 묶음 보상 구조** | 고난도 달릴 때마다 누적 보상, 마지막에 특별 보상 | 단계별 골드 보상 + 최종 장비 확정 |

**핵심 설계 원칙**: "**방치형 게임의 시간성 존중**" — 체인은 조급하게 달리는 구조가 아니다. 단계 사이 지연이 있고, 포기 불가능하며, 완주 시 확정 보상으로 인내를 보상한다.

## 상세 설계

### 1. 전체 구조 — 7체인 / 24단계

| # | 체인 ID | 체인 이름 | 단계 수 | 리전 티어 | 지역 이동 포함 | 세력 연결 | 최종 보상 |
|---|---------|----------|--------|-----------|----------------|-----------|----------|
| 1 | `chain_roadside_shrine` | 길가의 폐사당 | 2 | T1~T2 | ❌ (동일 리전) | 독립 | 철 투구 (T2) |
| 2 | `chain_windrunner_trail` | 질풍의 발자취 | 3 | T2~T3 | ✅ (1회 이동) | 독립 | 질풍의 가죽 부츠 (T3) |
| 3 | `chain_ironbound_pact` | 철갑의 서약 | 3 | T3 | ❌ | 공개(기사단형) | 사슬 흉갑 (T3) |
| 4 | `chain_blade_of_border` | 국경의 검 | 3 | T3~T4 | ✅ (1회 이동) | 공개(용병 연합형) | 강철 장검 (T3) |
| 5 | `chain_merchant_ledger` | 상인의 비망록 | 4 | T3~T4 | ✅ (2회 이동) | 지역(상인 조합형) | 황금 저울 (T3 용병단) |
| 6 | `chain_forge_masters` | 장인의 유산 | 4 | T4 | ✅ (1회 이동) | 지역(은세공 길드형) | 단련의 은반지 (T4) |
| 7 | `chain_soul_severance` | 혼을 끊는 자 | 5 | T4~T5 | ✅ (2회 이동) | 비밀(영혼 결사형) | **멸혼결 (T5 전설)** |

**총 단계 수**: 2+3+3+3+4+4+5 = **24단계** (로드맵 15~30 범위)
**단계 수 분포**: 2단(1), 3단(3), 4단(2), 5단(1) — 리듬 다양성 확보
**리전 티어 분포**: T1~T2(1) / T2~T3(1) / T3(1) / T3~T4(2) / T4(1) / T4~T5(1) — 초중반 진입 장벽 낮게, 후반 깊이 있게
**지역 이동 체인**: 7체인 중 5개 (71%). M3 로드맵의 "대륙을 횡단하며 3단계 서사" 감성 구현
**최종 보상 커버리지**: M2a 장비 10종 중 7종 매칭 (개인 6 + 용병단 1). 나머지 3종(깃발·뿔피리·수호자방패장식)은 엘리트 파밍(M2b)과 M4 이후 경로로 분리

### 2. 체인별 상세 설계

#### 체인 1: 길가의 폐사당 (`chain_roadside_shrine`)

**리전 티어**: T1~T2 (입문 리전 중 `region.tier ≤ 2`)
**세력 연결**: 독립 (어느 세력도 연계되지 않음 — 입문용 튜토리얼적 체인)
**발동 트리거**: 조사 지식 포인트 60 도달 시 `region_discoveries.hidden_quest`로 노출

| 단계 | 지역 | 유형 | 난이도 | 전투력 | 보상 | 서사 훅 |
|------|------|------|--------|--------|------|---------|
| 1 | 현재 리전 | 탐험 | 2 | 20~30 | 120G | "{region.name}의 폐사당 입구가 오랜 먼지에 덮여 있다. 용병 하나가 문고리에 새겨진 문양을 알아본다." |
| 2 (최종) | 현재 리전 | 토벌 | 3 | 40~55 | 300G + **철 투구** | "사당 깊은 곳, 옛 수호자의 뼈가 여전히 투구를 쓴 채 앉아 있다. {merc.name}은 예를 갖춰 투구를 건네받는다." |

**next_step_delay**: 단계 1 완료 후 **10분** 경과 → 단계 2 자동 노출

**의도**: 입문 플레이어에게 체인 시스템 첫 경험 제공. 동일 리전 내에서 완결되어 이동 비용 없음. 보상 철 투구(T2)는 입문 장비로 즉시 활용 가능.

| 5 (최종) | T5 리전 (단계 4 동일) | 토벌 | 5 | 190~220 | 1,500G + **멸혼결** | "옛 전사가 혼을 끊어 만들었다는 부적. 새로운 주인이 그것을 건네받는다. [if has_trait:brave]{merc.name}은 눈을 감지 않는다.[/if]" |

**next_step_delay**: 단계 1 → 2: **3시간**, 단계 2 → 3: **2시간**, 단계 3 → 4: **4시간**, 단계 4 → 5: **6시간**

**의도**: **엔드게임 체인**. 5단계 · 2회 이동 · 최고 임계 트리거로 진입 장벽이 높다. 보상 **멸혼결(T5 전설)** 은 엘리트 파밍에서도 낮은 확률로 획득 가능하나, 이 체인은 **확정 경로**. 서사 호흡이 긴 총 15시간의 느긋한 여정(대기 15시간 + 실제 플레이 시간 별도).

### 3. 발동 트리거 (`region_discoveries` 연동)

M3의 `region_discoveries` 테이블에 체인 발동 진입점을 추가한다.

```
discovery_type: 'hidden_quest'
discovery_data: {
  "chain_id": "chain_roadside_shrine",
  "start_step": 1,
  "protagonist_hint": "polish"  -- 선택: 특정 직업군 용병이 파티에 있으면 서사 인트로 변형
}
```

**knowledge_threshold 분포**:

| 체인 | 임계 | 근거 |
|------|------|------|
| 길가의 폐사당 | 60 | 입문 — 조사 초중반 도달 |
| 질풍의 발자취 | 70 | 중급 진입 |
| 철갑의 서약 | 75 | 중급 |
| 국경의 검 | 75 | 중급 |
| 상인의 비망록 | 80 | 중상급 |
| 장인의 유산 | 85 | 상급 |
| 혼을 끊는 자 | 95 | 엔드 |

**발동 리전 선정 원칙**:
- 체인 1~2는 **T1~T3 어느 리전에도 배치 가능** (페이즈 3-2 벌크 생성 시 복수 리전에 배치하여 유저별 진입점 다양화)
- 체인 3~6은 **테마에 맞는 리전에 배치** (예: 국경의 검 → 국경 인접 리전, 상인의 비망록 → 대상로/항구 리전)
- 체인 7은 **2~3개 특정 T4 리전에만 배치** (엔드게임 희소성)

**중복 체인 방지**:
- 1개 리전에 동일 체인의 `hidden_quest` discovery는 1개만 배치
- 완주한 체인은 같은 리전에서 재발동되지 않음 (`FactionStateRepository` 패턴 유사 — `completedChains: Set<String>`을 `user` Hive 박스에 저장)
- 다른 리전에 같은 `chain_id`가 배치되어 있으면 여전히 그 리전에서 발동 가능? → **아니오**. chain_id는 유저 단위로 1회 완주 후 영구 완료 (복수 리전에 배치는 "어느 리전에서든 먼저 조건 달성한 유저가 시작" 의미)

### 4. `next_step_delay` 설계

**단위**: 초 단위 정수(`int`), `chain_quests.next_step_delay` 필드에 저장. Dart 측 `Duration(seconds: …)` 변환.
**시간 기준**: **실제 시간**. `ChainQuestProgress.currentStepAvailableAt: DateTime`에 기록하여 gameTickProvider(1초)가 도달 여부 체크.
**시간 가속**: `SettingsKeys` 시간 가속 설정에 **포함** (개발·테스트 편의). 기존 퀘스트/이동/건설과 동일 방식으로 endTime 비례 재계산.

**왜 실제 시간 기준인가:**
- 방치형 게임의 특성 — "세계가 숨쉬는 감성"은 실제 시간 경과에서 나옴
- 플레이어가 앱을 꺼도 체인이 진행되는 편안함
- 게임 틱 시간 기준으로 하면 대기 시간을 우회하기 위해 시간 가속을 남용하게 됨

**지연 분포 표**:

| 체인 티어 | 단계 간 지연 범위 | 체인 전체 누적 지연 |
|----------|-----------------|-------------------|
| 입문 (T1~T2) | 10~30분 | 10분 |
| 중급 (T2~T4) | 30분~2시간 | 1.5~5시간 |
| 상급 (T4) | 1.5~3시간 | 6~7시간 |
| 엔드 (T4~T5) | 3~6시간 | 15시간 |

**마지막 단계 지연**: 없음 (`next_step_delay = 0` 또는 NULL). 최종 보상은 즉시 지급.

**완주 소요 감각**:
- 입문 체인: 10분 대기 + 2회 퀘스트 = 약 30~60분
- 엔드 체인: 15시간 대기 + 5회 퀘스트 + 이동 = 약 20시간 (실시간 기준, 플레이어는 간간이 확인)

### 5. 실패 재도전 규칙

- **단계 실패 시**: 해당 단계만 재도전. 재도전 시 **불이익 없음** (로드맵 "실패 패널티 최소화" 기조). 난이도·전투력·보상 그대로.
- **체인 포기**: **불가능.** 체인은 한 번 발동되면 완주해야 한다. 이유:
  - 포기 허용 시 "어려운 체인을 포기하고 재롤링 시도"가 어뷰징 경로가 됨
  - 체인은 1회성 서사이므로 "도중 포기 = 서사 실패" 감성
- **체인 휴면 상태**:
  - `currentStepAvailableAt` 경과 후 **7일간** 체인 단계 미수행 → 체인이 "휴면" 상태로 전환
  - 휴면 체인은 활동 로그에서 더 이상 강조되지 않지만 **언제든 해당 단계 수행 시 자동 재개**
  - 휴면은 포기가 아니다. 진행은 유지되며 단지 UI의 강조가 약해질 뿐
- **용병 전원 사망 시**: 체인 단계 실패로 처리. 단계 재도전 가능. 주인공 용병(§7) 사망 시 다음 단계 수행 파티 최고 기여자로 대체.

### 6. 최종 확정 보상 — 체인 ↔ 아이템 매핑 근거

| 체인 | 매칭 아이템 | 매칭 근거 |
|------|-----------|---------|
| 길가의 폐사당 | 철 투구 (T2) | 옛 수호자의 투구 — 사당 서사와 정합, 입문 장비 |
| 질풍의 발자취 | 질풍의 가죽 부츠 (T3) | 이름 직접 대응. 질주 감성 |
| 철갑의 서약 | 사슬 흉갑 (T3) | 철갑 — 방어 서사와 정합 |
| 국경의 검 | 강철 장검 (T3) | 전장의 부러진 검 → 새 주인에게 이어지는 검 |
| 상인의 비망록 | 황금 저울 (T3 용병단) | 저울의 받침대에 숨겨진 비망록 — 서사 정합 |
| 장인의 유산 | 단련의 은반지 (T4) | 은세공 장인의 유산 — 이름·컨셉 직접 대응 |
| 혼을 끊는 자 | 멸혼결 (T5 전설) | 이름·서사·전설 티어 완전 정합 |

**엘리트 파밍(M2b)과의 관계**:
- M2b 엘리트 드랍 테이블에서도 장비·유물은 **확률적**으로 나옴 (로드맵: 장비 15~25%, 유물 3~8%)
- 체인 최종 보상은 **확정 드랍**. 즉, 연계 퀘스트는 "이 장비를 확실히 원하는 유저"에게 경로를 제공
- 멸혼결처럼 전설 아이템은 엘리트에서도 나오지만 극히 희소. 체인 완주는 확정 획득 경로

**커버되지 않는 M2a 장비 3종**:
- 용병단 깃발 (T3) — 엘리트 드랍 or M4 세력 거점 상점
- 명예의 뿔피리 (T4) — 엘리트 드랍 or M4 세력 거점 상점
- 수호자의 방패 장식 (T5) — 엘리트 레어 드랍 (엔드게임 희소성 유지)

**페이즈 2-1 밸런스 검토 포인트** (사전 제시):
- 체인 7개 완주에 소요되는 플레이 시간 vs 엘리트 반복 파밍으로 동등 장비 수집 시간
- 체인 보상 골드 총합(약 9,450G) + 확정 장비 vs 엘리트 파밍 기대 골드
- 각 체인의 단계 보상이 거리 비용을 충분히 상회하는가 (체인 5·6·7 특히)

### 7. Q-2 결론 — 대표 용병(`merc.*`) 선정 규칙

**결론: (e) 체인 시작 시점에 대표 용병 1명을 자동 고정 + 사망 시 폴백**

**동작**:
1. 체인 단계 1 완료 시 해당 단계 파티 내 **최고 기여 용병**(partyPower 비중 최대)을 `ChainQuestProgress.protagonistMercId`에 저장
2. 이후 모든 단계의 서사 렌더링에서 `merc.*`은 이 용병 기준으로 바인딩
3. 주인공 용병이 **사망** 또는 **방출**되면: 다음 단계 수행 시점의 파티 최고 기여자로 **자동 대체**하고 `protagonistMercId` 갱신. 활동 로그에 "이야기의 주인공이 바뀌었다" 기록
4. 주인공이 해당 단계 파티에 포함되지 않은 경우라도 서사 `merc.*`은 여전히 저장된 주인공 기준 (예: 조사나 특정 임무로 다른 용병을 쓰더라도 체인 주인공은 변하지 않음). 이것이 오히려 "이 이야기는 누구의 이야기인가"를 명확히 함

**왜 (b)나 (d)가 아닌가**:
- (b) 단계별 최고 기여: 서사 일관성 낮음. "1단계는 A의 이야기, 2단계는 B의 이야기"가 됨 → 소설처럼 읽히지 않음
- (d) 체인 전체 누적 기여: 매 단계 완료 시 전체 누적을 재계산해야 함. 주인공이 중간에 바뀌는 느낌. 복잡성 대비 효용 낮음
- (e) 고정: 단순성 + 일관성. "김철수 용병의 이야기"로 유저가 인식. 주인공 용병이 사망하면 서사적으로 "후계자가 잇는" 감성이 됨

**TemplateEngine 반영**:
- `merc.*` 네임스페이스는 체인 퀘스트 서사 렌더 시 `protagonistMercId` 기준
- 일반 퀘스트 서사(체인 외)는 TemplateEngine 기존 권장대로 파티 내 최고 기여자 (체인이 아닌 일회성 서사)

**페이즈 4-2 spec 반영 필요사항**:
- `ChainQuestProgress` Hive 모델에 `protagonistMercId: String` 필드 추가
- `ChainQuestService.onStep1Complete()` 시 자동 바인딩 로직
- `ChainQuestService.getProtagonist()` 폴백 로직 (사망/방출 체크 → 다음 단계 파티 최고 기여자로 갱신)

### 8. 공존 정책 힌트 (페이즈 1-6 상세화 예고)

파견 화면에서 연계 퀘스트가 표시될 때의 규칙 **힌트만** 제시. 최종 UI 결정은 페이즈 1-6.

- **표시 순서 (최상단부터)**: 연계 진행 단계 > 세력 전용(M1) > 엘리트(M2b) > 일반
- **체인 단계 강조**:
  - 좌측 사이드바 색상: 보라/금 (체인 티어별 그라데이션)
  - 상단 배지: "🔗 연계 {N}/{M}" 진행도 바
  - 이름 아래: 체인 이름 서브텍스트 "길가의 폐사당 · {N}/{M}단계"
- **타 지역 이동 필요 단계**: 카드 전체에 "📍 {target_region.name}으로 이동하여 수행" 오버레이 + 비활성 상태
- **next_step_delay 대기 중**: 카드 잠금. "💭 {남은 시간} 후 다음 단서가 드러납니다" 표시 (실제 시간 카운트다운)

## 현재 시스템과의 연관

### 영향받는 기존 시스템

| 시스템 | 영향 내용 |
|--------|---------|
| `region_discoveries` | `discovery_type = 'hidden_quest'` 행 추가 (페이즈 3-2). 기존 faction_clue 발견과 동등 레벨 신규 타입 |
| `QuestGenerator` | 체인 단계를 일반 퀘스트 풀과 다른 별도 경로로 주입. 파견 화면 최상단 슬롯 예약 필요 |
| `QuestCompletionService` | 체인 단계 완료 감지 → 다음 단계 자동 생성 로직 호출. 최종 단계 완료 시 확정 아이템 인벤토리 추가 |
| `gameTickProvider` | `currentStepAvailableAt` 도달 여부 1초 간격 체크 → 다음 단계 노출 |
| `TravelService` | 체인 단계의 `target_region_id`가 현재 리전과 다르면 이동 필요 안내 |
| `UserData` Hive 박스 | `completedChains: Set<String>` 필드 추가 (완주 체인 중복 방지) |
| `chainQuestProgress` Hive 박스 | **신규 박스** (M3 로드맵 명시). 체인별 진행 상태 |
| TemplateEngine | 체인 단계 서사 텍스트가 `{merc.name}`, `[if joined_faction:...]`, `[pick]` 사용 |

### 호환성 리스크

- **낮음**: 기존 `ActiveQuest`와 체인 단계 퀘스트의 구조 차이는 `isChainStep: bool` 플래그만 추가하면 해결. 기존 퀘스트 완료 플로우 재사용
- **중간**: 파견 화면에서 최상단 슬롯 예약으로 기존 퀘스트 5~8개 슬롯 중 1개를 대체하는지, 추가로 표시하는지 결정 필요 → 페이즈 1-6에서 확정
- **낮음**: `protagonistMercId` 용병이 체인 도중 제3자(예: 여행 이벤트로 사망) 조건에서 활동 로그 가독성 — 로그 메시지로 명확히 표기 필요

### 신규 인프라 (페이즈 4-2 spec에서 구체화)

- `ChainQuestData` Freezed 정적 모델 — `chain_id`, `region_id`, `step`, `target_region_id`, `name`, `description`, `combat_power`, `duration_minutes`, `reward_gold`, `reward_items`(JSONB), `final_reward`(bool), `next_step_delay`(int, 초)
- `ChainQuestProgress` Hive 모델 — `chainId`, `currentStep`, `status`(active/completed/dormant), `startedAt`, `protagonistMercId`, `currentStepAvailableAt`, `stepFailureCount`(재도전 추적용, UI 힌트만)
- `ChainQuestService` — 발동/단계 진행/완료/휴면/폴백 로직
- `UserData.completedChains` 확장 — 완주 체인 ID Set

## data-generator 지시사항

> 이 기획서는 페이즈 3-0-1(`types/chain-quest.md` 타입 스펙)과 페이즈 3-1(`chain_quests` 벌크 생성)의 입력이 된다.

- **대상 타입**: `chain-quest` (신규 — 타입 스펙 작성 필요)
- **대상 테이블**: `chain_quests` (신규 Supabase 테이블)
- **생성 수량**: **24행** (체인 7개, 각 2~5단계)
- **톤/세계관 가이드**:
  - 한국어 판타지 톤. "특성 쌓는 김전사"/"용마검전" 레퍼런스 감성 유지
  - 서사 훅(`description`)은 1~2문장, TemplateEngine 문법 활용(`{merc.name}`, `{region.name}`, `{merc.job}`, `[if joined_faction:<id>]...[/if]`, `[pick]A|B[/pick]`)
  - 고유명사는 저작권 금칙 준수 (한국 웹소설 고유명사 미사용)
- **구조적 제약**:
  - 7개 `chain_id`는 §2 표 그대로 고정
  - 각 체인의 단계 수·지역 이동 유무·세력 연결 §2 표 그대로 준수
  - `combat_power`, `reward_gold`, `duration_minutes`, `next_step_delay`는 §2 각 체인 표의 값을 범위 중간값으로 채우고 balance-designer 검토 대상으로 표시
  - `reward_items` JSONB: 단계별 장비 드롭 없음 (최종 단계만 확정 아이템). 중간 단계는 NULL 또는 `{}`
  - `final_reward`: bool, 최종 단계만 true
  - 최종 단계의 `reward_items`에 §6 표의 M2a 장비 ID 정확히 기입 (예: `equip_helmet_iron_helm`)
  - 1단계 `target_region_id` = `region_id` (현재 리전 시작)
  - 지역 이동 포함 단계: `target_region_id ≠ region_id`
- **수치 출처**:
  - combat_power, reward_gold, next_step_delay: 본 기획서 §2 + 페이즈 2-1 balance-designer 검토
  - duration_minutes: 기존 난이도별 소요 시간 + 거리 보정 공식 활용
- **특수 요구**:
  - 체인 1~2는 복수 리전에 `region_discoveries.hidden_quest` 배치 (페이즈 3-2에서 처리)
  - 체인 3~7은 테마 맞는 리전 소수에만 배치
  - 주인공 용병 바인딩은 런타임 로직, 데이터 생성 단계에서 다룰 필요 없음

## 오픈 질문

- **Q-1 (체인 휴면 기간 7일 적절성)**: 7일 미수행 후 휴면은 너무 관대한가 / 너무 빡빡한가. 방치형 특성 고려 시 **14일 이상**이 더 적합할 수 있음. → **페이즈 2-1 또는 구현 단계 경험치 기반 조정**

- **Q-2 (인벤토리 가득 시 확정 보상 처리)**: 체인 최종 단계 완료 시 인벤토리(M2a 정의)가 가득 차면 확정 아이템을 어떻게 처리하는가.
  - (a) 드롭 손실 — 유저 좌절 큼
  - (b) "우편함" 시스템으로 임시 보관 (M2a에 없는 시스템)
  - (c) 빈 자리 확보 후 완료 단계 수행 가능 — 사전 차단
  - → **권장 (c)**: 최종 단계 수행 전 "인벤토리에 1칸 여유가 필요합니다" 경고. 페이즈 4-2 spec에서 확정

- **Q-3 (체인 단계 중 지역 이동 대기 UX)**: 단계 2가 타 리전에서 수행되는 체인에서, 유저가 해당 리전으로 이동하는 동안 `next_step_delay`가 흘러도 되는가.
  - (a) 이동 중에도 delay 흐름 (기본 권장 — 이동 시간을 활용)
  - (b) 이동 중에는 delay 정지 (형평성 차원)
  - → **권장 (a)**: 실제 시간 기반 delay이므로 이동 중에도 흐름. 이것이 자연스러운 리듬

- **Q-4 (동일 리전 다중 체인 발동 순서)**: 1개 리전에 2개 체인의 `hidden_quest` discovery가 배치되고 임계값이 동일하면 동시 발동되는가 / 순차 발동되는가.
  - (a) 모두 동시 발동 — 파견 화면 상단이 혼잡
  - (b) 우선순위 지정(chain_id 또는 임계값 차이)으로 1개씩
  - → **권장 (b)**: 페이즈 3-2에서 `region_discoveries`의 임계값을 서로 다르게 배치(예: 75, 80)하여 자연 순차 발동. 페이즈 1-6 공존 정책에서 상세화

- **Q-5 (체인 단계 실패 시 보상 지급 여부)**: 단계 실패 시에도 소정의 참가 보상(XP 등) 지급이 정착 유도에 도움되는가.
  - → **권장**: 실패 시 일반 퀘스트 실패와 동일 처리 (보상 0, 용병 부상). 체인 단계라고 특별 대우 없음. 실패 재도전 때 다시 정상 보상.

- **Q-6 (세력 연결 매칭의 구체적 세력 ID)**: §2에서 "공개 세력(기사단형)", "지역 세력(상인 조합형)" 등 추상적으로만 표기. 실제 14개 세력 중 어느 세력과 매칭할 것인가. → **페이즈 1-4(서사 매트릭스) 또는 페이즈 3-1(데이터 생성) 때 `factions` 테이블의 실제 id 참조하여 확정**. 이 기획서는 컨셉 레벨에서 멈춤
