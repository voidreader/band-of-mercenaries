## 1. 개요

M8.5의 5 시스템(생활권 대시보드·솔로 의뢰·감정 반응·히든 스탯·전투 기억)이 모두 **개인 게임 진행**의 가시화였다면, 주간 기여도 랭킹은 그 진행을 **다른 플레이어와 비교 가능한 형태**로 한 번 더 가시화한다. 같은 게임을 플레이하는 다른 용병단들 사이에서 "나의 이번 주 활약은 어디쯤인가"를 가볍게 확인할 수 있는 비공격적 경쟁 채널을 도입한다.

**PvP 손실 없음** 원칙이 가장 중요하다. 다른 사용자의 데이터를 읽지도 변경하지도 않는다. 단순히 자기 점수를 Supabase에 업로드하고, top 100 익명 목록을 lookup하여 보여준다. 자기 게임 진행에 다른 사용자가 영향을 주는 일은 없으며, 사용자가 랭킹에 참여하지 않아도 게임 자체는 완전 동작한다.

핵심 결정 사항:
1. **MVP 범위**: 5항목 점수 합산 / 익명 랭킹 / 보상 없음 / 보존 정책 단순(현재 주 + 지난 주 1개). 친구 비교·시즌·보상 시스템은 M9 이후 위임.
2. **5 점수 항목**: 지역 안정화 (M7 dangerScore 감소 + infraTier 증가) / 세력 의뢰 (faction_named 완수 × difficulty) / 전투 활약 (M8b decisive scene + crit + 솔로 의뢰 대성공) / 제작 (M5 crafted 아이템 tier 합산) / 위업 (주중 BandAchievement 발급 수 × tier).
3. **주간 사이클**: **UTC 월요일 0:00** 갱신 (KST 월요일 09:00). 글로벌 일관성 + i18n 대비.
4. **업로드 빈도**: 5분 throttle. dirty flag 누적 → 5분마다 Supabase RPC 호출 + 강제 갱신 버튼.
5. **인증 전제**: 업로드는 Supabase 익명 인증 세션이 있을 때만 수행한다. 현재 앱은 Supabase client 초기화만 존재하므로, 페이즈 4 #6에서 `signInAnonymously()` 부트스트랩을 명시한다. 인증 실패 시 로컬 점수 집계와 랭킹 조회는 유지하되 업로드는 보류한다.
6. **익명성**: 사용자 ID 부분 표시 (`u_{first_8_chars}`). 닉네임·프로필 시스템 도입 없음. M9에서 검토.
7. **노출 위치 2개**: 정보 탭 메인 화면 (`WeeklyRankingScreen`) — top 100 + 자기 5항목 분해 / 홈 화면 작은 위젯 — 자기 순위 + 상위 X% (탭 시 정보 탭 점프).
8. **신뢰 모델**: MVP는 클라이언트 신뢰 (위조 가능). RLS는 본인만 자기 행 INSERT/UPDATE, 모든 사용자 SELECT 가능. M9에서 서버측 사건 검증 위임.

**핵심 설계 원칙**:
- **"비파괴적 비교"** — 다른 사용자의 게임에 영향을 줄 수 있는 어떤 채널도 만들지 않는다. 데이터는 일방향: 클라이언트 → Supabase. 다른 사용자 데이터는 read-only lookup만.
- **"종합이 아니라 분해"** — 단일 합산 점수만 보여주면 의미가 부족하다. 5항목 분해를 함께 표시하여 "내가 어떤 활동을 많이 했는가"를 가시화한다.
- **"가벼운 채널"** — 보상·시즌·친구·프로필 같은 무거운 시스템 도입 없음. M8.5 MVP는 "있으면 좋고, 없어도 게임 동작" 부가 기능. 정보 탭에 진입하지 않으면 사용자는 랭킹 존재를 거의 인지하지 못한다 (홈 위젯만 작게 표시).
- **"5 항목은 M3~M8b 시스템의 통합 가시화"** — 본 문서는 신규 시스템을 만들지 않는다. 기존 시스템의 이벤트를 모아 점수로 요약할 뿐.

## 3. 상세 설계

### 3.1 5 점수 항목 정의

#### 3.1.1 표

| 항목 | 카운트 사건 | 1회당 점수 | 출처 시스템 |
|------|----------|---------------|----------|
| **지역 안정화** | M7 핵심 7리전의 `dangerScore` 감소 1회 + `unlockedFlags` 1개 추가 + `infraTier` 1단계 상승 | 50 / 200 / 500 | M7 거점화 (페이즈 4 #1·#4) |
| **세력 의뢰** | `faction_named` 의뢰 완수 (성공·대성공만) | difficulty × 50 (1~5 → 50~250) | M8a 페이즈 4 #1 세력 수직 절편 |
| **전투 활약** | M8b decisive scene 기여 점수 ≥ 5 + critical 발동 + 솔로 의뢰 대성공 | 100 / 3 / 750 | M8b CombatSimulator (페이즈 4 #1), 페이즈 2 #5 보정 |
| **제작** | M5 crafted 아이템 1개 | tier × 30 (1~5 → 30~150) | M5 페이즈 4 #2 제작 시스템 |
| **위업** | 주중 발급된 `BandAchievement` 1개 | category별 100~500 (chain 300 / settlement_event 500 / reputation_rank 200 / elite_unique 400 / craft_first_rare 100 / memorial 100 / region_pacified 250 / infrastructure_growth 500) | M6 페이즈 4 #1 위업·연대기 |

본 표의 점수 강도는 페이즈 2 #5 시뮬레이션 검증값을 반영한 확정 default다.

#### 3.1.2 합 점수 계산

```
total_score = stabilization_score
            + faction_quest_score
            + combat_score
            + crafting_score
            + achievement_score
```

각 항목의 주간 누적 × 가중치 단순 합. 페이즈 2 #5에서 항목별 주 cap을 도입했으며, 세부 cap은 `WeeklyContributionScoreConfig.weeklyCategoryCaps`가 관리한다.

#### 3.1.3 예시 시뮬레이션 (페이즈 2 #5 검증 입력)

평균 신규 유저 1주일 플레이 (~10시간) 예상 점수:
- 지역 안정화: dangerScore decay 5회 + flag toggle 1개 + infraTier 1단계 = 50×5 + 200 + 500 = 950
- 세력 의뢰: faction_named 3회 (avg difficulty 2~3) = 125×3 = 375
- 전투 활약: decisive scene 10회 + critical 100회 + 솔로 0회 = 100×10 + 3×100 + 0 = 1300
- 제작: crafted 3개 (T1 1회 + T2 2회) = 30 + 60×2 = 150
- 위업: 2개 (chain 1 + reputation_rank 1) = 300 + 200 = 500

**예상 합**: 3,275 점

베테랑 유저 1주일 플레이 (~20시간):
- 지역 안정화: 2,500
- 세력 의뢰: 1,500
- 전투 활약: 5,000
- 제작: 1,000
- 위업: 1,500

**예상 합**: 11,500 점

신규~베테랑 분포 ~3배. 페이즈 2 #5에서 시뮬레이션 정밀화.

### 3.2 카운트 사건 정확한 매핑

#### 3.2.1 지역 안정화

`RegionStateRepository`의 다음 메서드에 trailing 추가:
- `addDangerScore({regionId, delta, source, ref})` — delta < 0 (위험도 감소)일 때 `weeklyScoreBuffer['stabilization'] += 50`
- `toggleFlag({regionId, flag, ref})` — 신규 flag 추가 시 `+= 200`
- `_evaluateInfrastructureTransition` — tier 진입 시 `+= 500`

대상 region 한정 없음 (모든 region 통계). 다만 M7 7리전만 카운트하는 것도 검토 가능 (페이즈 2 #5 결정).

#### 3.2.2 세력 의뢰

`QuestCompletionService` 본체 — 세력 지명/세력 전용 의뢰가 성공·대성공으로 끝났을 때:
- `pool.specialFlags['faction_named'] == true`를 1차 후보로 사용
- 실제 구현에서는 M8a 세력 수직 절편의 기존 `QuestPool` 필드(`factionTag`, specialFlags, named hook)를 확인해 정확한 predicate를 페이즈 4 #6에서 고정
- `weeklyScoreBuffer['faction'] += difficulty × 50`

#### 3.2.3 전투 활약

M8b `CombatSimulator.simulate`는 순수 계산 성격을 유지한다. 직접 `UserData`를 수정하지 않고, 결과 모델에 `weeklyContributionDelta`를 실어 `QuestCompletionService`가 한 번에 반영한다.

- 결정적 장면 기여자 score ≥ 5인 mercenary 1명당 `weeklyContributionDelta['combat'] += 100`
- critical 발동 (해당 라운드 mercenary critical 통과) 1회당 `+= 3` (페이즈 2 #5 보정값)
- `CombatSimulationResult.weeklyContributionDelta`는 페이즈 4 #6에서 HiveField 15를 사용한다. HiveField 13은 히든 스탯 `hiddenStatEvents`, 14는 전투 기억 `battleMemoryEvents`가 예약한다.

`QuestCompletionService`:
- `CombatSimulationResult.weeklyContributionDelta`를 `UserData.weeklyScoreBuffer`에 합산
- 솔로 의뢰 (`pool.partySizeMax == 1`) 대성공 1회당 `weeklyScoreBuffer['combat'] += 750` (페이즈 2 #5 보정값)

#### 3.2.4 제작

`CraftingService.craft(recipeId)` 본체 — 아이템 생성 직후:
- `weeklyScoreBuffer['crafting'] += tier × 30`

#### 3.2.5 위업

`AchievementService.grant(templateId, mercSnapshot?, regionId?, payload?)` 본체 trailing:
- 위업 카테고리별 score 매트릭스 lookup → `weeklyScoreBuffer['achievement'] += matrix[category]`

#### 3.2.6 weeklyScoreBuffer

`UserData`에 주간 점수 필드 4개를 추가한다. 현재 `foreignStallVisitCount`가 HiveField 27을 사용하고, 페이즈 4 #1 생활권 대시보드가 `shortGoalPinId`/`longGoalPinId`로 HiveField 28~29를 예약한다. 따라서 주간 기여도 필드는 30부터 시작한다.

```dart
@HiveField(30)
Map<String, int> weeklyScoreBuffer = {
  'stabilization': 0,
  'faction': 0,
  'combat': 0,
  'crafting': 0,
  'achievement': 0,
};

@HiveField(31)
DateTime? weeklyScoreWeekStartUtc;

@HiveField(32)
DateTime? weeklyScoreLastUploadedAt;

@HiveField(33)
bool weeklyScoreDirty = false;
```

`weeklyScoreBuffer`는 5 점수 키만 담는다. ISO 문자열이나 dirty flag를 `Map<String, int>`에 섞으면 타입이 깨지므로 주간 시작 시각·업로드 시각·dirty flag는 별도 HiveField로 분리한다. 이 필드까지 추가한 뒤 `UserData`의 다음 가용 HiveField는 34.

### 3.3 주간 사이클

#### 3.3.1 UTC 월요일 0:00 갱신

매주 UTC 월요일 0:00을 주간 경계로 정의. KST 기준 월요일 09:00.

```dart
DateTime currentWeekStart() {
  final nowUtc = DateTime.now().toUtc();
  final daysSinceMonday = (nowUtc.weekday - DateTime.monday + 7) % 7;
  return DateTime.utc(nowUtc.year, nowUtc.month, nowUtc.day - daysSinceMonday);
}
```

#### 3.3.2 주간 경계 감지

앱 시작 시 + 매 의뢰 완료 직후 검사:
```dart
final weekStart = currentWeekStart();
if (userData.weeklyScoreWeekStartUtc == null ||
    !userData.weeklyScoreWeekStartUtc!.isAtSameMomentAs(weekStart)) {
  // 새 주 시작 → 지난 주 점수를 서버에 final upload 후 buffer reset
  await uploadIfDirty(force: true);
  userData.weeklyScoreBuffer = freshWeeklyScoreBuffer();
  userData.weeklyScoreWeekStartUtc = weekStart;
  userData.weeklyScoreDirty = true;
}
```

#### 3.3.3 데이터 보존

Supabase `weekly_contributions` 테이블 (§3.5)에 클라이언트별 행 1개 + 지난 주 1개 = 최대 2주 보존. pg_cron으로 2주 이상 자동 삭제.

#### 3.3.4 시간 가속 정합

현재 게임은 설정에서 시간 가속 가능 (개발용). 주간 사이클은 실제 시간(`DateTime.now().toUtc()`) 사용. 시간 가속과 무관.

### 3.4 업로드 빈도 — 5분 throttle

#### 3.4.1 dirty flag 패턴

```dart
void incrementScore(String category, int delta) {
  userData.weeklyScoreBuffer[category] =
    (userData.weeklyScoreBuffer[category] ?? 0) + delta;
  userData.weeklyScoreDirty = true;
  scheduleUploadIfDirty();
}
```

`scheduleUploadIfDirty()`는 timer 기반 5분 throttle:
- 마지막 업로드 후 5분 미경과 → schedule만 (실행 안 함)
- 5분 경과 → Supabase RPC 호출 + `weeklyScoreDirty = false` + `weeklyScoreLastUploadedAt = nowUtc`

#### 3.4.2 강제 갱신 버튼

`WeeklyRankingScreen` 우상단 [⟳ 새로고침] 버튼. throttle 무시하고 즉시 업로드 + 랭킹 재조회. 사용자 자율 제어.

#### 3.4.3 오프라인 처리

업로드 실패 시 `weeklyScoreDirty = true` 유지. 다음 5분 후 또는 강제 갱신 시 재시도. 실패가 누적되어도 `weeklyScoreBuffer`는 클라이언트 영속, 점수 손실 없음.

### 3.5 Supabase 신규 테이블 — `weekly_contributions`

#### 3.5.1 스키마

| 컬럼 | 타입 | 제약 | 의미 |
|------|------|------|------|
| `user_id` | UUID | NOT NULL REFERENCES auth.users(id) | Supabase 익명 인증 user ID |
| `week_start_utc` | DATE | NOT NULL | UTC 월요일 0:00의 날짜 (예: 2026-05-18) |
| `stabilization_score` | INT | NOT NULL DEFAULT 0 CHECK >= 0 | |
| `faction_score` | INT | NOT NULL DEFAULT 0 CHECK >= 0 | |
| `combat_score` | INT | NOT NULL DEFAULT 0 CHECK >= 0 | |
| `crafting_score` | INT | NOT NULL DEFAULT 0 CHECK >= 0 | |
| `achievement_score` | INT | NOT NULL DEFAULT 0 CHECK >= 0 | |
| `total_score` | INT | GENERATED ALWAYS AS (`stabilization_score + faction_score + combat_score + crafting_score + achievement_score`) STORED | 자동 계산 |
| `updated_at` | TIMESTAMPTZ | NOT NULL DEFAULT now() | 마지막 업데이트 시각 |
| PRIMARY KEY | (user_id, week_start_utc) | | 사용자×주 유일 |

GENERATED 컬럼으로 total_score 자동 계산 → 클라이언트가 total을 따로 보낼 필요 없음.

#### 3.5.2 DDL + 인덱스

```sql
CREATE TABLE weekly_contributions (
  user_id UUID NOT NULL REFERENCES auth.users(id),
  week_start_utc DATE NOT NULL,
  stabilization_score INT NOT NULL DEFAULT 0 CHECK (stabilization_score >= 0),
  faction_score INT NOT NULL DEFAULT 0 CHECK (faction_score >= 0),

    stabilization_score
    + faction_score
    + combat_score
```

#### 3.5.5 RPC 함수 (MVP 채택)

직접 INSERT/UPDATE 대신 RPC `upsert_weekly_score(p_stab INT, p_fac INT, p_com INT, p_cra INT, p_ach INT)`:

```sql
CREATE FUNCTION upsert_weekly_score(
  p_stab INT, p_fac INT, p_com INT, p_cra INT, p_ach INT
) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_week_start DATE := date_trunc('week', timezone('UTC', now()))::date;
BEGIN
  IF auth.uid() IS NULL THEN
    RAISE EXCEPTION 'authentication required';
  END IF;

  IF p_stab < 0 OR p_fac < 0 OR p_com < 0 OR p_cra < 0 OR p_ach < 0 THEN
    RAISE EXCEPTION 'weekly contribution scores must be non-negative';
  END IF;

  INSERT INTO weekly_contributions (
    user_id, week_start_utc,
    stabilization_score, faction_score, combat_score, crafting_score, achievement_score
  ) VALUES (
    auth.uid(),
    v_week_start,
    p_stab, p_fac, p_com, p_cra, p_ach
  )
  ON CONFLICT (user_id, week_start_utc) DO UPDATE SET
    stabilization_score = EXCLUDED.stabilization_score,
    faction_score = EXCLUDED.faction_score,
    combat_score = EXCLUDED.combat_score,
    crafting_score = EXCLUDED.crafting_score,
    achievement_score = EXCLUDED.achievement_score,
    updated_at = now();
END;
$$;

GRANT EXECUTE ON FUNCTION upsert_weekly_score(INT, INT, INT, INT, INT) TO authenticated;
```

RLS 정책만으로 직접 UPSERT도 가능하지만, MVP는 RPC를 채택한다. 이유는 주차 계산·인증 가드·이상치 검증 hook을 서버 함수 한 곳에 모을 수 있기 때문이다.

#### 3.5.6 익명 인증 부트스트랩

현재 앱은 Supabase client 초기화와 정적 데이터 조회 중심이며, 활성 Supabase Auth 세션을 전제로 하는 흐름이 없다. 페이즈 4 #6 명세에는 다음 작업을 반드시 포함한다.

- 앱 시작 또는 `WeeklyContributionService` 최초 업로드 시 `SupabaseInitializer.client.auth.currentUser` 확인
- 사용자가 없으면 `signInAnonymously()` 호출
- 익명 인증 성공 후 RPC 업로드 수행
- 익명 인증 실패 시 `weeklyScoreDirty = true` 유지, 로컬 점수 집계 지속, 랭킹 화면에는 "업로드 대기" 상태 표시
- top 100 조회는 SELECT 정책이 열려 있으므로 인증 실패 시에도 가능하게 유지한다. 단, 본인 하이라이트는 `auth.uid()`가 없으면 표시하지 않는다.

### 3.6 신뢰 모델 — MVP 클라이언트 신뢰

#### 3.6.1 위조 가능성 인지

MVP는 클라이언트가 보낸 점수를 그대로 신뢰한다. 사용자가 앱을 디컴파일하여 임의 점수를 보낼 수 있다. M8.5 범위에서는 다음 이유로 허용:
- 보상이 없으므로 위조 인센티브 약함
- 신뢰 모델 구현은 서버측 사건 검증 필요 → M9+ 대규모 작업
- "재미 가시화" MVP가 핵심, 부정행위 방지는 부가 시스템

#### 3.6.2 부정행위 자연 억제

- 익명 표시(`u_xxxxxxxx`) — 명성 추구 인센티브 약함
- 보상 없음 — 경쟁 의욕 자연 한정
- 자기 5항목 분해 함께 표시 — 비현실적 점수는 자기 시선에서도 어색

#### 3.6.3 M9 위임 — 서버측 사건 검증

서버측 검증은 다음 채널로 구현:
- Supabase Edge Function: 클라이언트는 점수가 아니라 사건 ID만 전송
- 서버에서 사건 유효성 검증 (예: 의뢰 완수 → ActiveQuest 상태 + 시간 정합)
- 검증 통과 시 서버측 점수 가산

M8.5 MVP에서는 미구현. 페이즈 4 #6 spec에서 M9 hook만 정의.

```

    ↓ trailing fail-soft

    ↓ weeklyScoreDirty = true
  scheduleUploadIfDirty (5분 throttle)
    ↓ 5분 경과
  Supabase RPC upsert_weekly_score
    ↓
[Supabase]
  weekly_contributions UPSERT (PK: user_id × week_start_utc)
    ↓ total_score auto-compute
  (다음 lookup 시점에 top 100 정렬 가능)

[클라이언트 - 랭킹 화면 열림]
  SELECT user_id, total_score
    FROM weekly_contributions
    WHERE week_start_utc = current_week()
    ORDER BY total_score DESC LIMIT 100
    ↓
  자기 위치 lookup + 5항목 분해 표시

[주간 경계 — UTC 월요일 0:00]
  클라이언트 앱 시작/의뢰 완수 직후 검사
    ↓ currentWeekStart() != weeklyScoreWeekStartUtc
  지난 주 점수 final upload (안전 마진)
    ↓
  buffer 0 reset + weeklyScoreWeekStartUtc 갱신
```

### 4.2 호환성 검토

- **기존 UserData**: HiveField 30~33 추가는 기본값으로 호환. 기존 유저는 `weeklyScoreWeekStartUtc == null`이면 현재 주 buffer를 새로 시작한다.
- **기존 Supabase 정적 동기화 테이블**: 영향 없음. `weekly_contributions`는 사용자별 데이터 테이블이며 `data_versions`/`SyncService.allTables` 동기화 대상이 아니다.
- **익명 인증**: 현재 앱에는 Supabase Auth 세션 부트스트랩이 없으므로 페이즈 4 #6에서 별도 추가해야 한다. anon key만으로는 RLS의 `auth.uid()` 기반 쓰기가 통과하지 않는다.
- **시간 가속**: §3.3.4 — 시간 가속과 주간 사이클은 독립.

### 4.3 호환성 리스크

- **중간**: Supabase 익명 인증 미사용 시 RLS 정책 작동 불가. 페이즈 4 #6 spec에서 익명 인증 활성화 검증.
- **중간**: 신뢰 모델 (§3.6) — MVP는 위조 허용. 운영 도구에서 이상치 탐지 (`total_score > 100,000`) 필요. operation-bom 작업 위임.
- **낮음**: pg_cron 미설치 환경 — Supabase Pro+에서 사용 가능. 무료 tier에서는 별도 처리 필요.
- **낮음**: 5분 throttle은 timer 기반. 앱 백그라운드 시점에 의뢰 완료가 누적되면 다음 포그라운드 진입 시 일괄 업로드.
- **중간**: top 100 lookup 빈도 — 사용자가 정보 탭 자주 열면 RLS SELECT 호출 폭증. throttle 또는 캐시 정책 페이즈 4 #6 결정. 권장: 5분 캐시.

| 이상치 탐지 (operation-bom) | **낮음** | M9 신뢰 모델 전까지 부가 |
| 닉네임 시스템 | **낮음** | M9 위임 |

## 6. data-generator 지시사항

본 문서는 텍스트 데이터 생성을 요구하지 않는다. 5항목 가중치 매트릭스는 페이즈 2 #5에서 수치 확정, 신규 테이블은 페이즈 3 #5 또는 페이즈 4 #6에서 SQL 인라인.

- **대상 타입**: 별도 타입 스펙 불요
- **대상 테이블**:
  - `weekly_contributions` (사용자 데이터, 마이그레이션 시드 0행)
- **생성 수량**: 0행. 5항목 라벨·설명·표시 순서·색상은 Dart 정적 상수로 관리
- **톤/세계관 가이드**: 본 마일스톤은 시스템 데이터, 텍스트 톤 없음
- **구조적 제약**:
  - `weekly_contributions` GENERATED 컬럼 정합
  - RLS 정책 본인 INSERT/UPDATE + 모든 SELECT
  - pg_cron 권한
- **수치 출처**: 페이즈 2 #5에서 5항목 가중치 검증
- **특수 요구**:
  - 페이즈 3 #5 산출물 — Supabase 마이그레이션 SQL + RLS + pg_cron + RPC
- **검증**:
  - RLS 정책 본인 외 INSERT/UPDATE 거부 확인
  - GENERATED 컬럼 자동 계산 확인

## 7. 오픈 질문

- **Q-1 (5항목 정확한 가중치)**: §3.1.1 권장값. 페이즈 2 #5 시뮬레이션 검증 — 5항목 비중이 균등하지 않으면 한 영역만 플레이 유도. **권장**: 페이즈 2 #5 위임.
- **Q-2 (지역 안정화 카운트 범위)**: §3.2.1 — 모든 region vs M7 7리전만. **권장**: 페이즈 2 #5에서 시뮬레이션 후 결정. M7 7리전 한정 권장 (M8.5 더스트플레인 중심).
- **Q-3 (Supabase RPC 채택 여부)**: §3.5.5 — **결정**: MVP는 RPC 채택. 직접 UPSERT는 사용하지 않는다.
- **Q-4 (Supabase 익명 인증 활성화)**: 현재 앱은 익명 인증 세션을 자동 생성하지 않는다. **결정**: 페이즈 4 #6에서 `signInAnonymously()` 부트스트랩을 구현한다.
- **Q-5 (top 100 캐시 정책)**: §4.3 5분 캐시 권장. 클라이언트 메모리 또는 Hive 캐시. **권장**: 페이즈 4 #6 결정. Riverpod `AutoDispose` + 5분 timer 추천.
- **Q-6 (자기 위치 표시 정책)**: top 100 안에 자기가 있으면 하이라이트, 밖이면 별도 라인 "(나) 234위 / 1,247명". **결정**: §3.7.1 명시. 페이즈 4 #6 정밀화.
- **Q-7 (보상 시스템 도입 시점)**: M8.5 MVP는 보상 없음. M9+에서 검토. **결정**: M8.5는 보상 없음 (본 문서 명시).
- **Q-8 (이상치 탐지 정책)**: §4.3 — 클라이언트 위조 점수 (예: 100만점) 탐지. operation-bom 모니터링 또는 Supabase trigger. **권장**: M9 보안 시스템 작업으로 위임.
- **Q-9 (지난 주 점수 표시 정책)**: §3.7.1 [지난 주 보기] 토글. 지난 주 최종 점수만 표시 (5항목 분해 미포함). **결정**: 토글로 단순 표시.
