## 개요

M2a "아이템의 태동" 마일스톤의 핵심 기능인 **정수(Essence)** 시스템을 설계한다. 정수는 용병 1명의 특정 스탯을 영구적으로 강화하는 희귀 소비형 아이템이며, 동시에 M6에서 용병 티어 승급 재료로도 재사용되는 **다목적 희귀 재화**다. 본 문서는 정수의 티어 체계, 효과 곡선, 사용 상한, 소비 UX, 저장 방식, 사망/방출 시 처리, 상한 초과 엣지 케이스를 확정하여 이후 balance-designer 수치 시뮬레이션과 spec-writer 구현 명세의 기반을 제공한다.

핵심 설계 철학:
1. **희귀성 최우선** — 정수는 엘리트를 잡는다고 무조건 드랍되지 않으며, 기본이 희귀한 재화다.
2. **티어 계단** — 저티어부터 최상위까지 5단계 희소성 계단을 구축하여 장기 목표를 형성한다.
3. **위험과 보상의 대칭** — 투자한 정수는 사망·방출 시 전액 소멸한다. 육성 대상 선택에 무게를 부여한다.
4. **M6와의 정합성** — 정수로 용병을 무한 강화할 수 없다. 티어 상한은 승급으로만 돌파한다.

## 레퍼런스 분석

### 게임 속 바바리안으로 살아남기 (웹소설) — 정수의 감성 원점
- 몬스터에게서 드물게 얻는 정수를 "육체에 각인"하여 영구 스탯을 강화하는 설정. 주인공의 성장이 정수 수집과 직결되는 감성.
- **차용 포인트**: "정수 = 희귀 결정체, 소비 = 영구 신체 강화"의 서사를 그대로 가져온다. 4종(힘/지혜/수호/민첩)을 스탯 4축에 1:1 매핑하고, 소비 시 "각인(inscription)" 연출을 채용한다.

### Melvor Idle — 티어 계층과 장기 성장
- 모든 자원이 Normal → Elite 계층으로 분리되어 장기 파밍 목표를 형성. 상위 자원은 하위 자원의 **배수 효율**이 아닌 **곡선 증폭**으로 설계됨.
- **차용 포인트**: 정수 T1~T5를 선형이 아닌 **가속 곡선**(+1/+2/+4/+7/+11)으로 설계하여 고티어 파밍의 감동을 수치로 체감하게 한다.

### Diablo 시리즈 — "charm" 누적 한계
- Grand Charm 슬롯 제한, 인벤토리 공간 제한 등 무한 강화 방지 장치 다수.
- **차용 포인트**: 용병 티어별 스탯 축당 영구 보너스 상한을 두어 "정수만으로 T5 전력 달성"을 차단하고 M6 승급 시스템의 의미를 보존한다.

### Kingdom of Loathing — 프리뷰 팝업 표준
- 소비 아이템 사용 시 현재 스탯과 사용 후 스탯을 비교 표시하는 프리뷰 UX가 정착.
- **차용 포인트**: 정수 소비는 반드시 프리뷰 팝업을 거치고, 상한 초과 시 손실량을 명시적으로 안내한다.

## 상세 설계

### 1. 정수 티어 체계 (20종)

스탯 4축 × 티어 5단계 = **총 20종**.

| slot | 한국어 (T1 기준) | T2 | T3 | T4 | T5 | 대응 스탯 |
|---|---|---|---|---|---|---|
| `essence_str_t1~t5` | 힘의 정수 | 힘의 정수(상) | 힘의 정수(정) | 힘의 정수(강) | 힘의 정수(극) | STR |
| `essence_int_t1~t5` | 지혜의 정수 | … | … | … | … | INT |
| `essence_vit_t1~t5` | 수호의 정수 | … | … | … | … | VIT |
| `essence_agi_t1~t5` | 민첩의 정수 | … | … | … | … | AGI |

구체적인 명칭 수식어(상/정/강/극 등)는 산출물 3 초기 아이템 세트 문서에서 확정한다. 본 문서는 slot 체계와 티어 존재를 고정한다.

**로드맵 "정수 4종" 문구 재해석**:
- 로드맵(라인 239, 262)의 "정수 4종"은 **T1 대표 4종**을 의미하는 것으로 해석한다.
- M2a의 검증용 최소 데이터는 T1 4종이며, T2~T5는 로드맵 문구를 확장하여 본 기획서에서 20종 전체를 설계한다. 실제 데이터 생성 수량은 페이즈 3 계획 수립 시 확정.

### 2. 1회 사용 효과 — 가속 증가 곡선

정수 1개 사용 시 해당 스탯의 `permanent_*` 값이 다음만큼 증가한다 (권장 초안, balance-designer 페이즈 2에서 최종 확정):

| 티어 | 영구 스탯 증가량 |
|---|---|
| T1 | **+1** |
| T2 | **+2** |
| T3 | **+4** |
| T4 | **+7** |
| T5 | **+11** |

**곡선 특성**:
- 비선형 가속 곡선. T5 1개 = T1 11개 효과 (단순 비례가 아님).
- 고티어 정수의 희소 가치가 수치 자체로 체감되도록 설계.
- 저티어 정수도 "양으로 커버"의 유효한 경로를 제공 (T1도 +1로 유의미).

**구체 수치는 balance-designer 페이즈 2에서 확정**:
- 인플레이션 시뮬레이션을 통해 `+1/+2/+4/+7/+11`이 적정한지, 더 가파른 곡선(+1/+3/+6/+11/+18 등)이 필요한지 검토.
- 본 기획서는 **곡선 형태(가속 증가)**를 고정하며, 수치는 시뮬레이션 결과에 따라 조정 가능.

### 3. 용병 1명당 사용 상한 — 티어 연동

스탯 축(STR/INT/VIT/AGI) **각각**에 대해 용병 티어별로 영구 보너스 상한이 있다. 상한에 도달하면 더 이상 정수를 유효하게 사용할 수 없다 (프리뷰 경고 후 사용자 판단).

| 용병 티어 | 축당 `permanent_*` 상한 (권장 초안) |
|---|---|
| T1 | **+10** |
| T2 | **+20** |
| T3 | **+40** |
| T4 | **+70** |
| T5 | **+120** |

**상한의 의미**:
- "정수로 현재 티어의 성장만 가능, 티어 돌파는 승급으로만" 원칙을 코드로 강제.
- T1 용병의 STR을 T1 기준 base 8에서 +10까지 올리면 effective는 약 18. 이는 T3 base 근처로, "T1 정수 최대 강화 ≈ T3 용병 초기치" 수준의 환산 목표.
- 구체 상한 값은 balance-designer 페이즈 2에서 확정. 본 기획서는 "티어 연동 상한 정책"만 고정.

**4개 축은 독립적으로 카운트**:
- STR이 상한 도달해도 INT/VIT/AGI는 독립적으로 계속 사용 가능.
- 용병 1명당 총 잠재 영구 보너스 = T1 용병의 경우 4 × +10 = +40 상당.

**승급 후 상한 확장**:
- M6에서 용병이 승급(T1→T2)하면 상한도 새 티어 기준으로 확장 (+10 → +20).
- 기존 permanent 값은 유지되며, 남은 잔량은 새 상한 기준으로 재계산됨 (예: T1에서 STR +10 도달 → T2 승급 후 STR 잔량 +10 추가 획득 가능).
- 세부 승급 시 permanent 유지/재설정 정책은 M6 기획에서 확정.

### 4. 소비 UX

#### 4-1. 진입 경로 (양방향)

두 가지 경로를 모두 지원한다.

**경로 A: 용병 상세 오버레이에서 진입 (Pull 방식)**
1. 용병 카드 탭 → 용병 상세 오버레이 진입
2. 스탯 섹션에 "정수 사용" 버튼 표시
3. 버튼 탭 → 해당 용병이 사용할 수 있는 정수 목록 팝업 (인벤토리에서 필터링하여 표시)
4. 정수 선택 → 프리뷰 팝업 → 확인 → 적용

**경로 B: 인벤토리에서 진입 (Push 방식)**
1. 인벤토리 화면 → 소모품 카테고리 → 정수 아이템 선택
2. 아이템 상세 팝업의 "사용" 버튼
3. 용병 선택 팝업 (사용 가능/상한 도달 용병 구분 표시)
4. 용병 선택 → 프리뷰 팝업 → 확인 → 적용

**공통 원칙**:
- 양 경로 모두 **프리뷰 팝업 경유** 필수 (실수 방지).
- 양 경로에서 동일한 상한 판정·프리뷰 로직 사용 (spec-writer 페이즈 4에서 공통 컴포넌트화).

#### 4-2. 프리뷰 팝업 구성

프리뷰 팝업은 다음 정보를 표시한다:

```
[정수 사용 확인]

대상 용병: 김철수 (T2, Lv3)
사용 아이템: 힘의 정수(정) — T3 정수

현재 STR:        12 (base 10 + permanent +2)
사용 후 STR:     16 (base 10 + permanent +6)
상한 잔량:       +20 → +16 (남은 강화 여력)

effective STR:   13.4 → 17.4  (레벨 보너스 20% 반영)

[취소]  [사용]
```

**상한 초과 시 경고**:
- 잔량 < 정수 효과일 때 예시:
```
⚠ 상한 초과 경고

대상 용병: 김철수 (T1, Lv3)
사용 아이템: 힘의 정수(극) — T5 정수 (+11)
상한 잔량:       +5 (T1 용병 STR 상한)
적용 효과:       +5 (6 포인트는 손실됩니다)

정말 사용하시겠습니까?

[취소]  [손실 감수하고 사용]
```

#### 4-3. 소비 결과

**성공 소비 시**:
- `Mercenary.permanent_*` 값이 `min(현재 + 정수 효과, 티어 상한)`만큼 증가
- 해당 정수 아이템이 인벤토리에서 1개 감소 (수량 0이 되면 목록에서 제거)
- 활동 로그 기록: "{용병 이름}이(가) {정수 이름}을(를) 각인했다. STR +{실제 적용값}"
- 짧은 "각인" 연출 (스탯 수치 펄스 애니메이션 + 활동 로그 추가)

**상한 도달 시 후속 정수 사용 처리**:
- 이미 상한에 완전히 도달한 스탯에 해당하는 정수는 용병 상세 경로에서 "사용 버튼 비활성" + 툴팁 "STR 상한에 도달했습니다"
- 인벤토리 → 용병 선택 경로에서도 해당 용병을 "상한 도달" 표시로 비활성화
- 단, 잔량이 0이 아닌 경우(부분 소비 가능)에는 경고만 표시하고 사용은 허용

#### 4-4. 다중 소비 (M2a 범위 외)

- 한 번에 여러 정수를 일괄 소비하는 UI는 M2a에서 구현하지 않음. 후속 마일스톤(M2b 또는 그 이후)에서 도입 가능.
- M2a에서는 1회 사용당 1개 정수만 소비.

### 5. 저장 방식 — 별도 `permanent_*` 필드 신설

기존 `Mercenary.str/intelligence/vit/agi`(int, HiveField 4~7)는 **변경하지 않는다**. 정수 기여분을 추적하기 위해 별도 필드를 신설한다.

**Mercenary 모델 확장** (spec-writer 페이즈 4에서 HiveField 번호 확정):
```dart
@HiveField(18) int permanentStr       // 정수 누적 기여분 (초기 0)
@HiveField(19) int permanentIntelligence
@HiveField(20) int permanentVit
@HiveField(21) int permanentAgi
```

**기존 스탯과의 관계**:
- `str`, `intelligence`, `vit`, `agi` 필드는 용병 기본 스탯(base). 변동 없음.
- `permanentStr` 등은 "정수로 획득한 영구 보너스". 별도 카운트.
- 사용 상한 판정: `permanent_* ≤ tierCap(용병티어)` 체크.

### 6. effective 스탯 공식 — 순수 가산

기존 `effectiveStr` 공식에 `permanentStr`을 **비증폭 가산**으로 추가한다.

**기존 공식**:
```
effectiveStr = (baseStr × (1 + levelBonus)) × 피로디버프?
```

**확장 공식**:
```
effectiveStr = (baseStr × (1 + levelBonus) + permanentStr) × 피로디버프?
```

**핵심**:
- `permanentStr`은 레벨 보너스에 **곱해지지 않는다** (비증폭 가산).
- 피로 디버프는 `(baseStr × 레벨보너스 + permanentStr)` **전체**에 적용 (정수 기여분도 피로 영향을 받음).
- 단순·예측가능한 공식으로 밸런스 시뮬레이션 용이.

**UI 표시 원칙**:
- 용병 상세 스탯 섹션에 base와 permanent를 분리 표시 가능 (예: "STR 12 (+3)")
- 단순 표시가 필요한 곳(파견 카드 등)은 effective만 표시
- 표시 포맷은 spec-writer 페이즈 4의 UI 명세에서 확정

**다른 공식(옵션 A/C/D)은 배제**:
- A(permanent도 레벨 증폭): 인플레이션 폭주 위험 + 상한 판정 공식 복잡화
- C(절반 증폭): balance-designer가 두 배수 독립 튜닝 필요 → 밸런스 비용 증가
- D(피로 디버프 제외): 피로 시스템 의미 희석

### 7. 사망·방출 시 처리 — 전액 소멸 (하드코어)

**사망 시**:
- 용병 사망 → `permanent_*` 전액 소멸
- 용병 객체 자체가 영구 제거되므로 별도 처리 불필요
- 활동 로그에 사망 로그와 함께 "투입 정수 N개 소실" 같은 추가 문구 권장 (UI 결정은 spec-writer 단계)

**방출 시**:
- 용병 방출 → `permanent_*` 전액 소멸
- 환원/회수 기능 **없음**. 퇴직금만 지급(기존 로직 유지, `인건비 × 레벨`).
- 방출 확인 다이얼로그에 "투입한 정수 N개를 영구히 잃습니다" 경고 표시 (용병이 정수를 먹은 경우에만)

**설계 의도**:
- 정수 투자의 무게감을 극대화. "어느 용병에게 정수를 먹일지" 결정이 전략의 핵심.
- 사망 리스크를 회피하기 위해 정수 먹인 용병을 고난도 퀘스트에 무리하게 보내지 않는 플레이스타일 유도.
- 방출 악용 차단: "T5 정수 여러 개 먹였다가 방출로 환원" 같은 우회 경로 불가.

**다른 옵션(B/C/D/E)은 배제**:
- B(방출 시 부분 환원): 환원 공식·티어 다운그레이드 처리 복잡
- C(사망 시 50% 회수): 긴장감 저하
- D(기념비 계승): 기획 범위 확장
- E(방출 Lock): 사용자 선택권 제한

### 8. 상한 초과 시 처리 — 프리뷰 경고 + 사용자 판단

**흐름**:
1. 사용자가 정수 사용 시도 → 잔량 계산
2. 정수 효과 > 잔량 → 프리뷰 팝업에 **손실 예정량 명시**
3. 사용자가 "손실 감수하고 사용" 버튼으로 명시적 확인 시에만 소비
4. 적용 효과 = `min(정수 효과, 잔량)`. 초과분은 손실 처리되고 정수는 소멸

**옵션 A(자동 clamp) 대비 C의 장점**:
- 사용자가 손실을 "인지한 상태"에서 선택 → 불만 최소화
- 저티어 용병에게 "어쩔 수 없이" 고티어 정수를 쓰는 전략적 선택지 보존

**옵션 B(차단) 대비 C의 장점**:
- 잔량이 적더라도 "일부라도 올리고 싶다"는 선택 가능
- 인벤토리에 쌓여 쓸 수 없는 정수 → 불만 방지

**옵션 D(용병 티어별 정수 티어 제한) 배제 이유**:
- 저티어 용병의 강화 전략 차단 → "자유 장착/자유 사용" 철학과 충돌

### 9. 드랍 정책 가이드 (M2b 선반영)

본 기획서는 M2a 정수 사용 인프라를 정의하지만, 정수의 획득 경로(드랍)는 M2b에서 구현된다. M2b 설계 시 다음 원칙을 전달한다:

- **기본 희귀** — 엘리트 몬스터를 처치해도 정수가 무조건 드랍되지 않는다. 엘리트 1회 킬당 정수 드랍률은 보수적으로 설계 (balance-designer가 M2b에서 확정).
- **티어 매칭** — T2 엘리트는 T1~T2 정수가 주로 드랍, T5 엘리트는 T3~T5 정수가 드랍되는 방향. 구체 드랍 테이블은 M2b 담당.
- **수급 목표** — M2b 로드맵(라인 353)에 "정수 5~10개를 모아 용병 1명의 영구 스탯을 의미 있게 올리는 3~5시간 플레이" 언급. 이를 기준으로 드랍률 역산.
- **수동 지급 경로** — M2a 단계에서는 operation-bom의 **유저별 수동 지급 인터페이스**로 정수를 공급하여 인벤토리·사용 UX·인플레이션 시뮬레이션을 검증한다.

### 10. M6 승급 재료 재사용 정책 (선반영)

M6 용병 티어 업그레이드(로드맵 라인 756, 781)에서 정수가 승급 재료로 재사용된다. 본 기획서는 M2a 범위의 "영구 스탯 강화 경로"만 구현하되, 설계 시 다음 원칙을 M6가 이어받을 수 있도록 선반영한다:

- **동일 아이템 풀 재사용** — 승급용 별도 정수를 만들지 않는다. M2a에서 정의한 20종 정수가 그대로 M6 승급 재료가 된다.
- **소비 정책 견제** — M2a의 상한 시스템(질문 3)이 "정수를 무한정 스탯 강화에 쓸 수 없음"을 보장하므로, 자연스럽게 승급용 비축이 유도된다.
- **선택 긴장감** — 플레이어는 "T5 정수 1개를 지금 스탯 강화에 쓸지(+11 즉시), 승급 비축할지(나중에 새 티어 상한 확보)"를 결정한다. M6에서 승급 비용 설계 시 이 트레이드오프를 인지해야 한다.
- **스탯 재설정 정책** — 승급 시 base 스탯이 새 티어 기준으로 재설정된다면, `permanent_*`를 유지할지 초기화할지 M6 기획에서 결정한다 (본 기획서 범위 외).

## 구현 우선순위 제안

본 문서는 **페이즈 1 산출물 2/3**이며, M2a 핵심 기능 "정수 영구 스탯 강화"의 규칙 정의다. 우선순위 **높음**.

### 즉시 후속 착수

1. **검증용 초기 아이템 세트 컨셉** (산출물 3, `[content]20260418_initial_item_set.md`) — 정수 20종의 명칭·수식어·서사, 개인 장비 6~8종과 용병단 장비 3~4종의 테마.

### balance-designer 페이즈 2 의존 과제

1. **정수 효과 수치 확정** — 권장 초안 +1/+2/+4/+7/+11이 적정한지, 아니면 더 가파른/완만한 곡선이 필요한지 인플레이션 시뮬레이션으로 판정.
2. **용병 티어별 상한 확정** — 권장 초안 +10/+20/+40/+70/+120이 적정한지, T5 용병의 최종 전력 목표치에 맞춰 조정.
3. **M2b 드랍률 역산 기초자료** — 정수 5~10개로 "의미 있는 강화" 체감 기준 확정 (M2b 드랍 테이블 설계 입력).
4. **M6 승급 비용 가이드** — 승급에 필요한 정수 수량이 M2a 스탯 강화 소비와 양립 가능한지 선제 시뮬레이션.

### data-generator 페이즈 3 의존 과제

1. **essence 타입 스펙 작성** (`.claude/skills/data-generator/types/essence.md`) — 본 기획서 규약을 data-generator가 파싱할 수 있도록 타입 스펙 문서화.
2. **정수 20종 벌크 생성** — 4축 × 5티어 매트릭스 전체 데이터 생성, `items` 테이블 투입.

### spec-writer 페이즈 4 의존 과제

1. **Mercenary 모델 확장** — HiveField 18~21 (permanent_*) 추가, migration 포함.
2. **effectiveXxx 공식 확장** — `(base × 레벨보너스) + permanent` 적용.
3. **EssenceService** — 상한 판정, 적용 효과 계산, 소비 처리 (순수 정적 서비스).
4. **프리뷰 팝업 UI** — 현재/사용 후 스탯 비교, 상한 초과 경고, 손실 명시.
5. **용병 상세 오버레이 확장** — "정수 사용" 버튼 + 스탯 분리 표시.
6. **인벤토리 UI** — 소모품 카테고리 + 정수 아이템 사용 경로.
7. **용병 방출 다이얼로그 확장** — 정수 소실 경고 추가.

## data-generator 지시사항

본 기획서는 정수 시스템의 **규칙**을 정의하며, 실제 정수 20종의 명칭·설명 등 개별 데이터 생성은 **산출물 3(초기 아이템 세트 컨셉)** + balance-designer 페이즈 2 수치가 모두 확정된 후 페이즈 3에서 수행된다.

**페이즈 3 data-generator 호출 시 필요한 선행 조건**:
- 타입 스펙: `.claude/skills/data-generator/types/essence.md` 작성 완료 (M2a 페이즈 3 착수 전 필수)
- 입력 기획서: `[content]20260418_essence_system.md`(본 문서) + `[content]20260418_initial_item_set.md`(산출물 3) + `Docs/balance-design/20260418_essence_inflation.md`(페이즈 2 산출물 2)
- 수량: 정수 **20종** (스탯 4축 × 티어 5단계)
- 대상 테이블: `items` (category=consumable, slot=essence_*)
- 톤/세계관 가이드: 게임 속 바바리안으로 살아남기 풍의 "결정체·각인" 서사, 한국어, T티어가 오를수록 신비·고대성 강조
- 구조적 제약: 각 티어별 `effect_json.permanent_stat_gain.{stat}` 값이 본 기획서 2항의 곡선에 부합, tier 필드가 T1~T5 매칭
