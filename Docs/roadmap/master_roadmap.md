## 현재 상태 요약

### 완료된 시스템

- 핵심 게임 루프 (모집 → 이동 → 파견 → 결과)
- 용병 시스템 (85개 직업, 5티어, STR/INT/VIT/AGI 4스탯)
- 트레잇 시스템 Phase 1~5 + Phase A + Phase B (109개 트레잇, 획득/진화/삭제/시설 연계)
- 시설 시스템 12종 (Lv25, 건설 큐, OGame 스타일)
- 이동 시스템 (199개 리전, 여행 이벤트 12종)
- 퀘스트 시스템 (4유형, 5난이도, 자동 갱신)
- 명성/랭크 시스템 (F~A, 티어 잠금 해제)
- 지역 조사 시스템 (지식 포인트, 발견 트리거)
- 세력 핵심 시스템 (14개 세력, 발견/가입/탈퇴/평판/충돌)
- 아이템/장비 인프라, 엘리트 몬스터, 드랍 테이블
- 연계 퀘스트, 지역 변형, 퀘스트 서사, 이동 선택지, 다이얼로그 큐
- 활동 로그, 방치형 보상, 시간 가속
- operation-bom 운영 도구 (정적 데이터 CRUD, 버전 관리)

### M3 완료 후 재설계 판단

M3까지 완료된 상태에서 테스트 플레이를 진행한 결과, 현재 게임은 "짜임새 있게 채워진 세계에서 활동하는 용병단"보다 "파견 → 보상 → 모집/시설/장비 → 다시 파견"의 반복으로 체감된다. 따라서 M3 이후의 기존 로드맵은 폐기하고, 다음 기준으로 새 마일스톤을 설계한다.

- **초반 2시간을 먼저 날카롭게 만든다.** 전 대륙 컨텐츠 밀도보다 시작 거점 1개의 체감 완성도를 우선한다.
- **파견은 허드렛일에서 지역 사건 해결로 상승해야 한다.** 같은 파견 시스템이라도 의뢰의 위계, 서사, 지역 반응이 달라져야 한다.
- **용병 성장은 세계 반응으로 이어져야 한다.** 강한 용병은 단순히 성공률이 높은 카드가 아니라 칭호, 위업, 지명 의뢰, 연대기의 주인공이 되어야 한다.
- **아이템은 완제품 획득과 제작 완성을 병행한다.** Melvor Idle처럼 여러 활동에서 재료를 모아 장비·아티팩트를 완성하는 장기 목표를 제공한다.
- **세력 심화는 후순위로 내린다.** 세력 시설·상점·전용 트레잇은 시작 거점과 지역 생활권의 재미가 검증된 뒤 2~3개 세력부터 재도입한다.

## 설계 원칙

### 마일스톤 운영 원칙

1. **매 마일스톤 = 기획 + 구현 + 데이터 + 밸런스 스냅샷** — 기능만 만들고 끝나지 않는다.
2. **기획자 선행** — 현재 마일스톤 구현 중에 기획자는 다음 마일스톤의 컨텐츠/데이터를 설계한다.
3. **마일스톤 간 밸런스 체크포인트** — 각 마일스톤 완료 시 balance-designer로 경제 시뮬레이션을 수행하고, 다음 마일스톤 진입 전 수치를 확정한다.
4. **통합 가능한 시스템은 통합** — trait Phase D(특수 임무)와 world expansion Phase 4(연계 퀘스트)는 M3에서 하나로 합친다. 서사 텍스트와 이동 선택지 이벤트도 M3에서 공통 템플릿 엔진으로 통합한다.
5. **릴리스 테마 기반** — 매 마일스톤이 플레이어 관점에서 의미 있는 변화를 전달한다.
6. **누적 플레이 가능 기준** — 각 마일스톤 종료 시점에 "게임을 처음부터 시작하여 해당 마일스톤까지의 경험을 연속으로 플레이할 수 있는 데이터 분량"을 함께 확보한다. 데이터 채우기를 M6에 몰지 않는다.
7. **재미 검증 우선** — M4 이후에는 신규 시스템 수보다 "플레이어가 용병단의 성장을 체감하는가"를 완료 기준의 최상위에 둔다.

### 데이터 설계 원칙

- **구조화된 데이터는 JSONB 대신 정규화 테이블을 사용한다** — operation-bom에서 일반 CRUD 폼으로 편집할 수 있고, DB 레벨에서 검증이 가능하며, 기획자가 JSON 문법을 몰라도 된다.
- 기존에 동작하는 JSONB 필드(effect_json, discovery_data 등 스키마가 타입별로 가변적인 것)는 유지한다.
- **신규 추가하는 데이터는 정규화 우선.**
- **동일 테이블의 스키마 확장은 최초 1회에 몰아서 수행한다** — `quest_pools` 테이블에 필요한 `faction_tag`와 `sector_type` 필드는 M1에서 한 번에 추가하고, M3은 데이터만 채우도록 한다. table-config·Freezed 모델·SyncService 마이그레이션을 중복하지 않는다.

### 우선순위 기준

- **첫 2시간을 기준으로 판단한다** — 신규 유저가 초반에 이해하고 체감할 수 없는 기능은 후순위로 내린다.
- **장소·사건·보상 연결을 우선한다** — 랜덤 퀘스트 수를 늘리는 대신, 특정 거점에서 발생하는 고정 사건과 지역 반응을 먼저 만든다.
- **용병단 위상 변화를 우선한다** — 수치 보너스보다 칭호, 위업, 지명 의뢰, 지역 신뢰도처럼 플레이어가 기억하는 변화를 먼저 배치한다.
- **전체 확장보다 수직 절편을 우선한다** — 199개 리전을 얕게 채우기보다 시작 거점 1개와 주변 생활권 5~10개 리전을 깊게 만든다.
- **경제 충격이 큰 변경은 격리한다** — 용병 티어 업그레이드처럼 전체 골드 수급·지출 곡선을 재설계하는 변경은 재미 검증 이후 후순위로 다룬다.

## 마일스톤 전체 구조

```
M1  "세력의 영향력"       ── 세력이 실제 게임에 영향을 미치기 시작
  ↓
M2a "아이템의 태동"       ── 아이템 인프라 + 정수(영구 스탯 강화) 경로 개시
  ↓
M2b "엘리트의 시대"       ── 엘리트 몬스터 + 드랍 테이블, 반복 파밍 성립
  ↓
M3  "깊어지는 세계"       ── 연계 퀘스트 + 지역 변형 + 서사/선택지 통합 텍스트 계층
  ↓
M4  "첫 거점과 초반 2시간" ── 시작 마을·허드렛일·고정 사건·마을 인정 루프
  ↓
M5  "재료와 제작"         ── 활동별 재료 수집 + 레시피 + 장비·아티팩트 완성
  ↓
M6  "이름을 얻는 용병단"   ── 위업·칭호·간판 용병·지명 의뢰·연대기
  ↓
M7  "지역 생활권 확장"     ── 시작 거점 주변 5~10개 리전의 사건·특산품·상태 변화
  ↓
M8  "세력 재도입"         ── 2~3개 세력만 수직 절편으로 거점·상점·전용 보상 구현
  ↓
M9  "완성과 균형"         ── 데이터 확장·stub 정리·전체 밸런싱·최종 폴리싱
```

### 의존 관계

```
M2a (아이템 인프라) ──→ M2b (엘리트 드랍)     : 아이템 모델·인벤토리가 있어야 드랍 판정이 가능하다
M2a (아이템 인프라) ──→ M3 (연계 퀘스트 보상) : 아이템이 있어야 연계 퀘스트 최종 보상이 성립한다
M3 (서사/체인/지역 변형) ──→ M4 (첫 거점)      : 고정 사건과 지역 반응을 기존 서사·체인 인프라 위에 얹는다
M4 (첫 거점)       ──→ M5 (재료와 제작)       : 시작 생활권의 활동이 재료 출처가 되어야 제작 목표가 의미를 가진다
M4~M5              ──→ M6 (위상 성장)         : 사건 해결과 제작 완성이 위업·칭호·지명 의뢰로 기록된다
M4~M6              ──→ M7 (지역 확장)         : 검증된 초반 생활권 패턴을 주변 리전으로 확장한다
M1 + M4~M7         ──→ M8 (세력 재도입)       : 세력은 지역과 용병단 위상이 잡힌 뒤 큰 세계의 후원자로 재등장한다
M4~M8              ──→ M9 (완성과 균형)       : 재미 검증이 끝난 축을 대상으로 데이터 확장과 최종 밸런싱을 수행한다
```

기존 M4(세력 전용 시설·상점·전용 트레잇), 기존 M5(용병 관계), 기존 M6(티어 업그레이드·전체 데이터 채우기)는 본 로드맵에서 제외하고 후순위 아이디어로 보관한다. 새 로드맵에서는 초반 재미 검증이 끝나기 전까지 대규모 데이터 확장과 14세력 완결형 구현을 진행하지 않는다.

## M1: 세력의 영향력

### 목표

현재 세력 시스템은 "발견하고 가입하면 텍스트가 보이는" 수준이다. M1이 끝나면 세력 가입이 퀘스트 보상, 성공률, 경제 수치에 실제로 영향을 미친다. 플레이어가 "어떤 세력에 가입할까"를 진지하게 고민하기 시작하는 지점이다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **세력 패시브 효과 연동** | `passive_bonus_json`이 실제 게임 수치에 반영된다 (보상 배수, 성공률, 회복 시간 등) |
| **세력 퀘스트 태그 + 전용 퀘스트** | 일반 퀘스트에 세력 태그가 부여되며 완료 시 평판을 획득한다. 가입 세력 전용 고난도 퀘스트가 생성된다 |
| **파견 상성 시스템** | 퀘스트 유형별 유리한 직업/트레잇 조합이 존재하며 파견 선택에 전략적 깊이를 더한다 |
| **명성 등급별 보너스** | 랭크 상승 시 해금 외에 추가 보너스(보상 +%, 모집 할인 등)를 제공한다 |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 14개 세력의 패시브 보너스를 게임 메커니즘에 매핑하는 효과 타입을 정의한다.
- 세력 태그 퀘스트의 확률 테이블을 설계한다 (거점 거리별 보정 포함).
- 세력 전용 퀘스트의 난이도/보상을 설계한다 (일반 대비 +1~2 난이도, 평판 +5~10).
- 상성 매트릭스를 설계한다: 직업군 × 퀘스트 유형별 보정값.
- 명성 등급(F~A)별 보너스 수치를 확정한다.

**구현 (Flutter — band_of_mercenaries)**

- `PassiveBonusService`를 신규 추가한다. 패시브 보너스 데이터를 조회하여 기존 서비스(QuestCalculator, RecruitmentService, 회복 로직 등)에 보정값을 주입한다.
- `QuestGenerator`를 확장한다. 퀘스트 생성 시 세력 태그 확률을 판정하고 가입 세력 전용 퀘스트 슬롯을 관리한다.
- `QuestCompletionService`를 확장한다. 태그 퀘스트 완료 시 `FactionStateRepository.addReputation()`을 호출한다.
- `QuestCalculator`를 확장한다. 직업/트레잇 × 퀘스트 유형 상성 보정을 적용한다.
- `ReputationService`를 확장한다. 랭크별 보너스를 적용한다.
- 파견 화면에 세력 태그 배지와 상성 힌트 UI를 추가한다.

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블을 생성한다:
  - `faction_passive_bonuses` (기존 JSONB 필드를 정규화 테이블로 전환)
  - 상성 테이블 (`quest_synergy_matrix` 등, 신규 테이블 사용 시)
- **`quest_pools` 스키마를 한 번에 확장한다** — `faction_tag` 필드와 `sector_type` 필드(village/ruins/hidden, 기본 null)를 M1에서 동시에 추가한다. `sector_type` 데이터 입력은 M3에서 수행하지만, 스키마·Freezed 모델·SyncService·table-config 마이그레이션은 M1에서 일괄로 끝낸다.
- `ranks` 테이블에 보너스 필드를 추가한다.
- `table-config.ts`에 신규 테이블과 확장 필드를 반영한다.
- 데이터 입력 후 `data_versions`를 퍼블리시한다.

**데이터 입력 (operation-bom에서 수행)**

- 14개 세력의 패시브 보너스 데이터를 입력한다 (`faction_passive_bonuses` 테이블).
- 세력 전용 퀘스트 풀을 추가한다 (세력당 3~5개, 총 42~70개).
- 상성 보정 데이터를 입력한다.
- `ranks` 테이블: 등급별 보너스 값을 입력한다.

**밸런스 체크 (balance-designer)**

- 세력 패시브가 경제에 미치는 영향을 시뮬레이션한다 (보상 +15%가 골드 인플레이션을 유발하는지).
- 상성 보정이 성공률 분포를 어떻게 바꾸는지 검증한다.
- 명성 보너스의 누적 효과를 검증한다.

### 누적 플레이 기준 (M1 종료 시)

- 신규 유저가 F랭크에서 시작하여 세력 단서 발견 → 가입 → 전용 퀘스트 수행 → 평판 누적 → 랭크 업까지 30~60분 연속 플레이가 가능한 수준으로 세력 전용 퀘스트 풀이 채워져 있어야 한다.

### 완료 점검표

**기능 동작**

- [ ] 세력 가입 후 퀘스트 보상이 패시브 보너스만큼 실제로 변하는가?
- [ ] 세력 단서를 보유한 리전에서 퀘스트 생성 시 태그가 확률적으로 붙는가?

- [ ] 가입 세력의 전용 퀘스트가 파견 화면에 별도 표시되는가?
- [ ] 파견 화면에서 유리한 용병 조합에 대한 시각적 힌트가 보이는가?
- [ ] 랭크 E 이상에서 해당 등급의 보너스가 적용되는가?

**데이터 & 운영 도구**

- [ ] operation-bom에서 세력 전용 퀘스트를 생성/수정/삭제할 수 있는가?

**밸런스 & 회귀**

- [ ] balance-designer로 경제 시뮬레이션 1회 이상 수행, 이상치 없음 확인
- [ ] 세력 미가입 상태에서 기존 게임 루프가 이전과 동일하게 동작

### M1 완료 후 게임 상태

> 플레이어가 세력에 가입하면 퀘스트 보상이 실제로 달라진다. 특정 세력 근처에서 태그 퀘스트를 수행하며 평판을 올릴 수 있고, 충분한 평판이 쌓이면 전용 고난도 퀘스트에 도전할 수 있다. 파견 시 "이 퀘스트에는 어떤 용병이 유리한가"를 고려하는 전략적 깊이가 생긴다. 명성이 올라갈수록 추가 보너스가 붙어 성장 동기가 강화된다. 운영팀은 operation-bom에서 세력 퀘스트, 상성 데이터, 랭크 보너스를 자유롭게 편집하고 앱에 반영할 수 있다.

## M2a: 아이템의 태동

### 목표

게임에 골드 이외의 새로운 보상 축을 도입하는 인프라를 깐다. 아이템 모델, 인벤토리, 장비 장착/해제, 정수 사용(영구 스탯 강화)까지를 구현한다. 드랍 경로는 없지만 operation-bom에서 수동 지급으로 검증이 가능하며, 인플레이션 곡선을 사전에 시뮬레이션할 수 있다.

### 분할 근거

v1에서는 M2에 아이템 인프라와 엘리트 몬스터를 묶었다. 두 영역은 각각 다음과 같은 작업을 포함한다.

- 아이템 인프라: Freezed 모델 3종, Hive 박스 신설, effective 스탯 재계산, 인벤토리 UI, 용병 장비 슬롯 UI, 정수 소모 연출
- 엘리트 파이프라인: 엘리트 출현 로직, 퀘스트 생성 분기, 전투 난이도 조정, 드랍 엔진, 반복 파밍 설계

둘이 같은 마일스톤이면 인프라를 검증할 아이템 출처가 없고, 드랍 엔진을 검증할 아이템 풀이 없어 교차 테스트가 막힌다. 분할하면 M2a 종료 시점에 정수 인플레이션을 선행 검증하고, M2b 작업 범위를 드랍 엔진에 집중할 수 있다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **아이템/장비 인프라** | 아이템 모델, 인벤토리 Hive 박스, 장비 장착/해제, 정수 사용(영구 스탯 강화), 인벤토리 UI |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 아이템 분류 체계를 확정한다: 개인 장비(무기/방어구/장신구) + 용병단 장비(군기/유물) + 소모품(정수/일반).
- 정수 종류 및 영구 스탯 강화 수치를 설계한다 (힘/수호/생명/상급).
- 장비 스탯 보정 범위, 용병단 장비 전체 효과 범위를 정한다.
- 검증용 최소 데이터 세트: 정수 4종, 개인 장비 6~8종, 용병단 장비 3~4종.

**구현 (Flutter — band_of_mercenaries)**

- `ItemData` 정적 모델(Freezed)을 작성한다: id, name, category, slot, tier, description 등.
- `InventoryItem` Hive 모델을 작성한다: itemId, quantity, equippedTo(용병 ID).
- `inventory` Hive 박스를 신설한다.
- `InventoryRepository`를 작성한다: 아이템 추가/제거/장착/해제 CRUD.
- `ItemEffectService`를 작성한다: 장착 장비의 스탯 보정을 기존 effectiveStr/effectiveAgi 등에 반영한다.
- `EssenceService`를 작성한다: 정수 사용 시 용병 영구 스탯이 증가하고 정수가 소멸한다.
- 인벤토리 UI: 아이템 목록 화면(카테고리별 필터), 아이템 상세 팝업.
- 용병 상세 오버레이에 장비 슬롯(무기/방어구/장신구)을 추가한다.
- `SyncService`를 확장한다: `items` 테이블 동기화.

**운영 도구 (operation-bom)**

- Supabase에 `items` 테이블을 생성한다.
- `table-config.ts`에 `items` 정의를 추가한다. category/slot/tier 셀렉트 필드를 지원한다.
- **검증용 수동 지급 인터페이스를 추가한다** — 특정 유저에게 아이템을 직접 추가하는 관리자 액션을 구현한다. 드랍 엔진(M2b) 없이 인벤토리·장착·정수 소모 전체 경로를 검증할 수 있다.
- `data_versions`에 행을 추가한다.

**데이터 입력 (operation-bom에서 수행)**

- 정수 4종(힘/수호/생명/상급)을 입력한다.
- 개인 장비 6~8종(티어 2~5 분포)을 입력한다.
- 용병단 장비 3~4종을 입력한다.

**밸런스 체크 (balance-designer)**

- 정수 누적 사용 시 영구 스탯 인플레이션 시뮬레이션 — 정수 N개 획득 가정 시 스탯 상승폭 × 성공률 변화율.
- 장비 스탯 보정이 기존 성공률 밸런스를 얼마나 변경하는지 측정.
- 용병단 장비가 전체 용병에게 적용될 때의 총합 영향.

### 누적 플레이 기준 (M2a 종료 시)

- M1 범위의 플레이에 더해, operation-bom에서 정수·장비를 수동 지급받은 가정 아래 용병 장비 세팅을 실험할 수 있어야 한다. 엘리트가 없어도 "만약 정수 10개를 모으면 어떤 성장 곡선이 되는가"를 시뮬레이션할 수 있다.

### 완료 점검표

- [ ] 인벤토리 화면에서 보유 아이템 목록을 카테고리별로 확인할 수 있다.
- [ ] 용병 상세 오버레이에 장비 슬롯이 보이고 장착/해제가 동작한다.
- [ ] 장비 장착 시 용병의 effective 스탯이 실제로 변한다.
- [ ] 정수를 용병에게 사용하면 영구 스탯이 증가하고 정수가 소멸한다.
- [ ] 용병단 장비 장착 시 전체 용병에게 효과가 적용된다.
- [ ] 앱 재시작 후 인벤토리와 장착 상태가 Hive에서 정상 복원된다.
- [ ] operation-bom에서 아이템을 생성/수정/삭제할 수 있다.
- [ ] operation-bom의 수동 지급 인터페이스로 정수·장비를 유저에게 전달할 수 있다.
- [ ] balance-designer로 정수 사용 인플레이션 시뮬레이션 완료.
- [ ] M1 기능 회귀 이상 없음.

### M2a 완료 후 게임 상태

> 용병마다 무기·방어구·장신구 슬롯이 열려 있다. 플레이어는 아직 아이템을 스스로 획득할 수는 없지만, 인벤토리 화면과 장착 UI가 완비되어 있다. 용병단 유물을 장착하면 전체 전력이 상승한다. 운영팀은 테스트 단계에서 아이템을 수동 지급하여 장비 성장 곡선을 검증한다. 다음 M2b에서 엘리트 몬스터가 첫 번째 공식 드랍 경로로 열린다.

## M2b: 엘리트의 시대

### 목표

M2a의 아이템 인프라 위에 엘리트 몬스터를 올린다. 지역 조사로 엘리트의 존재를 해금하고, 강한 용병을 파견하여 토벌하고, 드랍 판정으로 정수·장비·용병단 유물을 획득한다. M2b가 끝나면 "이 지역에 머물며 강철 골렘을 반복 사냥해서 힘의 정수를 모은다"는 새로운 플레이 패턴이 성립한다.

### 선행 조건

- M2a 아이템 인프라 — 드랍 엔진이 인벤토리에 아이템을 넣을 수 있어야 한다.
- 지역 조사 시스템 (완료)
- `region_discoveries.discovery_type = elite` 인프라 (완료)

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **엘리트 몬스터** | 지식 임계값으로 해금, 반복 출현, 고난도 토벌 퀘스트, 드랍 판정 |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 엘리트 몬스터 카탈로그를 설계한다: 리전 티어별 전투력, 출현 확률, 보상 구성.
- 드랍 테이블 가이드라인: 아이템별 드랍률(정수 40~60%, 장비 15~25%, 용병단 장비 3~8%).
- 검증용 최소 데이터: 엘리트 10~15종.

**구현 (Flutter — band_of_mercenaries)**

- `EliteMonsterData` 정적 모델(Freezed)을 작성한다.
- `EliteLootEntry` 정적 모델을 작성한다: elite_id, item_id, drop_rate.
- 엘리트 출현 로직 — `RegionState.triggeredDiscoveries`에 엘리트 discovery가 존재하면 퀘스트 갱신 시 확률 판정(티어별 8~15%)으로 엘리트 퀘스트를 생성한다.
- 엘리트 퀘스트 — 기존 `ActiveQuest`에 `isElite` 플래그를 추가하고, 전투력 100~200, 소요시간 1.5~2배로 설정한다.
- 드랍 판정 — 퀘스트 완료 시 `EliteLootService`가 각 드랍 항목별 독립 확률 판정으로 인벤토리에 아이템을 추가한다.
- 파견 화면에 엘리트 퀘스트 시각적 구분을 추가한다 (아이콘, 강조 색상, 최소 파견 인원 권장 표시).
- 퀘스트 완료 팝업에 아이템 드랍 결과를 표시한다.
- `SyncService`를 확장한다: `elite_monsters`, `elite_loot_tables` 테이블 동기화.

**운영 도구 (operation-bom)**

- Supabase에 `elite_monsters`, `elite_loot_tables` 테이블을 생성한다.
- `table-config.ts`에 2개 테이블 정의를 추가한다.
- `elite_loot_tables` 편집 화면: elite_id와 item_id는 이름 참조 표시, drop_rate는 0.0~1.0 범위 검증.
- `data_versions`에 행을 추가한다.

**데이터 입력 (operation-bom에서 수행)**

- 엘리트 몬스터 10~15종(리전 티어별 분포)을 입력한다.
- 드랍 테이블 — 엘리트별 드랍 항목 및 확률.
- `region_discoveries`에 엘리트 관련 discovery 데이터를 추가한다.

**밸런스 체크 (balance-designer)**

- 엘리트 전투력 vs 플레이어 파티 전투력 분포.
- 드랍률 × 엘리트 출현 확률 × 파밍 주기 — 아이템 획득 기대 시간 검증.
- M2a에서 시뮬레이션한 정수 인플레이션에 엘리트 파밍 속도를 대입하여 실제 현실 수치를 재조정한다.

### 누적 플레이 기준 (M2b 종료 시)

- 신규 유저가 M1 흐름을 거친 후 Tier 2~3 리전에서 엘리트 1~2종을 해금, 반복 사냥으로 정수 5~10개를 모아 용병 1명의 영구 스탯을 의미 있게 올리는 3~5시간 플레이가 가능해야 한다. 이를 위해 Tier 2~3 리전 일부에 엘리트 데이터가 배치되어 있어야 한다.

### 완료 점검표

- [ ] 지식 임계값 도달 후 해당 리전에서 퀘스트 갱신 시 엘리트 퀘스트가 확률적으로 출현한다.
- [ ] 엘리트 퀘스트가 파견 화면에서 일반 퀘스트와 시각적으로 구분된다.
- [ ] 엘리트 토벌 성공 시 드랍 판정이 돌아가고 획득 아이템이 인벤토리에 추가된다.
- [ ] 퀘스트 완료 팝업에 드랍된 아이템이 표시된다.
- [ ] 엘리트를 처치해도 이후 퀘스트 갱신에서 다시 확률적으로 출현한다 (반복 파밍 가능).
- [ ] operation-bom에서 엘리트 몬스터와 드랍 테이블을 편집할 수 있다.
- [ ] data_versions 퍼블리시 후 Flutter 앱에서 2개 테이블 데이터가 정상 동기화된다.
- [ ] balance-designer로 정수 파밍 실제 수치 시뮬레이션 완료, M2a 시뮬레이션과 교차 검증.
- [ ] M1 기능 회귀 이상 없음.

### M2b 완료 후 게임 상태

> 플레이어는 지역 조사로 엘리트 몬스터의 존재를 발견하고, 강력한 용병을 파견하여 토벌에 도전한다. 성공하면 정수, 장비, 드물게는 용병단 유물을 획득한다. 정수로 용병의 영구 스탯을 올리고, 장비를 장착하여 더 어려운 엘리트에 도전하는 새로운 성장 루프가 생긴다. "이 리전에 머물며 강철 골렘을 반복 사냥해서 힘의 정수를 모을까, 아니면 다음 리전으로 넘어갈까"라는 전략적 판단이 등장한다.

## M3: 깊어지는 세계

### 목표

지역 조사의 최종 보상을 완성한다. 지식을 쌓으면 다단계 서사 퀘스트가 열리고, 끝까지 조사하면 지역 자체가 변형된다. 퀘스트 결과와 이동 중 이벤트에 서사 텍스트가 붙어 세계가 살아있는 느낌을 준다. M3이 끝나면 "이 지역을 끝까지 파고들면 어떤 이야기가 나올까"라는 호기심이 게임을 이끄는 동력이 된다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **숨겨진 연계 퀘스트** | 지식 임계값 도달 시 2~5단계 체인 퀘스트 발동. 지역 이동 포함 가능. 최종 보상으로 확정 아이템. trait Phase D(특수 임무)를 통합 |
| **지역 변형** | 지식 최고 임계값 도달 시 섹터가 마을/유적지/숨겨진 섹터로 변형. 변형된 섹터에 전용 퀘스트 풀 |
| **퀘스트 서사 + 이동 선택지 (통합 텍스트 계층)** | 퀘스트 결과마다 조합형 서사 텍스트 생성. 이동 완료 시 확률적으로 선택지 이벤트 발동. 두 영역은 `TemplateEngine` 공유 모듈을 통해 변수 치환·조건 분기·트레잇 기반 숨겨진 옵션을 동일한 방식으로 처리 |

### 통합 근거 (서사 + 선택지)

v1에서는 퀘스트 서사 텍스트를 M3에, 이동 이벤트 선택지를 M5에 분리 배치했다. 그러나 두 영역은 다음을 공유한다.

- 변수 치환(용병 이름, 직업명, 지역명)
- 트레잇/스탯 기반 숨겨진 분기
- operation-bom 템플릿 편집기
- 활동 로그 연동

이를 한 마일스톤에 묶으면 `TemplateEngine` 공유 모듈을 1회 설계하고 두 영역이 재사용한다. 분리하면 편집기·검증 툴·조건 분기 엔진을 두 번 만들게 된다.

### 선행 조건

- M2a 아이템 인프라 — 연계 퀘스트 최종 보상, 유적지 전용 퀘스트 드랍에 아이템을 사용한다.
- 지역 조사 + `region_discoveries` (완료) — `discovery_type: hidden_quest`, `discovery_type: transform` 트리거.

### 작업 상세

**기획 (content-designer)**

- 체인 퀘스트 시나리오 5~8개(리전 티어별 분포), 2~5단계, 단계별 수행 지역/전투력/보상/서사 텍스트.
- 지역 이동을 포함하는 체인의 서사 흐름, 실패 시 재도전 가능 설계.
- 변형 대상 리전 15~25개 선정 (마을/유적지/숨겨진 섹터 유형별 컨텐츠).
- 서사 템플릿: 퀘스트 유형(4) × 결과(4) 매트릭스, 유형×결과별 3~5개 변형, 총 48~80개.
- 이동 선택지 이벤트 시나리오 10~15종, 선택지 2~3개, 선택지별 결과 확률, 트레잇 기반 숨겨진 선택지.
- **공존 정책 정의** — 가입 세력 전용 퀘스트(M1)와 연계 퀘스트(M3)가 동시에 존재할 때 파견 화면 정렬 순서·강조 방식·UI 슬롯 규칙을 확정한다. 기본 원칙: 연계 진행 중 단계는 최상단 고정, 그 아래 가입 세력 전용, 그 아래 엘리트, 나머지는 일반 퀘스트.

**구현 (Flutter — band_of_mercenaries)**

공유 텍스트 엔진:
- `TemplateEngine` 모듈 신규 — 변수 치환 `{merc_name}/{job}/{region}`, 조건 분기, 트레잇 기반 숨겨진 옵션 평가. M3 내 서사와 선택지가 모두 이 엔진을 호출한다.

연계 퀘스트:
- `ChainQuestData` 정적 모델(Freezed): chain_id, region_id, step, target_region_id, name, description, combat_power, duration_minutes, reward_gold, reward_items, final_reward, next_step_delay.
- `ChainQuestProgress` Hive 모델: chainId, currentStep, status(active/completed), startedAt.
- `chainQuestProgress` Hive 박스를 신설한다.
- `ChainQuestService` — 발견 트리거 시 1단계 자동 생성, 단계 완료 시 다음 단계 생성, 최종 단계에서 확정 보상.
- 파견 화면 — 연계 퀘스트를 상단 고정, 강조 색상, 진행도 표시, 타 지역 "이동 필요" 안내.
- 실패 시 해당 단계 재도전 가능.
- `SyncService` 확장 — `chain_quests` 테이블 동기화.

지역 변형:
- `RegionState` 모델 확장 — `sectorChanges` (Map<int, String>) 추가.
- 지식 임계값 도달 + transform 트리거 시 변형을 기록한다.
- 이동 화면에서 변형된 섹터를 특수 아이콘/이름으로 구분한다.
- `QuestGenerator` 확장 — 섹터 변형 타입에 따라 전용 퀘스트 풀에서 생성 (M1에서 추가한 `sector_type` 필드 활용).
- 변형 발견 시 팝업과 활동 로그 기록.

퀘스트 서사:
- `QuestNarrativeData` 정적 모델(Freezed): quest_type, result_type, template_text.
- `QuestNarrativeService` — 완료 시 템플릿 조회 → `TemplateEngine` 치환 → 서사 텍스트 출력.
- 퀘스트 완료 팝업에 서사 텍스트 영역을 추가한다.
- `SyncService` 확장 — `quest_narratives` 테이블 동기화.

이동 선택지:
- `TravelChoiceEventData` 정적 모델(Freezed).
- `TravelEventService` 확장 — 선택지 이벤트 풀 확률 판정.
- 이벤트 발생은 이동 완료 시 "회상" 형식으로 제시한다 (이동 중단 없음).
- 선택지 UI — 상황 텍스트 + 선택 버튼, 결과 즉시 적용 + 활동 로그.
- 트레잇 기반 숨겨진 선택지는 `TemplateEngine`이 평가한다.
- `SyncService` 확장 — `travel_choice_events` 테이블 동기화.

온보딩:
- 연계 퀘스트 최초 발동 시 튜토리얼 팝업 1회 표시 ("이 퀘스트는 여러 단계로 이어집니다").
- 첫 선택지 이벤트 조우 시 안내 한 줄.

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블 3개 — `chain_quests`, `quest_narratives`, `travel_choice_events`.
- `table-config.ts`에 3개 테이블 정의를 추가한다.
- `chain_quests` 편집 — chain_id로 그룹핑하여 체인 단위 조회/편집.
- `quest_narratives` 편집 — quest_type × result_type 필터, 템플릿 변수 미리보기.
- `travel_choice_events` 편집 — 선택지별 결과 확률 인라인 편집, 확률 합산 검증.
- `data_versions`에 신규 테이블 행을 추가한다.

**데이터 입력 (operation-bom에서 수행)**

- 연계 퀘스트 체인 5~8개 (총 15~30단계).
- `region_discoveries`에 hidden_quest + transform 트리거 추가.
- 변형 대상 15~25개 리전의 transform 데이터.
- 마을/유적지 전용 퀘스트 풀 (유형별 10~15개, `quest_pools`의 `sector_type` 필드 활용).
- 서사 템플릿 48~80개.
- 이동 선택지 이벤트 10~15종.
- 연계 퀘스트 최종 보상 아이템이 items 테이블에 존재하는지 확인한다.

**밸런스 체크 (balance-designer)**

- 연계 퀘스트 확정 보상 강도 vs 엘리트 반복 파밍 시간 효율 비교.
- 유적지 전용 퀘스트 난이도/보상 프리미엄 적절성.
- 마을 퀘스트 보상이 "안전한 대안"으로서 적절한 수준인지.
- 이동 선택지 기대값 — 위험/안전 선택 모두 합리적인지.

### 누적 플레이 기준 (M3 종료 시)

- 신규 유저가 M1~M2b를 거친 후 연계 퀘스트 2개 이상을 완주하고, 지역 변형을 1회 이상 경험하며, 이동 중 선택지 이벤트를 여러 번 겪는 5~8시간 플레이가 가능해야 한다. 서사 템플릿이 모든 quest_type × result_type 조합에 최소 1개씩 존재한다.

### 완료 점검표

**연계 퀘스트**

- [ ] 지식 임계값 도달 시 1단계 연계 퀘스트가 자동 생성된다.
- [ ] 파견 화면에서 연계 퀘스트가 상단 고정, 진행도 표시된다.
- [ ] 타 지역 수행 단계에서 "이동 필요" 안내, 해당 지역 도착 후 파견 가능하다.
- [ ] 단계 실패 시 해당 단계만 재도전 가능하다.
- [ ] 최종 단계 완료 시 확정 아이템이 인벤토리에 추가된다.
- [ ] 완료된 체인은 재발동이 방지된다.
- [ ] 가입 세력 전용 퀘스트(M1)와 연계 퀘스트가 동시에 파견 목록에 있을 때 공존 정책대로 정렬된다.

**지역 변형**

- [ ] 지식 최고 임계값 도달 시 변형 팝업과 활동 로그가 기록된다.
- [ ] 이동 화면에서 변형된 섹터가 아이콘/이름으로 시각적 구분된다.
- [ ] 마을/유적지/숨겨진 섹터에서 각각 전용 퀘스트가 생성된다.

**퀘스트 서사 + 이동 선택지**

- [ ] 퀘스트 완료 팝업에 서사 텍스트가 표시된다.
- [ ] 서사 텍스트에 용병 이름, 직업명, 지역명이 정확히 치환된다.
- [ ] 동일 유형/결과 조합에서 여러 변형 중 랜덤으로 선택된다.
- [ ] 이동 완료 시 확률적으로 선택지 이벤트가 표시된다.
- [ ] 선택지별 결과(골드/아이템/평판/상태)가 즉시 반영된다.
- [ ] 특정 트레잇 보유 시 숨겨진 선택지가 추가 표시된다.
- [ ] 기존 자동 여행 이벤트(12종)와 선택지 이벤트가 공존한다.

**데이터 & 운영 도구**

- [ ] operation-bom에서 연계 퀘스트를 체인 단위로 조회/생성/수정할 수 있다.
- [ ] operation-bom에서 서사 템플릿을 유형×결과로 필터링하여 편집할 수 있다.
- [ ] operation-bom에서 선택지 이벤트의 결과 확률 합산이 검증된다.
- [ ] data_versions 퍼블리시 후 Flutter 앱에서 신규 데이터 정상 동기화 확인.

**밸런스 & 회귀**

- [ ] balance-designer로 연계 퀘스트 보상 vs 엘리트 파밍 시간 효율 비교 수행.
- [ ] M1 + M2a + M2b 기능 회귀 이상 없음.

### M3 완료 후 게임 상태

> 지역을 깊이 조사하면 숨겨진 이야기가 열린다. "봉인 문서 해독" 연계 퀘스트를 따라 대륙을 횡단하며 3단계 서사를 진행하면, 마지막에 고대의 유물을 확정으로 얻는다. 리전을 끝까지 파면 폐광이 마을로 바뀌거나, 안개가 걷히며 고대 유적이 드러난다. 변형된 섹터에서는 전용 퀘스트가 생성되어 새로운 도전거리가 생긴다. 모든 퀘스트 결과에 "용병 김철수가 적 대장과 일대일 결투를 벌여 승리했다" 같은 서사가 붙고, 이동 중에는 "부상당한 여행자" 이벤트를 만나 도와줄지·무시할지·소지품을 뒤질지를 선택한다. 숫자 나열이 아닌 이야기로 느껴진다.

## M4: 첫 거점과 초반 2시간

### 목표

M4는 M3까지 만든 시스템을 시작 거점 1개에 다시 묶어, 신규 유저가 첫 2시간 안에 "보잘것없는 용병단이 마을에서 인정받기 시작했다"는 변화를 체감하게 만든다. 이 마일스톤은 전 대륙 도시 시스템이 아니라 **첫 거점 MVP**를 구현한다. 동시에 현재 199개 리전과 리전당 10개 섹터 구조를 축소하여, 컨텐츠를 얕게 흩뿌리지 않고 의미 있는 장소에 재배치한다. 마을/도시는 맵 장식이 아니라 파견, 조사, 모집, 제작, 고정 사건을 한 장소에서 연결하는 플레이어 성장의 거울이다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **지역/섹터 축소** | 전체 리전을 199개에서 40개로 축소하고, 기본 섹터 수를 10개에서 4개로 줄인다. 특수 지역만 최대 6개 섹터를 사용한다 |
| **시작 거점 1개** | 모든 신규 유저는 Tier 1 지역 `더스트플레인`의 1번 섹터 `더스트빌`에서 시작한다 |
| **마을/도시 섹터 구현** | 마을이 있는 지역은 1번 섹터를 마을/도시로 고정하고, 해당 섹터 선택 시 하단에 마을 내 방문 거점을 표시한다 |
| **허드렛일 의뢰 계층** | 짐 나르기, 야간 순찰, 약초 채집, 실종자 수색, 도적 흔적 조사처럼 낮은 위상의 초반 의뢰를 추가한다 |
| **마을 신뢰도** | 용병단 명성과 별개로 시작 거점의 지역 신뢰도를 누적한다 |
| **고정 사건 라인** | 첫 거점의 문제를 해결하는 60~120분 분량의 고정 사건을 제공한다 |
| **거점 상태 변화** | 사건 해결에 따라 상점, 대장간, 의뢰 등급, NPC 반응이 변화한다 |

### 설계 방향

M4의 첫 거점은 Tier 1 지역 `더스트플레인`의 마을 `더스트빌`이다. 모든 신규 유저는 `더스트빌`에서 시작한다. `더스트플레인`은 기존 Tier 1 지역군(초원/해안 중심)을 재편하여 만든 시작 지역이며, 광산 마을을 품은 변방 지역으로 정의한다. 광산은 재료 수집과 제작의 출처가 되고, 변방 마을은 허드렛일부터 시작하는 위상 상승을 자연스럽게 만든다.

#### 지역 축소 정책

현재 199개 리전은 초반 재미를 검증하기에 너무 넓다. M4에서는 전체 리전을 40개로 축소하고, 기존 컨텐츠를 새 지역 구조에 맞게 재배치한다. 티어는 1~10 구조를 유지한다.

티어별 리전 수는 다음 원칙을 따른다.

| 티어 | 리전 수 | 배치 의도 |
|------|---------|----------|
| Tier 1 | 3개 | 시작 구간. `더스트플레인`을 포함한다 |
| Tier 2 | 4개 | 초반 생활권 확장 |
| Tier 3 | 5개 | 초반 후반부와 첫 전문화 구간 |
| Tier 4 | 5개 | 중반 진입 |
| Tier 5 | 5개 | 중반 핵심 |
| Tier 6 | 5개 | 중후반 확장 |
| Tier 7 | 4개 | 고난도 진입 |
| Tier 8 | 4개 | 고난도 핵심 |
| Tier 9 | 3개 | 최상위 직전 |
| Tier 10 | 2개 | 최종권. 희소성을 유지한다 |

기존 `regions`, `region_discoveries`, `quest_pools`, `elite_monsters`, `chain_quests`, `factions`의 지역 참조는 새 40개 리전 구조에 맞게 재매핑한다. 삭제되는 지역의 컨텐츠는 폐기하지 않고, 새 생활권 또는 후속 마일스톤의 데이터 후보로 보관한다.

#### 섹터 축소 정책

기존 리전당 10개 섹터는 초반 이동 판단을 과도하게 분산시킨다. M4에서는 기본 섹터 수를 4개로 줄이고, 컨텐츠 밀도가 높은 특수 지역만 최대 6개 섹터를 허용한다.

섹터 규칙은 다음과 같다.

- 일반 지역은 4개 섹터를 사용한다.
- 특수 지역은 5~6개 섹터를 사용할 수 있다.
- 마을/도시가 있는 지역은 반드시 1번 섹터를 마을/도시로 사용한다.
- `더스트플레인`의 1번 섹터는 `더스트빌`이다.
- 폐광은 마을 내부 거점이 아니라 `더스트플레인`의 별도 섹터로 구현한다.
- 지역 간 이동은 기존 **지역 선택** UI가 담당한다. 특정 섹터를 선택해야만 다른 지역으로 이동할 수 있는 구조는 M4에서 도입하지 않는다.

`더스트플레인`의 MVP 섹터 구성은 다음을 권장한다.

| 섹터 | 이름 | 역할 |
|------|------|------|
| 1 | 더스트빌 | 시작 마을. 촌장 집, 대장간, 약초상 방문 가능 |
| 2 | 폐광 | 첫 위험 지역. 조사, 재료, 고정 사건, 첫 엘리트 후보 |
| 3 | 마른 초원 | 허드렛일, 약초 채집, 야간 순찰 |
| 4 | 먼지로 덮인 길 | 호위 의뢰, 여행자 조우 |

특수 확장이 필요하면 `더스트플레인`은 5~6섹터 지역으로 승격할 수 있지만, M4 MVP에서는 4섹터를 기본으로 한다.

#### 마을 섹터 방문 UI

**이동** 화면은 현재 위치, **지역 선택**, **섹터 선택** 구역을 유지한다. 단, 선택한 섹터가 마을/도시인 경우 하단에 "마을 내 방문" 영역을 추가한다. 이 영역은 파견 메뉴를 대체하지 않으며, 마을 내부의 장소 기반 상호작용만 제공한다. 마을 내 방문은 시간의 소모가 없다.

`더스트빌`의 M4 방문 거점은 다음과 같다.

| 장소 | 기능 | 초반 역할 |
|------|------|----------|
| 촌장 집 | 고정 사건 진행 | 마을 문제의 서사 허브 |
| 낡은 대장간 | 제작 목표 제시 | M5 제작 시스템의 진입점 |
| 약초상 | 조사·채집 의뢰 제공 | 조사와 재료 수집 연결 |

`의뢰소`는 별도 방문 거점으로 두지 않는다. 파견 메뉴가 이미 의뢰 목록과 파견 실행을 담당하므로, 의뢰소를 추가하면 기능이 중복된다. 허드렛일과 초급 의뢰는 파견 메뉴에서 별도 계층으로 노출한다. `폐광`은 마을 내부 방문 거점이 아니라 지역 내 섹터로 구현한다.

#### 마을 거점별 MVP 기능

마을 내 방문 거점은 단순 설명 화면이 아니라 초반 루프를 연결하는 소규모 기능을 가진다. M4에서는 복잡한 상점/제작 시스템 전체를 구현하지 않고, 각 거점마다 하나의 명확한 플레이 목적을 제공한다.

| 거점 | M4 MVP 기능 | 해금/상태 변화 | 연결 시스템 |
|------|-------------|----------------|-------------|
| 촌장 집 | 고정 사건 진행, 마을 신뢰도 보상 확인, 다음 목표 안내 | 고정 사건 단계와 신뢰도 구간에 따라 대화·보상·요청이 변화한다 | 고정 사건, 마을 신뢰도, 활동 로그 |
| 낡은 대장간 | 첫 제작 목표 미리보기, 수리/복원 의뢰, 제작 재료 요구량 표시 | 초기에는 닫혀 있거나 제한 상태이며, 신뢰도 또는 사건 진행으로 기본 기능이 열린다 | M5 제작, 재료 수집, 폐광 섹터 |
| 약초상 | 약초 채집 의뢰 안내, 조사 보너스/회복 보조, 재료 획득 힌트 제공 | 신뢰도 상승 후 회복 보조 또는 채집 의뢰 보상이 개선된다 | 조사, 허드렛일 의뢰, 부상/피로 회복 |

거점별 상세 기능은 다음과 같이 정의한다.

**촌장 집**

- 현재 고정 사건 단계와 다음 목표를 표시한다.
- 마을 신뢰도 단계와 다음 단계 보상을 표시한다.
- 고정 사건 단계가 진행될 때 짧은 대화 또는 사건 브리핑을 제공한다.
- 사건 완료 후 마을 반응 텍스트와 활동 로그를 기록한다.
- 기능 버튼은 `상황 듣기`, `신뢰도 확인`, `보상 받기` 정도로 제한한다.

**낡은 대장간**

- M4에서는 완전한 제작 UI를 만들지 않는다. 대신 M5 제작 시스템의 목표를 예고하는 "첫 제작 목표"를 표시한다.
- 예시 목표는 `광부의 단검` 또는 `수습대장의 검`이다.
- 필요한 재료 목록과 현재 보유량을 표시하되, 실제 제작은 M5에서 완성한다. 단, M4에서 "수리 의뢰" 형태의 간단한 보상 지급은 허용한다.
- 폐광 사건 진행 또는 마을 신뢰도 상승 후 대장간 기능이 개선된다.
- 기능 버튼은 `제작 목표 보기`, `수리 의뢰 확인`, `재료 힌트 보기` 정도로 제한한다.

**약초상**

- 약초 채집 계열 허드렛일 의뢰와 조사 목표를 안내한다.
- 현재 지역에서 얻을 수 있는 약초/접착 수액/회복 재료의 힌트를 제공한다.
- 신뢰도 상승 후 피로 회복 또는 부상 회복 보조를 제한적으로 제공한다. M4에서는 무료 남용을 막기 위해 1회성 보상 또는 쿨다운 보상으로 제한한다.
- 약초상 의뢰는 파견 메뉴에 노출되며, 약초상 방문은 해당 의뢰의 맥락과 보상을 설명하는 역할을 한다.
- 기능 버튼은 `채집 정보 보기`, `회복 보조 받기`, `재료 힌트 보기` 정도로 제한한다.

이 3개 거점은 모두 시간 소모가 없는 정보·보상·해금 인터랙션으로 시작한다. 실제 시간 소모 행동은 파견, 조사, 이동, 건설 같은 기존 시스템이 담당한다.

### 작업 상세

**기획 (content-designer / balance-designer)**

- 40개 리전 축소안을 확정한다. Tier 1은 3개, Tier 10은 2개로 고정하고, Tier 2~9는 총량 35개 안에서 적절히 배분한다.
- 기존 199개 리전의 컨텐츠를 새 40개 리전으로 재배치하거나 후순위 후보로 보관한다.
- 기본 4섹터, 특수 지역 최대 6섹터 규칙에 맞춰 섹터 구조를 재설계한다.
- 시작 지역 `더스트플레인`과 시작 마을 `더스트빌`의 분위기, NPC 4~6명, 방문 거점 3개를 정의한다.
- 촌장 집, 낡은 대장간, 약초상의 MVP 기능과 상태 변화 문구를 정의한다.
- 초반 2시간 플레이 흐름을 분 단위로 설계한다: 첫 모집 → 허드렛일 → 조사 → 재료 발견 → 첫 위험 의뢰 → 마을 인정.
- 고정 사건 1개를 설계한다. 예: 폐광길 재개방, 실종된 광부 수색, 야간 습격 방어.
- 허드렛일 의뢰 10~15개와 사건 의뢰 5~8개를 설계한다.
- 마을 신뢰도 구간별 해금과 보상 수치를 정의한다.

**구현 (Flutter — band_of_mercenaries)**

- `regions` 데이터와 관련 정적 데이터를 40개 리전 기준으로 재정렬한다. 기존 ID 참조가 많은 경우, 명세 단계에서 ID 유지/신규 ID 부여 정책을 확정한다.
- 이동 화면의 섹터 그리드를 기본 4개 섹터 기준으로 조정하고, 지역별 섹터 수를 데이터 기반으로 렌더링한다.
- 마을/도시 섹터 메타데이터를 추가한다. `더스트플레인` 1번 섹터는 `더스트빌`로 고정한다.
- 마을 섹터 선택 시 하단에 "마을 내 방문" 영역을 표시한다.
- `더스트빌` 방문 거점으로 촌장 집, 낡은 대장간, 약초상을 추가한다.
- 촌장 집 화면에서 고정 사건 단계, 마을 신뢰도, 다음 목표를 표시한다.
- 낡은 대장간 화면에서 첫 제작 목표, 필요 재료, 재료 획득 힌트를 표시한다.
- 약초상 화면에서 채집 의뢰 맥락, 회복 보조, 재료 획득 힌트를 표시한다.
- `QuestGenerator`에 거점 고정 의뢰 또는 `fixed_quest` 풀을 추가한다. 랜덤 의뢰와 별도로 초반 사건 의뢰를 노출한다.
- 마을 신뢰도 저장소를 추가한다. MVP에서는 `regionStates` 확장 또는 신규 `settlementStates` 박스를 검토한다.
- 고정 사건 진행 상태를 저장한다. 기존 `chainQuestProgress`를 재사용하거나 `settlement_event_progress` 모델을 신설한다.
- 결과 다이얼로그와 활동 로그에 마을 반응 문구를 추가한다.

**운영 도구 (operation-bom)**

- 40개 리전 재편 데이터를 편집하고 검증할 수 있게 한다.
- 리전별 섹터 수(4~6), 마을/도시 섹터 여부, 마을 방문 거점을 편집할 수 있게 한다.
- 방문 거점별 기능 타입, 상태 문구, 버튼 라벨, 해금 조건을 편집할 수 있게 한다.
- 거점, 방문 장소, 고정 의뢰, 지역 신뢰도 보상을 편집할 수 있는 최소 테이블 구성을 추가한다.
- MVP 단계에서는 1개 거점만 입력해도 동작해야 한다.
- `data_versions`에 신규 또는 확장 테이블을 반영한다.

**데이터 입력**

- 리전 40개. Tier 1은 3개, Tier 10은 2개로 고정한다.
- 리전별 섹터 4개 기본값. 특수 지역만 5~6개를 사용한다.
- 기존 컨텐츠 재배치 매핑표 1개. `region_discoveries`, `quest_pools`, `elite_monsters`, `chain_quests`, `factions` 참조를 포함한다.
- 시작 지역 `더스트플레인` 1개.
- 시작 마을 `더스트빌` 1개.
- 마을 방문 거점 3개: 촌장 집, 낡은 대장간, 약초상.
- 방문 거점별 상태 문구 8~12개.
- 방문 거점별 버튼/행동 정의 6~9개.
- 폐광 섹터 1개.
- NPC 반응 텍스트 20~30개.
- 허드렛일 의뢰 10~15개.
- 고정 사건 의뢰 5~8개.
- 마을 신뢰도 보상 4단계.

**밸런스 체크**

- 첫 30분 안에 실패해도 회복 가능한 골드 흐름을 보장한다.
- 첫 2시간 안에 신뢰도 단계가 최소 2회 상승해야 한다.
- 고정 사건 최종 의뢰는 준비 없이 어렵지만, 조사·제작·용병 성장을 거치면 60~80% 성공률이 되도록 조정한다.

### 누적 플레이 기준 (M4 종료 시)

- 신규 유저가 2시간 안에 시작 거점에서 허드렛일을 수행하고, 마을 신뢰도를 올리고, 고정 사건 1개를 해결하며, "이 마을은 내 용병단 때문에 달라졌다"는 변화를 확인해야 한다.
- 파견, 조사, 모집, 시설, 서사, 이동 선택지가 한 거점 안에서 서로 연결되어야 한다.

### 완료 점검표

- [ ] 전체 리전이 40개로 축소되고, Tier 1은 3개, Tier 10은 2개로 고정된다.
- [ ] 모든 일반 리전은 4개 섹터를 기본으로 사용하고, 특수 지역만 5~6개 섹터를 사용한다.
- [ ] 기존 지역 참조 데이터가 새 40개 리전 구조에 맞게 재배치된다.
- [ ] 모든 신규 유저가 `더스트플레인`의 `더스트빌`에서 시작한다.
- [ ] 마을/도시가 있는 지역의 1번 섹터가 마을/도시로 표시된다.
- [ ] **이동** 화면에서 `더스트빌` 섹터 선택 시 촌장 집, 낡은 대장간, 약초상 방문 기능이 하단에 표시된다.
- [ ] 폐광은 마을 내부 거점이 아니라 `더스트플레인`의 별도 섹터로 표시된다.
- [ ] 시작 거점 1개가 이동/홈/정보 UI에서 명확히 표시된다.
- [ ] 마을 방문 UI에서 장소와 신뢰도 상태를 확인할 수 있다.
- [ ] 촌장 집에서 고정 사건 단계, 마을 신뢰도, 다음 목표를 확인할 수 있다.
- [ ] 낡은 대장간에서 첫 제작 목표, 필요 재료, 재료 힌트를 확인할 수 있다.
- [ ] 약초상에서 채집 의뢰 맥락, 회복 보조, 재료 힌트를 확인할 수 있다.
- [ ] 허드렛일 의뢰가 랜덤 의뢰와 구분되어 노출된다.
- [ ] 의뢰소는 마을 방문 거점으로 중복 구현하지 않고, 파견 메뉴가 의뢰 기능을 담당한다.
- [ ] 고정 사건 진행 상태가 저장되고 재접속 후에도 유지된다.
- [ ] 사건 해결 후 NPC 반응, 의뢰 등급, 상점/대장간 상태 중 최소 2개가 변화한다.
- [ ] 첫 2시간 플레이테스트에서 신뢰도 상승 2회 이상, 고정 사건 해결 1회 이상이 가능하다.
- [ ] M1~M3 기능 회귀 이상 없음.

### M4 완료 후 게임 상태

> 플레이어는 40개로 재편된 대륙의 Tier 1 지역 `더스트플레인`에서 시작한다. 1번 섹터 `더스트빌`에 들어가 촌장 집, 낡은 대장간, 약초상을 방문하고, 파견 메뉴에서 마을의 짐 나르기, 야간 순찰, 폐광 조사 같은 허드렛일을 맡는다. 반복 파견 끝에 마을 신뢰도가 오르고, 닫혀 있던 대장간이 열리며, 별도 섹터인 폐광을 둘러싼 고정 사건이 진행된다. 사건을 해결하면 마을은 용병단을 다르게 대하고, 플레이어는 "내가 이곳에서 출발했다"는 기억을 갖는다.

## M5: 재료와 제작

### 목표

M5는 단순 완제품 드랍 중심의 보상 구조를 "여러 활동에서 재료를 모아 장비와 아티팩트를 완성하는 구조"로 바꾼다. 완제품 획득은 유지하지만, 핵심 장비와 아티팩트는 파견, 조사, 엘리트, 이동 선택지, 고정 사건에서 얻는 재료를 조합해 완성한다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **재료 아이템 분류** | 광석, 가죽, 약초, 유물 파편, 몬스터 부산물, 지역 특산품 등 제작용 아이템을 추가한다 |
| **드랍 출처 연결** | 의뢰 유형, 조사 발견, 엘리트, 이동 선택지, 고정 사건이 서로 다른 재료를 제공한다 |
| **제작 레시피** | 특정 재료 조합으로 장비·소모품·아티팩트를 제작한다 |
| **아티팩트 조립** | 여러 파편을 모아 용병단 장비 또는 지역 고유 아티팩트를 완성한다 |
| **완제품 병행 획득** | 엘리트·체인 퀘스트는 완제품을 낮은 확률 또는 확정 보상으로 제공할 수 있다 |

### 설계 방향

M5는 Melvor Idle식 수집 목표를 이 게임의 파견·조사 구조에 맞게 변환한다. 플레이어는 "어디서 무엇을 해야 어떤 재료가 나오는지"를 학습하고, 한 장비를 완성하기 위해 여러 활동을 순환한다.

초반 대표 제작 목표는 다음 3개를 권장한다.

| 목표 | 의미 | 재료 출처 |
|------|------|----------|
| 낡은 용병단 깃발 복원 | 용병단 정체성의 첫 상징 | 허드렛일, 마을 신뢰도 보상, 폐광 사건 |
| 광부의 단검 또는 수습대장의 검 | 첫 희귀 개인 장비 | 폐광 조사, 도적 의뢰, 대장간 해금 |
| 폐광의 유물 조각 | 첫 지역 아티팩트 | 고정 사건 단계, 첫 엘리트, 숨겨진 발견 |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 재료 분류 체계와 희귀도 규칙을 정의한다.
- 시작 거점 전용 재료 8~12개를 설계한다.
- 초반 제작 레시피 8~12개를 설계한다.
- 완제품 드랍과 재료 제작의 시간 효율을 비교한다.

**구현 (Flutter — band_of_mercenaries)**

- `items` 테이블의 category/slot/effect 구조를 재료와 레시피에 맞게 확장한다.
- `crafting_recipes` 정적 테이블 또는 JSONB 기반 MVP를 추가한다.
- `CraftingService`를 추가하여 재료 보유량, 제작 가능 여부, 제작 결과를 계산한다.
- 인벤토리 UI에 재료 탭과 제작 가능 표시를 추가한다.
- 거점 대장간 또는 제작 장소에서 레시피를 확인하고 제작한다.
- 퀘스트 완료, 조사 완료, 이동 선택지 결과, 엘리트 드랍에 재료 보상을 연결한다.

**운영 도구 (operation-bom)**

- 제작 레시피 편집 UI를 추가한다.
- 아이템별 주요 획득처를 관리할 수 있게 한다.
- 검증 쿼리로 "레시피 재료가 실제 획득 가능한가"를 확인한다.

**데이터 입력**

- 재료 20~30종.
- 제작 레시피 10~15개.
- 시작 거점 전용 아티팩트 1~2개.
- 기존 퀘스트·조사·엘리트 보상에 재료 드랍 연결.

**밸런스 체크**

- 첫 제작 성공은 30~45분 안에 발생해야 한다.
- 첫 희귀 장비 완성은 90~150분 사이에 가능해야 한다.
- 완제품 드랍은 기쁘지만, 제작 루트를 무력화하지 않는 속도로 제한한다.

### 누적 플레이 기준 (M5 종료 시)

- 신규 유저가 M4의 시작 거점 흐름을 플레이하면서 재료를 모아 첫 제작을 완료하고, 2~3시간 안에 첫 희귀 장비 또는 지역 아티팩트 조각을 완성할 수 있어야 한다.

### 완료 점검표

- [ ] 재료 아이템이 인벤토리에서 별도 구분된다.
- [ ] 제작 레시피가 UI에서 재료 충족/부족 상태를 명확히 표시한다.
- [ ] 파견, 조사, 엘리트, 이동 선택지 중 3개 이상이 재료 출처로 연결된다.
- [ ] 첫 제작 목표 3개가 실제 플레이로 달성 가능하다.
- [ ] 완제품 드랍과 제작 루트가 서로 공존한다.
- [ ] M1~M4 기능 회귀 이상 없음.

### M5 완료 후 게임 상태

> 플레이어는 폐광에서 광석을 얻고, 도적 의뢰에서 가죽끈을 얻고, 약초상 의뢰로 접착 수액을 구한다. 이 재료를 낡은 대장간에 가져가 첫 희귀 장비를 완성한다. 장비는 단순 드랍이 아니라 여러 활동을 묶는 목표가 되며, 플레이어는 "다음 재료는 어디서 얻지?"라는 계획을 세운다.

## M6: 이름을 얻는 용병단

### 목표

M6는 강한 용병과 성장한 용병단이 세계에서 다르게 취급받는 체감을 만든다. 특정 용병이 강해졌을 때 단순 성공률 상승을 넘어 칭호, 위업, 지명 의뢰, 연대기 기록이 발생해야 한다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **위업 기록** | 사건 해결, 첫 엘리트 토벌, 고정 사건 완주, 제작 완성 같은 주요 순간을 영구 기록한다 |
| **용병 칭호** | 특정 조건을 달성한 용병에게 지역·사건 기반 칭호를 부여한다 |
| **간판 용병** | 용병단을 대표하는 용병을 자동 선정하거나 수동 지정한다 |
| **지명 의뢰** | 강한 용병 또는 특정 칭호 보유 용병을 요구하는 의뢰가 등장한다 |
| **용병단 연대기** | 홈 또는 정보 탭에서 용병단의 주요 사건을 시간순으로 확인한다 |

### 설계 방향

M6의 기준은 "플레이어가 용병 이름을 기억하는가"이다. 용병단이 성장했음을 보여주는 가장 강한 장치는 추상 랭크보다 특정 용병의 일대기다.

칭호 예시는 다음과 같다.

| 칭호 | 획득 조건 | 효과 |
|------|----------|------|
| 마을의 은인 | 시작 거점 고정 사건 최종 의뢰 참여 및 성공 | 시작 거점 의뢰 성공률 소량 증가 |
| 폐광의 생존자 | 폐광 사건 중 부상 또는 대실패 후 생존 | 부상 회복 시간 감소 또는 VIT 소량 증가 |
| 첫 깃발을 든 자 | 용병단 깃발 복원 제작에 참여 | 용병단 장비 효과 소량 증가 |
| 도적길 추적자 | 도적 관련 의뢰 다수 성공 | hunt/raid 계열 초반 의뢰 보정 |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 위업 카테고리와 기록 조건을 정의한다.
- 초반 칭호 8~12개를 설계한다.
- 지명 의뢰 5~8개를 설계한다.
- 칭호 효과가 기존 트레잇과 중복되지 않도록 강도를 제한한다.

**구현 (Flutter — band_of_mercenaries)**

- `BandAchievement` 또는 `ChronicleEntry` 모델을 추가한다.
- `Mercenary`에 칭호 또는 titleIds 필드를 추가한다.
- 주요 서비스(퀘스트 완료, 제작 완료, 체인 완료, 조사 완료)에서 위업 기록 이벤트를 발행한다.
- 홈 화면에 최근 위업 1~3개를 표시한다.
- 용병 상세에 칭호 섹션을 추가한다.
- 지명 의뢰 생성 시 칭호/위업/간판 용병 조건을 평가한다.

**운영 도구 (operation-bom)**

- 칭호 데이터와 지명 의뢰 조건을 편집할 수 있게 한다.
- 위업 템플릿과 연대기 문구를 관리한다.

**데이터 입력**

- 위업 템플릿 20~30개.
- 용병 칭호 8~12개.
- 지명 의뢰 5~8개.
- 연대기 문구 30~50개.

**밸런스 체크**

- 칭호 효과는 플레이 감각을 주되 필수 최적해가 되지 않도록 제한한다.
- 지명 의뢰는 조건이 너무 좁아 노출되지 않는 상황을 피한다.
- 사망한 용병의 칭호와 위업은 유지하여 상실감이 기록으로 남게 한다.

### 누적 플레이 기준 (M6 종료 시)

- 신규 유저가 3~5시간 안에 최소 1명의 용병 이름과 칭호를 기억해야 한다.
- 시작 거점 사건을 해결한 용병이 연대기에 남고, 해당 용병을 요구하거나 우대하는 지명 의뢰가 1회 이상 등장해야 한다.

### 완료 점검표

- [ ] 주요 사건이 용병단 연대기에 기록된다.
- [ ] 용병 칭호가 조건에 따라 부여되고 상세 화면에 표시된다.
- [ ] 간판 용병을 확인하거나 지정할 수 있다.
- [ ] 지명 의뢰가 특정 칭호/위업/간판 용병 조건으로 등장한다.
- [ ] 사망 또는 방출된 용병의 기록이 연대기에 남는다.
- [ ] M1~M5 기능 회귀 이상 없음.

### M6 완료 후 게임 상태

> 플레이어는 단순히 강한 카드를 보유한 것이 아니라 "폐광의 생존자 김철수"를 보유한다. 마을 사람들은 그 이름을 기억하고, 새로운 의뢰는 그 용병을 지명한다. 용병단은 수치상 강해지는 것뿐 아니라 세계 안에서 이름을 얻는다.

## M7: 지역 생활권 확장

### 목표

M7은 M4의 시작 거점 패턴을 주변 5~10개 리전으로 확장한다. 전 대륙을 얕게 채우지 않고, 시작 거점 주변에 재료, 사건, 위험, 이동 목적이 있는 생활권을 만든다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **생활권 리전 묶음** | 시작 거점 주변 5~10개 리전을 하나의 초반 생활권으로 정의한다 |
| **지역 특산품** | 리전별로 고유 재료 또는 높은 드랍률을 가진 재료를 배치한다 |
| **지역 상태 변화** | 사건 해결에 따라 리전 위험도, 의뢰 유형, 조사 발견이 변화한다 |
| **마을 인프라 성장** | 시작 거점의 방문 거점, 대장간, 시장 상태가 주변 사건 해결에 따라 개선된다 |
| **이동 목적 강화** | 이동은 퀘스트 갱신이 아니라 필요한 재료·사건·위협을 찾아가는 선택이 된다 |

### 작업 상세

**기획**

- 시작 거점 주변 리전 5~10개를 선정한다.
- 각 리전에 특산 재료, 고정 발견, 반복 의뢰 성격을 부여한다.
- 지역 상태 변화 규칙을 설계한다. 예: 도적 소탕 후 호위 의뢰 증가, 폐광 안정화 후 광석 드랍 증가.
- 생활권 내 중간 목표와 최종 목표를 설계한다.

**구현**

- `regionStates` 또는 신규 지역 상태 모델에 위험도/안정도/해금 상태를 추가한다.
- `QuestGenerator`가 지역 상태에 따라 퀘스트 풀 가중치를 조정한다.
- 이동 화면에서 생활권 목적지를 읽기 쉽게 표시한다.
- 거점 상세 화면에서 주변 리전 상태를 요약한다.

**데이터 입력**

- 생활권 리전 5~10개.
- 지역 특산 재료 10~20개.
- 고정 발견 20~40개.
- 지역 상태별 퀘스트 풀 30~50개.

### 누적 플레이 기준 (M7 종료 시)

- 신규 유저가 시작 거점 밖으로 이동할 때 "필요한 재료를 얻기 위해", "사건을 해결하기 위해", "위험도를 낮추기 위해"라는 명확한 목적을 가져야 한다.
- 5~8시간 안에 초반 생활권의 주요 사건 2개 이상, 제작 목표 2개 이상, 지역 상태 변화 2개 이상을 경험해야 한다.

### 완료 점검표

- [ ] 시작 생활권 5~10개 리전이 데이터로 정의되어 있다.
- [ ] 리전별 특산품 또는 고유 발견이 존재한다.
- [ ] 지역 상태가 퀘스트 생성 또는 보상에 영향을 준다.
- [ ] 거점 UI에서 주변 생활권 상태를 확인할 수 있다.
- [ ] 이동 목적이 재료/사건/상태 변화와 연결된다.
- [ ] M1~M6 기능 회귀 이상 없음.

### M7 완료 후 게임 상태

> 플레이어는 더 이상 아무 리전이나 이동하지 않는다. 폐광의 철광석, 숲의 약초, 옛길의 도적 흔적처럼 목적이 있는 장소를 찾아간다. 용병단이 사건을 해결하면 주변 지역의 의뢰와 보상이 변하고, 시작 거점은 점점 살아난다.

## M8: 세력 재도입

### 목표

M8은 기존 M4에서 계획했던 세력 심화 요소를 축소된 수직 절편으로 재도입한다. 14세력 전체를 한 번에 완성하지 않고, 시작 생활권과 자연스럽게 연결되는 2~3개 세력만 먼저 깊게 구현한다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **대표 세력 2~3개** | 시작 생활권 사건과 관련된 세력만 우선 구현한다 |
| **세력 거점/접촉점** | 거대한 세력 본부보다 초반 생활권의 대리인, 지부, 후원자 형태로 등장한다 |
| **세력 지명 의뢰** | 용병단 위업과 지역 신뢰도에 반응하여 세력이 의뢰를 맡긴다 |
| **세력 상점 축소판** | 세력당 5~8개 상품만 제공하고, 제작 재료·레시피·아티팩트와 연결한다 |
| **세력 전용 보상** | 전용 시설 14종 대신 대표 세력별 보상 1~2개를 구현한다 |

### 설계 방향

세력은 추상 보너스 제공자가 아니라 "큰 세계가 용병단을 알아보기 시작했다"는 신호로 작동해야 한다. 시작 거점에서 이름을 얻은 용병단에게 세력의 지부가 접근하고, 세력 의뢰는 지역 생활권을 더 큰 세계와 연결한다.

### 작업 상세

**기획**

- 시작 생활권과 가장 잘 맞는 대표 세력 2~3개를 선정한다.
- 각 세력의 등장 계기, 지명 의뢰, 상품, 보상, 갈등 관계를 설계한다.
- 기존 M4 문서의 세력 시설·상점·트레잇 설계를 축소 적용한다.

**구현**

- `FactionData`에 생활권 접촉점 또는 지부 정보를 추가한다.
- 세력 상점은 기존 인벤토리/제작 시스템과 연결한다.
- 세력 의뢰는 지명 의뢰와 지역 상태 조건을 함께 평가한다.
- 전용 보상은 시설 대량 확장보다 패시브 보정, 레시피, 칭호, 아티팩트로 우선 구현한다.

**데이터 입력**

- 대표 세력 2~3개.
- 세력 지명 의뢰 10~15개.
- 세력 상점 상품 15~24개.
- 세력 레시피 또는 아티팩트 3~6개.
- 세력 반응 텍스트 30~50개.

### 누적 플레이 기준 (M8 종료 시)

- 플레이어가 시작 생활권에서 위업을 쌓은 뒤, 적어도 1개 세력이 용병단을 주목하고 지명 의뢰를 맡기는 경험을 해야 한다.
- 세력 보상은 단순 수치 보너스가 아니라 제작, 칭호, 지역 사건, 용병단 위상과 연결되어야 한다.

### 완료 점검표

- [ ] 대표 세력 2~3개가 생활권 사건과 연결되어 등장한다.
- [ ] 세력 지명 의뢰가 위업/신뢰도/칭호 조건으로 노출된다.
- [ ] 세력 상점 상품이 제작 재료 또는 레시피와 연결된다.
- [ ] 세력 보상이 용병단 위상 또는 지역 상태에 영향을 준다.
- [ ] 14세력 전체 확장을 전제로 하지 않아도 MVP가 완결된다.
- [ ] M1~M7 기능 회귀 이상 없음.

### M8 완료 후 게임 상태

> 마을에서 이름을 얻은 용병단에게 모험가 길드 지부와 상인 연합 대리인이 접근한다. 이들은 단순한 보너스 제공자가 아니라 지역 사건을 더 큰 세계와 연결하는 의뢰자다. 플레이어는 세력의 물건을 사는 것이 아니라 세력의 눈에 들어 더 큰 무대로 초대받는다.

## M9: 완성과 균형

### 목표

M9는 M4~M8에서 검증된 재미 축을 기준으로 데이터를 확장하고, stub 기능을 실제 동작으로 연결하며, 경제·전투·성장 밸런스를 통합 조정한다. 기존 v2의 "전체 데이터 채우기"는 이 시점에서만 재개한다.

### 포함 작업

| 축 | 핵심 변화 |
|----|----------|
| **데이터 확장** | 시작 생활권 패턴을 다른 리전군으로 확장한다 |
| **stub 시설 연동** | 대장간, 연구소, 게시판 등 기존 stub 시설을 제작·칭호·고정 사건과 연결한다 |
| **티어 업그레이드 재검토** | 용병단 위상과 제작 경제가 안정된 뒤 승급 시스템을 다시 설계한다 |
| **전체 밸런스 튜닝** | 초반 2시간, 5시간, 20시간, 장기 플레이 수지를 통합 검증한다 |
| **품질 마무리** | UI 일관성, 활동 로그 톤, 온보딩, 엣지 케이스, 데이터 검증 도구를 정리한다 |

### 작업 상세

**데이터 확장**

- 생활권 단위로 리전 묶음을 추가한다.
- 고정 사건, 재료 출처, 제작 목표, 지명 의뢰를 세트로 입력한다.
- 199개 리전을 모두 같은 밀도로 채우지 않고, 플레이 구간별 핵심 생활권부터 완성한다.

**stub 시설 연동**

| 시설 | M9 작업 |
|------|---------|
| 대장간 | 제작 레시피, 장비 강화, 아티팩트 복원과 연결한다 |
| 연구소 | 고급 트레잇 조건 완화, 특수 조사, 제작법 해석과 연결한다 |
| 게시판 | 고정 사건 힌트, 지명 의뢰 확률, 생활권 사건 추적과 연결한다 |

**티어 업그레이드 재검토**

- 기존 v2의 M6 티어 업그레이드는 M9 이후 후보로 보관한다.
- 승급은 단순 스탯 상승보다 위업, 칭호, 제작 재료, 용병단 등급을 요구하는 장기 목표로 재설계한다.
- 승급 경제는 제작 경제와 충돌하지 않도록 별도 시뮬레이션한다.

**밸런스 체크**

- 초반 2시간: 허드렛일 → 마을 인정 → 첫 제작 → 첫 사건 해결.
- 5시간: 생활권 이동 목적, 재료 목표, 첫 칭호, 지명 의뢰.
- 20시간: 세력 재도입, 아티팩트 완성, 지역 상태 변화 누적.
- 장기: 데이터 반복 피로, 골드·재료 인플레이션, 성공률 상한, 용병 사망/상실 체감.

### 누적 플레이 기준 (M9 종료 시)

- 신규 유저가 첫 2시간에 재미를 느끼고, 5~8시간에 생활권 목표를 이해하며, 20시간 이후에도 제작·위업·세력·지역 확장 목표를 가진다.
- 게임은 "랜덤 의뢰 반복"이 아니라 "장소, 사건, 재료, 용병, 세력이 연결된 용병단 성장 시뮬레이션"으로 체감되어야 한다.

### 완료 점검표

- [ ] 첫 2시간 플레이테스트에서 이탈 지점과 목표 혼란이 크게 줄어든다.
- [ ] 생활권 단위 데이터 확장 지침이 정립된다.
- [ ] 대장간·연구소·게시판 stub이 실제 시스템과 연결된다.
- [ ] 티어 업그레이드 재도입 여부와 조건이 별도 문서로 확정된다.
- [ ] 경제·전투·성장·제작·위업 밸런스 시뮬레이션을 완료한다.
- [ ] operation-bom에 데이터 검증 대시보드 또는 검증 쿼리가 준비된다.
- [ ] M1~M8 전체 회귀 테스트 통과.

### M9 완료 후 게임 상태

> 게임은 더 이상 넓은 맵에 흩어진 랜덤 의뢰 묶음이 아니다. 플레이어는 변방 마을에서 시작해 생활권의 문제를 해결하고, 재료를 모아 장비와 아티팩트를 완성하며, 특정 용병을 전설처럼 기억하고, 세력의 눈에 들어 더 큰 세계로 나아간다. 전체 데이터 확장은 이 검증된 구조를 반복 가능한 패턴으로 넓힌다.

### 마일스톤별 작업 흐름 (반복)

```
1. 기획 확정
   └─ content-designer / balance-designer로 컨텐츠/수치 설계
   └─ M4 이후에는 "초반 2시간 재미 검증" 관점에서 범위 축소 여부를 먼저 판단
   └─ 기획자 검토 및 확정

2. 개발 명세서 생성
   └─ spec-writer로 마일스톤별 개발 명세서 생성
   └─ verify-spec으로 검증

3. 구현
   └─ implement-spec 또는 implement-agent로 Flutter + operation-bom 동시 구현
   └─ 코드 리뷰

4. 데이터 입력
   └─ operation-bom에서 기획 확정된 데이터 입력
   └─ 누적 플레이 기준을 만족하는 분량 확보
   └─ data_versions 퍼블리시

5. 밸런스 체크포인트
   └─ balance-designer로 시뮬레이션
   └─ 이상치 수정 → 재시뮬레이션

6. 완료 점검
   └─ 마일스톤 점검표 전체 항목 확인
   └─ 이전 마일스톤 회귀 테스트
   └─ 누적 플레이 기준 실제 검증 (처음부터 해당 마일스톤까지 연속 플레이)
   └─ 다음 마일스톤 진입 결정
```

### Supabase 테이블 누적 추이

M3 완료 이후의 테이블 증가는 "전 대륙 데이터 확장"이 아니라 시작 거점과 초반 생활권을 검증하기 위한 최소 구조를 우선한다. 정확한 테이블명은 각 마일스톤 명세에서 확정한다.

| 시점 | 예상 변화 |
|------|----------|
| M3 완료 기준 | `chain_quests`, `quest_narratives`, `travel_choice_*` 등 M3 신규 테이블 반영 완료 |
| M4 후 | 40개 리전 재편, 기본 4섹터 구조, 거점/장소/고정 의뢰/마을 신뢰도 관련 테이블 또는 기존 테이블 확장 |
| M5 후 | 제작 레시피, 재료 출처, 아티팩트 조립 관련 테이블 또는 `items` 확장 |
| M6 후 | 위업, 칭호, 지명 의뢰, 연대기 템플릿 관련 테이블 |
| M7 후 | 생활권, 지역 상태, 특산품 연결 데이터 확장 |
| M8 후 | 대표 세력 2~3개의 지부/상점/의뢰 데이터 확장 |
| M9 후 | 검증된 구조를 기준으로 데이터 확장 및 검증 대시보드 추가 |

### Hive 박스 누적 추이

M3 완료 이후에는 기존 박스 확장을 우선 검토하고, 별도 수명주기나 watch 스트림이 필요한 데이터만 신규 박스로 분리한다.

| 시점 | 예상 변화 |
|------|----------|
| M4 후 | `regionStates` 확장 또는 `settlementStates` 신규 박스 |
| M5 후 | `inventory` 확장. 제작 진행 상태가 필요하면 별도 제작 상태 박스 검토 |
| M6 후 | `mercenaries` 칭호 필드 확장 또는 `chronicle` 신규 박스 |
| M7 후 | `regionStates`에 지역 상태/생활권 상태 확장 |
| M8 후 | `factionStates`에 세력 지부 방문/상점 이력 확장 가능 |
| M9 후 | 성능과 복원 안정성을 기준으로 박스 분리 여부 재검토 |

## 마일스톤 실행 가이드

각 마일스톤을 시작할 때 다음 절차를 따른다.

```bash
# 1. 기획 확정 후 개발 명세서 생성
/spec-writer @Docs/roadmap/master_roadmap.md  # M{N} 섹션 참조

# 2. 명세서 검증
/verify-spec @Docs/spec/M{N}_spec.md

# 3. 구현 실행
/implement-spec @Docs/spec/M{N}_spec.md
# 또는
/implement-agent @Docs/spec/M{N}_spec.md
```

각 마일스톤 완료 시 다음을 수행한다.

1. 해당 마일스톤의 **완료 점검표** 전 항목 확인.
2. **이전 마일스톤 회귀 테스트** 수행.
3. **balance-designer로 밸런스 스냅샷** 생성.
4. **누적 플레이 기준** 실제 검증 (처음부터 해당 마일스톤까지 연속 플레이).
5. `content_status.md` 업데이트.
6. 다음 마일스톤 진입 결정.

## 다음 단계

M1~M3은 완료된 기준으로 유지한다. 다음 작업은 새 M4의 기획 확정이다.
