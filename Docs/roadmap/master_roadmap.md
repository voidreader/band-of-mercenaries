### 미구현 시스템 (이 로드맵의 범위)

16개 시스템을 6개 마일스톤에 배치하여 순차 구현한다.

## 설계 원칙

### 마일스톤 운영 원칙

1. **매 마일스톤 = 기획 + 구현 + 데이터 + 밸런스 스냅샷** — 기능만 만들고 끝나지 않음
2. **기획자 선행** — 현재 마일스톤 구현 중에 기획자는 다음 마일스톤의 컨텐츠/데이터 설계
3. **마일스톤 간 밸런스 체크포인트** — 각 마일스톤 완료 시 balance-designer로 경제 시뮬레이션, 다음 마일스톤 진입 전 수치 확정
4. **통합 가능한 시스템은 통합** — trait Phase D(특수 임무)와 world expansion Phase 4(연계 퀘스트)는 M3에서 하나로 합침
5. **릴리스 테마 기반** — 매 마일스톤이 플레이어 관점에서 의미 있는 변화를 전달

### 데이터 설계 원칙

- **구조화된 데이터는 JSONB 대신 정규화 테이블 사용** — operation-bom에서 일반 CRUD 폼으로 편집 가능, DB 레벨 검증, 기획자가 JSON 문법 불필요
- 기존에 동작하는 JSONB 필드(effect_json, discovery_data 등 스키마가 타입별로 가변적인 것)는 유지
- **신규 추가하는 데이터는 정규화 우선**

### 우선순위 기준

- **의존 관계를 기본 골격으로** — 다른 시스템의 선행 조건이 되는 것을 먼저
- **독립 기능은 게임 경험 임팩트 순으로 배치** — 의존 체인 사이의 빈 슬롯에 끼워넣기

## 마일스톤 전체 구조

```
M1 "세력의 영향력"     ── 세력이 실제 게임에 영향을 미치기 시작
  ↓
M2 "아이템의 시대"     ── 새로운 보상 축(아이템) 도입 + 엘리트 몬스터
  ↓
M3 "깊어지는 세계"     ── 연계 퀘스트 + 지역 변형 + 서사
  ↓
M4 "세력의 심화"       ── 전용 시설/거점/상점으로 세력 완결
  ↓
M5 "용병의 성장"       ── 용병 개성 강화 + 전투 깊이 + 선택지
  ↓
M6 "완성과 균형"       ── 전체 데이터 채우기 + 밸런싱 + 최종 폴리싱
```

### 의존 관계

```
M1 (세력 영향력) ──→ M4 (세력 심화)     : 패시브/태그가 있어야 시설/상점이 의미 있음
M2 (아이템)      ──→ M3 (연계 퀘스트)   : 아이템이 있어야 연계 퀘스트 보상 가능
M2 (아이템)      ──→ M4 (세력 상점)     : 아이템이 있어야 상점에서 판매 가능
M1~M5 전체       ──→ M6 (완성과 균형)   : 모든 시스템이 있어야 전체 밸런싱 가능
M5 (용병 성장)   ── 독립                 : M1 이후 언제든 가능, M4 이후 권장
```

## M1: 세력의 영향력

### 목표

현재 세력 시스템은 "발견하고 가입하면 텍스트가 보이는" 수준이다. M1이 끝나면 세력 가입이 퀘스트 보상, 성공률, 경제 수치에 실제로 영향을 미치게 된다. 플레이어가 "어떤 세력에 가입할까"를 진지하게 고민하기 시작하는 지점.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **세력 패시브 효과 연동** | `passive_bonus_json`이 실제 게임 수치에 반영 (보상 배수, 성공률, 회복 시간 등) |
| **세력 퀘스트 태그 + 전용 퀘스트** | 일반 퀘스트에 세력 태그 부여 → 완료 시 평판 획득. 가입 세력 전용 고난도 퀘스트 생성 |
| **파견 상성 시스템** | 퀘스트 유형별 유리한 직업/트레잇 조합 → 파견 선택에 전략적 깊이 |
| **명성 등급별 보너스** | 랭크 상승 시 해금 외 추가 보너스 (보상 +%, 모집 할인 등) |

### 작업 상세

**기획 (content-designer / balance-designer)**

- 14개 세력의 패시브 보너스를 게임 메커니즘에 매핑하는 효과 타입 정의
- 세력 태그 퀘스트의 확률 테이블 설계 (거점 거리별 보정 포함)
- 세력 전용 퀘스트의 난이도/보상 설계 (일반 대비 +1~2 난이도, 평판 +5~10)
- 상성 매트릭스 설계: 직업군 × 퀘스트 유형별 보정값
- 명성 등급(F~A)별 보너스 수치 확정

**구현 (Flutter — band_of_mercenaries)**

- `PassiveBonusService` 신규: 패시브 보너스 데이터 조회 → 기존 서비스(QuestCalculator, RecruitmentService, 회복 로직 등)에 보정값 주입
- `QuestGenerator` 확장: 퀘스트 생성 시 세력 태그 확률 판정 + 가입 세력 전용 퀘스트 슬롯
- `QuestCompletionService` 확장: 태그 퀘스트 완료 시 `FactionStateRepository.addReputation()` 호출
- `QuestCalculator` 확장: 직업/트레잇 × 퀘스트 유형 상성 보정
- `ReputationService` 확장: 랭크별 보너스 적용
- 파견 화면에 세력 태그 배지 + 상성 힌트 UI

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블 생성:
  - `faction_passive_bonuses` (기존 JSONB → 정규화 테이블)
  - 상성 테이블 (신규 테이블 사용 시)
- `table-config.ts`에 신규 테이블 추가
- 세력 전용 퀘스트 풀 CRUD: `quest_pools`에 세력 태그 필드 추가 → table-config 반영
- `ranks` 테이블에 보너스 필드 추가 → table-config 반영
- 데이터 입력 후 data_versions 퍼블리시

**데이터 입력 (operation-bom에서 수행)**

- 14개 세력의 패시브 보너스 데이터 입력 (`faction_passive_bonuses` 테이블)
- 세력 전용 퀘스트 풀 추가 (세력당 3~5개, 총 42~70개)
- 상성 보정 데이터 입력
- `ranks` 테이블: 등급별 보너스 값 입력

**밸런스 체크 (balance-designer)**

- 세력 패시브가 경제에 미치는 영향 시뮬레이션 (보상 +15%가 골드 인플레이션을 유발하는지)
- 상성 보정이 성공률 분포를 어떻게 바꾸는지 검증
- 명성 보너스의 누적 효과 검증

### 완료 점검표

**기능 동작**

- [ ] 세력 가입 후 퀘스트 보상이 패시브 보너스만큼 실제로 변하는가?
- [ ] 세력 단서를 보유한 리전에서 퀘스트 생성 시 태그가 확률적으로 붙는가?
- [ ] 태그 퀘스트 완료 → 평판 +1~2 → 도감에서 평판 바 변화 확인
- [ ] 가입 세력의 전용 퀘스트가 파견 화면에 별도 표시되는가?
- [ ] 파견 화면에서 유리한 용병 조합에 대한 시각적 힌트가 보이는가?
- [ ] 랭크 E 이상에서 해당 등급의 보너스가 적용되는가?

**데이터 & 운영 도구**

- [ ] operation-bom에서 세력 전용 퀘스트를 생성/수정/삭제할 수 있는가?
- [ ] operation-bom에서 상성 데이터를 편집할 수 있는가?
- [ ] operation-bom에서 랭크 보너스를 편집할 수 있는가?
- [ ] data_versions 퍼블리시 후 Flutter 앱에서 신규 데이터가 정상 동기화되는가?

**밸런스 & 회귀**

- [ ] balance-designer로 경제 시뮬레이션 1회 이상 수행, 이상치 없음 확인
- [ ] 세력 미가입 상태에서 기존 게임 루프가 이전과 동일하게 동작

### M1 완료 후 게임 상태

> 플레이어가 세력에 가입하면 퀘스트 보상이 실제로 달라진다. 특정 세력 근처에서 태그 퀘스트를 수행하며 평판을 올릴 수 있고, 충분한 평판이 쌓이면 전용 고난도 퀘스트에 도전할 수 있다. 파견 시 "이 퀘스트에는 어떤 용병이 유리한가"를 고려하는 전략적 깊이가 생긴다. 명성이 올라갈수록 추가 보너스가 붙어 성장 동기가 강화된다. 운영팀은 operation-bom에서 세력 퀘스트, 상성 데이터, 랭크 보너스를 자유롭게 편집하고 앱에 반영할 수 있다.

## M2: 아이템의 시대

### 목표

게임에 골드 이외의 새로운 보상 축을 도입한다. 아이템(장비/정수/소모품)이라는 인프라를 깔고, 엘리트 몬스터를 통해 아이템을 획득하는 첫 번째 경로를 만든다. M2가 끝나면 "이 지역에 머물며 엘리트를 반복 사냥해서 정수를 모은다"는 새로운 플레이 패턴이 성립한다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **아이템/장비 인프라** | 아이템 모델, 인벤토리(Hive), 장비 장착/해제, 정수 사용(영구 스탯 강화), 인벤토리 UI |
| **엘리트 몬스터** | 지역 지식 임계값으로 해금, 반복 출현, 고난도 토벌 퀘스트, 드랍 판정 |

### 선행 조건

- 지역 조사 시스템 (완료) — 지식 임계값이 엘리트 출현 트리거
- `region_discoveries`의 `discovery_type: elite` (완료) — 엘리트 발견 이벤트 트리거 인프라

### 작업 상세

**기획 (content-designer / balance-designer)**

- 아이템 분류 체계 확정: 개인 장비(무기/방어구/장신구) + 용병단 장비(군기/유물) + 소모품(정수/일반)
- 정수 종류 및 영구 스탯 강화 수치 설계 (힘/수호/생명/상급)
- 엘리트 몬스터 카탈로그: 리전 티어별 전투력, 출현 확률, 보상 구성
- 드랍 테이블 설계: 아이템별 드랍률 가이드라인 (정수 40~60%, 장비 15~25%, 용병단 장비 3~8%)
- 장비 스탯 보정 범위, 용병단 장비 전체 효과 범위
- 검증용 최소 데이터 세트: 정수 4종, 개인 장비 6~8종, 용병단 장비 3~4종, 엘리트 10~15종

**구현 (Flutter — band_of_mercenaries)**

아이템 인프라:
- `ItemData` 정적 모델 (Freezed): id, name, category, slot, tier, description 등
- `InventoryItem` Hive 모델: itemId, quantity, equippedTo(용병 ID)
- `inventory` Hive 박스 신설
- `InventoryRepository`: 아이템 추가/제거/장착/해제 CRUD
- `ItemEffectService`: 장착 장비의 스탯 보정 → 기존 effectiveStr/effectiveAgi 등에 반영
- `EssenceService`: 정수 사용 → 용병 영구 스탯 증가 + 정수 소멸
- 인벤토리 UI: 아이템 목록 화면 (카테고리별 필터), 아이템 상세 팝업
- 용병 상세 오버레이에 장비 슬롯(무기/방어구/장신구) 추가
- `SyncService` 확장: `items` 테이블 동기화 추가

엘리트 몬스터:
- `EliteMonsterData` 정적 모델 (Freezed)
- `EliteLootEntry` 정적 모델: elite_id, item_id, drop_rate
- 엘리트 출현 로직: `RegionState.triggeredDiscoveries`에 엘리트 discovery 존재 → 퀘스트 갱신 시 확률 판정(티어별 8~15%)으로 엘리트 퀘스트 생성
- 엘리트 퀘스트: 기존 `ActiveQuest`에 `isElite` 플래그 추가, 전투력 100~200, 소요시간 1.5~2배
- 드랍 판정: 퀘스트 완료 시 `EliteLootService`가 각 드랍 항목별 독립 확률 판정 → 인벤토리에 아이템 추가
- 파견 화면에 엘리트 퀘스트 시각적 구분 (아이콘, 강조 색상, 최소 파견 인원 권장 표시)
- 퀘스트 완료 팝업에 아이템 드랍 결과 표시
- `SyncService` 확장: `elite_monsters`, `elite_loot_tables` 테이블 동기화 추가

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블 3개 생성:
  - `items` (아이템 마스터 데이터)
  - `elite_monsters` (엘리트 몬스터 정의)
  - `elite_loot_tables` (드랍 테이블)
- `table-config.ts`에 3개 테이블 정의 추가 → 자동 CRUD UI 생성
- `items` 편집 화면: category/slot/tier 셀렉트 필드
- `elite_loot_tables` 편집 화면: elite_id → 엘리트 이름 참조 표시, item_id → 아이템 이름 참조 표시, drop_rate 범위 검증 (0.0~1.0)
- `data_versions`에 3개 테이블 행 추가

**데이터 입력 (operation-bom에서 수행)**

- 정수 4종 (힘/수호/생명/상급) 데이터 입력
- 개인 장비 6~8종 (티어 2~5 분포) 데이터 입력
- 용병단 장비 3~4종 데이터 입력
- 엘리트 몬스터 10~15종 (리전 티어별 분포) 데이터 입력
- 드랍 테이블: 엘리트별 드랍 항목 및 확률 입력
- `region_discoveries`에 엘리트 관련 discovery 데이터 추가

**밸런스 체크 (balance-designer)**

- 정수의 영구 스탯 강화가 장기적으로 인플레이션을 유발하는지 시뮬레이션

## M3: 깊어지는 세계

### 목표

지역 조사의 최종 보상을 완성한다. 지식을 쌓으면 다단계 서사 퀘스트가 열리고, 끝까지 조사하면 지역 자체가 변형된다. 퀘스트 결과에 서사 텍스트가 붙어 세계가 살아있는 느낌을 준다. M3이 끝나면 "이 지역을 끝까지 파고들면 어떤 이야기가 나올까"라는 호기심이 게임을 이끄는 동력이 된다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **숨겨진 연계 퀘스트** | 지식 임계값 도달 → 2~5단계 체인 퀘스트 발동. 지역 이동 포함 가능. 최종 보상으로 확정 아이템. trait Phase D(특수 임무)를 통합 |
| **지역 변형** | 지식 최고 임계값 도달 → 섹터가 마을/유적지/숨겨진 섹터로 변형. 변형된 섹터에 전용 퀘스트 풀 |
| **퀘스트 서사 텍스트** | 퀘스트 결과마다 용병 이름/직업/유형에 따른 조합형 짧은 이야기 생성 |

### 선행 조건

- M2 아이템 인프라 — 연계 퀘스트 최종 보상, 유적지 전용 퀘스트 드랍에 아이템 사용
- 지역 조사 + `region_discoveries` (완료) — `discovery_type: hidden_quest`, `discovery_type: transform` 트리거

### 작업 상세

**기획 (content-designer)**

- 체인 퀘스트 시나리오 5~8개 (리전 티어별 분포), 2~5단계, 단계별 수행 지역/전투력/보상/서사 텍스트
- 지역 이동을 포함하는 체인의 서사 흐름, 실패 시 재도전 가능 설계
- 변형 대상 리전 15~25개 선정 (마을/유적지/숨겨진 섹터 유형별 컨텐츠)
- 서사 템플릿: 퀘스트 유형(4) × 결과(4) 매트릭스, 유형×결과별 3~5개 변형, 총 48~80개

**구현 (Flutter — band_of_mercenaries)**

연계 퀘스트:
- `ChainQuestData` 정적 모델 (Freezed): chain_id, region_id, step, target_region_id, name, description, combat_power, duration_minutes, reward_gold, reward_items, final_reward, next_step_delay
- `ChainQuestProgress` Hive 모델: chainId, currentStep, status(active/completed), startedAt
- `chainQuestProgress` Hive 박스 신설
- `ChainQuestService`: 발견 트리거 시 1단계 자동 생성, 단계 완료 → 다음 단계 생성, 최종 단계 → 확정 보상
- 파견 화면: 연계 퀘스트 상단 고정, 강조 색상, 진행도 표시, 타 지역 "이동 필요" 안내
- 실패 시 해당 단계 재도전 가능
- `SyncService` 확장: `chain_quests` 테이블 동기화

지역 변형:
- `RegionState` 모델 확장: `sectorChanges` (Map<int, String>) 추가
- 지식 임계값 도달 + transform 트리거 → 변형 기록
- 이동 화면: 변형된 섹터를 특수 아이콘/이름으로 구분
- `QuestGenerator` 확장: 섹터 변형 타입에 따라 전용 퀘스트 풀에서 생성
- 변형 발견 시 팝업 + 활동 로그

퀘스트 서사:
- `QuestNarrativeData` 정적 모델 (Freezed): quest_type, result_type, template_text
- `QuestNarrativeService`: 완료 시 템플릿 조회 → 변수 치환 → 서사 텍스트
- 퀘스트 완료 팝업에 서사 텍스트 영역 추가
- `SyncService` 확장: `quest_narratives` 테이블 동기화

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블 2개: `chain_quests`, `quest_narratives`
- `quest_pools`에 sector_type 필드 추가 (village/ruins/hidden)
- `table-config.ts`에 2개 테이블 + quest_pools 필드 확장 반영
- `chain_quests` 편집: chain_id로 그룹핑하여 체인 단위 조회/편집
- `quest_narratives` 편집: quest_type × result_type 필터, 템플릿 변수 미리보기
- `data_versions`에 신규 테이블 행 추가

**데이터 입력 (operation-bom에서 수행)**

- 연계 퀘스트 체인 5~8개 (총 15~30단계)
- `region_discoveries`에 hidden_quest + transform 트리거 추가
- 변형 대상 15~25개 리전의 transform 데이터
- 마을/유적지 전용 퀘스트 풀 (유형별 10~15개)
- 서사 템플릿 48~80개
- 연계 퀘스트 최종 보상 아이템이 items 테이블에 존재하는지 확인

**밸런스 체크 (balance-designer)**

- 연계 퀘스트 확정 보상 강도 vs 엘리트 반복 파밍 시간 효율 비교
- 유적지 전용 퀘스트 난이도/보상 프리미엄 적절성
- 마을 퀘스트의 보상이 "안전한 대안"으로서 적절한 수준인지

## M4: 세력의 심화

### 목표

M1에서 "세력이 게임에 영향을 주기 시작"했다면, M4에서는 세력이 완결된 진행 시스템이 된다. 전용 시설을 건설하고, 거점을 방문하여 상점에서 아이템을 구매하고, 세력 전용 트레잇을 획득한다. M4가 끝나면 "어떤 세력과 손을 잡느냐"가 용병단의 장기 방향성을 결정하는 핵심 선택이 된다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **세력 전용 시설 14종** | 가입 세력마다 고유 시설 건설 가능. 탈퇴 시 잠금(레벨 보존). 기존 시설 시스템 위에 확장 |
| **세력 거점 + 상점** | 각 세력의 거점 리전 지정. 거점 방문 시 세력 상점 이용 가능. 평판 구간별 상품 범위 확장 |
| **세력 전용 트레잇** | 거점 방문 + 평판 조건 충족 시 획득 가능한 세력 고유 트레잇 |

### 선행 조건

- M1 세력 패시브/퀘스트 태그 — 평판 축적 수단이 있어야 상점/트레잇 해금에 의미 있음
- M2 아이템 인프라 — 상점에서 판매할 아이템이 존재해야 함

### 작업 상세

**기획 (content-designer / balance-designer)**

- 14개 세력별 전용 시설 상세 설계 (이름, 설명, 최대 레벨, 비용 공식, 효과 공식)
- 기존 시설과의 시너지/중복 검토
- 14개 세력별 거점 리전 1~2개 지정
- 평판 구간별(11~30, 31~60, 61~90, 91~100) 상품 범위, 세력별 8~15개 상품, 가격 체계

## M5: 용병의 성장

### 목표

M1~M4는 세계와 세력을 풍부하게 만들었다. M5는 시선을 용병 개인에게 돌린다. 용병 간의 관계가 생기고, 티어 업그레이드로 성장 천장이 열리며, 이동 중 선택지 이벤트로 플레이어의 의사결정이 늘어난다. M5가 끝나면 용병 하나하나가 "내가 키운 캐릭터"로 느껴지는 애착이 생긴다.

### 포함 시스템

| 시스템 | 핵심 변화 |
|--------|----------|
| **용병 간 상호작용** | 공동 파견 횟수 추적 → 우정/라이벌 트레잇 자동 발생. 파견 조합에 전략적 깊이 (trait Phase C) |
| **용병 티어 업그레이드** | 최대 레벨 도달 용병을 상위 티어로 승급. 골드 + 정수 + 조건 필요. 경제 재설계 포함 (trait Phase E 일부) |
| **이동 이벤트 선택지** | 이동 중 텍스트 선택지 이벤트 (FTL/Darkest Dungeon 스타일). 위험한 선택 = 높은 보상 + 부상 리스크 |

### 선행 조건

- M2 아이템 인프라 — 티어 업그레이드에 정수 소모, 선택지 이벤트 보상에 아이템 포함
- 트레잇 시스템 전체 (완료) — 우정/라이벌 트레잇이 기존 엔진 위에 동작
- 이동 시스템 + 여행 이벤트 (완료) — 선택지 이벤트가 기존 프레임워크를 확장

### 작업 상세

**기획 (content-designer / balance-designer)**

- 상호작용: 공동 파견 임계값, 관계 유형(우정/라이벌/멘토), 관계별 효과, 관계 트레잇 6~10종
- 티어 업그레이드: 승급 조건(T1→T2 ~ T4→T5), 승급 후 변화(레벨 리셋, 스탯 재설정, 트레잇/장비 유지), 직업 변환 규칙, 인건비 변화
- 선택지 이벤트: 15~25종 시나리오, 선택지 2~3개, 선택지별 결과 확률, 트레잇 기반 숨겨진 선택지

**구현 (Flutter — band_of_mercenaries)**

용병 간 상호작용:
- `Mercenary` 모델에 `coDispatchMap` 필드 추가 (Map<String, int>)
- `MercenaryStatService` 확장: 퀘스트 완료 시 파견 용병 쌍별 카운트 갱신
- `MercenaryRelationService` 신규: 임계값 도달 → 관계 트레잇 자동 부여
- `QuestCalculator` 확장: 관계에 따른 성공률 보정
- 파견 화면: 용병 선택 시 관계 아이콘 표시
- 용병 상세 오버레이에 "관계" 섹션

용병 티어 업그레이드:
- `TierUpgradeService` 신규: 승급 조건 판정, 비용 차감, 직업 전환, 스탯 재설정
- `Mercenary` 모델 확장: 승급 이력 필드
- 용병 상세에 "승급" 버튼 + 조건 충족 현황
- 승급 성공 연출: 축하 팝업 + 전/후 비교 카드 + 활동 로그

이동 이벤트 선택지:
- `TravelChoiceEventData` 정적 모델 (Freezed)
- `TravelEventService` 확장: 선택지 이벤트 풀 확률 판정
- 이벤트 발생: 이동 완료 시 "회상" 형식으로 제시 (이동 중단 없음)
- 선택지 UI: 상황 텍스트 + 선택 버튼, 결과 즉시 적용 + 활동 로그
- 트레잇 기반 숨겨진 선택지
- `SyncService` 확장: `travel_choice_events`, `tier_upgrade_requirements` 동기화

**운영 도구 (operation-bom)**

- Supabase에 신규 테이블 2개: `tier_upgrade_requirements`, `travel_choice_events`
- 관계 트레잇: 기존 `traits` 테이블에 추가
- `table-config.ts`에 신규 테이블 정의
- `tier_upgrade_requirements` 편집: 티어별 조건 구조화 폼
- `travel_choice_events` 편집: 선택지별 결과 확률 인라인 편집, 확률 합산 검증
- `data_versions`에 신규 테이블 행 추가

**데이터 입력 (operation-bom에서 수행)**

- 관계 트레잇 6~10종 + 충돌/시너지 관계 데이터
- 티어 승급 조건 4단계 데이터
- 직업 승급 매핑 (85개 직업의 상위 티어 대응)
- 선택지 이벤트 15~25종 (이벤트당 선택지 2~3개 × 결과 확률)

**밸런스 체크 (balance-designer — 최중요)**

- 티어 업그레이드 경제 시뮬레이션: T1→T5 전체 비용 vs 골드 수급, 승급 현실 소요 기간, 승급 후 인건비 vs 순수익

- 선택지 이벤트 기대값: 위험/안전 선택 모두 합리적인지

## M6: 완성과 균형

### 목표

M1~M5에서 모든 시스템을 구현했다. M6은 게임을 "완성품"으로 만드는 마일스톤이다. 199개 리전에 컨텐츠를 채우고, 전체 시스템 간 경제 밸런스를 맞추고, 빈 구석과 stub을 정리한다. M6이 끝나면 "처음부터 끝까지 플레이해도 빈 곳이 없는" 싱글 플레이어 완성판이 된다.

### 포함 작업

| 축 | 핵심 변화 |
|----|----------|
| **전체 데이터 채우기** | 199개 리전의 발견 이벤트, 퀘스트 풀, 엘리트, 연계 퀘스트, 변형 데이터를 티어별 밀도 기준으로 채움 |
| **전체 밸런스 튜닝** | 경제/전투/성장/트레잇을 통합 시뮬레이션하여 최종 수치 확정 |
| **stub 정리 + 품질 마무리** | 미구현 시설 stub 실제 연동, 엣지 케이스 수정, UX 개선 |

### 선행 조건

- M1~M5 전체 완료

|----------|---------|-------------------|--------|-----------|------|----------|

#### 축 2: 전체 밸런스 튜닝

경제 밸런스:
- 골드 수입 곡선 (퀘스트 + 방치 + 이벤트 + 선택지 총합)
- 골드 지출 곡선 (파견비 + 인건비 + 시설 + 트레잇 삭제 + 상점 + 승급)
- 구간별 순 수지 (초반 F~E / 중반 D~C / 후반 B~A)
- 아이템 획득 속도 (정수 파밍 주기)

전투 밸런스:
- 파티 전투력 vs 난이도별 적 전투력 분포
- 상성 + 장비 + 패시브 + 관계 합산 시 성공률 상한/하한
- 엘리트 전투력 vs 풀파티 — 도전적이지만 불가능하지 않은 수준

성장 밸런스:
- XP 곡선: Lv1→Lv5 도달 기간 (티어별)
- 승급 비용 vs 골드 축적 → 현실 소요 기간
- 명성 곡선: F→A 도달 기간, 랭크 체류 시간 분포

트레잇 밸런스:
- 획득 임계값, 진화 조건의 현실 달성 시점
- 세력 트레잇 강도

밸런스 작업 방식:
1. balance-designer로 전체 경제 시뮬레이션 (1일/1주/1개월/3개월)
2. 이상치 조정 → 재시뮬레이션 (최대 3회 반복)
3. 확정 수치를 operation-bom에서 일괄 반영
4. 실제 플레이 테스트로 체감 검증

#### 축 3: stub 정리 + 품질 마무리

stub 시설 실제 연동:

| 시설 | 현재 상태 | M6 작업 |
|------|----------|---------|
| 대장간 | 장비 강화 효과 stub | M2 아이템과 연동 — 장비 강화(스탯 증가) |
| 연구소 | 특수 트레잇 해금 stub | M4 세력 트레잇 + 기존 트레잇 연동 — 고급 트레잇 조건 완화 |
| 게시판 | 특수 의뢰 알림 stub | M3 연계 퀘스트 연동 — 발견 확률 상승, 미발견 체인 힌트 |

품질 마무리:
- 전체 UI 일관성 검토
- 활동 로그 메시지 톤 일관성
- 엣지 케이스 수정 (승급 중 파견, 탈퇴 중 전용 퀘스트, 대량 인벤토리 성능 등)
- 온보딩 힌트 (신규 시스템 첫 접촉 시 안내)
- 시설 이정표 기능 연동 (Lv5/10/15/20/25 해금)

운영 도구 (operation-bom):
- 대량 데이터 입력 효율화: 리전 단위 일괄 편집 뷰
- 밸런스 수치 일괄 수정 도구
- 데이터 검증 대시보드: 리전별 컨텐츠 채움률, 데이터 무결성 체크, 미입력 리전 하이라이트

### 마일스톤별 작업 흐름 (반복)

```
1. 기획 확정
   └─ content-designer / balance-designer로 컨텐츠/수치 설계
   └─ 기획자 검토 및 확정

   └─ spec-writer로 마일스톤별 개발 명세서 생성

3. 구현
   └─ implement-spec 또는 implement-agent로 Flutter + operation-bom 동시 구현
   └─ 코드 리뷰

4. 데이터 입력
   └─ operation-bom에서 기획 확정된 데이터 입력

   └─ balance-designer로 시뮬레이션
   └─ 이상치 수정 → 재시뮬레이션

6. 완료 점검
   └─ 마일스톤 점검표 전체 항목 확인
   └─ 이전 마일스톤 회귀 테스트
```

## 마일스톤 실행 가이드

각 마일스톤을 시작할 때:

```bash
# 1. 기획 확정 후 개발 명세서 생성
/spec-writer @Docs/roadmap/master_roadmap.md  # M{N} 섹션 참조

# 2. 명세서 검증
/verify-spec @Docs/spec/M{N}_spec.md

# 3. 구현 실행
/implement-spec @Docs/spec/M{N}_spec.md
# 또는
/implement-agent @Docs/spec/M{N}_spec.md
```

각 마일스톤 완료 시:
1. 해당 마일스톤의 **완료 점검표** 전 항목 확인
2. **이전 마일스톤 회귀 테스트** 수행
3. **balance-designer로 밸런스 스냅샷** 생성
4. `content_status.md` 업데이트
5. 다음 마일스톤 진입 결정
