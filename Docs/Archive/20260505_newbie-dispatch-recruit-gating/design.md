# 신규 유저 파견·모집 하향 밸런스 분석 리포트

> 작성일: 2026-05-05
> 유형: 수치 조정 제안 (코드 로직 변경 포함)
> 분석 대상: `QuestGenerator.generateQuests()` · `RecruitmentService.selectTier()`
> 게이팅 방식: **명성 등급 기반 (옵션 A)** — 사용자 합의 완료

---

## 1. 배경

M5까지 개발이 진행되며 신규 유저 진입 1시간의 난도가 과하다는 판단. 시작 region 3 (regionTier 1) 기준 일반 풀에 난이도 3 의뢰가 23.6% 비중으로 노출되며, 4인 파티(T1~T2 추정 power ~60)로는 사망률 15%×사용자 수만큼 손실이 누적된다. 모집 또한 T3+ 등용 시 인건비가 시작 골드 회수 곡선을 초과한다.

본 리포트는 **명성 등급 기반(F/E/D+)** 으로 신규 단계를 정의하고, 파견 풀과 모집 티어 분포에 단계별 게이트를 적용하는 안을 다룬다. `ranks.unlock_tier` 컬럼이 이미 존재하지만 코드에서 미사용이므로, 데이터 변경 없이 **로직만 수정**해도 적용 가능하다.

## 3. 데이터 분석

### 3.1 사망 기댓값 시뮬레이션 (시간당)

가정: 정보망 0레벨, 6슬롯, 평균 의뢰 30분, 1시간 12회 진행, 4인 파티, 의뢰별 4명 모두 투입.

| 단계 | d1 노출 회수 | d2 노출 회수 | d3 노출 회수 | 사망 기댓값/시간 |
|------|----------------|----------------|----------------|---------------------|
| **현재 (게이팅 없음)** | 5.42 | 3.74 | 2.84 | **1.71명** |

산식 예시 (d3): `12회 × (22/93) × 0.15 사망률 × 4명 = 1.70명`

시작 4명 중 절반이 1시간 내 손실되는 위험 곡선. 신규 유저 이탈 요인.

### 3.2 모집 인건비 손익 분석

T3 등용 시 시간당 인건비 50G. 파견 1회 의뢰비 평균 ~17G(d1) + 보상 1배수 = 신규 유저는 T3 1명만으로도 평균 의뢰 차익이 음수로 전환.

T2 인건비 25G — d1 보상 곡선과 균형. T1 인건비 10G — 가장 안전.

→ 신규 단계는 T2까지만 등용 허용이 합리적.

### 3.3 명성 진행 시뮬레이션 (자연 졸업 곡선)

거점 사건 보상:
- step 1 (성공): rep ~5 + trust +10
- step 2 (성공): rep ~5 + trust +15
- step 6 완료: **rep +500** (사건 종결 보너스)

허드렛일 보상: d1 성공 rep ~2~5/회

**F → E 진입 (명성 300):**
- 사건 step 1, 2 완료 후 일반 d1 의뢰 약 50~100회 누적 시 도달
- 또는 사건 step 6 완료 1회로 단번에 점프

**E → D 진입 (명성 2000):**
- 사건 종료(+500) 이후 일반 d2 의뢰 누적으로 1~2시간 정상 플레이

자연스러운 마일스톤 곡선이며, 사건 진행과 결합 시 **F = 사건 step 1~2 단계**, **E = 사건 step 3~6 진행 단계**, **D+ = 사건 졸업 후 정상 운영** 으로 매핑된다.

| T3+ 등용 시 인건비 폭주 | 50G+/시간 vs d1 보상 ~10G/회 | 골드 마이너스 진입 |
| 신규 단계 보호막 부재 | `ranks.unlock_tier` 컬럼 정의됐으나 미사용 | 진입 학습 곡선 단절 |

## 5. 플레이어 체감 분석

**현재 신규 1시간 시퀀스(예상):**
1. 시작 4인 파티(T1×2~3 + T2×1~2)로 d1 의뢰 진행 → 정상
2. d2~d3 의뢰가 매 슬롯에 1~3개 노출 → 보상 매력에 끌려 도전
3. d3 진행 중 1~2명 부상/사망 → 회복 대기 또는 영구 손실
4. 모집 시 T3 15% 확률 등용 → 인건비 부담 가중
5. 30분~1시간 안에 자원 압박 → "어디서 잘못한 건지" 학습 신호 없음

**제안 적용 후 시퀀스:**
1. 시작 파티로 d1 의뢰 6슬롯 안정 진행
2. 사건 step 1, 2 완료 → 명성 +25 + 거점 신뢰도 +25 (사건 진행감)
3. 거점 사건 step 6 종결 → 명성 +500 → E 진입 (의뢰 d2 풀 살짝 열림, T2 모집 가능)
4. d2 적응 + 명성 누적 → D 진입 (정상 분포 풀 해금)
5. 진행이 단계별로 보상으로 가시화됨 → 학습 곡선 연속

## 6. 조정 제안 (옵션 A — 명성 등급 기반)

### 6.1 모집 티어 확률 — `RecruitmentService.selectTier()`

| 단계 | 명성 | T1 | T2 | T3 | T4 | T5 |
|------|------|------|------|------|------|------|
| **F (신규)** | 0~299 | **100%** | 0% | 0% | 0% | 0% |
| **E (초보)** | 300~1999 | **90%** | **10%** | 0% | 0% | 0% |
| **D+ (정상)** | 2000+ | 45% | 30% | 15% | 8% | 2% |

D+ 단계는 현재 분포 유지. `recruitBonus`/`extraHighTierBoost` 계산도 D+ 단계에서만 적용한다 (F/E 단계에서는 보너스가 적용되어도 잠긴 티어가 풀리지 않도록 `unlock_tier` 캡 강제).

### 6.2 파견 풀 weighted sampling — `QuestGenerator.generateQuests()`

기존 균등 shuffle을 가중 샘플링으로 변경. 단계별 weight:

| 단계 | d1 weight | d2 weight | d3 weight | 6슬롯 d2 기댓값 | 6슬롯 d3 기댓값 |
|------|-----------|-----------|-----------|----------------|----------------|
| **F** | 1.0 | 0 | 0 | 0 | 0 |
| **E** | 1.0 | 0.25 | 0 | 0.88 | 0 |
| **D+** | 1.0 | 1.0 | 1.0 (현재) | 1.87 | 1.42 |

E 단계 d2 비율 산식:
```
effective_d1 = 42 × 1.0 = 42
effective_d2 = 29 × 0.25 = 7.25
d2_ratio = 7.25 / (42 + 7.25) = 14.7%
6_slot_d2_expected = 6 × 0.147 = 0.88
```

D+ 단계는 weight 변경 없이 현재 균등 분포 유지.

### 6.3 게이트 결정 함수

```
신규 단계(rank): UserData.reputation 기준 ranks 테이블에서 grade 조회
  - grade == 'F'        → tier=F (모집 cap=1, 파견 d2/d3 weight=0)
  - grade == 'E'        → tier=E (모집 cap=2, 파견 d2 weight=0.25, d3 weight=0)
  - grade in ['D','C','B','A'] → tier=D+ (현재 분포 유지)
```

### 6.4 변경 전/후 비교 요약

| 항목 | 현재 | 제안 (F) | 제안 (E) | 제안 (D+) |
|------|------|----------|----------|-----------|
| 모집 T1 비율 | 45% | 100% | 90% | 45% |
| 모집 T3+ 가능 | O | X | X | O |
| 파견 d2 6슬롯 | 1.87 | 0 | 0.88 | 1.87 |
| 파견 d3 6슬롯 | 1.42 | 0 | 0 | 1.42 |
| 사망 기댓값/시간 | 1.71명 | 0.24명 | 0.32명 | 1.71명 |

## 7. 시뮬레이션

### 7.1 신규 1시간 사망 곡선 비교

가정: 정보망 0레벨 6슬롯, 30분 평균 진행, 12회 회전, 4명 파티 모두 투입.

```
현재(게이팅 없음): d1=5.4 + d2=3.7 + d3=2.8 노출
  사망 = 5.4×0.05×4 + 3.7×0.10×4 + 2.8×0.15×4
       = 1.08 + 1.48 + 1.68 = 1.71명/시간

F 단계: d1=12 노출
  사망 = 12 × 0.05 × 4 = 0.24명/시간

E 단계: d1≈11.0 + d2≈1.0 노출
  사망 = 11.0×0.05×4 + 1.0×0.10×4 = 0.32명/시간

D+ 단계: 현재와 동일 1.71명/시간
```

신규 단계에서 사망 기댓값 **약 5배 감소**. D+ 진입 시 자연스럽게 정상 위험으로 복귀.

### 7.2 골드 흐름 시뮬레이션 (1시간)

가정: 시작 골드 1000G, 의뢰 1회당 평균 인건비 + 파견비 + 보상.

| 단계 | 평균 의뢰 순수익(d1) | 평균 의뢰 순수익(d2) | 1시간 누적(추정) |
|------|----------------------|----------------------|-------------------|
| F (12회 d1) | +8G | — | +96G |
| E (10.6회 d1 + 1.4회 d2) | +8G | +20G | +113G |
| D+ (5.4 d1 + 3.7 d2 + 2.8 d3, 사망 변수 제외) | +8G | +20G | +210G |

D+ 진입 시 보상 곡선이 자연스럽게 상승. 신규 단계는 **안정적인 +96~+113G/시간**으로 유지.

## 8. 후속 작업

본 리포트는 코드 로직 변경 명세로 이어진다. 데이터(quest_pools, ranks)에는 변경이 없으므로 data-generator는 사용하지 않는다.

**권장 후속:**
- `/spec-writer @Docs/balance-design/[balance]20260505_newbie-dispatch-recruit-gating.md`로 개발 명세서 작성
- 명세에 포함될 핵심 변경:
  1. `RecruitmentService.selectTier()` — `currentReputation` 인자 추가, ranks 조회로 단계 판별, F/E/D+ 분기 분포 적용, F/E 단계에서 `recruitBonus`/`extraHighTierBoost` 캡 강제
  2. `QuestGenerator.generateQuests()` — `currentReputation` 인자 추가, 단계별 weight 테이블 적용, 균등 shuffle을 가중 샘플링으로 교체
  3. 호출부 갱신: `quest_provider.dart`의 generateQuests/fillQuests/_refreshExpiredQuests 3곳 + `mercenary_provider`의 모집 호출부에 `userData.reputation` 전달
  4. 단위 테스트: 명성 0/300/2000 경계에서 분포 회귀 테스트

**검증 포인트:**
- 명성 299 → 300 경계에서 게이팅 즉시 해제 (rank 진입 직후 다음 generateQuests 호출 시점)
- 사건 step 6 완료 직후 +500 명성으로 F → D 점프 시나리오에서도 정상 동작
- 정보망 시설로 슬롯이 늘어나도 비율 유지 (weight 기반이므로 자동 충족)
