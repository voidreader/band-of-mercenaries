# ecc 토큰 효율 항목 도입 영향 분석

**작성일**: 2026-04-22
**상위 문서**: [ecc_integration_review.md](ecc_integration_review.md)
**목적**: ecc 저장소의 토큰 효율 관련 항목(5~10번)이 현재 `band-of-mercenaries` 프로젝트 워크플로우에 어떤 영향을 미치는지 항목별로 분석하고, 도입 순서를 권장한다.

---

## 분석 대상

| 번호 | 항목 | 유형 |
|---|---|---|
| 5 | `context-budget` | skill |
| 6 | `search-first` | skill |
| 7 | `strategic-compact` | skill |
| 8 | `ck` | skill (SessionStart 훅 포함) |
| 9 | `agent-harness-construction` | skill |
| 10 | `harness-optimizer` | agent |

## 현재 워크플로우 컨텍스트

- **spec-pipeline**: 기획 문서 → `spec-writer` → `verify-spec` → `implement-agent` 또는 `implement-spec`
- **implement-agent 파이프라인**: `planner` → `coder` → `dart-build-resolver` (조건부) → `verifier` + `flutter-reviewer` (병렬)
- **implement-spec 경량 경로**: main 직접 구현 + 빌드 게이트에서 `dart-build-resolver` 위임
- **milestone-runner**: 마일스톤 단위(M1~M6) 4페이즈 체크포인트 기반 진행
- **로드된 구성**: agent 7개(analyzer/architect/coder/planner/verifier/dart-build-resolver/flutter-reviewer) + skill 13개

## 항목별 영향 분석

### 5. `context-budget` — 독립 감사 도구

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | 없음. 사용자 명시 호출 (`/context-budget`) |
| **상시 비용** | 없음 (필요 시만 로드) |
| **통합 포인트** | 마일스톤 완료 시점 / 월 1회 정기 감사 |
| **이득** | 현재 20개 agent+skill 중 사용 빈도 낮은 항목 식별 → 아카이브 기준 데이터 확보 |
| **리스크** | 감사 결과를 실행하지 않으면 무의미 (수동 프로세스) |

**영향 요약**: 가장 안전. 현재 워크플로우를 건드리지 않고 "어디에 토큰이 낭비되는지" 가시화하는 독립 도구. 이후 항목 도입 판단 근거가 되므로 **가장 먼저 도입하는 게 합리적**.

**검토 포인트**:
- 감사 결과를 실제 정리 작업으로 이어갈 주기(월 1회/마일스톤 완료 시) 합의 필요
- 감사 결과 문서화 위치: `Docs/ecc-audits/YYYY-MM-DD.md` 등 일관된 경로 정의

### 6. `search-first` — 구현 전 탐색 단계 추가

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | `planner` 이전 단계로 삽입 가능. 매번 트리거 시 오버헤드 |
| **상시 비용** | skill 로드 + researcher agent 호출 시 토큰 |
| **통합 포인트** | `spec-writer` 또는 `planner` 앞 (새 의존성·큰 기능 추가 시만) |
| **이득** | 재발명 방지. pub.dev 패키지 중복 구현 예방 |
| **리스크** | 우리 프로젝트는 스택이 이미 고정(Supabase/Hive/Riverpod) → 매 태스크마다 탐색은 과함. researcher agent 별도 추가 필요 |

**영향 요약**: 프로젝트 특성상 **제한적 이득**. 대부분 작업이 기존 스택 내부 로직 추가라 "탐색할 대상"이 적음. 새 기능이 외부 패키지 도입을 수반할 때만 사용자 명시 호출로 운영하는 게 맞음. 자동 편입 비권장.

**검토 포인트**:
- researcher agent를 별도 도입할지, 기존 planner가 탐색 로직을 내재화할지 결정
- 트리거 조건(외부 패키지 도입 시만) 명시 필요. 자동 트리거는 과함

### 7. `strategic-compact` — 컨텍스트 압축 전략

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | 없음. compact 시점 자동 전략 적용 |
| **상시 비용** | 낮음 |
| **통합 포인트** | 자동 compact 훅 |
| **이득** | 장시간 마일스톤 작업 시 유용 |
| **리스크** | 거의 없음. 단, **현재 우리 파이프라인은 이미 에이전트 위임 구조라 main 컨텍스트가 적게 쌓임** → 체감 이득 작을 수 있음 |

**영향 요약**: `implement-agent`가 이미 subagent로 컨텍스트를 분산시키므로, main 컨텍스트 누적이 크지 않음. **이득은 제한적이지만 무해**. 도입 가치는 중하.

**검토 포인트**:
- 현 파이프라인의 main 컨텍스트 누적량을 `context-budget`으로 먼저 측정
- 누적이 적으면 도입 보류, 큰 마일스톤에서 누적이 관찰되면 도입

### 8. `ck` — 세션 간 체크포인트

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | SessionStart 훅 설치 필요 (settings.json 변경) |
| **상시 비용** | SessionStart 훅이 매 세션 시작 시 1회 실행 |
| **통합 포인트** | `milestone-runner`와 **기능 중첩 주의** |
| **이득** | 세션 간 컨텍스트 재로딩 비용↓. 마일스톤 중단/재개 시 재탐색 불필요 |
| **리스크** | **`milestone-runner`가 이미 체크포인트 기반으로 4개 페이즈 관리 중** → 책임 경계 혼란 가능 |

**영향 요약**: 가장 큰 고려사항. 우리는 이미 `Docs/milestone-runs/`에 마일스톤 단위 체크포인트가 있음. ck를 도입한다면 **역할 분리**가 필요:

- **milestone-runner**: 마일스톤 단위 (기획/밸런스/데이터/스펙 4페이즈)
- **ck**: 구현 세션 단위 (하루 끝낼 때 현재 상태 저장 → 다음날 복원)

이 경계를 합의하지 않고 도입하면 혼란. **도입 전 경계 정의 필수**.

**검토 포인트**:
- ck의 저장 경로가 `milestone-runner`의 상태 파일과 충돌하지 않는지 확인
- SessionStart 훅이 기존 자동화와 상호작용하는지 점검
- 세션 단위 체크포인트의 실질적 사용 시나리오 검증 (그냥 git commit으로 충분한지)

### 9. `agent-harness-construction` — 설계 원리 참조 문서

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | 없음 (지식 문서) |
| **상시 비용** | skill 로드 시 토큰 |
| **통합 포인트** | 기존 agent 프롬프트 리팩토링 시 |
| **이득** | 자체 agent (`planner`/`coder`/`verifier`) 프롬프트 품질 향상 기준 |
| **리스크** | **참조만 하고 리팩토링 실행이 없으면 무의미** |

**영향 요약**: 그 자체로는 워크플로우에 영향 없음. 10번 `harness-optimizer`와 쌍으로 도입해야 실질적 효과. 단독 도입 비권장.

**검토 포인트**:
- 9+10 쌍으로 도입 시 정기 튜닝 주기 합의 (분기별 1회 등)
- 튜닝 산출물 아카이브 경로 정의

### 10. `harness-optimizer` — 하네스 품질 점수화 + 개선

| 항목 | 내용 |
|---|---|
| **워크플로우 간섭** | `/harness-audit` 명령 체계 추가 도입 필요 |
| **상시 비용** | 감사 실행 시 토큰 (수 분 단위 실행) |
| **통합 포인트** | 마일스톤 완료 시 / 분기별 품질 감사 |
| **이득** | 하네스 품질 KPI 추적 가능. before/after 델타 측정 |
| **리스크** | `/harness-audit` 등 관련 명령/훅 함께 도입해야 하므로 **도입 복잡도 높음** |

**영향 요약**: 장기적으로 가장 큰 가치이지만, 부속 명령 체계까지 함께 도입해야 함. `context-budget`으로 먼저 감사 근거를 쌓은 뒤 도입 고려.

**검토 포인트**:
- `/harness-audit` 명령의 구현 요구사항 전체 파악 후 결정
- 감사 점수 기록 위치 및 추적 주기 정의
- 5번(`context-budget`) 도입 3개월 후 재검토가 합리적

## 핵심 결정 포인트

1. **`context-budget` 먼저 돌려서 현재 상태 측정**
   - 실제 블로트가 있는지 확인 후 다른 항목 판단
   - 감사 결과가 "현재 최적"이면 나머지 항목들의 도입 우선순위가 크게 낮아짐

2. **`ck`는 `milestone-runner`와 경계 정의 먼저**
   - 없이 도입하면 이중 체크포인트 혼란
   - 현재 `milestone-runs/`로 충분한지 실제 사용 경험 후 판단

3. **9+10 쌍 원칙은 재고됨** (2026-04-23 수정)
   - 9번(참조 문서)은 단독 도입해도 유효 — 수동 리팩토링 가이드로 기능함
   - 10번(정기 감사 프로세스)은 에이전트 규모·협업 체계가 커질 때 별도 판단
   - 5번 감사 결과에 구체 harness 이슈가 있으면 9번 즉시 도입 가능

4. **프로젝트 특성 고려사항**
   - 우리 spec-pipeline이 이미 subagent 기반으로 main 컨텍스트를 절약하는 구조 → 일부 토큰 효율 도구의 한계 이득은 작을 수 있음
   - 마일스톤 단위 장기 작업이 많음 → 세션 간 상태 보존은 가치 큼 (ck)
   - 기술 스택 고정 → 탐색 기반 도구(search-first)는 필요성 낮음
