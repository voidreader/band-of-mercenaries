# 노출 빈도·획득 페이스 밸런스 분석 리포트

> 작성일: 2026-05-13
> 유형: 통합 시뮬레이션 + 페이스 검증 + α 적절성 검증
> 분석 대상: M6 위업(#1) + 칭호(#2) + 지명 의뢰(#3) 3 시스템 통합 페이스

## 개요

M6 페이즈 1의 세 산출물(위업·연대기 / 칭호·간판 / 지명 의뢰)과 페이즈 2 #1(칭호 효과 수치 검증) 결과를 통합 시뮬레이션하여 다음을 검증한다.

1. **시간당 평균 파견 수 8회 가정**의 정합성 — Supabase 실데이터 + GameConstants.baseQuestCount = 6 기반
2. **roadmap "3~5시간 안 1명 이상 칭호 기억"** 충족 검증 (위업·칭호·지명 의뢰 모두)
3. **지명 의뢰 가중치 α=3 적절성** — 일반 풀 203행 대비 등장 확률 정량 계산
4. **시간당 발급 1~3개 무게감 보존** 검증
5. **보수/평균/적극 3종 페이스 모드** 시뮬레이션
6. **페이즈 4 #1·#2·#3 명세 입력용 최종 페이스 표** 확정

본 리포트의 핵심 결론:
- **시간당 평균 파견 8회 가정 ✓** — GameConstants.baseQuestCount 6 + 의뢰 평균 소요 20min + 정보망 시설 보너스 고려 시 정확
- **3~5h 종료 조건 모두 충족** — 평균 페이스 유저 기준 3h 시점에 위업 3~4 + 칭호 2~3 + 지명 의뢰 2회 자연 누적
- **α=3 적절** — 매 1h 갱신 시 named 의뢰 등장 확률 ~24%, 시간당 0.24회 등장 → 4h에 1회 자연
- **무게감 보존 ✓** — 평균 페이스 시간당 발급 1~3개. 적극 페이스도 시간당 최대 3~4개로 무게감 희석 없음
- **조정 권고 1건**: M5 시점 신규 유저 적극 모드(12 파견/h)에서 5h 시점 칭호 6~7개 보유 → 약간 빠를 수 있음. 단, 사용자 다수는 평균 모드 예상 — **수치 조정 없이 모니터링 권장**

## 데이터 분석

### 2.1 시간당 평균 파견 수 8회 가정 검증

**검증 모델**:

- 동시 슬롯 6개 (시설 무) ~ 11개 (정보망 Lv25)
- 의뢰 평균 소요 시간 = (difficulty 1~3 균등 가정) (10+20+30)/3 = **20min**
- 1시간 동안 한 슬롯에서 평균 1시간/20min = **3회 파견** (이상)
- 동시 슬롯 6개 가정 시 1시간 최대 18회 파견 (이상)
- 실제로는 슬롯 모두 동시 가동 불가 (보유 용병 수 제한). 평균 가용 슬롯 ~**3~4개**

**현실 보정 시뮬레이션** (신규 유저 ~5시간):

| 단계 | 신규 유저 가용 용병 수 | 동시 파견 가능 | 시간당 파견 (소요 20min 평균) |
|------|--------------------|--------------|---------------------------|
| 0~1h | 1~3명 | 1~2 의뢰 | 3~6회 |
| 1~3h | 3~5명 | 2~3 의뢰 | 6~9회 |
| 3~5h | 5~8명 | 3~4 의뢰 | 9~12회 |
| 5~10h | 8~12명 | 4~6 의뢰 | 12~18회 |

→ **시간당 8회 가정은 1~3h 시점의 신규 유저 페이스**. 더 적극적 사용자(5h+)는 12회/h 이상 가능.

**조정 권고**: 페이즈 2 #1 시뮬레이션 그대로 유지하되 본 리포트에서 **3종 페이스 모드** 도입:
- **보수 (느린 유저)**: 4 파견/h — 신중하게 의뢰 선택, 안전 위주
- **평균 (기준)**: 8 파견/h — 페이즈 2 #1 가정과 동일
- **적극 (빠른 유저)**: 12 파견/h — 가용 용병 적극 활용

이 모드 분류는 본 리포트 §6 시뮬레이션에 활용.

### 2.2 지명 의뢰 가중치 α=3 정량 검증

#3 산출물 §3.1 권장 α=3을 일반 풀 크기 203 대비 등장 확률로 계산.

**가정**:
- 신규 유저 region tier 1~2에서 quest_pools sampling
- region tier 매칭 후 활성 풀 ~50~80 (전체 203의 약 25~40%)
- named 의뢰 7개 중 hook 통과 시: M5 시점 평균 **2~3개** 매칭 (#3 §3.2 분포 기준)

**가중치 계산** (활성 풀 = 70, named 매칭 = 3 가정):

```
일반 의뢰 가중치 합계: (70 - 3) × 1 = 67
named 의뢰 가중치 합계: 3 × (1 + α) = 3 × 4 = 12
총 가중치: 67 + 12 = 79
```

**1개 슬롯 sampling 시 named 확률**: 12 / 79 = **15.2%**

**6개 슬롯 sampling 시 named 최소 1개 등장 확률** (병행 sampling, 중복 허용 안 함):
```
P(0 named in 6 slots) ≈ (67/79) × (66/78) × (65/77) × (64/76) × (63/75) × (62/74)
                    ≈ 0.848 × 0.846 × 0.844 × 0.842 × 0.840 × 0.838 ≈ 0.358
P(≥1 named in 6 slots) ≈ 1 - 0.358 = 0.642 = 64.2%
```

→ **매 1h 갱신 시 64% 확률로 1개 named 의뢰 등장**

**1시간당 named 의뢰 등장 횟수 기대값**:
```
E[named per refresh] = 6 × (12 / 79) = 0.91 ≈ 0.9
```

→ **매 갱신마다 평균 0.9개 named 의뢰 등장**. 24h 쿨다운 고려 시 7개 의뢰 중 매번 다른 의뢰가 노출됨.

**1시간 = 1회 갱신 → 시간당 ~0.9회 등장 → 1.1시간 만에 평균 1회 등장**

**페이스 평가**:
- roadmap "3~5h 안 1회 등장" 충족 ✓ (1.1h 만에 평균 1회)
- 너무 자주 등장 위험: 시간당 ~0.9회 = 거의 매 갱신마다 노출됨. 사용자 입장에서 "매번 지명 의뢰가 있다" 느낌
- 24h 쿨다운으로 동일 의뢰 재등장 방지되지만 7개 풀이라 회전 빠름

**조정 검토**:
- α=3 → α=2로 미세 하향?
  - α=2 시: 1개 슬롯 named 확률 9/76 = 11.8%, 6개 슬롯 ≥1 등장 ≈ 51%
  - 1.9h 만에 평균 1회 등장
- α=3 → α=1로 하향?
  - α=1 시: 6/73 = 8.2%, 6개 슬롯 ≥1 ≈ 39%
  - 2.5h 만에 평균 1회

**권고**: **α=3 유지**. 사유:
- 신규 유저 3h 시점에 지명 의뢰 2~3회 등장 — roadmap "3~5h 안 1회 이상" 충분히 충족
- "매번 노출 느낌"은 24h 쿨다운으로 자연 분산 (7개 풀 회전)
- α=2로 낮추면 보수 유저는 3~4h 만 1회 → 빠듯
- α=3은 평균/적극 사용자에게 자연, 보수 사용자에게 안전 마진

### 2.4 지명 의뢰 24h 쿨다운 회전 검증

**시뮬레이션** (평균 페이스 + 24h 추적):

→ **자연 동작, 문제 없음** ✓

### 2.5 동시 발생 시나리오 — dialog 폭주 검증

**최악의 경우**: 한 퀘스트 완료로 다음이 동시 발생:
- 체인 완주 (chain_completed dialog, high)
- 위업 발급 (Achievement dialog, high, 칭호 인라인 통합)
- 명성 등급 진입 (RankUp dialog, critical, 위업 인라인 통합)

이론적으로 3개 dialog 큐잉. 큐 우선도 정렬:
1. RankUp (critical) — 위업 인라인
2. Chain (high)
3. Achievement (high) — 칭호 인라인

**검증**:
- critical → high 순차 표시. 사용자 입장에서 "큰 사건 → 작은 사건" 자연 흐름
- 1개 사건에 3개 dialog는 드문 경우(파견 시 명성 진입 임계점 + 체인 완주 동시)
- 무게감 보존: 사용자가 "이번 의뢰가 특별했다"고 인식하는 momentum

→ **자연 OK** ✓

또 다른 폭주 시나리오 — **TitleUnlockedDialog (b)·(c) 칭호 발급**:
- raid_count 20 임계 도달 시 도적길 추적자 칭호 (TitleUnlockedDialog)
- 동시에 체인 완주 (ChainCompletedDialog)
→ 두 dialog 순차. 무게감 OK.

### 2.6 페이즈 2 #1 시간당 8회 가정 재검증

## 문제점

### 3.1 적극 페이스 사용자 5h 시점 칭호 6~7개 → 약간 빠른 페이스 위험

§2.3 적극 페이스 시뮬레이션에서 5h 시점 칭호 보유가 평균 페이스 7h 수준. roadmap "3~5h 안 1명 이상 칭호 기억"은 충족하지만, 적극 유저는 "칭호 너무 많이 받음" 인상 가능.

**근거**: 시간당 신규 발급 2~3개 (적극) vs 1~3개 (평균). 적극 페이스 발급 빈도 약 1.5배.

**위험도**: 낮음. 적극 유저는 게임 깊이 탐구를 즐기는 유저이므로 칭호 다수 보유가 보상감으로 작동. 다이얼로그 큐로 폭주 방지.

### 3.2 보수 유저 raid_count 20 도달 ~20h — M6 종료 외부

§2.3 보수 페이스 시뮬레이션에서 #4 도적길 추적자(raid 20 임계) 도달이 ~20h. M6 누적 플레이 종료 조건은 5h 시점이지만, 보수 유저는 M6 기간 후반(M7+ 진입 시)에야 행동 지표 칭호 첫 발급.

**위험도**: 낮음. 보수 유저는 위업 기반 칭호 fallback으로 5h 시점 1~2개 칭호 보유 → roadmap 충족. 행동 지표 칭호는 "장기 목표"로 자연 위치.

### 3.3 일반 의뢰 풀 quest_type 분포 불균등 — raid 25%, escort/hunt 32~34%

§1.2 실측: raid 25%, escort 32%, hunt 34%. raid 비율이 약간 낮음.

**영향**: #4 도적길 추적자(raid 임계)가 #7 호위(escort) 대비 약간 느린 페이스. 페이즈 2 #1 §5.2 임계 조정(raid 20)으로 일부 보완됨.

**조정 권고**: 페이즈 2 #1 임계 그대로 유지. M9 컨텐츠 균형 조정 시 raid 풀 확장 검토.

### 3.4 신규 유저 1h 시점 위업·칭호·지명 의뢰 거의 0개 발급

§2.3 평균 페이스 1h 시점에 위업 0~1, 칭호 0~1, 지명 의뢰 0~1. 신규 유저가 "M6 시스템 체감"하기까지 30min~1h 학습 시간 필요.

**위험도**: 낮음. roadmap 종료 조건은 3~5h 기준이며 1h는 학습 단계로 자연. 단, 페이즈 4 명세 작성 시 첫 위업이 30min~1h 시점에 자연 발급되도록 보장 필요 (#1 산출물 §2.4 검증 OK).

## 플레이어 체감 분석

### 4.1 신규 유저 0~5h 체감 흐름

**0~30min**: M5 폐광길 사건 진행. 첫 제작 1~2회. **위업 0개, 칭호 0개** — 학습 단계.

**30min~1h**: 첫 깃발 복원 완료 시 craft_first_rare 위업 + #3 첫 깃발을 든 자 칭호 발급. **첫 명예 순간**.

**1~2h**: 폐광길 step 1~3 완료. settlement_event 진행. **위업 2~3 / 칭호 1~2 / 지명 의뢰 1~2회**. "이 게임이 내 용병단을 기억한다"는 인지.

**2~3h**: 폐광길 step 4~6 완료. #1 마을의 은인 칭호 + 거점 소속 위업 + 명성 D 진입(#10 이름을 알린 자). **3시간 시점 칭호 2~3개 보유**. roadmap 종료 조건 충족.

**3~5h**: 안정기. 일반 의뢰 + 체인 일부 + 추가 위업. **칭호 3~4개 / 지명 의뢰 3~6회**. 사용자가 "특정 용병의 이야기"를 명확히 기억하는 시점.

### 4.2 적극 유저 0~5h 체감

5h 시점에 칭호 5~6 + 위업 7~8. M6 시스템 풀스택 80% 도달. 사용자는 "이미 다 봤다"는 느낌일 수 있으나, 다음 단계 목표 = 행동 지표 칭호(#4·#5·#6·#7) + 엔드 칭호(#11) + 지명 의뢰 다양화. **장기 목표 충분**.

### 4.3 보수 유저 0~5h 체감

5h 시점에 칭호 1~2 + 위업 2~3. M6 시스템 첫 맛 단계. roadmap 종료 조건은 충족하지만 깊이는 미흡. **5h ~ 15h 구간에 자연 확장**. M6 누적 플레이 한도(roadmap 명시 없음, M7 진입 시점) 안에서 충분히 다 경험 가능.

### 4.4 무게감 vs 페이스 trade-off

**무게감 보존 정책 검증**:
- 평균 페이스 시간당 발급 1~3개 — "이번 시간에 1~2개 특별 순간" 인지 가능
- 다이얼로그 큐의 critical/high 우선도 + 인라인 통합으로 dialog 폭주 회피
- 24h 쿨다운으로 지명 의뢰 회전 다양화

**페이스 보존 정책 검증**:
- 위업·칭호·지명 의뢰 3 시스템이 시간당 1~3개씩 발급 → 사용자가 매 시간 "뭔가 일어남" 인지
- 단조로움 회피

→ **trade-off 균형 ✓**

- **Q-3 (보수 유저 hook 보강)**: 보수 유저(시간당 4 파견)가 5~10h 구간에 행동 지표 칭호 도달 못할 위험. 페이즈 4 #2 명세에서 별도 보조 hook(예: total_dispatch_count 50 등 더 낮은 임계 칭호 추가) 검토. **권장**: M6 MVP 변경 없음 — 보수 유저는 위업 기반 칭호로 fallback 충분
- **Q-4 (페이즈 3 스킵 확정)**: 본 리포트 §스킵 검토 권장. milestone-runner 체크포인트에서 사용자 결정
- **Q-5 (지명 의뢰 가중치 α 동적 조정)**: 사용자별 hook 매칭 풀 크기에 따라 α를 동적 조정할지(예: 3개 매칭이면 α=3, 1개 매칭이면 α=5). **권장**: M6 MVP 정적 α=3 — 동적 조정은 M9+ 검토
