## 개요

전투 보고서 MVP는 M8a에서 실제 턴 기반 전투 시뮬레이터를 구현하지 않고도 "의뢰 결과가 전투처럼 읽히는 경험"을 제공하기 위한 보고서 계층이다. 기존 성공률 계산과 결과 판정을 그대로 사용하고, 완료된 의뢰의 맥락을 해석하여 요약과 상세 로그를 한 번 생성한 뒤 저장한다.

보고서는 기존 `QuestNarrativeService`의 한 줄 서사를 대체하지 않는다. M8a에서는 세력·지명·엘리트·연계 의뢰 같은 중요한 의뢰에만 추가로 붙는 확장 기록으로 정의한다.

## 레퍼런스 분석

| 레퍼런스 | 차용 메커니즘 | 적용 방식 |
|---------|--------------|----------|
| Battle Brothers — 전투 후 로그 | 누가 죽였고 누가 다쳤는지 짧은 사건으로 기억된다 | 보고서 상세 로그에 활약·부상·결정타를 남긴다 |
| Darkest Dungeon — 결과 화면 대사 | 짧은 문장이 원정의 감정을 만든다 | 요약은 2~3문장으로 빠르게 읽힌다 |
| Football Manager — 경기 리포트 | 시뮬레이션 수치를 사람이 읽는 기사처럼 바꾼다 | 기존 성공률/결과를 보고서 템플릿으로 번역한다 |
| Slay the Spire — Run History | 결과가 기록으로 보존되어 나중에 다시 볼 수 있다 | 완료 후 재접속해도 동일한 보고서를 보여준다 |

## 적용 범위

### 보고서 생성 대상

| 의뢰 유형 | M8a 적용 | 이유 |
|----------|----------|------|
| 세력 지명 의뢰 | 필수 | M8a 핵심 루프 |
| 기존 지명 의뢰 | 필수 | M6 지명 의뢰의 의미를 강화 |
| 엘리트 의뢰 | 필수 | 전투성·강적성이 가장 높다 |
| 연계 퀘스트 | 필수 | 기억해야 할 사건이다 |
| 세력 전용 의뢰 | 권장 | 대표 3세력 의뢰에 우선 적용 |
| 일반 의뢰 | 제외 | UI 피로도와 템플릿 폭증 방지 |
| 더스트빌 허드렛일 | 제외 | 초반 반복 루프를 무겁게 만들지 않는다 |

### 결과별 보고서 정책

| 결과 | 요약 톤 | 상세 로그 구성 |
|------|---------|----------------|
| 대성공 | 전열이 안정적이고 결정적 장면이 선명하다 | 진입 → 주도권 확보 → 결정타 → 후속 효과 |
| 성공 | 위험이 있었지만 목적을 달성한다 | 진입 → 위기 → 대응 → 완료 |
| 실패 | 목표 달성 실패와 손실을 함께 기록한다 | 진입 → 균열 → 후퇴 → 부상/손실 |
| 대실패 | 사망·중상·명성 손상을 과장 없이 기록한다 | 진입 → 붕괴 → 치명적 장면 → 후퇴 |

## 톤 가이드

전투 보고서는 영웅 서사보다 "용병단 기록"에 가깝게 쓴다. 과도한 문학적 표현보다, 어떤 일이 벌어졌고 누가 눈에 띄었는지 명확히 남긴다.

### 공통 톤

- 현재형보다 완료 기록형 문장을 사용한다.
- 용병 이름, 직업, 적 이름, 지역 이름을 최소 1개 이상 포함한다.
- 한 문장에 사건 하나만 담는다.
- 사망이나 부상은 과장하지 않고 건조하게 기록한다.
- 결과가 실패일 때도 플레이어를 조롱하지 않는다.
- 템플릿은 같은 의뢰를 반복해도 다른 장면이 나오도록 키워드 조합을 허용한다.

### 세력별 톤

| 세력 | 톤 | 문장 재료 |
|------|----|----------|
| 모험가 길드 | 기록·관찰·지도 | 장부, 좌표, 흔적, 지도, 탐사선 |
| 상인 연합 | 계약·호송·손익 | 수레, 계약서, 봉인, 포대, 통행료 |
| 전사 길드 | 전열·결투·무공 | 방패, 창끝, 숨, 전열, 결정타 |
| 엘리트 의뢰 | 긴장·강적·상처 | 포효, 돌진, 균열, 피로, 후퇴 |
| 연계 퀘스트 | 사건·전환·후일담 | 증언, 표식, 봉인, 서약, 마을 반응 |

## 길이 규칙

### 요약 보고서

요약 보고서는 결과 다이얼로그에서 바로 보이는 짧은 텍스트다.

| 중요도 | 길이 | 대상 |
|--------|------|------|
| 일반 중요 | 2~3문장 | 세력 지명 의뢰, 기존 지명 의뢰 |
| 높은 중요 | 3~4문장 | 엘리트, 연계 퀘스트 핵심 단계 |
| 매우 높은 중요 | 4문장 이내 | 유니크 엘리트, 체인 최종 단계, 세력 신뢰 보상 의뢰 |

### 상세 보고서

상세 보고서는 접을 수 있는 별도 영역 또는 별도 화면에서 보여준다.

| 중요도 | 줄 수 | 구조 |
|--------|------|------|
| 일반 중요 | 4~5줄 | 진입 / 위기 / 활약 / 결과 |
| 높은 중요 | 5~7줄 | 진입 / 적 전술 / 주인공 활약 / 파티 보조 / 결과 |
| 매우 높은 중요 | 6~8줄 | 전장 분위기 / 1차 충돌 / 위기 / 결정적 장면 / 피해 / 후일담 |

요약과 상세는 같은 seed로 생성한다. 이미 생성된 보고서는 재렌더링하지 않는다.

## 보고서 구성 요소

### 입력 맥락

`CombatReportService`는 다음 정보를 입력받아 보고서를 생성한다.

| 입력 | 용도 |
|------|------|
| `ActiveQuest` | 의뢰명, 타입, 리전, 세력 태그, 지명/엘리트/체인 여부 |
| 파티 용병 목록 | 주인공, 보조 활약자, 부상자 후보 선정 |
| `QuestResult` | 결과별 톤과 상세 구조 선택 |
| `QuestPool` | 적 이름, 지역 상태 효과, 세력 조건 |
| `EliteMonster` | 강적 이름과 전장 키워드 |
| `FactionData` | 세력별 톤과 색상 |
| `RegionState` | dangerLevel, unlockedFlags, 인프라 맥락 |
| seed | 완료 시점 보고서 고정 |

### 출력 구조 후보

페이즈 4에서 정확한 모델을 정하되, 컨텐츠 관점의 최소 출력은 다음과 같다.

```text
CombatReport
- summary: String
- details: List<String>
- protagonistMercId: String?
- featuredMercIds: List<String>
- toneTags: List<String>
- createdAt: DateTime
```

`summary`는 결과 다이얼로그에 보이고, `details`는 상세 보고서에 보인다. `toneTags`는 템플릿 선택과 향후 M8.5 감정 반응·히든 스탯의 연결점으로 남긴다.

## 노출 규칙

### 결과 다이얼로그

기존 `QuestResultDialog`에는 이미 결과, 보상, 엘리트 드랍, 서사 텍스트가 표시된다. M8a에서는 다음 순서를 권장한다.

1. 결과 헤더
2. 보상·손실
3. 기존 서사 텍스트
4. 전투 보고서 요약 카드
5. **상세 전투 보고서** 버튼 또는 접기 영역

요약 카드는 보고서 대상 의뢰에만 표시한다. 일반 의뢰에는 표시하지 않는다.

### 연대기 또는 기록 접근

M8a에서 완료 보고서 목록 화면까지 만들 필요는 없다. 다만 완료 직후 재접속해도 같은 보고서를 볼 수 있어야 하므로, 완료 퀘스트 결과가 사라지는 현재 흐름을 페이즈 4에서 점검한다.

권장안은 다음 중 하나다.

| 옵션 | 설명 | M8a 권장 |
|------|------|----------|
| ActiveQuest에 보고서 필드 추가 | 완료 직후 결과 다이얼로그와 활동 로그에서 사용 | 높음 |
| `combatReports` Hive 박스 신설 | 장기 기록과 검색 가능 | 중간, M8b 확장 후보 |
| ActivityLog에만 저장 | 구현이 쉽지만 상세 구조가 약하다 | 낮음 |

M8a는 `ActiveQuest` 필드 추가를 우선 검토하고, M8b에서 전투 스냅샷이 필요해지면 `combatReports` 박스로 확장한다.

## 템플릿 데이터 설계

### 템플릿 그룹

페이즈 3 #5의 80~120개 템플릿은 다음 그룹으로 나눈다.

| 그룹 | 수량 목표 | 설명 |
|------|----------|------|
| quest_type × result | 32개 | 4 quest_type × 4 result × 2문장 변형 |
| 세력별 톤 | 24개 | 3세력 × 4 result × 2변형 |
| 엘리트 전투 | 20개 | 일반 엘리트/유니크/괴물 부위/부상 반응 |
| 연계 퀘스트 | 16개 | 체인 중간/최종, 증언·서약·봉인 계열 |
| 결정적 장면 | 20~28개 | 주인공 활약, 보조, 후퇴, 방패, 추격 |

### 키워드 데이터

키워드는 템플릿과 별도 테이블 또는 JSON 배열로 둘 수 있다.

| 카테고리 | 수량 목표 | 예시 |
|----------|----------|------|
| 전장 분위기 | 10~15개 | 먼지 낀 광장, 해안 절벽, 부서진 성벽 |
| 적 유형 | 8~12개 | 도굴꾼 대장, 늪지 추적자, 상단 습격대 |
| 결정적 장면 | 12~18개 | 방패로 길을 열었다, 후열을 끊었다 |
| 손실·부상 장면 | 6~10개 | 무릎이 꺾였다, 후퇴 신호가 늦었다 |

## 현재 시스템과의 연관

| 시스템 | 재사용 | 확장 후보 |
|--------|--------|----------|
| `QuestNarrativeService` | 주인공 선정, TemplateEngine 사용 방식 | `CombatReportService`에서 helper 일부 재사용 |
| `ActiveQuest.renderedNarrative` | 1회 렌더 후 저장 패턴 | `combatReportSummary`, `combatReportDetails` 필드 후보 |
| `QuestResultDialog` | 결과 표시 위치 | 요약 카드와 상세 접기 영역 추가 |
| `TemplateEngine` | `{merc.name}`, `{region.name}` 등 변수 렌더 | combat 전용 namespace 후보 |
| `ActivityLog` | 결과 요약 저장 | 보고서 생성 여부를 짧게 표시 |
| M8.5 히든 스탯 | 아직 미구현 | `toneTags`와 featuredMercIds가 후속 연결점 |

## 구현 우선순위 제안

| 우선순위 | 항목 | 이유 |
|----------|------|------|
| 높음 | 보고서 생성 대상과 길이 규칙 | UI 피로도와 데이터량을 결정한다 |
| 높음 | 요약/상세 구조 | M8a 완료 기준에 직접 연결된다 |
| 높음 | 1회 생성 후 저장 정책 | 재접속 동일성 요구사항을 만족한다 |
| 중간 | 보고서 템플릿 80~120개 | 읽는 재미의 핵심이지만 페이즈 3 데이터 생성 대상이다 |
| 낮음 | 완료 보고서 목록 화면 | M8.5 기록/대시보드에서 다루는 편이 좋다 |

## data-generator 지시사항

- **대상 타입**: 신규 타입 스펙 필요. 후보명 `combat-report-template`
- **대상 테이블**: 신규 후보 `combat_report_templates`, `combat_report_keywords`
- **생성 수량**:
  - 전투 보고서 템플릿 80~120개
  - 전장/적/결정적 장면 키워드 30~50개
- **톤/세계관 가이드**:
  - 결과를 과장된 영웅담으로 만들지 않는다.
  - 용병 이름과 직업이 자연스럽게 들어갈 수 있는 문장 구조를 사용한다.
  - 실패·대실패 템플릿은 조롱하지 않는다.
  - 세력별 문체 차이를 유지한다.
- **구조적 제약**:
  - 일반 의뢰용 템플릿은 최소화한다.
  - 세력·지명·엘리트·연계 의뢰에 우선 배정한다.
  - 요약 템플릿과 상세 라인 템플릿을 구분한다.
  - seed 기반 재현이 가능하도록 랜덤 선택 단위를 명확히 둔다.
- **수치 출처**: 페이즈 2 #3에서 노출 빈도와 저장 범위를 확정한다.
