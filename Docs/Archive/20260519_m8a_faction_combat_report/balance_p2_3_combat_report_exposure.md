## 현재 상태

현재 퀘스트 결과에는 `QuestNarrativeService`가 생성한 `renderedNarrative`가 1회 저장된다. 이 서사는 `ActiveQuest.renderedNarrative`에 저장되며, 결과 다이얼로그에서 짧은 텍스트로 표시된다.

M8a 전투 보고서는 이 서사를 대체하지 않고, 중요한 의뢰에만 추가되는 요약·상세 기록이다. 일반 의뢰 전체에 적용하면 결과 확인 시간이 길어져 방치형 루프의 속도가 둔해진다.

Supabase 실데이터 기준 현재 지명 의뢰는 7개, 기존 세력 전용 의뢰는 98개, M8a 추가 세력 지명 의뢰는 12개 계획이다.

## 데이터 분석

### 보고서 대상 후보 규모

| 유형 | 현재/계획 수량 | 보고서 적용 |
|------|---------------|------------|
| 기존 지명 의뢰 | 7개 | 적용 |
| M8a 세력 지명 의뢰 | 12개 | 적용 |
| 기존 세력 전용 의뢰 | 98개 | 대표 3세력 우선 적용 |
| 엘리트 의뢰 | 동적 생성 | 적용 |
| 연계 퀘스트 | 26단계 이상 | 최종 단계 + 핵심 단계 적용 |
| 일반 의뢰 | 301개 이상 | 제외 |

M8a에서 실제 신규 템플릿이 필요한 핵심 대상은 20~40개 의뢰·상황이다. 템플릿 80~120개는 같은 대상에 여러 변형을 제공하기에 충분하다.

### 결과 확인 시간 추정

기존 결과 다이얼로그는 보상, 드랍, 서사를 빠르게 확인하는 구조다. 전투 보고서 요약이 항상 길게 보이면 반복 피로가 커진다.

권장 읽기 시간은 다음과 같다.

| UI 요소 | 길이 | 예상 읽기 시간 |
|---------|------|---------------|
| 기존 서사 | 1~2문장 | 3~6초 |
| 보고서 요약 | 2~4문장 | 6~12초 |
| 상세 보고서 | 4~8줄 | 15~30초 |

보고서 대상 의뢰 비율은 전체 완료 의뢰의 15~25% 안에 머물러야 한다. 이 범위를 넘으면 플레이어가 매번 상세 텍스트를 닫는 행동을 반복하게 된다.

## 문제점

### 1. 세력 전용 의뢰 98개 전체 적용 시 과다

기존 세력 전용 의뢰 98개 전체에 보고서를 적용하면 템플릿 분산이 넓어지고, M8a 대표 세력의 선명도가 약해진다. M8a에서는 대표 3세력의 전용 의뢰와 신규 세력 지명 의뢰를 우선 적용한다.

### 2. 상세 보고서 자동 펼침은 반복 피로를 만든다

상세 보고서가 결과 다이얼로그에서 항상 펼쳐지면 결과 확인 시간이 길어진다. 요약만 기본 노출하고, 상세는 버튼 또는 접기 영역으로 둔다.

### 3. 장기 저장 모델을 과하게 만들면 M8b와 충돌한다

M8b는 실제 턴 기반 전투 스냅샷을 저장할 가능성이 높다. M8a에서 별도 `combatReports` 박스를 완성형으로 만들면 M8b 설계가 경직된다. M8a는 `ActiveQuest` 저장 필드 또는 가벼운 보고서 객체를 우선한다.

## 조정 제안

### 보고서 생성 대상 확정안

| 대상 | M8a 정책 | 근거 |
|------|----------|------|
| M8a 세력 지명 의뢰 12개 | 항상 생성 | 마일스톤 핵심 |
| 기존 지명 의뢰 7개 | 항상 생성 | M6 지명 의뢰의 기억성 강화 |
| 엘리트 의뢰 | 항상 생성 | 강적 전투 체감 |
| 유니크 엘리트 | 항상 생성 + 상세 기본 강조 | 중요한 전투 |
| 연계 퀘스트 | 최종 단계 항상, 중간 핵심 단계 선택 | 모든 단계 적용은 과다 |
| 대표 3세력 기존 전용 의뢰 | 평판 31 이상 또는 고급 트랙만 생성 | 세력 신뢰 이후 체감 강화 |
| 일반 의뢰 | 생성 안 함 | 반복 피로 방지 |
| 허드렛일·조사 전용 | 생성 안 함 | 빠른 루프 유지 |

### 노출 빈도 목표

| 구간 | 총 의뢰 완료 대비 보고서 비율 | 기준 |
|------|-----------------------------|------|
| M8a 진입 직후 | 10~15% | 세력 접촉 직후는 일반 의뢰 병행이 많다 |
| 세력 지명 의뢰 집중 | 20~25% | 보고서 체감 구간 |
| 장기 반복 | 15~20% | 피로도 방지 상한 |

보고서 대상 의뢰가 30%를 넘으면 일반 의뢰 제외 정책을 다시 강화한다.

### 길이 확정안

| 중요도 | 요약 | 상세 | 기본 UI |
|--------|------|------|---------|
| 일반 세력 지명 | 2문장 | 4줄 | 요약만 표시, 상세 접힘 |
| 신뢰 세력 지명 | 3문장 | 5~6줄 | 요약 표시, 상세 버튼 강조 |
| 엘리트 | 3문장 | 5~7줄 | 요약 표시, 드랍 아래 상세 버튼 |
| 유니크 엘리트 | 4문장 | 6~8줄 | 요약 표시, 상세 버튼 강조 |
| 연계 최종 | 3~4문장 | 6~8줄 | 요약 표시, 상세 버튼 강조 |

상세 보고서는 자동 펼침하지 않는다. 단, 최초 1회 튜토리얼성 보고서만 자동 펼침을 허용할 수 있다.

### 저장 범위 확정안

M8a에서는 다음 필드를 `ActiveQuest` 또는 완료 결과 저장 모델에 추가하는 방향을 권장한다.

| 필드 | 타입 | 목적 |
|------|------|------|
| `combatReportSummary` | `String?` | 결과 다이얼로그 요약 |
| `combatReportDetails` | `List<String>` 또는 JSON string | 상세 보고서 |
| `combatReportSeed` | `int?` | 재현성과 디버깅 |
| `combatReportTags` | `List<String>` 또는 JSON string | M8.5 히든 스탯 연결 후보 |
| `combatReportProtagonistMercId` | `String?` | 주인공 용병 기록 |

별도 `combatReports` Hive 박스는 M8b에서 실제 전투 스냅샷 구조가 확정된 뒤 도입한다.

## 템플릿 수량 제안

페이즈 3 #5의 템플릿은 96개를 기준으로 한다. 목표 범위 80~120개 안에서 관리하기 쉽고, 12개 단위로 배분된다.

| 그룹 | 수량 | 설명 |
|------|------|------|
| 공통 quest_type × result 요약 | 24개 | 4개 유형 × 성공/대성공/실패 계열 변형 |
| 대표 세력별 요약 | 24개 | 3세력 × 4결과 × 2변형 |
| 엘리트 상세 라인 | 16개 | 일반/유니크, 강적, 부상, 결정타 |
| 연계 퀘스트 상세 라인 | 12개 | 중간·최종 단계 |
| 결정적 장면 라인 | 20개 | 주인공 활약, 보조, 후퇴, 방패, 추격 |

키워드는 40개를 기준으로 한다.

| 키워드 | 수량 |
--------|------|
| 전장 분위기 | 12개 |
| 적 유형 | 10개 |
| 결정적 장면 | 12개 |
| 손실·부상 장면 | 6개 |

## 플레이어 체감 분석

전투 보고서의 목표는 매번 읽게 만드는 것이 아니라, 중요한 의뢰를 완료했을 때 "이 의뢰는 기록으로 남길 만했다"는 감각을 주는 것이다. 따라서 기본 노출은 짧고, 상세는 플레이어가 원할 때 펼치게 한다.

보고서가 붙은 의뢰는 파견 전에도 카드에서 작게 표시하는 것이 좋다. 예를 들어 "보고서 생성" 배지를 붙이면 플레이어가 의미 있는 의뢰를 선택한다는 기대를 갖는다.

## 시뮬레이션

### 1시간 플레이 예시

가정은 다음과 같다.

- 총 의뢰 완료: 25회
- 일반 의뢰: 17회
- 세력 지명 의뢰: 4회
- 기존 지명 또는 엘리트: 2회
- 연계 퀘스트 핵심 단계: 1회
- 세력 전용 고급: 1회

| 항목 | 횟수 | 보고서 생성 |
|------|------|-------------|
| 일반 의뢰 | 17 | 0 |
| 세력 지명 의뢰 | 4 | 4 |
| 기존 지명/엘리트 | 2 | 2 |
| 연계 핵심 | 1 | 1 |
| 세력 전용 고급 | 1 | 1 |
| 합계 | 25 | 8 |

보고서 비율은 32%로 높다. 이 집중 구간은 허용하되, 장기 평균은 세력 지명 의뢰 쿨다운과 일반 의뢰 병행으로 20% 안팎으로 낮아져야 한다.

### 장기 평균 예시

세력 지명 의뢰 쿨다운 18~24시간, 기존 지명 의뢰 24시간을 적용하면 같은 의뢰 반복 보고서가 줄어든다. 2~3시간 플레이 기준 보고서 생성률은 15~22%로 수렴한다.

## data-generator 수치 가이드

- **대상 타입**: 신규 타입 스펙 필요. 후보명 `combat-report-template`
- **대상 테이블**: 신규 후보 `combat_report_templates`, `combat_report_keywords`
- **수치 범위**:
  - 템플릿 총량: 96개 기준, 허용 80~120개
  - 키워드 총량: 40개 기준, 허용 30~50개
  - 요약 문장 수: 2~4문장
  - 상세 줄 수: 4~8줄
  - 보고서 대상 의뢰 장기 평균: 전체 완료의 15~25%
- **외래 키 제약**:
  - `quest_type`: `explore`, `escort`, `hunt`, `raid`
  - `result_type`: `greatSuccess`, `success`, `failure`, `criticalFailure`에 대응
  - `faction_id`: 대표 3세력 우선
- **balance 근거**: 일반 의뢰를 제외하고 중요 의뢰만 대상으로 삼아 결과 확인 피로도를 억제한다.
