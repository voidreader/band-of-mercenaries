# Band of Mercenaries

`Band of Mercenaries`는 용병단 단장이 되어 용병을 모집하고, 지역을 이동하며, 의뢰를 수행해 명성과 자원을 쌓는 텍스트 기반 전략 시뮬레이션 모바일 게임이다. Flutter로 개발하며, Supabase에서 정적 데이터를 동기화하고 Hive에 플레이어 데이터를 저장한다. 이 저장소는 기획, 명세 작성, 코드 구현, 검증, 문서화를 AI 에이전트와 함께 진행하는 AI 주도 개발 프로젝트이기도 하다.

이 프로젝트는 방치형 게임의 느린 성장감과 텍스트 전략 게임의 선택지를 결합하는 것을 목표로 한다. 플레이어는 직접 전투를 조작하기보다 어떤 용병을 어디에 보내고, 어떤 위험을 감수하며, 어떤 지역과 세력에 개입할지 결정한다.

> **참고:** 이 프로젝트는 현재 활발히 개발 중인 개인 게임 프로젝트이다. 공개 저장소에는 게임 클라이언트와 설계 문서가 포함되며, 운영 도구와 배포 환경은 별도 프로젝트 또는 비공개 환경에서 관리될 수 있다.

## AI 주도 개발

이 프로젝트는 AI를 보조 도구가 아니라 개발 프로세스의 중심 협업자로 두고 진행한다. 사람이 게임의 방향, 우선순위, 최종 판단을 정하고, AI 에이전트는 기획 구체화, 명세 작성, 코드 구현, 리팩터링, 테스트 검토, 문서 정리를 반복적으로 수행한다.

AI 주도 개발 방식은 다음 영역에서 활용한다.

- **기획 확장:** 게임 루프, 신규 시스템, 밸런스 방향, 콘텐츠 구조를 문서화하고 대안을 비교한다.
- **명세 작성:** 구현 전에 요구사항, 데이터 모델, Provider 흐름, UI 동작, 검증 기준을 정리한다.
- **코드 구현:** Flutter feature 모듈, 도메인 서비스, Hive 모델, Supabase 정적 데이터 모델을 기존 아키텍처에 맞춰 작성한다.
- **품질 검토:** 테스트, 정적 분석, 코드 리뷰 문서, 회귀 위험 점검을 통해 변경 사항을 검증한다.
- **문서 관리:** `Docs` 디렉토리와 README를 통해 개발 의도, 설계 결정, 마일스톤 결과를 지속적으로 기록한다.

이 저장소의 `AGENTS.md`, `CLAUDE.md`, `.agents/skills` 문서는 AI 에이전트가 프로젝트 규칙과 개발 절차를 이해하도록 돕는 작업 지침이다. 따라서 이 프로젝트는 게임 클라이언트인 동시에 AI와 함께 장기 소프트웨어 프로젝트를 운영하는 실험 기록이기도 하다.

## 게임 개요

게임의 핵심은 용병단 운영이다. 플레이어는 제한된 골드와 인력으로 용병을 고용하고, 각 지역에서 생성되는 의뢰에 적합한 인원을 파견한다. 파견은 실제 시간 흐름에 따라 진행되며, 완료 후 보상·피해·명성·서사 결과가 반영된다.

핵심 루프는 다음 흐름으로 구성된다.

```text
용병 모집 -> 지역 이동 -> 의뢰 선택 -> 파견 대기 -> 결과 확인 -> 성장 및 반복
```

게임은 다음 감각을 중심으로 설계한다.

- 플레이어가 직접 조작하지 않는 시간에도 용병단이 움직이는 방치형 성장감.
- 지역, 세력, 의뢰, 용병 특성이 얽히는 운영 판단.
- 반복 파견 속에서 용병이 이력과 칭호를 얻는 누적 서사.
- 성공률, 보상, 위험도, 이동 거리 사이에서 발생하는 전략적 선택.

## 주요 특징

게임은 단순한 퀘스트 반복이 아니라, 용병단과 세계 상태가 함께 변하는 장기 운영 구조를 지향한다. 현재 구현은 용병 모집·파견·이동·시설·제작·세력·위업·전투 보고서 등 여러 시스템으로 확장되어 있다.

- **용병단 운영:** 용병은 직업, 티어, 능력치, 특성, 상태, 레벨, 칭호를 가진다. 피로, 부상, 사망, 방출 같은 상태 변화가 장기 운영의 비용으로 작동한다.
- **파견 의뢰:** 의뢰는 지역, 난이도, 유형, 세력, 지명 조건, 체인 단계, 지역 상태에 따라 생성된다. 성공률은 파티 전력, 역할 상성, 특성, 보너스, 위험 요소를 반영한다.
- **지역 이동:** 지역과 섹터 간 이동 시간이 존재하며, 이동 중 선택지 이벤트가 발생한다. 일부 지역은 인접성 데이터와 거점 인프라에 따라 이동 비용이 달라진다.
- **지역 상태 변화:** 의뢰 완료, 체인 완주, 엘리트 처치, 시간 경과가 지역의 위험도와 해금 플래그에 영향을 준다. 지역이 안정되거나 위협 상태로 변하면 의뢰 구성도 달라진다.
- **마을과 생활권:** 특정 거점은 신뢰도와 인프라 단계를 가진다. 플레이어의 행동에 따라 약초상, 낡은 대장간, 외래 좌판, 생활권 정보 같은 기능이 열리고 강화된다.
- **제작과 인벤토리:** 재료는 의뢰, 조사, 엘리트 전리품, 이동 선택지, 체인 보상 등에서 획득한다. 레시피는 신뢰도, 체인 진행, 지역 플래그, 인프라 단계에 따라 해금된다.
- **세력 시스템:** 세력은 접촉점, 평판, 가입 상태, 반응 텍스트, 세력 상점, 지명 의뢰를 가진다. 플레이어의 파견 결과가 세력 관계와 보상에 영향을 준다.
- **위업과 칭호:** 특정 사건은 용병단 연대기에 위업으로 기록된다. 위업, 행동 통계, 상태 변화, 세력 평판은 용병 칭호를 해금할 수 있다.
- **전투 보고서:** 중요 의뢰는 결과 요약과 상세 전투 로그를 생성한다. 엘리트, 체인, 지명 의뢰, 세력 의뢰는 더 풍부한 보고서와 서사 템플릿을 사용한다.
- **정적 데이터 동기화:** 지역, 직업, 특성, 의뢰 풀, 세력, 아이템, 제작 레시피, 전투 템플릿 등은 Supabase에서 관리하고 앱 시작 및 포그라운드 복귀 시 버전 기반으로 동기화한다.

## 플레이 경험

플레이어는 하단 탭을 오가며 용병단의 현재 상황을 확인하고 다음 행동을 결정한다. 각 화면은 하나의 운영 축을 담당하며, 화면 전환은 Flutter 앱 내부 상태를 기준으로 처리한다.

주요 화면은 다음 역할을 가진다.

- **이동:** 현재 위치에서 갈 수 있는 지역과 섹터를 확인하고 이동을 시작한다.
- **파견:** 현재 위치의 의뢰 목록을 보고 파티를 편성해 출발시킨다.
- **홈:** 골드, 현재 위치, 진행 중인 의뢰, 간판 용병, 활동 로그를 확인한다.
- **모집:** 새 용병을 고용하고 기존 용병의 상태와 상세 정보를 확인한다.
- **시설:** 야영지 시설과 건설 큐를 관리한다.
- **정보:** 세력 도감, 연대기, 게임 내 정보성 화면으로 진입한다.

## 세계와 데이터

게임의 세계는 지역, 섹터, 세력, 의뢰 풀, 체인 퀘스트, 정적 보상 테이블로 구성된다. 지역은 단순 배경이 아니라 의뢰 생성 조건과 위험도, 이동 거리, 발견 요소, 거점 성장에 영향을 주는 단위로 사용된다.

정적 데이터는 서버에서 관리하고, 앱은 필요한 데이터를 로컬 캐시에 저장한다. 이 구조를 통해 코드 배포 없이도 의뢰 풀, 밸런스 값, 텍스트 템플릿, 아이템, 세력 콘텐츠를 조정할 수 있다.

현재 정적 데이터는 다음 범주를 포함한다.

- 지역, 섹터, 인접 지역, 난이도, 랭크.
- 직업, 용병 임금, 특성, 특성 충돌, 특성 진화, 특성 시너지.
- 의뢰 유형, 의뢰 풀, 지명 의뢰 조건, 의뢰별 재료 드랍.
- 이동 이벤트, 이동 선택지, 선택지 결과.
- 아이템, 제작 레시피, 엘리트 몬스터, 전리품 테이블.
- 세력, 세력 접촉점, 세력 반응, 세력 상점.
- 체인 퀘스트, 위업 템플릿, 칭호, 전투 보고서 템플릿.
- 전투 스킬, 상태 이상, 적 유형, 숨겨진 통계, 전투 기억 템플릿.

## 기술 구조

클라이언트는 Flutter와 Dart로 작성한다. 상태 관리는 Riverpod을 사용하며, 사용자 진행 데이터는 Hive에 저장한다. Supabase는 정적 데이터 저장소로 사용하고, 앱은 `data_versions` 기반으로 변경된 테이블만 내려받는다.

주요 기술 스택은 다음과 같다.

- **Flutter / Dart:** 모바일 중심 UI와 게임 클라이언트 구현.
- **Flutter Riverpod:** 게임 상태, 정적 데이터, 화면 상태, 이벤트 큐 관리.
- **Hive:** 사용자 데이터, 용병, 의뢰, 활동 로그, 지역 상태, 세력 상태, 위업 등 로컬 영속화.
- **Supabase:** 정적 데이터 테이블과 버전 기반 동기화.
- **freezed / json_serializable:** 정적 데이터 모델과 JSON 파싱 모델 생성.
- **hive_generator:** Hive 어댑터 생성.
- **riverpod_generator:** 일부 Provider 코드 생성.

앱 시작 흐름은 다음과 같다.

```text
Hive 초기화
-> Supabase 초기화
-> 정적 데이터 동기화
-> 로컬 캐시 로딩
-> Region 마이그레이션
-> ProviderScope 기반 앱 실행
-> 1초 게임 틱으로 완료 처리
```

## 저장소 구성

저장소는 Flutter 앱, 정적 데이터 샘플, 설계 문서, 작업 산출물로 나뉜다. 실제 게임 코드는 `band_of_mercenaries` 디렉토리 아래에 있다.

주요 디렉토리는 다음과 같다.

```text
.
├── band_of_mercenaries/      # Flutter 게임 클라이언트
├── Docs/                     # 기획, 밸런스, 명세, 리뷰 문서
├── Json/                     # 정적 데이터 또는 변환용 JSON 자료
├── scripts/                  # 데이터 및 보조 작업 스크립트
├── AGENTS.md                 # Codex 작업 지침
├── CLAUDE.md                 # Claude 작업 지침
└── CHANGELOG.md              # 주요 변경 이력
```

앱 내부 구조는 feature 단위로 나뉜다. 각 feature는 가능한 한 `view`, `domain`, `data` 계층으로 분리한다.

```text
band_of_mercenaries/lib/
├── app.dart                  # 앱 셸, 하단 탭, 전역 다이얼로그 큐
├── main.dart                 # 초기화, 동기화, 마이그레이션 진입점
├── core/                     # 공통 모델, 데이터 로더, Provider, 테마, 도메인 서비스
├── features/                 # 게임 기능별 모듈
└── shared/                   # 공용 위젯
```

주요 feature 모듈은 다음과 같다.

- `home`: 야영지 홈, 활동 로그, 간판 용병 카드.
- `movement`: 지역 이동, 이동 거리 계산, 생활권 점프.
- `quest`: 의뢰 생성, 파견, 완료 처리, 전투 보고서, 전투 시뮬레이터.
- `mercenary`: 용병 모집, 상세 정보, 특성, 숨겨진 통계.
- `facility`: 시설과 건설 큐.
- `inventory`: 아이템, 재료, 장비, 인벤토리 UI.
- `crafting`: 제작 서비스, 레시피 상태, 낡은 대장간.
- `investigation`: 지역 조사, 발견, 지역 상태 변화.
- `settlement`: 마을 방문, 약초상, 촌장 집, 외래 좌판.
- `chain_quest`: 연계 퀘스트 진행.
- `info`: 세력 도감과 정보 화면.
- `achievement`: 위업과 연대기.
- `title`: 칭호와 간판 용병.
- `settings`: 개발 및 설정성 옵션.

## 설계 원칙

프로젝트는 작동하는 프로토타입에서 시작했지만, 장기적으로는 콘텐츠를 많이 추가할 수 있는 구조를 지향한다. 이를 위해 게임 규칙, 정적 데이터, 사용자 영속성, UI 표시를 가능한 한 분리한다.

핵심 설계 원칙은 다음과 같다.

- 정적 콘텐츠는 Supabase 테이블과 JSON 모델에 두고, 클라이언트는 이를 로컬 캐시로 사용한다.
- 사용자 진행 상태는 Hive에 저장하고, 앱 재시작 후에도 동일한 용병단 상태를 복원한다.
- 전역 이벤트 팝업은 다이얼로그 큐로 관리하여 랭크업, 체인 완료, 위업 해금, 이동 선택지 같은 이벤트가 서로 덮어쓰지 않게 한다.
- 도메인 서비스는 가능한 한 순수 계산과 콜백 주입을 사용해 Provider 순환 의존을 줄인다.
- 화면 전환은 모바일 앱의 예측 가능성을 위해 상태 기반 렌더링을 우선한다.
- 같은 UI 패턴이 반복되면 `shared/widgets` 또는 테마 계층으로 이동한다.

## 문서

`Docs` 디렉토리에는 게임 설계와 구현 과정에서 작성한 문서가 들어 있다. 공개 저장소에서는 코드만큼이나 개발 의도를 파악하는 자료로 사용할 수 있다.

문서 범주는 다음과 같다.

- `Docs/spec`: 기능 명세와 구현 계획.
- `Docs/balance-design`: 밸런스 값, 경제 곡선, 보상 구조 검토.
- `Docs/Requirement`: 초기 요구사항과 개선 요청.

## 현재 상태

이 저장소는 완성된 상용 게임이 아니라, 기능을 단계적으로 추가하는 개발 중 프로젝트이다. 핵심 루프는 구현되어 있으며, 이후 마일스톤을 거치며 세력, 지역 상태, 전투 보고서, 생활권, 숨겨진 통계 같은 시스템이 추가되었다.

현재 중점은 다음 방향에 있다.

- 용병 개인의 이력과 감정, 숨겨진 통계를 더 풍부하게 표현한다.
- 세력과 지역 상태가 의뢰 생성 및 보상 구조에 더 강하게 영향을 주도록 확장한다.
- 전투 보고서를 통해 자동 전투 결과를 읽을 만한 이야기로 만든다.
- 생활권 단위의 장기 변화와 거점 성장을 플레이어가 체감할 수 있게 한다.
- 정적 데이터와 밸런스 데이터를 안정적으로 운영할 수 있는 파이프라인을 다듬는다.

## 라이선스

현재 저장소에는 별도 라이선스 파일이 포함되어 있지 않다. 코드, 문서, 데이터의 재사용 조건은 추후 라이선스가 명시되기 전까지 저장소 소유자의 별도 허가를 기준으로 한다.
